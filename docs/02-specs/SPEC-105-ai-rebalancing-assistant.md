## 2. Overview

### Purpose

Given a **contribution amount** ("tenho R$X para aportar este mês"), produce **explainable
guidance on where to focus the new money** — suggested allocation _areas_ and optional **named
candidate assets** that fit them, each with reasoning, risks/trade-offs, and assumptions —
**never a transaction order** (FR-011, FR-014).

### Business Value

This is the assistant's most actionable moment: the investor has cash and a decision to make.
It turns the deterministic portfolio facts (SPEC-104) plus the new contribution into copilot
guidance — areas to consider and real, grounded candidate tickers to research — while the
non-advice guardrail keeps the final decision with the user. It is the second consumer of the
published **Fact Builder** seam, and the "tenho R$X pra aportar" path the Conversational
Copilot (SPEC-108) later routes into.

### Scope

**In scope**

- A **rebalancing engine** that reuses the published SPEC-104 **Fact Builder** (`BuildFacts`) for
  the portfolio/allocation/macro facts, adds the **contribution amount**, the **live FII market
  universe** (the quotes refreshed by the SPEC-006 ingestion worker — current yields/prices/sectors),
  and a **deterministically computed contribution split**, then calls the gated `Insighter`
  (SPEC-005) once for the `rebalancing` task — so explainability (FR-013) and non-advice (FR-014)
  hold **by construction**.
- **Suggested areas** (e.g. Fixed Income, Logistics FIIs) reasoned from **all the context the
  system has** — the full portfolio (every class), current allocation, investor profile, macro
  indicators, and the live FII market data — each carrying a **computed suggested share of the
  contribution** (`suggested_share_bps`, half-up, Σ = 100%).
- **Named candidate assets** (optional) grounded in the system's market-data universe (SPEC-006),
  each with reasoning/risks/assumptions — with a deterministic **grounding guard** that drops any
  candidate naming a ticker the system does not actually know (facts computed, names never invented).
- A **deterministic contribution allocator**: the suggested per-area percentages are **computed**
  (not generated by the LLM), so the same inputs always yield the same split; the Insighter only
  **explains** them. A % split across _areas_ is a consideration, never a transaction order (FR-014).
- `POST /rebalancing` (auth-scoped); no new persisted entities.

**Out of scope**

- Explicit **numeric target allocations** per class (the assistant reasons _qualitatively_ over
  the profile + current allocation; precise targets are a deferred follow-up, shared with the
  SPEC-104 open question).
- Health Score (SPEC-106), Projections (SPEC-107), Chat (SPEC-108).
- Persisting rebalancing history; any change to the SPEC-005 gates or a new LLM adapter.
- A **named-product** universe beyond FIIs. Fixed income is **market-aware** via the live macro
  indicators (SELIC/CDI/IPCA price every CDB/LCI/Tesouro — already in `BuildFacts`), so FI gets
  full live area guidance; but it has no public ticker universe, so FI candidates are **type-level**
  ("pós-fixado ≥100% CDI", "Tesouro IPCA+"), not a specific named product. **Stocks and ETFs are
  out of the MVP** (not ingested, not held). Named **ticker** candidates are therefore FIIs only.

## 3. Functional Requirements

### FR-1051 — Contribution input

Accepts a contribution amount as the request input — an integer in **centavos** (`int64`),
validated `> 0` at the edge. Identity comes from the session context (BR-1043); the contribution
amount is the **only** client-supplied value.

#### Acceptance Criteria

- [ ] `contribution_centavos` ≤ 0, missing, or non-integer (float/string) → `400` with `{"error":…}`.
- [ ] A valid positive amount is accepted and reaches the engine as `int64` centavos.

### FR-1052 — Rebalancing facts (reuse the Fact Builder + market context + computed split)

Builds the deterministic fact snapshot by **reusing** `engine.BuildFacts(ctx, userID)` (SPEC-104),
then augments it with: the `contribution_centavos`; the **live FII market universe** (the quotes
the SPEC-006 worker refreshes — ticker, sector, price, dividend yield); and the **computed
contribution split** (FR-1053a). Facts are computed, never invented; money is centavos and rates
basis points, never float.

#### Acceptance Criteria

- [ ] The portfolio/allocation/macro/profile facts are obtained via the published `BuildFacts`
      seam (not recomputed).
- [ ] The facts include the contribution amount, the FII market universe (current, from ingestion),
      and the computed per-area split — all as integers/structured values.
- [ ] Same inputs → same facts (deterministic; no float anywhere).

### FR-1053 — Suggested areas reasoned from the full context (FR-011)

Generates **explainable area suggestions** — where to focus the contribution — reasoned from **all
the data the system has**: the full portfolio (every class — FII, fixed income, stocks, ETFs),
the current allocation, the investor profile (risk/objectives/horizon), the macro indicators, and
the **live FII market data** (current yields/prices refreshed by ingestion).

#### Acceptance Criteria

- [ ] Output includes ≥1 suggested area when facts permit; each area carries a human-readable
      explanation (FR-013) that references the relevant facts (allocation gap, profile, market).
- [ ] Areas reflect the profile (e.g. a conservative profile skews toward Fixed Income), the
      current allocation, **and** current market conditions — for **FIIs** the live quote/dividend
      data, and for **fixed income** the live macro indicators that price it (SELIC/CDI/IPCA, e.g.
      a high-SELIC environment favouring post-fixed) — not generic boilerplate.
- [ ] An **empty portfolio + a contribution** still yields first-investment area guidance (unlike
      SPEC-104 insights, which short-circuit on an empty portfolio).

### FR-1053a — Computed contribution split (the suggested %)

A **deterministic allocator** computes a suggested **share of the contribution per area**
(`suggested_share_bps`, integer basis points, half-up, **Σ = 10 000**) from the current allocation
vs the profile-implied direction and the contribution amount. The percentages are **computed, not
generated** by the LLM (binding constraint: facts computed, not generated); the Insighter only
**explains** them. A % split across _areas_ is a **consideration**, not a transaction order (FR-014).

#### Acceptance Criteria

- [ ] Each suggested area carries a computed `suggested_share_bps`; the shares sum to exactly
      `10 000` (reconciliation invariant, half-up rounding via `internal/platform/money`).
- [ ] The split is deterministic: same portfolio + profile + contribution → same split.
- [ ] The split is **computed before** the LLM call and passed as a fact; the LLM never alters the
      numbers — it explains the computed split (FR-1055 gate still applies to the text).
- [ ] The response carries the split as integer bps (and/or the centavos amount per area), never a float.

### FR-1054 — Named candidate assets + grounding guard (FR-011)

Lists **named candidate assets** that fit each suggested area, each with reasoning, risks/
trade-offs, and assumptions. By default no per-asset number is shown; an **illustrative within-area
share** per candidate is **opt-in** (request flag `include_asset_shares`, default off) — computed,
framed as a consideration with the disclaimer. A deterministic **grounding guard** drops any
surfaced candidate whose ticker is not present in the system's market-data universe — the assistant
never invents a ticker.

#### Acceptance Criteria

- [ ] **FII** candidates are drawn from / validated against the SPEC-006 FII universe; a candidate
      naming an unknown ticker is **dropped** before reaching the user (test-enforced). **Fixed
      income** candidates are **type-level** (e.g. "pós-fixado ≥100% CDI"), grounded in the live
      macro, with no specific named product (no ingested FI product universe).
- [ ] Each surfaced candidate carries reasoning + risks/trade-offs + assumptions (FR-013/FR-020).
- [ ] Any per-candidate share shown is **computed** and presented as an _illustrative consideration_
      within its area — never an imperative, a share quantity, a price target, or a guarantee (FR-014).
- [ ] Naming a candidate is **not** an order (FR-014); the gate still strips any order signature.

### FR-1055 — Explainability & Non-Advice, by construction (FR-013/FR-014)

All user-facing text is emitted **only** through the gated `Insighter` (SPEC-005). The engine does
not re-implement or bypass the gates and constructs no AI text itself.

#### Acceptance Criteria

- [ ] Every returned area/candidate carries an explanation; the response carries the non-advice
      disclaimer.
- [ ] An output bearing a transaction order is rejected by the gate (fail-closed), never reaching
      the user — verified end-to-end.

### FR-1056 — API

`POST /rebalancing` returns the suggested areas (each with its computed `suggested_share_bps`),
the named grounded candidates listed inside their area, the disclaimer, and an `available` flag.
Auth-protected; identity from context. An optional `include_asset_shares` flag (default false)
opts into the per-candidate illustrative share (D6).

#### Acceptance Criteria

- [ ] `POST /rebalancing` with `{contribution_centavos}` → `200` with areas (+ computed shares),
      candidates, and disclaimer; `include_asset_shares:true` adds the per-candidate illustrative share.
- [ ] Unauthenticated → `401`; invalid amount → `400`; documented in `api/openapi.yaml` (drift test).

### FR-1057 — Graceful Degradation

A full LLM outage returns `200` with `available:false` and empty suggestions (friendlier than a
503). Money/rates never cross the wire as floats.

#### Acceptance Criteria

- [ ] LLM fully unavailable → `200 available:false`, empty areas/candidates, disclaimer present.

### FR-1058 — Observability

The `POST /rebalancing` route is traced; fact-building reuses the `insight.facts` span; the
Insighter records its generation span. **No PII** (no facts, figures, profile, contribution
amount, or generated text) on spans/logs (BR-505).

#### Acceptance Criteria

- [ ] A span-content test asserts no fact values / contribution amount / generated text on spans.

### FR-1059 — Documentation

Closeout updates `README` (the endpoint), `CHANGELOG`, `api/openapi.yaml`, and a PT-BR lesson.

#### Acceptance Criteria

- [ ] `docs/lessons/SPEC-105-aula.html` produced; OpenAPI in lockstep.

## 4. User Flows

### Main Flow

1. The authenticated investor submits a contribution amount (`POST /rebalancing`).
2. The engine reuses `BuildFacts` for the portfolio facts, adds the amount + a grounded candidate
   universe, and calls the gated Insighter for the `rebalancing` task.
3. The grounding guard drops any candidate with an unknown ticker.
4. The system returns suggested areas + grounded candidates (each explained) + the disclaimer.

### Alternative Flows

- **Empty portfolio:** still returns first-investment area guidance for the contribution.
- **LLM outage:** `200 available:false`, empty suggestions, disclaimer present.
- **Invalid amount:** `400` before any LLM call.

## 5. Business Rules

### BR-1051

Contribution is **`int64` centavos, `> 0`**, parsed at the edge (parse-don't-validate); it never
crosses the JSON boundary as a float.

### BR-1052

The engine **reuses** the published `engine.BuildFacts` seam (SPEC-104) for portfolio facts — it
does not duplicate the dashboard/profile/macro reads.

### BR-1053

**Candidate names are computed, not generated:** a surfaced candidate must name a ticker the
system actually knows (the SPEC-006 universe); unknown tickers are dropped deterministically.

### BR-1056

**Suggested percentages are computed, not generated.** The per-area split (`suggested_share_bps`)
is produced by a deterministic allocator in `internal/platform/money` (half-up, sign-aware, Σ =
10 000), **before** the LLM call, and passed as a fact; the Insighter explains it but never
produces or alters the numbers. A **percentage split across areas/sections is a consideration**,
not a transaction order — it carries no share quantity, price/entry-exit target, imperative
buy/sell, or guarantee, so it is outside the FR-014 order signature and the disclaimer still applies.

### BR-1054

User-facing text is emitted **only** through the gated `Insighter`; the gates (FR-013/FR-014) are
inherited from SPEC-005 and **never re-coded** in this feature.

### BR-1055

Identity comes from the session context (`auth.UserID(ctx)`); per-user scoping is inherited via
the Fact Builder's reads. No client-supplied `user_id` is trusted.

## 6. Domain Model

### Value objects

- **Contribution** — `int64` centavos, constructor validates `> 0`, returns a sentinel on invalid.
- The `rebalancing` **`insight.Task`** (a new task value; SPEC-005 is task-agnostic).

### Deterministic allocator (the computed split)

- **`ContributionSplit`** — a pure function `Split(current []ClassSlice, profile, contribution) →
  []AreaShare` computing `suggested_share_bps` per area (half-up, sign-aware, **Σ = 10 000**),
  mirroring the dashboard's `money.ShareBps` discipline (SPEC-103). Lives in
  `internal/platform/money` (or the rebalancing domain) — no SQL/HTTP/LLM, fully unit-tested.
- **AreaShare** — `Area` (class/sector) + `suggested_share_bps` + `suggested_amount_centavos`.

### Computed / produced (no persistence)

- **Rebalancing** result — `Areas []Area`, `Candidates []Candidate`, `Disclaimer`, `Available`,
  where each `Area` carries the computed `suggested_share_bps` + the gated explanation. (Whether
  areas/candidates are modelled as gated `insight.Insight` items tagged by category, joined to the
  computed shares, is **D3**.)
- **Candidate** — a grounded suggestion: ticker + sector + explanation + optional illustrative share.
- No new tables.

## 7. Ports

### Reused (consumed)

- **`engine.FactBuilder` / `BuildFacts(ctx, userID)`** — the published SPEC-104 seam (portfolio facts).
- **`insight.Insighter`** — the gated chain (SPEC-005); a new `rebalancing` task.

### New input port

- **`UniverseReader`** — the grounded FII candidate universe (tickers per sector, with dividend
  yield), backed by market data (SPEC-006). **Requires a new market-data read** (`ListFIIQuotes` /
  by-sector) — today only `GetFIIQuoteByTicker` exists. (See **D2**.)

```go
// Consumer-defined; satisfied by a market-data adapter at the edge.
type UniverseReader interface {
    ListFIIUniverse(ctx context.Context) ([]marketdata.FIIQuote, error)
}
```

## 9. Edge Cases

- **Empty portfolio + contribution** → first-investment area guidance; the computed split follows
  the profile-implied direction (no current allocation to rebalance against).
- **Computed split rounding** → the per-area `suggested_share_bps` always reconcile to `10 000`
  exactly (the half-up remainder is absorbed deterministically, like the dashboard's allocation).
- **No FII universe yet** (fresh DB, ingestion not yet run) → FII candidates omitted; areas + the
  macro-priced fixed-income guidance still produced.
- **LLM names an unknown / hallucinated ticker** → dropped by the grounding guard (BR-1053).
- **LLM emits an order** ("compre 100 cotas de X") → rejected by the gate (FR-014), never surfaced.
- **Contribution ≤ 0 / float / missing** → `400`, no LLM call.
- **Cancelled request** mid-generation → abort, not a silent degrade.

---

## 10. Security Considerations

- **Auth:** all routes require a valid session; identity from context (SPEC-003).
- **Prompt-injection surface:** holdings/profile values + the contribution flow into the facts; the
  non-advice gate is the backstop, and the grounding guard bounds candidate names to real tickers.
- **Money integrity:** contribution is `int64` centavos end to end; never a float on the wire.
- **No PII in telemetry** (BR-505): no facts, contribution amount, or generated text on spans/logs.

---

## 11. Observability

- **Traces:** the `POST /rebalancing` route span; the reused `insight.facts` span (no content); the
  Insighter's `insight.generate` span (SPEC-005).
- **Metrics (optional):** `rebalancing.requests` by outcome (ok / degraded / empty-universe).
- **Logs:** structured, at the edges; never the contribution amount, facts, or generated text.

## 12. Testing Strategy

### Unit Tests

- Contribution parsing (`> 0`, float/string rejected).
- **Contribution allocator** — `suggested_share_bps` per area sums to exactly `10 000`
  (reconciliation invariant), half-up rounding, deterministic (table-driven); empty-portfolio
  split follows the profile-implied direction; no float.
- Fact augmentation (contribution + universe + computed split added; reuses `BuildFacts`;
  deterministic; no float).
- **Grounding guard** — a candidate with an unknown ticker is dropped; a known one is kept.
- Engine: areas produced; empty-portfolio still guides; degradation → `available:false`; AI text
  **only via the Insighter** (marker test).
- Handler: identity-from-context, `400` on bad amount, `401`, degraded `200`.

### Integration Tests (gated by `testing.Short()` + `TEST_DATABASE_URL`)

- Real Postgres + the **gated fake Insighter**: seed holdings + quotes (a universe) + profile,
  `POST /rebalancing`, assert **every area/candidate carries an explanation**, the **disclaimer is
  present** (gates hold end to end), candidates are **grounded** in the seeded universe, and
  per-user isolation holds.

### Quality gate

- `task vet` + `task test:short` green; gated integration green with a DB; gofmt clean;
  hexagonal-reviewer + go-correctness-reviewer pass; `/security-review` (AI-output safety).

## 14. Decisions (resolved)

| # | Decision | Recommendation |
| - | -------- | -------------- |
| D1 | Where the engine lives | A new feature package **`internal/rebalancing`** consuming the published `engine.FactBuilder` seam + the `Insighter` (mirrors SPEC-108's own `internal/chat`). Keeps features separate; the insight core stays pure. (Alt: another service inside `internal/insight/engine` — heavier coupling.) |
| D2 | Named candidates in the MVP? | **Resolved: YES.** Include named FII candidates, adding a small `ListFIIUniverse` read over the existing `fii_quotes` table (SPEC-006). FI candidates stay type-level (no product universe). |
| D3 | Candidate output shape | Model areas **and** candidates as gated `insight.Insight` items tagged by category (`area` / `candidate`, ticker in the title), so everything flows through the **existing** SPEC-005 gate unchanged; split them in the response DTO. (Alt: a richer `Candidate` struct that also passes the gate — more fields, more surface.) |
| D4 | Target allocations | **Qualitative** for the MVP — reason over the profile + current allocation; **defer** explicit numeric per-class targets (shared open question with SPEC-104). |
| D5 | Degraded HTTP shape | `200` with `available:false` + empty suggestions on a full LLM outage (consistent with SPEC-104 FR-1047). |
| D6 | Granularity of the computed % | **Resolved.** Default output = **area/section-level %** with **named candidates listed inside each area** (no per-asset number by default). A **per-asset illustrative %** (e.g. "~15% to HGLG11 as a consideration") is **opt-in** — surfaced only when explicitly requested via a request flag (`include_asset_shares`), which is the natural path the Conversational Copilot (SPEC-108) sets when the user asks "quanto em cada um?". The allocator computes both; the per-asset value is always framed as a consideration with the disclaimer, never an imperative/quantity. |
| D7 | Does live market data tilt the computed % number, or only the narrative? | **Resolved: narrative only (A).** The computed split is a deterministic function of **allocation + profile** only — visibly mechanical, reproducible. **Live market data (SELIC/CDI/IPCA, FII yields) tilts the LLM's _explanation_**, not the raw number. A market-tilted number stays a documented fast-follow (a deterministic rule-table over rate bands), out of the MVP. |

## 15. Open Questions (deferred, not blocking)

- **Explicit numeric target allocations** (per class, from the risk profile) would sharpen both the
  rebalancing areas and SPEC-104's allocation insights — a shared future input, likely alongside
  SPEC-107.
- **Named-product universe beyond FIIs.** Fixed income is already market-aware via macro, but
  naming specific products needs a product universe — **Tesouro Direto** is a finite, public,
  nameable set (Tesouro Selic / IPCA+ / Prefixado) that a future SPEC-006 ingestion could add,
  turning FI candidates from type-level into named products. Stocks/ETFs need their own ingestion.
- **Contribution source** — SPEC-105 takes the amount per request; a saved monthly contribution
  (profile/projection) could prefill it later (SPEC-107 / SPEC-108).
