Arquitetura Hexagonal — Ports & Adapters

Você já domina handler → DTO → service → repository. Esta aula é a ponte: o hexagonal não é um mundo novo — é a mesma ideia com a direção das dependências invertida e as interfaces (ports) pertencendo ao núcleo. Vamos chegar nesse "aha" usando o código real do YieldForge.

1. Ponto de partida: o que você já sabe

Você já construiu dezenas de aplicações em camadas. O fluxo é familiar e bom:

ARQUITETURA EM CAMADAS (o que você já domina) HTTP request │ ▼ ┌──────────┐ recebe o request, lê o DTO, │ Handler │ chama o service, devolve a resposta └────┬─────┘ │ depende de ▼ ┌──────────┐ regra de negócio / caso de uso │ Service │ orquestra; chama o repositório └────┬─────┘ │ depende de ▼ ┌──────────┐ fala com o Postgres │Repository│ (struct concreta, com *sql.DB dentro) └────┬─────┘ │ ▼ Banco de dados A SETA SEMPRE APONTA PARA BAIXO: cada camada IMPORTA a camada de baixo. O Service IMPORTA o Repository concreto. Tudo flui em direção ao banco.

Isso funciona. Mas em algum momento você sente um incômodo: o Service — onde mora a regra de negócio, a parte mais valiosa e mais testável do sistema — importa o repositório concreto, que por sua vez importa database/sql, drivers, e detalhes do Postgres. Para testar a regra, você precisa de um banco (ou de um mock pesado). Para trocar o Postgres por outra coisa, você mexe no que o service importa. O centro da sua aplicação depende das bordas.

O incômodo em uma frase Na arquitetura em camadas, a dependência aponta para fora: o núcleo (regra de negócio) conhece e importa os detalhes externos (SQL, HTTP, SDKs). A arquitetura hexagonal faz uma única cirurgia: ela inverte essa seta. O núcleo passa a não conhecer ninguém de fora — são as bordas que conhecem o núcleo. Guarde essa frase; é o aha desta aula inteira.

2. A metáfora do hexágono

"Hexagonal" é só um desenho — o número de lados não importa (são seis por tradição, para caber vários adaptadores em volta). O que importa é a divisão dentro vs. fora:

MUNDO EXTERNO (bordas, adapters) ┌─────────────────────────────────────────────┐ │ │ │ HTTP ┌───────────────────────┐ Postgres│ │ handler ─┤ ├─ adapter │ │ (driving)│ NÚCLEO PURO │ (driven) │ │ │ domínio + service │ │ │ CLI ─┤ + PORTS (interfaces)├─ Yahoo │ │ worker │ │ adapter │ │ (driving)└───────────────────────┘ (driven) │ │ ▲ a dependência ▲ │ │ │ APONTA PARA DENTRO │ │ │ quem CHAMA a app a app CHAMA │ │ (lado esquerdo) (lado direito) │ └─────────────────────────────────────────────┘ REGRA DE OURO: tudo de fora depende do núcleo; o núcleo NÃO depende de nada de fora. (no YieldForge: o domínio não importa SQL, HTTP nem SDK)

Dentro do hexágono vive a lógica que é a razão de o software existir: as regras de negócio, os tipos de domínio, os casos de uso. Fora vivem os detalhes substituíveis: qual banco, qual protocolo HTTP, qual provedor de dados de mercado, qual LLM. A fronteira entre os dois é desenhada por interfaces que pertencem ao núcleo — os ports.

A regra de ouro, no CLAUDE.md do projeto Não é uma escolha de estilo — é uma regra binding de arquitetura do YieldForge: "Dependency direction: domain core imports NO SQL, HTTP, or framework types. Ports (interfaces) live with the feature; adapters implement them at the edge." A seta aponta para dentro, sempre.

3. O vocabulário: quatro palavras, definições limpas

Antes da ponte, vamos fixar quatro termos separadamente. Os exemplos vêm do pacote internal/marketdata (dados de mercado: cotações de FIIs + indicadores macro), porque é o código mais recente do projeto.

Domain — o núcleo puro

São os tipos e regras que representam o problema, sem nenhuma tecnologia. Um value object como Ticker que se valida no construtor (parse, don't validate): um Ticker inválido não pode existir. Repare: zero import de SQL, HTTP ou rede.

// internal/marketdata/ticker.go
// Ticker is a validated B3 ticker (parse-don't-validate: an invalid Ticker cannot exist).
type Ticker struct{ value string }

func ParseTicker(s string) (Ticker, error) {
    v := strings.ToUpper(strings.TrimSpace(s))
    if !tickerPattern.MatchString(v) {
        return Ticker{}, fmt.Errorf("parse ticker %q: %w", s, ErrInvalidTicker)
    }
    return Ticker{value: v}, nil
}

Port — a interface (o "contrato")

Um port é uma interface Go que o núcleo define e da qual ele depende — mas que ele não implementa. É o "buraco" no hexágono por onde o mundo externo se conecta. No YieldForge, o port de leitura de dados de mercado é:

// internal/marketdata/ports.go
// MarketDataProvider reads current data from an external source — the FR-018 seam that keeps
// the provider swappable by config (BR-601).
type MarketDataProvider interface {
    FetchFIIQuotes(ctx context.Context, tickers []Ticker) (map[Ticker]FIIQuote, error)
    FetchMacroIndicator(ctx context.Context, ind Indicator) (MacroIndicator, error)
}

Note que MarketDataProvider mora no mesmo pacote do domínio (package marketdata), fala só a linguagem do domínio (Ticker, FIIQuote) e não menciona Fundamentus, Yahoo ou HTTP. É um contrato escrito pelo cliente, não pelo fornecedor.

Adapter — a implementação na borda

Um adapter é uma struct concreta que satisfaz um port e que conhece uma tecnologia específica. Aqui é onde o net/http, o parsing de HTML e a URL do Fundamentus vivem — confinados. O adapter importa o domínio; o domínio jamais importa o adapter.

// internal/marketdata/fundamentus/fundamentus.go
// HTTP/parsing live here only; the core stays pure (BR-601).
type Adapter struct {
    baseURL string
    client  *http.Client    // ← o detalhe HTTP fica AQUI, na borda
    clock   clock.Clock
}

func (a *Adapter) FetchFIIQuotes(ctx context.Context, tickers []marketdata.Ticker)
    (map[marketdata.Ticker]marketdata.FIIQuote, error) { /* ...scrape, parse... */ }

Service / use-case — o orquestrador (também no núcleo)

O service continua existindo e continua sendo o orquestrador da regra de negócio — mas agora ele vive dentro do hexágono e depende só de ports, nunca de implementações concretas. Veja o service de autenticação: seus campos são todos interfaces (UserRepository, PasswordHasher) e o Clock:

// internal/auth/service.go
// Service ... depends only on ports (repositories, the password hasher) and the Clock, so it is
// pure application logic — unit-testable with hand-written fakes, no DB or HTTP.
type Service struct {
    users    UserRepository    // ← port (interface), não a struct do Postgres
    sessions SessionRepository // ← port
    hasher   PasswordHasher    // ← port
    clock    clock.Clock       // ← port
}
"Accept interfaces, return structs" É a convenção da casa, e ela é o hexagonal em uma frase. O construtor do service aceita interfaces (ports) — então não sabe (nem se importa) se por trás está o Postgres real ou um fake de teste. E os adapters retornam structs concretas (ex. fundamentus.New(...) devolve um *Adapter). O ponto onde interface e struct se encontram é o seam — a costura — onde você troca implementações sem tocar no núcleo.

4. Os dois lados: driving vs. driven

Nem todo port é igual. Há dois lados no hexágono, e confundi-los é a fonte de metade da confusão inicial. A pergunta que decide o lado é simples: quem chama quem?

 Driving (lado esquerdo)Driven (lado direito)
Quem chama? O mundo externo chama a aplicação ("me dê as cotações"). A aplicação chama o mundo externo ("Postgres, salve isto").
Sinônimo primary / inbound / "quem dirige" secondary / outbound / "o que é dirigido"
Adapter típico Handler HTTP, comando CLI, worker agendado, consumidor de fila. Repositório (DB), provedor externo, cache, relógio, e-mail.
No YieldForge O apiHandler em internal/transport/http; o worker de ingestão que dispara FetchFIIQuotes. MarketDataProvider (Fundamentus/Yahoo), FIIQuoteRepository (Postgres), Clock, Insighter (LLM).
Por que o handler HTTP é só um adapter (e não "o topo do mundo") Na cabeça de camadas, o handler é o topo da pilha — onde tudo começa. No hexagonal, o handler é só um dos vários adaptadores de entrada possíveis, do lado de fora, igualzinho ao Postgres está do outro lado. O HTTP é um detalhe: amanhã o mesmo caso de uso pode ser disparado por um worker agendado (é exatamente o caso da ingestão de mercado) ou por um CLI, sem o núcleo perceber. O núcleo não sabe — e não deve saber — como foi chamado.

5. A PONTE: de-para camadas → hexagonal

Aqui está o coração da aula. Quase tudo que você conhece continua existindo; muda a posição de cada peça em relação à fronteira e a direção da seta.

Em camadas (o que você sabe)No hexagonal (o equivalente)O que de fato muda
Handler / Controller Driving adapter (adaptador de entrada) Praticamente nada de código — muda o papel: deixa de ser "o topo" e vira "um adaptador da borda esquerda". Vive em internal/transport/http, fora do núcleo.
DTO (request/response) Continua sendo DTO, na borda Nada muda: o DTO segue traduzindo JSON da rede ↔ value objects do domínio (ParseTicker). DTO é detalhe de transporte; nunca entra no núcleo.
Service (regra de negócio) Service / use-case, agora no núcleo O método e a lógica são os mesmos. A virada: ele passa a depender de ports (interfaces), não de structs concretas. Era users *PostgresUserRepo; vira users UserRepository.
Repository (1 struct concreta com *sql.DB) Port (interface no núcleo) + Adapter (struct na borda) A peça se parte em duas. A interface (FIIQuoteRepository) sobe para o núcleo, escrita na linguagem do domínio. A implementação Postgres desce para a borda (marketdata/postgres).
Cliente de API externa (struct concreta) Driven port + adapter Idem: interface no núcleo (MarketDataProvider), implementação concreta na borda (Fundamentus, Yahoo). Trocar de provedor = trocar o adapter, sem tocar no núcleo.
main / DI container Composition root (a fiação) O lugar — e o único lugar — que importa os adaptadores concretos e os injeta nos services. No YieldForge é o cmd/api + as factory.
A grande virada, isolada (leia duas vezes) Em camadas: o Service importa o repositório concreto. Seta: núcleo → fora.
No hexagonal: o Service define a interface e o adaptador a implementa. Seta: fora → núcleo.
É literalmente a mesma chamada repo.GetFIIQuoteByTicker(...) no service. A diferença é quem possui o tipo de repo: antes era o pacote do banco (e o service o importava); agora é o pacote do domínio (e o banco o implementa). Isso é a Inversão de Dependência — o "D" do SOLID — e é tudo que o hexagonal faz de novo.

Os mesmos dois fluxos, lado a lado, para o "salvar uma cotação":

EM CAMADAS HEXAGONAL ───────── ───────── Service ──importa──▶ PgRepo Service ──usa──▶ FIIQuoteRepository (PORT, no núcleo) │ (struct concreta) │ ▲ │ │ │ implementa ▼ ▼ │ Postgres (em runtime) postgres.FIIQuoteRepository (ADAPTER, borda) │ seta núcleo → banco ▼ (núcleo conhece o banco) Postgres seta banco → núcleo (banco conhece o núcleo; o núcleo só conhece a interface que ele mesmo definiu)

6. Como o YieldForge faz isso — o caso marketdata

Vamos ver os quatro conceitos juntos, num exemplo real e bonito: a montagem do MarketDataProvider de FIIs. O dado de uma cotação de FII vem de duas fontes gratuitas diferentes — fundamentos (preço, DY, P/VP, segmento) do Fundamentus, e o último dividendo do Yahoo. O projeto compõe as duas atrás do mesmo port.

Layout físico: a feature dona do port, os adaptadores ao lado

internal/marketdata/            ← NÚCLEO PURO (package marketdata)
├── ticker.go  quote.go  macro.go  sector.go   domínio (value objects)
├── ports.go                                   PORTS: MarketDataProvider, *Repository, TickerSource
├── errors.go                                  sentinelas (ErrProviderUnavailable, ...NotFound)
├── fake.go                                    Fake provider determinístico (CI/dev)
├── fii/composite.go           ← ADAPTER: compõe duas fontes num provider
├── fundamentus/fundamentus.go ← ADAPTER: scrape HTML do Fundamentus (net/http)
├── yahoo/yahoo.go             ← ADAPTER: API do Yahoo (último dividendo)
├── bcb/bcb.go                 ← ADAPTER: SGS do Banco Central (macro)
└── postgres/postgres.go       ← ADAPTER: repositórios sobre PostgreSQL

Repare na física: o port mora com a feature (ports.go, dentro do pacote do domínio) e cada adaptador mora num subpacote ao lado, na borda. É a regra de arquitetura do projeto tornada estrutura de diretórios.

O port "Composite": interfaces minúsculas, definidas pelo consumidor

O fii.Composite precisa de duas coisas — fundamentos e dividendos. Mas ele não importa os adaptadores concretos; ele declara dois ports minúsculos, do tamanho exato do que consome, e depende deles:

// internal/marketdata/fii/composite.go
// It depends on small consumer-defined interfaces, not the concrete adapters, so
// either source can be swapped behind the marketdata port (BR-601).

type Fundamentals interface {  // ← port de 1 método, definido por quem CONSOME
    FetchFIIQuotes(ctx context.Context, tickers []marketdata.Ticker)
        (map[marketdata.Ticker]marketdata.FIIQuote, error)
}
type LastDividends interface { // ← idem, e "best-effort"
    FetchLastDividend(ctx context.Context, t marketdata.Ticker)
        (centavos int64, date *time.Time, err error)
}

type Composite struct {
    fundamentals Fundamentals  // ← guarda interfaces, não *fundamentus.Adapter
    dividends    LastDividends
}

Esse é o princípio "interfaces small + consumer-defined" do CLAUDE.md em ação: o Composite não diz "me dê o adaptador do Yahoo"; ele diz "me dê qualquer coisa que saiba buscar o último dividendo". Em produção, é o Yahoo; num teste, é um fake de três linhas.

A prova de que o adapter implementa o port: a checagem em tempo de compilação

Como o Go satisfaz interfaces estruturalmente (sem implements), o projeto usa um truque para garantir, na compilação, que cada adaptador honra seu port:

// internal/marketdata/postgres/postgres.go
// Compile-time checks that the adapters satisfy the marketdata ports.
var (
    _ marketdata.FIIQuoteRepository = FIIQuoteRepository{}
    _ marketdata.MacroRepository    = MacroRepository{}
)

Se alguém mudar a assinatura do port e esquecer de atualizar o adaptador, o build quebra — não um teste em runtime, o build. A direção da dependência aqui é explícita: o pacote postgres (borda) importa marketdata (núcleo). Nunca o contrário.

O Fake: o que ports tornam quase grátis

Porque MarketDataProvider é uma interface pequena, escrever uma implementação determinística para testes e dev é trivial — e ela é o provedor default, para a app rodar sem rede nem configuração:

// internal/marketdata/fake.go
// Fake is a deterministic MarketDataProvider for tests, CI, and offline/dev — it is the
// default MARKETDATA_PROVIDER so the zero-config app never hits the network (FR-611).
type Fake struct{ At time.Time }

var _ MarketDataProvider = Fake{}   // ← também provado em compile-time

Onde tudo se amarra: o composition root

O núcleo e os adaptadores nunca se conhecem diretamente. Quem os apresenta é a fiação — o único lugar que importa os concretos e escolhe o adaptador por configuração:

// pseudo-fiação (composition root, em cmd/api / factory) — escolhe por config
var provider marketdata.MarketDataProvider
switch cfg.MarketDataProvider {
case "fake":
    provider = marketdata.Fake{}
default: // "live"
    fund := fundamentus.New(cfg.FundamentusBaseURL, cfg.Timeout, clk)
    yh   := yahoo.New(cfg.YahooBaseURL, cfg.Timeout, clk)
    provider = fii.New(fund, yh)   // ← compõe os dois adaptadores num só port
}
// daqui pra baixo, o worker só conhece `provider marketdata.MarketDataProvider`
O ecossistema de ports do YieldForge (todos os provedores externos) O marketdata não é exceção — é a regra. O CLAUDE.md lista os seams que devem permanecer atrás de ports: Insighter (o LLM, SPEC-005), MarketDataProvider (SPEC-006), os repositórios, e o Clock. Cada um é uma interface no núcleo de sua feature, com adaptadores trocáveis por config. Você verá o mesmo padrão em auth (UserRepository, PasswordHasher) e nas features futuras (portfolio/, profile/, projection/).

7. Por que vale a pena (no contexto deste projeto)

Inverter a seta dá trabalho — um arquivo de interface a mais, um pouco de cerimônia na fiação. Para um produto com as restrições binding do YieldForge, o retorno é direto:

BenefícioO que ele resolveExemplo concreto no YieldForge
Swappability por config (custo zero) Restrição #4 do PRD: tudo free/local, todo provedor trocável por .env. Trocar o LLM (INSIGHTER_PROVIDER=ollama|groq|fake) ou o dado de mercado (MARKETDATA_PROVIDER) é uma linha de config, nunca uma mudança de código.
Testabilidade Testar a regra de negócio sem banco, sem rede, sem LLM. O Fake (3 dezenas de linhas) entra no lugar do adaptador real; o CI roda a feature inteira offline e de graça. Fakes escritos à mão — sem gomock (convenção da casa).
Domínio puro e determinístico Mesmos inputs → mesmo resultado (PRD: mesmo Health Score reproduzível). Tempo vem do Clock (não time.Now()); dinheiro é int64 centavos. A staleness de uma cotação (StaleAfter) é testável sem sleep frágil.
Confinamento do "feio" Scraping de HTML, drivers, SDKs — coisas frágeis — não contaminam o núcleo. O parsing defensivo do Fundamentus (colunas achadas por palavra-chave, degradação para ErrProviderUnavailable) fica todo dentro de fundamentus/.
O seam do futuro Multi-agente CIO + ferramentas MCP (PRD G9/G10) sem reescrever o núcleo. O Insighter é a fronteira onde, amanhã, um orquestrador de agentes se pluga — atrás da mesma interface, herdando os mesmos guardrails. "Keep it intact" é regra de arquitetura.
Quando NÃO vale a cerimônia Hexagonal não é gratuito, e o YieldForge não o aplica cegamente. Código cross-cutting que não é regra de negócio — config, logging, o servidor HTTP, buildinfo — vive em internal/platform/ sem o ritual de port+adapter completo. A pergunta é sempre: "este detalhe é substituível e eu quero proteger o núcleo dele?". Se sim (banco, LLM, dado de mercado), vira port. Se é mera infra de plataforma, fica simples.

8. Armadilhas comuns na tradução

Erro nº 1: colocar a interface do lado do adaptador Vindo de camadas, é tentador criar package postgres e definir ali a interface Repository, e o service importar postgres.Repository. Isso é camadas com um nome chique — a seta ainda aponta para fora. O port tem que morar no núcleo, no pacote da feature (package marketdata), escrito na linguagem do domínio. Quem implementa é que importa o núcleo, não o contrário.
Erro nº 2: vazar tipos da borda para o núcleo Se o seu port retorna *sql.Rows, ou recebe um *http.Request, ou devolve o JSON cru do provedor, você furou a fronteira — o detalhe externo entrou no núcleo. O port do YieldForge fala só domínio: FetchFIIQuotes(ctx, []Ticker) (map[Ticker]FIIQuote, error). Nenhum Ticker, FIIQuote ou erro aqui sabe que o Postgres ou o Fundamentus existem.
Erro nº 3: confiar em user_id que veio do payload Específico do projeto, mas crucial: a identidade vem do contexto autenticado, nunca do DTO de request (CLAUDE.md). O adaptador HTTP (driving) extrai o usuário da sessão e o passa adiante; o núcleo escopa com WHERE user_id = $1 usando o ID do contexto. Um port que aceitasse userID vindo do cliente seria um buraco de segurança disfarçado de boa arquitetura. (No marketdata, aliás, não há escopo por usuário — dado de mercado é global, BR-603 — e isso também é uma decisão de domínio que o port reflete.)

9. Harness Engineering: como as regras viram trilhos

Este projeto tem propósito duplo: aprender Go/backend e aprender engenharia de harness — dar à IA (e aos humanos) trilhos que baixam a taxa de erro. A arquitetura hexagonal só se mantém honesta se algo a impede de erodir a cada commit. No YieldForge, várias dessas defesas agiram nas specs que você já viu.

O subagente hexagonal-reviewer

É um revisor especializado que conhece a regra de ouro e cobra a direção da dependência. A pergunta que ele faz a cada PR: "algum pacote de núcleo importou SQL, HTTP ou um SDK?". Foi essa lente que garantiu, na SPEC-005, que o core de insight e os adaptadores não importassem OpenTelemetry (só a factory amarra), e que mantém o marketdata core sem nenhum net/http.

A checagem em tempo de compilação como trilho

Aquele var _ marketdata.FIIQuoteRepository = FIIQuoteRepository{} não é decoração — é um hard rule barata. Ele transforma "esqueci de implementar um método do port" de um bug de runtime num erro de compilação. O harness prefere sempre o trilho mais barato e mais cedo: o compilador pega antes do teste, que pega antes do review, que pega antes da produção.

O doc-comment que cita a BR/FR

Toda peça desta aula tem um comentário citando o SPEC/BR que ela implementa — (SPEC-006 BR-601) no ports.go, (BR-603) no quote.go. Isso fecha o ciclo da SDD: do documento ao código há um traço direto e auditável. Quando o revisor lê "the core stays pure (BR-601)" no topo do fundamentus.go, doc e código contam a mesma história — e qualquer divergência salta aos olhos.

Os hooks que agem ao fechar

  • gofmt: todo arquivo Go sai formatado — diffs sem ruído de estilo.
  • O backstop do Stop: ao encerrar uma tarefa, o gate exige build / vet / test -short / gofmt limpos. É o que prova, mecanicamente, que o núcleo ainda compila sem os adaptadores e que os fakes cobrem o CI sem banco e sem rede.
  • Bloqueio de imutabilidade: ADRs aceitos (como o ADR-0003, custo-zero, que abençoa os provedores trocáveis) não podem ser editados — só superados. A arquitetura tem memória.
Por que cada trilho baixa o erro A arquitetura hexagonal é uma disciplina, e disciplinas humanas falham sob pressão. O harness converte cada regra de "lembre-se de não importar SQL no núcleo" em algo estrutural: um revisor que cobra, um compilador que quebra, um doc-comment que denuncia, um hook que barra. Quanto mais a regra de ouro vira hard rule verificável, menos ela depende de alguém lembrar.

10. Conexão com AI Engineering

A arquitetura hexagonal não é só boa engenharia de software — no YieldForge ela é o que torna a IA confiável possível. Os dois recursos mais voláteis e perigosos do produto — o LLM e o dado de mercado externo — são exatamente os que ficam atrás de ports.

Conceito de AI EngineeringComo o hexagonal o viabiliza
O LLM como detalhe trocável O Insighter (SPEC-005) é um port de um método. Ollama local, Groq hospedado ou um fake determinístico entram pela mesma costura. O modelo — a parte mais cara e instável — vira configuração, não acoplamento.
Verification middleware (os gates) Como tudo é Insighter ⇒ Insighter, os portões de explicabilidade (FR-013) e não-aconselhamento (FR-014) viram um decorator que embrulha qualquer provider. Nenhuma saída de IA escapa da verificação — e isso só é possível porque o LLM está atrás de uma interface uniforme.
Grounding / Fact Builder "Fatos são computados, não gerados": o LLM raciocina sobre dados determinísticos. Esses fatos vêm, em parte, do marketdata desta aula — cotações e indicadores que o domínio puro produz, com dinheiro em centavos e tempo do Clock. O núcleo determinístico é o que ancora a IA na realidade.
O seam do multi-agente (CIO + MCP) Hoje há um provedor atrás do Insighter; amanhã pode haver um orquestrador de agentes raciocinando sobre os mesmos fatos, ainda atrás da mesma interface, ainda passando pelos mesmos gates. O port é a fronteira de extensão — mantida intacta desde o dia um, de propósito.

Handler vira driving adapter, repositório vira port + adapter, o service continua o orquestrador mas agora depende só de contratos. No YieldForge isso é o que torna LLM e dado de mercado trocáveis por config, o domínio puro e testável, e a IA verificável por construção.
