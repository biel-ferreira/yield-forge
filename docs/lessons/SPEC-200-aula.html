SPEC-200 — Fundação do Frontend (o "SPEC-001" do cliente web)

O backend inteiro (SPEC-001…108) só entrega valor por uma API. O SPEC-200 transforma esse contrato numa superfície web usável — e faz isso de um jeito em que as travas binding do produto (explicabilidade FR-013, não-conselho FR-014) e a regra dinheiro-nunca-float deixam de ser disciplina de backend e viram estrutura no cliente. Esta spec não entrega nenhuma tela de feature: ela é a fundação — a "costura" onde as telas SPEC-21x plugam. A grande lição não é React; é como o contrato OpenAPI vira a fronteira única entre front e back, como as guardas do produto viram componentes que o usuário vê, e como o shell já reserva o lugar do copiloto — preparando o terreno para a SPEC-215 e a visão multi-agente/MCP. Você não precisa dominar React para acompanhar.

1. A visão geral: a fundação que torna as travas do produto ESTRUTURAIS

Toda a Fase 1 do backend está pronta: perfil, carteira, dashboard, insights, rebalanceamento, health score, projeções e o copiloto conversacional. Mas nada disso é visível para o usuário — vive atrás de endpoints JSON. O SPEC-200 é o primeiro passo do track de frontend (SPEC-2xx): ele monta o cliente web sob web/ e tudo que cada tela futura vai depender. É o análogo frontend do SPEC-001 — o esqueleto, a fiação, as convenções — não uma feature.

O que ele entrega, em código real:

  • Um app Next.js 16 (App Router) + React 19 + TypeScript strict + Tailwind v4 sob web/, isolado do módulo Go, com CI path-scoped (só roda no que muda em web/**).
  • O design system Aurora como código — tokens em @theme no globals.css, três fontes self-hosted (Inter · Fraunces · IBM Plex Mono) — com as guardas binding como componentes estruturais: um InsightCard cujo prop explanation é obrigatório (FR-013) e um NonAdviceDisclaimer (FR-014). Nenhum botão de comprar/vender existe em lugar nenhum.
  • Um cliente de API tipado gerado de api/openapi.yaml (openapi-typescript + openapi-fetch + TanStack Query) — zero DTOs à mão — com um guardião de deriva (npm run check:api, o espelho do openapi_test.go do backend); o money.ts formata centavos/bps inteiros para pt-BR só na borda de renderização; e um transporte SSE de streaming pronto para o chat da SPEC-215.
  • Auth & sessão contra a SPEC-003 via um proxy same-origin do Next.js (o cookie HttpOnly funciona sem CORS); useSession resolve a identidade por /auth/me (nunca do estado do cliente).
  • O shell do app — sidebar / abas no mobile, top bar, e o launcher flutuante global do copiloto em toda tela — com rotas de feature "stubadas" (estados vazios) e boundaries de loading/erro compartilhados.

Por que "fundação estrutural" é a decisão-mãe desta spec A tentação numa base de frontend é entregar logo uma tela bonita. O SPEC-200 faz o oposto: entrega zero telas de produto e investe todo o esforço em tornar as regras do produto impossíveis de violar por acidente nas telas que virão. Se o InsightCard exige a explicação como prop obrigatório, nenhuma tela futura consegue mostrar um insight sem o "por quê". Se o único caminho para formatar dinheiro é o money.ts, nenhuma tela inventa um toFixed com float. É parse-don't-validate aplicado a props de componente — o mesmo princípio do backend, agora na borda visual. Cada tela SPEC-21x herda as travas de graça.

3. A jornada do usuário: do login à casca autenticada

O único fluxo vivo que o SPEC-200 entrega ponta a ponta é a autenticação e a casca vazia. É simples de descrever e revela as decisões de produto importantes:

1. Usuário abre o app ─▶ não autenticado ─▶ redireciona para /login 2. Envia credenciais ─▶ backend seta o cookie HttpOnly (via o proxy same-origin) 3. /auth/me resolve o usuário ─▶ o shell autenticado renderiza (sidebar + top bar + copiloto) 4. Recarrega a página ─▶ o cookie persiste a sessão; nada some 5. Clica em cada item do menu ─▶ rota "stubada": um ESTADO VAZIO honesto ("chega na SPEC-2xx") 6. Logout ─▶ o cookie é limpo, o cache é descartado ─▶ volta para /login Sessão inválida no meio do caminho: qualquer chamada protegida retorna 401 ─▶ estado limpo ─▶ redireciona, SEM vazar dado protegido

Repare em duas decisões de produto na jornada. A primeira: as rotas de feature são estados vazios honestos — não telas quebradas nem placeholders enganosos. O DashboardPage, por exemplo, renderiza um EmptyState que diz exatamente o que vem e quando:

// web/app/(app)/dashboard/page.tsx — o estado vazio é honesto, não um placeholder falso
export default function DashboardPage() {
  return (
    <EmptyState
      title="Painel"
      description="Seu resumo de patrimônio, alocação por setor, Health Score e insights
                   do copiloto chega na SPEC-212 (Dashboard)."
    />
  );
}

A segunda: a não há rota "Chat" no menu. O copiloto é um widget flutuante global, não uma página — uma decisão de produto deliberada (seção 8), registrada em código no nav.ts.

4. LIÇÃO PRINCIPAL — as travas viram UI: o InsightCard, o disclaimer, a ausência de botões

Esta é a ideia central da aula. As duas guardas binding do produto — explicabilidade (FR-013) e não-conselho (FR-014) — já eram garantidas no backend, dentro do Insighter gated. O SPEC-200 faz uma coisa poderosa: as transforma em componentes que o usuário vê e que o desenvolvedor não consegue burlar. É o espelho, na borda visual, do decorator Gated do backend.

4.1 — Explicabilidade por construção: o explanation é prop obrigatório

O InsightCard tem um slot fixo "Por quê" que sempre renderiza, e o texto dele vem de um prop explanation que é obrigatório no contrato de tipos. Um insight sem explicação é literalmente irrepresentável — o TypeScript recusa compilar. Não é disciplina do revisor; é o compilador.

// web/components/insight-card.tsx — o "por quê" é prop OBRIGATÓRIO (FR-013)
export interface InsightCardProps {
  children: ReactNode;         // a manchete do insight
  /**
   * O "por quê" humano. OBRIGATÓRIO por contrato — um insight sem explicação
   * é irrepresentável (FR-013), espelhando o Gated Insighter do backend.
   */
  explanation: ReactNode;      // ← NÃO é opcional. Omitir = erro de tipo.
  category?: string;
  attention?: boolean;
}

// ...e o corpo SEMPRE renderiza o bloco "Por quê":
<div className="mt-3 rounded-md border ...">
  <div className="... text-primary-tint">Por quê</div>
  <div className="... text-muted-strong">{explanation}</div>
</div>

4.2 — Não-conselho: o disclaimer não é opcional, e não há botão de ordem

O NonAdviceDisclaimer é um componente cuja presença é um contrato da tela, não um prop opcional. Ele carrega o texto de "conteúdo educacional, não recomendação" (e uma variante estendida para projeções). E — a parte mais importante — não existe nenhum componente de comprar/vender no design system. Não há <BuyButton>, <OrderCTA>, meta de preço ou campo de quantidade. A ausência é deliberada.

// web/components/non-advice-disclaimer.tsx — presença é contrato da tela (FR-014)
const DEFAULT_TEXT =
  "Isto é conteúdo educacional, não recomendação de investimento. As decisões são suas.";
const EXTENDED_TEXT =
  "As projeções são estimativas não garantidas... YieldForge não emite ordens de compra ou
   venda, metas de preço ou quantidades. Consulte um profissional habilitado antes de decidir.";
Por que "cor de número, nunca ação" é uma decisão de produto (e de UX) Um ganho ou perda no YieldForge aparece só como cor do texto (verde/vermelho) — nunca como um botão, uma seta piscante ou um fill que sugira "agir agora". O revisor frontend-reviewer vigia exatamente isto: "gain/loss aparecem apenas como cor da figura, nunca como ação ou fill". A razão é a trava FR-014 traduzida para UX: o produto é um copiloto que informa, não um corretor que induz. Um botão "Comprar" ao lado de um número verde é uma ordem disfarçada; a sua ausência estrutural é o que mantém o produto do lado certo da linha regulatória e ética. E, para acessibilidade, a cor nunca é o único sinal — sempre acompanha texto/seta (colorblind-safe).
O ÚNICO caminho de uma saída de IA até a tela do usuário: dado do backend ─▶ <InsightCard explanation={...}> ◀── o "por quê" é PROP OBRIGATÓRIO (FR-013) headline ┌─ "Por quê" ─────────────┐ ◀── slot fixo, SEMPRE renderiza └─────────────────────────┘ <NonAdviceDisclaimer /> ◀── presença é contrato da tela (FR-014) NÃO EXISTE no design system: <BuyButton> <SellButton> meta de preço campo de quantidade ganho/perda = SÓ cor de número (+ texto/seta), NUNCA uma ação. ← burlar é impossível por construção

5. Dinheiro em centavos inteiros, pt-BR só na borda (e o bug que isso quase causou)

A regra do backend "dinheiro nunca é float64; é int64 centavos, taxas em basis points" atravessa a fronteira JSON e entra na UI. No cliente, valores fluem por props e estado como inteiros (centavos e bps); a conversão para uma string localizada em pt-BR acontece só na borda de renderização, num único helper: o money.ts — o análogo cliente do internal/platform/money.

// web/lib/money.ts — a ÚNICA borda de formatação; inteiro entra, string pt-BR sai. Sem float.
export function formatCentavos(centavos: number): string {
  const reais = Math.trunc(Math.abs(centavos) / 100);   // divisão INTEIRA
  const cents = Math.abs(centavos) % 100;               // resto INTEIRO
  return `R$ ${reais.toLocaleString("pt-BR")},${String(cents).padStart(2, "0")}`;
}
// formatCentavos(1234567) === "R$ 12.345,67"     formatBps(1050) === "10,50%"

Repare: nunca há centavos / 100 gerando um 12345.67 flutuante — é Math.trunc(.../100) para os reais e % 100 para os centavos, tudo aritmética inteira. A vírgula/ponto do formato brasileiro é montada à mão sobre inteiros.

O bug real que a disciplina "pt-BR só na borda" quase deixou passar — pego pelos revisores Há um detalhe sutil e traiçoeiro no AllocationBar (a barra de alocação por setor, a assinatura visual do Aurora). A largura CSS de cada segmento precisa de um número com ponto decimal (width: "16.5%"). Mas o texto da legenda precisa do pt-BR com vírgula ("16,5%"). Se alguém, "para ser consistente", usasse a string pt-BR também na largura, o CSS width: "16,5%" seria inválido e o segmento colapsaria para largura zero — a barra sumiria, silenciosamente. typecheck, lint e build passam (é uma string válida em TS!); só o olho de um revisor de correção pega. Por isso existe um teste de regressão dedicado: 
// web/components/allocation-bar.test.tsx — a regressão que trava o bug para sempre
it("uses a dot-decimal CSS width for fractional shares", () => {
  // ... bps: 1650
  expect(segmentWidth(container)).toBe("16.5%");   // ← PONTO na largura (CSS válido)
});
it("shows the pt-BR percent (comma) in the legend text", () => {
  expect(screen.getByText("16,5%")).toBeInTheDocument();  // ← VÍRGULA no texto (pt-BR)
});
O código separa as duas preocupações com um comentário explícito: "width is layout geometry → a plain dot-decimal % (never the pt-BR display string)". A geometria de layout usa {s.bps / 100}% (ponto); a legenda usa formatShareBps(s.bps) (vírgula). Duas bordas, dois formatos.

6. Conexão com o backend: o contrato OpenAPI como fronteira

Esta seção é o coração da aula de frontend — e substitui a "ponte hexagonal" que a aula de backend ensina. No backend, a fronteira é o port (a interface). No frontend, a fronteira entre o cliente e o servidor é o contrato OpenAPI — o api/openapi.yaml — e ele é a fonte única da verdade para os tipos que cruzam a rede.

6.1 — Tipos gerados, nunca DTOs à mão

O cliente não escreve interfaces TypeScript para os payloads da API. Elas são geradas de api/openapi.yaml por openapi-typescript para um único arquivo web/lib/api/schema.ts, e consumidas por um wrapper de fetch tipado (openapi-fetch). Um User ou um Credentials no cliente são o mesmo tipo que o backend documenta — por construção. Escrever um DTO à mão convidaria exatamente a deriva que o openapi_test.go do backend combate.

// web/lib/auth/session.ts — os tipos vêm do schema GERADO, nunca escritos à mão
import type { components } from "@/lib/api/schema";
export type User        = components["schemas"]["User"];        // ← derivado do openapi.yaml
export type Credentials = components["schemas"]["Credentials"];

6.2 — O guardião de deriva: npm run check:api, o espelho do openapi_test.go

E se o backend mudar o contrato e ninguém regenerar os tipos do cliente? O check:api falha o build. Ele regenera os tipos de api/openapi.yaml na hora e compara com o schema.ts commitado — se diferirem, ele quebra e exige um gen:api. É o espelho cliente do teste de drift do backend, forçando os dois lados de volta ao lockstep:

// web/scripts/check-api-types.mjs — falha se schema.ts divergir de api/openapi.yaml
if (fresh !== committed) {
  console.error("✖ lib/api/schema.ts is out of date vs api/openapi.yaml.\n" +
                "  Run `npm run gen:api` and commit the result.");
  process.exit(1);   // ← quebra o build (roda no web-ci)
}
console.log("✓ API types are in lockstep with api/openapi.yaml");

6.3 — O frontend é APENAS UM cliente da API entre outros futuros

A propriedade mais importante desta arquitetura: o navegador é só mais um consumidor do mesmo contrato. Como o backend expõe tudo por uma API documentada e o frontend não tem lógica de domínio própria (a SPEC-200 declara: "o frontend não tem modelo de domínio próprio"), amanhã um app nativo, um servidor MCP ou um agente podem consumir a mesma API sem nenhuma reescrita. O front não é privilegiado; é um cliente.

web/lib/api/schema.ts (GERADO) ▲ gen:api / check:api (drift guard) │ api/openapi.yaml ──── a FONTE ÚNICA DA VERDADE do contrato ────┐ (fonte) │ ▼ Navegador ──▶ /api/* (mesma origem) ──▶ Next.js proxy (rewrite) ──▶ API Go (openapi-fetch) cookie HttpOnly same-origin API_PROXY_TARGET (backend) │ ▲ └─ hoje: o cliente web é APENAS UM consumidor ── amanhã: app nativo / MCP / agente ─┘ todos falam o MESMO contrato

7. Identidade vem do servidor: o proxy same-origin e o /auth/me

A regra do backend "identidade vem da sessão autenticada, nunca do payload" tem um espelho exato no cliente (BR-2001): o cliente nunca confia num user id guardado localmente como verdade — /auth/me é a autoridade.

7.1 — O proxy same-origin (a decisão D1)

Havia um problema clássico: a SPEC-003 usa um cookie de sessão HttpOnly SameSite=Lax. Se o app web (na Vercel) e a API estivessem em origens diferentes, o cookie não iria/voltaria sem CORS + SameSite=None — complexidade e risco. A decisão D1 resolve elegantemente: o navegador chama /api/* na própria origem do app, e o Next.js faz um rewrite (proxy) para o backend. O cookie SameSite=Lax funciona sem mudança nenhuma — sem CORS.

// web/next.config.ts — o proxy same-origin (D1): o cookie HttpOnly Lax funciona sem CORS
const nextConfig: NextConfig = {
  async rewrites() {
    return [{ source: "/api/:path*", destination: `${API_PROXY_TARGET}/:path*` }];
  },
};
// e o cliente sempre bate na PRÓPRIA origem, mandando o cookie:
export const api = createClient<paths>({ baseUrl: "/api", credentials: "include" });

7.2 — /auth/me é a autoridade; um 401 é distinto de um erro transitório

O useSession pergunta ao servidor quem é o usuário. Um 401 confirmado resolve para "não autenticado" (null); qualquer outra falha (servidor fora do ar) é um erro transitório, tratado diferente. Essa distinção é uma decisão de robustez fina: nunca ejetar um usuário de fato autenticado por causa de uma piscada de rede.

// web/lib/auth/session.ts — o servidor é a autoridade; 401 ≠ erro transitório
queryFn: async () => {
  const { data, response } = await api.GET("/auth/me");
  if (response.status === 401) return null;   // não autenticado (confirmado)
  if (!data) throw new Error("failed to load session");  // falha transitória → isError
  return data;
},
// web/components/auth/require-auth.tsx — três estados, não um só
//   loading            → placeholder neutro
//   isError (5xx/rede) → botão "Tentar novamente", NÃO um redirect (não ejeta quem está logado)
//   401 → null         → redirect p/ /login, SEM renderizar nada protegido
if (!isLoading && !isError && !isAuthenticated) router.replace("/login");
Por que o logout limpa TODO o cache No useLogout, o onSuccess faz qc.setQueryData(ME_KEY, null) e qc.clear() — descarta toda query protegida em cache. Sem isso, um dado da carteira do usuário A poderia piscar na tela para o usuário B que loga em seguida no mesmo navegador. É o análogo cliente do isolamento por usuário do backend: ao trocar de identidade, o estado da identidade anterior não pode sobreviver.

8. O shell do app e o slot global do copiloto (a ponte para a SPEC-215)

O shell autenticado é a moldura que toda tela SPEC-21x vai preencher: uma sidebar de 248px (Painel · Carteira · Insights · Saúde · Projeções · Perfil) que vira abas na base no mobile, uma top bar e a área de conteúdo. Tudo gated pela sessão via RequireAuth.

// web/app/(app)/layout.tsx — o shell: sidebar/abas + top bar + conteúdo + o launcher GLOBAL do copiloto
<RequireAuth>
  <div className="md:grid md:grid-cols-[248px_1fr]">
    <AppSidebar />
    <div className="flex min-h-screen flex-col">
      <TopBar />
      <main>{children}</main>      {/* aqui entra cada tela SPEC-21x */}
    </div>
  </div>
  <MobileTabBar />
  <CopilotLauncher />              {/* ← o botão flutuante do copiloto, em TODA tela */}
</RequireAuth>
A decisão de produto: o copiloto é um WIDGET global, não uma rota Há uma escolha de UX deliberada e importante: não existe uma página "Chat" no menu. O copiloto é um botão flutuante presente em todas as telas — para que Rafael possa perguntar sobre a carteira enquanto olha o dashboard ou as projeções, sem sair do contexto. O nav.ts registra isso em código: "Chat/Copiloto is deliberately absent — it is a global floating widget, not a route (SPEC-215)". O SPEC-200 entrega só o ponto de montagem + o estado aberto/fechado do launcher; o widget conversacional de verdade (turnos, streaming) é a SPEC-215. Por isso o painel aberto hoje mostra um estado honesto ("chega na SPEC-215") já com o NonAdviceDisclaimer — a guarda já está no lugar antes mesmo do chat existir.

É por isso que o SPEC-200 também construiu, na Fase 3, um transporte de streaming SSE (lib/stream.ts): quando o chat da SPEC-215 chegar, o encanamento de streaming de tokens já existe — o chat não vira retrabalho (decisão D5).

9. Como foi construído (visão geral, sem virar tutorial de React)

Você não precisa dominar React para acompanhar. O suficiente para entender a estrutura:

  • O shell do app (Next.js App Router). As páginas vivem em web/app/. O grupo (app)/ é a área autenticada (layout com sidebar); login/ e register/ ficam fora dele. Cada item de menu é uma pasta com uma page.tsx — hoje um estado vazio.
  • Os componentes do design system Aurora. Em web/components/: as guardas (insight-card.tsx, non-advice-disclaimer.tsx), as primitivas (ui/button, ui/card, ui/badge, ui/input) e a allocation-bar.tsx. Os tokens (cores, glow, fontes) vivem como custom properties no app/globals.css (Tailwind v4, sem tailwind.config.js).
  • O cliente tipado. web/lib/api/: schema.ts (gerado), client.ts (o fetch tipado). A sessão é web/lib/auth/session.ts. O dinheiro é web/lib/money.ts.
  • A sessão via /auth/me. Um hook useSession (TanStack Query) pergunta ao servidor quem está logado; RequireAuth gateia as rotas.

Se você quiser se aprofundar, os arquivos reais estão apontados acima. Mas a mensagem da aula é o porquê, não a mecânica de React: cada peça existe para tornar uma regra de produto estrutural.

11. Harness Engineering: como a máquina construiu e verificou isto

O projeto tem propósito duplo: além do produto, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. O SPEC-200 é especial porque estende o harness para um novo track: o Go tem seus revisores e hooks; agora o frontend ganha os seus. Veja como a máquina construiu e verificou a fundação.

O fluxo /spec-implement em fases — track-aware

A IA não atacou "faça o frontend" de uma vez. O PLAN-200 fatiou em 8 fases, cada uma deixando web/ compilando, lintando e type-checando verde e revisável sozinha. A ordem foi deliberada para de-riscar as duas costuras mais difíceis primeiro: o contrato tipado (Fase 3) e a auth cross-origin (Fase 4). E o /spec-implement foi feito track-aware: numa mudança em web/**, ele seleciona os revisores de frontend — os revisores Go não se aplicam a React/TS.

Os dois subagentes de revisão de frontend — e o bug real que pegaram

O harness Go tem o hexagonal-reviewer e o go-correctness-reviewer. O SPEC-200 introduziu os gêmeos de frontend, com divisão de trabalho clara:

  • frontend-reviewer — a lente de convenções + guardas do produto + design system. Ele cobra, em ordem de severidade: contrato-do-OpenAPI (nada de DTO à mão), dinheiro-sem-float (só via money.ts, nada de toFixed/parseFloat/R$ concatenado à mão), as guardas binding (o explanation obrigatório, o disclaimer presente, nenhum botão de ordem, ganho/perda só como cor), identidade-do-servidor e acessibilidade.
  • react-correctness-reviewer — a lente de bugs de correção: regras de hooks, deps de useEffect, setState em effect, fronteira client/server, hidratação, vazamento de listener/stream, races assíncronas, tipos inseguros. Ele roda typecheck/lint/build e lê o código de perto.
O bug que o typecheck/lint/build NÃO pegaram — mas o revisor de correção pegou O caso do AllocationBar (seção 5) é a prova viva de por que a revisão semântica importa. Usar a string pt-BR "16,5%" como largura CSS é uma string perfeitamente válida em TypeScript — então typecheck passa, lint passa, build passa. Mas no navegador o CSS width: "16,5%" é inválido e o segmento colapsa para largura zero: a barra some, sem erro nenhum. Nenhum gate mecânico pega isso — é preciso um revisor que entenda que largura de layout precisa de ponto decimal e legenda precisa de vírgula pt-BR. O react-correctness-reviewer pegou, e a correção virou um teste de regressão que trava o bug para sempre. Lição de harness: os gates mecânicos (typecheck/lint/build) são necessários, mas não suficientes — a lente semântica dos subagentes cobre a classe de bugs que compila e quebra em runtime.

Os hooks que agiram — prettier-edited e on-stop-web

  • prettier-edited (PostToolUse): roda o Prettier em todo arquivo editado sob web/ — o espelho do gofmt-edited do Go. Zero ruído de estilo nos diffs. Importante: ele pula o schema.ts gerado (não se reformata um artefato de geração — isso quebraria o check:api).
  • on-stop-web (Stop, só lembrete): ao encerrar, lembra de rodar o gate de web/ (typecheck/lint/build), atualizar o CHANGELOG e regerar os tipos se o contrato mudou.

12. Conexão com AI Engineering & Produto

Esta fundação parece "só encanamento de frontend", mas ela é onde as garantias de governança da IA deixam de ser regra de backend e viram algo que o usuário vê e sente. Vamos aos elos concretos.

12.1 — As guardas de IA viram UI tangível

No backend, a explicabilidade (FR-013) e a não-conselho (FR-014) são invisíveis — vivem no Insighter gated. O SPEC-200 as materializa: o "por quê" vira um bloco fixo que o usuário lê em todo insight; o disclaimer vira um rodapé presente em toda superfície de IA; a ausência de botões de ordem vira uma experiência que comunica "somos um copiloto, não um corretor". A promessa abstrata do produto vira percepção do usuário.

12.2 — O terreno pronto para o copiloto conversacional (SPEC-215)

O SPEC-200 não implementa o chat, mas prepara todo o terreno: o slot flutuante global já está em todas as telas; o transporte SSE de streaming (stream.ts) já existe para os tokens do chat; e o NonAdviceDisclaimer já está montado no painel do copiloto — então, quando a SPEC-215 ligar o motor conversacional (que no backend é o capstone SPEC-108, com grounding turno a turno e gates), a guarda de não-conselho já está no lugar na UI. O chat não vira retrabalho: ele preenche um slot preparado.

12.3 — A costura para a visão multi-agente / MCP

Lembre da propriedade da seção 6: o frontend é apenas um cliente do contrato OpenAPI, sem lógica de domínio própria. Isso é exatamente o que sustenta a visão de futuro do produto — o CIO multi-agente e o MCP. Como toda a inteligência e todos os fatos vivem atrás da API (o front só renderiza), um servidor MCP ou um agente podem consumir o mesmo contrato para orquestrar as mesmas ferramentas — sem tocar no frontend. O cliente web, o app nativo futuro e o agente são consumidores irmãos da mesma fronteira. Manter o front como "só um cliente" é o que deixa a porta aberta para a Fase 2/3.

A ideia que conecta tudo Uma fundação de frontend confiável para um produto de IA financeira é: as travas viram estrutura (não disciplina), o contrato é a fronteira única (não DTOs à mão), a identidade vem do servidor (não do cliente) e o dinheiro é inteiro até a borda (nunca float). O SPEC-200 entrega os quatro — e, ao fazê-lo, garante que toda tela SPEC-21x nasce com as guardas do produto embutidas, o contrato em lockstep e o copiloto ao alcance. A modéstia é a virtude: zero telas de produto, tudo investido em tornar o certo inescapável.
