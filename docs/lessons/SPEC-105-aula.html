SPEC-105 — AI Rebalancing Assistant (% Computada + Candidatos Aterrados)

Este é o momento mais acionável do copiloto: o investidor tem dinheiro novo e uma decisão a tomar. O assistente pega um aporte ("tenho R$X pra aportar este mês") e devolve onde focar — áreas sugeridas, cada uma com uma fatia computada do aporte, mais candidatos nomeados e aterrados — nunca uma ordem de transação (FR-011, FR-014). A lição central são duas travas de "computado, não gerado": a % por área é calculada por um allocator determinístico antes do LLM (que só a explica), e os nomes só sobrevivem se o ticker existir de verdade no universo de FIIs (a trava de aterramento). Números computados, nomes aterrados — o modelo raciocina, mas não inventa.

1. A visão geral: do aporte à orientação — áreas + candidatos, nunca uma ordem

O SPEC-104 entregou a primeira feature de IA (os insights de carteira) e — crucialmente — publicou uma seam: o BuildFacts(ctx, userID), a função determinística que monta o snapshot de fatos do investidor. O SPEC-105 é o segundo consumidor dessa seam, e ele a usa para o caso mais útil de todos: o investidor diz "tenho R$X pra aportar", e o assistente responde onde focar esse dinheiro novo.

O que ele entrega é um novo pacote-feature internal/rebalancing e um endpoint:

  • O Contribution — um value object int64 centavos, validado > 0 na borda (parse-don't-validate); é o único valor que o cliente fornece.
  • O ContributionSplit (a função pura Split) — o coração: um allocator determinístico que computa a fatia sugerida por área (suggested_share_bps, Σ = 10 000 exato) antes de qualquer LLM.
  • O Service.Rebalance — reusa BuildFacts, aumenta os fatos com o aporte + o universo de FIIs + o split computado, e chama o Insighter gated para explicar cada área e nomear candidatos — com a trava de aterramento descartando qualquer ticker que o sistema não conhece.
  • Um único endpoint POST /rebalancing. Nenhuma tabela, nenhuma migration — só uma nova query de leitura (ListFIIUniverse) sobre a tabela fii_quotes que já existe.

O endpoint novo (atrás da auth deny-by-default, fora da allowlist pública):

Por que esta spec é a "hora da verdade" do copiloto — e por que ela respeita FR-014 Um insight ("sua carteira está concentrada em logística") é observação. Um aporte ("tenho R$1.000") é uma decisão prestes a acontecer. É exatamente aqui que a tentação de "dar uma ordem" é máxima — e onde a regra binding nº 2 (copiloto, nunca conselheiro) precisa segurar firme. O truque de design é sutil: o assistente sugere uma % de divisão entre áreas ("60% renda fixa, 40% FIIs"), que é uma consideração, não uma ordem — não carrega quantidade de cotas, preço-alvo, imperativo de compra/venda nem garantia, então fica fora da assinatura de ordem do FR-014. E ainda assim o disclaimer de não-aconselhamento acompanha toda resposta, e a decisão final continua com o usuário.

2. LIÇÃO PRINCIPAL — a % é COMPUTADA, não gerada: o ContributionSplit

Esta é a ideia mais importante da aula. A regra binding nº 3 do PRD diz "fatos são computados, não gerados" — e aqui ela ganha sua forma mais ousada: a própria porcentagem sugerida é um fato computado. O LLM nunca produz o número; um allocator determinístico o calcula antes da chamada ao modelo, e o Insighter só explica o que já foi decidido pela matemática.

2.1 — A primitiva reutilizável: money.AllocateBps (Σ = 10 000 exato)

No coração do split está uma primitiva nova em internal/platform/money: AllocateBps distribui 10 000 basis points (= 100%) entre pesos, usando o método do maior resto para que o resultado some exatamente 10 000 — sem deriva de arredondamento. É a mesma disciplina da alocação do dashboard (SPEC-103):

// internal/platform/money/ratio.go
// AllocateBps distribui 10000 bps entre os pesos, proporcionalmente, com maior-resto
// para que a soma seja EXATAMENTE 10000 — sem deriva (SPEC-105 BR-1056). Pesos negativos
// viram 0; total zero → tudo zero. Empate no resto vai para o índice menor (determinístico).
func AllocateBps(weights []int64) []int {
    // ... floor de cada fatia (w*10000/total) ...
    leftover := 10000 - distributed
    // ordena por resto decrescente (estável: empate mantém ordem) e
    // distribui o leftover de 1 em 1 aos maiores restos:
    sort.SliceStable(rems, func(a, b int) bool { return rems[a].r > rems[b].r })
    for k := 0; k < leftover && k < len(rems); k++ { shares[rems[k].idx]++ }
    return shares
}
Por que "maior-resto" e não arredondar cada fatia isolada Se você arredondasse cada fatia independentemente (33,33% → 33%, três vezes), a soma daria 99% — falta 1 ponto. O método do maior resto primeiro faz o floor de todas as fatias, conta quantos bps sobraram, e distribui esse leftover de 1 em 1 para as fatias com o maior resto fracionário. O resultado sempre fecha em 10 000, e o empate vai para o índice menor — então a mesma entrada sempre dá a mesma saída. É o invariante de reconciliação: a soma das fatias é exatamente 100%, por construção, nunca "≈100%".

2.2 — O allocator de domínio: rebalanceamento-por-aporte

O Split (em internal/rebalancing/split.go) é uma função pura: dada a alocação atual, o perfil de risco e o aporte, devolve as fatias por área. A lógica é elegante — em vez de mandar vender o que está sobrando, ele direciona o dinheiro novo para as classes que estão ABAIXO do alvo implícito no perfil. É o "rebalanceamento por aporte": você corrige o desequilíbrio aportando mais onde falta, sem nunca emitir uma ordem de venda.

// internal/rebalancing/split.go — o split é uma função PURA dos fatos
func Split(current Allocation, risk profile.RiskProfile, contribution Contribution) []AreaShare {
    targets := targetShares(risk)                              // direção implícita no perfil
    futureTotal := current.TotalCentavos + contribution.Centavos()

    // Pesa cada classe por QUANTO ela está abaixo do alvo APÓS o aporte:
    weights := make([]int64, len(splitClasses))
    var anyGap bool
    for i, c := range splitClasses {
        targetValue := money.ApplyBps(futureTotal, targets[c])
        if gap := targetValue - current.ByClass[c]; gap > 0 {  // só os DÉFICITS pesam
            weights[i] = gap
            anyGap = true
        }
    }
    if !anyGap {                                               // rede de segurança
        for i, c := range splitClasses { weights[i] = int64(targets[c]) }
    }

    shares := money.AllocateBps(weights)                       // ← Σ = 10 000 exato
    // ... monta AreaShare{Class, SuggestedShareBps, SuggestedAmountCentavos=ApplyBps(aporte, share)}
}

A "direção implícita no perfil" é uma regra pequena e documentada — não uma promessa de alvo numérico (isso é deferido em D4/D7). Conservador inclina para renda fixa; agressivo para FIIs; perfil desconhecido lê como moderado:

// targetShares: a direção (bps, Σ=10000) que o aporte persegue — uma regra de DIREÇÃO, não um alvo prometido
func targetShares(risk profile.RiskProfile) map[dashboard.AssetClass]int {
    switch risk {
    case profile.RiskConservative:
        return map[...]int{ClassFII: 3000, ClassFixedIncome: 7000}  // inclina p/ renda fixa
    case profile.RiskAggressive:
        return map[...]int{ClassFII: 7000, ClassFixedIncome: 3000}  // inclina p/ FIIs
    default:  // moderado / não setado
        return map[...]int{ClassFII: 5000, ClassFixedIncome: 5000}
    }
}
Por que o número é COMPUTADO antes do LLM — e o que isso resolve (D7) Há duas formas de produzir um "60% renda fixa". A frágil: pedir ao LLM. Aí o número fica não-determinístico (varia entre chamadas), não-auditável (de onde veio o 60?) e perigosamente próximo de uma ordem. A robusta, escolhida aqui: o Split computa a fatia a partir de alocação + perfil — uma função visivelmente mecânica e reproduzível — e passa o resultado como um fato. O LLM recebe esse fato e só o explica ("a renda fixa recebe a maior fatia porque seu perfil é conservador e o ambiente de SELIC alta favorece pós-fixados"). A decisão D7 traça a linha com precisão: os dados de mercado ao vivo inclinam a NARRATIVA, não o número cru. O número é função de alocação + perfil; o mercado entra na explicação. Assim o split é determinístico (o mesmo aporte → o mesmo split), e a explicação fica rica e contextual.

2.3 — Por que uma % entre áreas é uma consideração, não uma ordem (BR-1056)

Esta é a justificativa de conformidade, e vale entendê-la a fundo. O FR-014 proíbe ordens de transação: quantidade de cotas, preço-alvo, imperativo de compra/venda, garantia. Um split percentual entre seções ("60% para renda fixa, 40% para FIIs") não carrega nenhuma dessas coisas — é um direcionamento qualitativo de onde considerar focar, com o disclaimer junto. Por isso ele fica fora da assinatura de ordem do FR-014, e o gate de não-aconselhamento (que ainda roda sobre o texto da explicação) o deixa passar. A fronteira é deliberada: a área é uma consideração; uma ordem por ativo nomeado com quantidade seria proibida — e é exatamente por isso que a fatia por candidato é opt-in e sempre enquadrada como ilustrativa (seção 7, D6).

3. A trava de aterramento: candidatos só sobrevivem se o ticker for real

Se a % é "computada, não gerada", os nomes são "aterrados, não inventados". Quando o assistente nomeia candidatos ("dê uma olhada em HGLG11, KNRI11..."), há um risco clássico de IA: a alucinação de ticker — o modelo cospe um código que parece um FII mas não existe, ou que o sistema não conhece. A trava de aterramento (grounding guard) é determinística e fail-closed: todo candidato é validado contra o universo real de FIIs (SPEC-006); se o ticker não está lá, ele é descartado antes de chegar ao usuário (BR-1053).

// internal/rebalancing/engine.go — buildCandidates: o grounding guard
known := tickerIndex(universe)   // mapa ticker-normalizado → cotação real (o universo SPEC-006)
res, err := s.insighter.Generate(ctx, ...)   // o LLM propõe candidatos pela porta gated
// ...
var candidates []Candidate
for _, in := range res.Insights {
    q, ok := known[normalizeTicker(in.Title)]
    if !ok {
        continue   // TRAVA DE ATERRAMENTO: o assistente nunca nomeia um ticker que o sistema não conhece
    }
    candidates = append(candidates, Candidate{
        Ticker: q.Ticker.String(), Sector: string(q.Sector),  // dados REAIS do universo
        Title: in.Title, Detail: in.Detail, Explanation: in.Explanation,
    })
}
O detalhe que torna isto à prova de alucinação: o ticker vem do universo, não do texto Repare em q.Ticker.String() e q.Sector: o ticker e o setor que vão para o usuário não são lidos do texto do LLM — eles vêm da cotação real (q) que o índice known devolveu. O LLM só serve como um "proponente" de qual ticker mencionar; a fonte da verdade é o universo. Se o modelo escrevesse "HGLG11" com um erro de digitação, ou inventasse "XPTO99", o normalizeTicker + lookup falharia e o candidato seria silenciosamente descartado. É a mesma filosofia da % computada, aplicada a nomes: o modelo aponta, o sistema confirma. Um teste dedicado prova que um ticker desconhecido é descartado e um conhecido é mantido.

E a renda fixa? Ela não tem um universo público de tickers (não há "ticker" de um CDB), então os candidatos de RF são type-level ("pós-fixado ≥100% CDI", "Tesouro IPCA+"), aterrados na macro ao vivo (SELIC/CDI/IPCA que já vêm no BuildFacts), sem um produto nomeado específico. Só os candidatos por ticker (FIIs) passam pela trava de aterramento — porque só FIIs têm um universo nomeável ingerido (SPEC-006).

4. Reúso da seam publicada: BuildFacts + reconstrução da alocação (BR-1052)

A decisão D6 do SPEC-104 — exportar o Fact Builder — paga dividendos aqui. O SPEC-105 não re-lê o dashboard, o perfil ou a macro: ele consome a seam publicada via uma interface consumer-defined, FactSource, satisfeita na fiação por *engine.Service (o motor do SPEC-104):

// internal/rebalancing/facts.go — a seam publicada que o engine REUSA (BR-1052)
type FactSource interface {
    BuildFacts(ctx context.Context, userID string) (insight.Facts, error)
}

O motor então aumenta esses fatos base com três coisas: o aporte, o universo de FIIs ao vivo, e o split computado. E há um detalhe esperto: para alimentar o allocator, ele precisa da alocação atual por classe — mas ele a reconstrói a partir dos próprios fatos (current_value_centavos × allocation_bps), em vez de fazer uma segunda leitura do dashboard:

// internal/rebalancing/facts.go — reconstrói a alocação SEM um segundo read do dashboard (BR-1052)
func allocationFromFacts(base insight.Facts) Allocation {
    total, _ := base["current_value_centavos"].(int64)
    byClass := map[dashboard.AssetClass]int64{}
    if bps, ok := base["allocation_bps"].(map[string]int); ok {
        for cls, share := range bps {
            byClass[dashboard.AssetClass(cls)] = money.ApplyBps(total, share)  // bps → centavos
        }
    }
    return Allocation{TotalCentavos: total, ByClass: byClass}
}

func assembleFacts(base insight.Facts, contribution Contribution, universe []marketdata.FIIQuote) (insight.Facts, []AreaShare) {
    risk, _ := base["risk_profile"].(string)                  // "" se perfil não setado → moderado
    split := Split(allocationFromFacts(base), profile.RiskProfile(risk), contribution)
    base["contribution_centavos"] = contribution.Centavos()
    base["fii_universe"]          = universeFacts(universe)   // ticker/setor/yield/preço
    base["suggested_split"]       = splitFacts(split)         // bps + centavos, NUNCA float
    return base, split
}
Por que reconstruir a alocação dos fatos em vez de reler o dashboard O dashboard já foi computado uma vez dentro do BuildFacts (que reusa a Compute do SPEC-103). Reler o dashboard aqui significaria um segundo caminho de computação — e o risco de divergência por arredondamento que o SPEC-104 tanto evitou. Reconstruir a alocação a partir de current_value_centavos e allocation_bps (que já estão nos fatos) mantém um único caminho: o número que a IA explica, o número do split e o número que a tela mostra são o mesmo. A renda fixa entra ao vivo via a macro que já está nos fatos base (SELIC/CDI/IPCA precificam todo CDB/Tesouro); só o universo de produtos nomeados é FII-only.

5. As travas por construção: texto de IA só pelo Insighter gated

Como no SPEC-104, as duas guardas binding — explicabilidade (FR-013) e não-aconselhamento (FR-014) — não são re-implementadas aqui (BR-1054). Elas já vivem no Insighter (SPEC-005) como decorators de gate. O motor de rebalanceamento garante que elas valem por uma propriedade arquitetural: ele só emite texto de usuário pela porta que tem os gates. Repare que tanto as áreas quanto os candidatos passam pelo mesmo insighter.Generate — não há um segundo gerador de strings:

// internal/rebalancing/engine.go — áreas: o LLM EXPLICA, mas os NÚMEROS vêm do split computado
for _, sh := range split {            // sh = AreaShare com suggested_share_bps JÁ calculado
    res, err := s.insighter.Generate(ctx, insight.InsightRequest{   // ← a ÚNICA via do texto
        Facts:  withFocus(facts, "area", string(sh.Class)),
        Task:   TaskRebalancing,
        UserID: userID,
    })
    if err != nil {
        if ctx.Err() != nil { return nil, 0, fmt.Errorf("rebalance: %w", ctx.Err()) }
        continue   // esta área degradou ou foi rejeitada pelo gate — pula (FR-1057)
    }
    succeeded++
    area := Area{Class: string(sh.Class),
        SuggestedShareBps:       sh.SuggestedShareBps,        // ← número COMPUTADO, não do texto
        SuggestedAmountCentavos: sh.SuggestedAmountCentavos}
    if len(res.Insights) > 0 {
        in := res.Insights[0]
        area.Title, area.Detail, area.Explanation = in.Title, in.Detail, in.Explanation  // ← texto gated
    }
    areas = append(areas, area)
}
A junção (join): número computado + texto gated, lado a lado O padrão-chave do motor é um join: para cada área, o suggested_share_bps vem do AreaShare computado (a fonte autoritativa), e o title/detail/explanation vêm da Insight gated. O número e a prosa nunca se misturam: o modelo jamais tem a chance de alterar a fatia, porque a fatia já está no area antes de o texto chegar. Mesmo que o LLM escrevesse "62%" no meio da explicação, o campo estruturado SuggestedShareBps continuaria sendo o valor que o allocator calculou. É assim que "o Insighter explica, mas nunca produz nem altera os números" (BR-1056) vira código.

O número é computado, o nome é aterrado, o texto é gated.

6. Estados graciosos: vazio AINDA orienta, outage → 200 available:false

O mundo real falha de formas variadas, e o motor representa cada uma como um estado, não um erro 500.

6.1 — Carteira vazia AINDA orienta (a diferença-chave do SPEC-104)

Esta é a distinção mais importante em relação ao SPEC-104. Lá, uma carteira vazia faz short-circuit (nada a analisar, sem chamada ao LLM). Aqui, é o contrário: uma carteira vazia + um aporte é justamente o caso do primeiro investimento — o investidor mais precisa de orientação. Então o motor não dá short-circuit: o Split simplesmente não acha nenhuma alocação atual para rebalancear, então segue a direção implícita no perfil (a rede de segurança if !anyGap da seção 2.2 entra), e o LLM explica como começar (FR-1053).

6.2 — Outage total → 200 available:false (D5)

Se o LLM está totalmente fora, o motor não devolve 503. Consistente com o SPEC-104, ele devolve 200 com available:false e sugestões vazias — o dashboard e a carteira continuam usáveis; só a faixa de rebalanceamento mostra "temporariamente indisponível". O Available é true sempre que ao menos uma área deu certo (resultado parcial); só vira false quando todas falharam:

// internal/rebalancing/engine.go — Available:false SÓ no outage total
return Rebalancing{
    Areas:      areas,
    Candidates: candidates,
    Disclaimer: insight.Disclaimer,
    Available:  succeeded > 0,   // false só quando TODA área falhou (outage total)
}, nil

E os candidatos são opcionais: se a chamada de candidatos degrada, isso não é fatal — o motor devolve as áreas mesmo assim (return nil, nil em buildCandidates). Como no SPEC-104, um cancelamento de contexto (ctx.Err() != nil) aborta em vez de degradar silenciosamente — distinção que separa degradação correta de degradação que esconde um problema.

6.3 — Candidato por-ativo só com include_asset_shares (D6)

Por padrão, nenhum número por candidato é mostrado — só áreas com %, e os candidatos nomeados dentro da área de FII. Uma fatia ilustrativa por candidato ("~15% para HGLG11, como consideração") é opt-in via o flag include_asset_shares (default off). Quando ligado, a fatia da área de FII é distribuída uniformemente entre os candidatos (maior-resto ao primeiro), sempre enquadrada como ilustrativa, nunca como imperativo:

// internal/rebalancing/engine.go — fatia por candidato: opt-in, ilustrativa, distribuída uniformemente
func assignIllustrativeShares(candidates []Candidate, fiiAreaBps int) {
    n := len(candidates)
    if n == 0 || fiiAreaBps <= 0 { return }
    base := fiiAreaBps / n
    remainder := fiiAreaBps - base*n
    for i := range candidates {
        candidates[i].IllustrativeShareBps = base
        if i < remainder { candidates[i].IllustrativeShareBps++ }   // maior-resto ao primeiro
    }
}

7. O endpoint POST /rebalancing e o contrato

O handler é deliberadamente magro: lê a identidade do contexto, faz o parse do aporte (o único input do cliente), chama o serviço, mapeia para o DTO. O dinheiro entra como campo inteiro (int64 centavos) — um corpo com float é rejeitado pelo decoder, então o ban do float vale na borda (BR-1051):

// internal/transport/http/rebalancing.go
type rebalancingRequest struct {
    ContributionCentavos int64 `json:"contribution_centavos"`  // inteiro → float é rejeitado no decode
    IncludeAssetShares   bool  `json:"include_asset_shares"`
}

func (h rebalancingHandler) postRebalancing(w http.ResponseWriter, r *http.Request) {
    userID, ok := callerID(w, r)   // identidade do contexto, nunca do payload (BR-1055)
    if !ok { return }
    var req rebalancingRequest
    if err := decodeJSON(w, r, &req); err != nil {
        writeError(w, http.StatusBadRequest, "invalid request body"); return
    }
    contribution, err := rebalancing.ParseContribution(req.ContributionCentavos)  // parse-don't-validate, > 0
    if err != nil {
        writeError(w, http.StatusBadRequest, "contribution_centavos must be a positive integer"); return
    }
    res, err := h.service.Rebalance(r.Context(), userID, contribution,
        rebalancing.Options{IncludeAssetShares: req.IncludeAssetShares})
    if err != nil { /* erro REAL de serviço → 500 */ }
    writeJSON(w, http.StatusOK, toRebalancingResponse(res))   // vazio E degradado → 200
}

O DTO de resposta aninha os candidatos dentro da área de FII (D6): como todos os candidatos com ticker são FIIs, eles pertencem àquela área; se FIIs não é uma área sugerida, os candidatos não aparecem. Cada item carrega um explanation sempre presente (o gate de explicabilidade garante), e a resposta carrega o disclaimer:

// internal/transport/http/rebalancing.go — candidatos aninhados na área de FII (D6)
type areaResponse struct {
    Class                   string              `json:"class"`
    SuggestedShareBps       int                 `json:"suggested_share_bps"`
    SuggestedAmountCentavos int64               `json:"suggested_amount_centavos"`
    Explanation             string              `json:"explanation"` // sempre presente — o gate garante

9. As 8 fases, uma a uma

Bottom-up, como nas specs anteriores — core/domínio primeiro, borda depois, cada fase compilando e revisável sozinha. Com três fios condutores: "fatos computados, não gerados" (aqui a % e os nomes), "guardas por construção" (texto só pelo Insighter) e "dinheiro é inteiro + reconcilia" (o split soma exatamente 10 000 bps, half-up).

Fase 1 — Domínio & o allocator determinístico (o coração)

A primitiva money.AllocateBps (Σ = 10 000 exato, half-up, maior-resto); o value object Contribution (int64 centavos, ParseContribution valida > 0, sentinela ErrInvalidContribution); o Split puro (pesos a partir da alocação + a regra de direção do perfil) e os tipos de resultado (Area, Candidate, Rebalancing) + a task rebalancing. O número principal foi provado antes de qualquer LLM entrar — testes table-driven de Σ = 10 000 em pesos difíceis, determinismo, vazio-segue-a-direção, zero-float.

Fase 2 — Persistência: o read do universo de FIIs

marketdata.ListFIIUniverse(ctx) — um SELECT sobre fii_quotes (mesmas colunas de GetFIIQuoteByTicker, sem WHERE), ordenado deterministicamente — e o port consumer-defined rebalancing.UniverseReader, satisfeito na borda pelo adapter Postgres. Sem mudança de schema — só uma query nova. Teste de integração gated (PG real: semeia cotações → list devolve).

Fase 3 — Fatos de rebalanceamento (aumenta o Fact Builder)

assembleFacts: reusa BuildFacts, depois adiciona contribution_centavos, o universo de FIIs (fii_universe: ticker/setor/yield/preço) e o split computado (suggested_split: bps + centavos). Reconstrói a alocação dos fatos (allocationFromFacts) sem um segundo read. Testes: fatos incluem aporte + universo + split; reusa BuildFacts (fake seam); determinístico; nenhum float; o split reconcilia a 10 000.

Fase 4 — O Engine (orquestração, grounding guard, gates por construção)

Service.Rebalance(ctx, userID, contribution, opts): monta os fatos, chama o Insighter por área (juntando o suggested_share_bps computado à explicação gated) e por candidatos (aplicando a trava de aterramento); carteira vazia → ainda orienta (sem short-circuit); degradação → Available:false; aborta em ctx.Err(); fatia por-ativo só com o flag (D6). Inclui o teste-chave: o motor emite texto de IA SÓ via o Insighter; e o teste do grounding guard (ticker desconhecido descartado).

Fase 5 — API (transporte)

transport/http/rebalancing.go: POST /rebalancing → parse {contribution_centavos, include_asset_shares?} (inteiro, nunca float; > 0 senão 400), identidade via auth.UserID(ctx); DTO (áreas + bps + candidatos aninhados + disclaimer + available); degradado → 200; erro de serviço → 500. Deps.Rebalancing, registro na routeTable, path + schema em api/openapi.yaml (drift verde) e a fiação no cmd/api.

Fase 6 — Observabilidade

A rota POST /rebalancing herda o span do otelhttp; um span rebalancing.facts para o fact-building sem PII (nenhum aporte/figura/texto gerado vai para telemetria); o Insighter grava seus spans insight.generate (sem conteúdo — BR-505). Um teste garante nenhum valor de fato / valor do aporte / texto gerado no span.

Fase 7 — Testes

Unitários sem banco: o allocator (Σ = 10 000, determinismo, direção, no float); parse do aporte; aumento de fatos; grounding guard; engine com fakes (só-via-Insighter, vazio-orienta, degrada, flag); handler (identidade-do-contexto, 400 aporte ruim, 401, degradado 200). Integração gated: Postgres real (TEST_DATABASE_URL, -p 1) + o fake Insighter — semeia holdings + cotações (um universo) + perfil, chama POST /rebalancing e asserta que toda área/candidato carrega explicação, o disclaimer está presente, os candidatos são aterrados no universo semeado, as áreas somam 10 000 e o isolamento por usuário vale.

Fase 8 — Documentação & aula

README.md (endpoint /rebalancing), CHANGELOG.md, api/openapi.yaml (path + schema), SPEC-105 + PLAN-105 marcados Done, índices e status do CLAUDE.md atualizados, e esta aula. 🙂

11. Harness Engineering: como a máquina construiu isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. Veja, no SPEC-105, como o harness construiu e verificou.

O fluxo /spec-implement, fase a fase — o número antes do LLM

A IA não atacou "faça o assistente de rebalanceamento" de uma vez. O PLAN-105 fatiou em 8 fases bottom-up, e a ordenação foi deliberada: a Fase 1 lança o allocator (AllocateBps + Split) antes de tudo, com testes table-driven provando Σ = 10 000 em pesos difíceis, determinismo e zero-float — sem nenhum LLM por perto. Isso significa que o "número principal" (a fatia) já estava provado e mecânico quando o engine (Fase 4) entrou para só orquestrar. É o oposto de pedir o número ao modelo: o harness força a matemática a ser escrita e testada primeiro, então não há como o LLM "vazar" para dentro do cálculo.

O hook determinístico block-layering

O hook block-layering (PostToolUse, exit 2) barra mecanicamente qualquer import de SQL/HTTP/SDK de vendor dentro de um core de feature. Importar os tipos de domínio puros de outras features (dashboard.AssetClass, marketdata.FIIQuote, profile.RiskProfile, insight.Facts) passa; se a IA tivesse tentado meter um database/sql ou um net/http dentro de internal/rebalancing, o hook teria bloqueado — sem revisão humana. É o que garante, na prática, que o Split ficou puro e o read do universo ficou na borda (atrás do UniverseReader), não embutido no domínio.

O subagente hexagonal-reviewer

O revisor especializado cobrou as regras mais sensíveis daqui: a direção de dependência (o grafo acíclico rebalancing → {insight, marketdata, dashboard, profile, money}, sem ciclo de volta), o reúso da seam publicada (a FactSource em vez de um segundo read do dashboard, BR-1052) e — o ponto mais crítico — a trava por construção: que o motor emite texto de usuário só via insighter.Generate, tanto para áreas quanto para candidatos, nunca construindo strings de IA por outro caminho (BR-1054).

O go-correctness-reviewer e a disciplina de "computado, não gerado"

Sendo a feature onde a % e os nomes são produzidos, a lente de correção vigiou os riscos nº 1 do plano (PLAN-105 §8): que a % lesse como ordem, que o LLM inventasse um ticker, que o texto burlasse os gates e que o split não reconciliasse. Ela garantiu que o suggested_share_bps dos campos estruturados vem do allocator (não parseado do texto do LLM); que a trava de aterramento usa q.Ticker do universo (não o título do LLM); que AllocateBps soma exatamente 10 000; que ctx.Err() aborta em vez de degradar; e o %w + prefixo minúsculo nos erros e o ctx como primeiro parâmetro.

A prova de que as travas valem — duas, desta vez É a estrutura que prova. (1) A %: o campo SuggestedShareBps da Area é preenchido do AreaShare computado antes de o texto do LLM chegar — o modelo nem recebe a chance de alterá-lo. (2) Os nomes: o Ticker do Candidate vem de q.Ticker (a cotação real do universo), não do título do LLM; um ticker desconhecido cai no continue. Some a isso o teste "nenhum texto de IA nasce fora do Insighter", o teste do grounding guard e o hook block-layering (sem provider cru no pacote) e você tem as guardas binding garantidas por construção, não por boa-fé — exatamente o que o PRD exige.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • block-layering: o bloqueio duro que provou que o rebalancing ficou livre de SQL/HTTP, ao mesmo tempo que permitiu o consumo dos tipos de domínio puros das features vizinhas.
  • O teste de drift do OpenAPI: ao adicionar POST /rebalancing na routeTable, o openapi_test.go exigiu a entrada correspondente em api/openapi.yaml — senão o build falharia (seção 7).
  • Bloqueio de imutabilidade: o SPEC-105 referencia ADR-0002/ADR-0003 e herda os gates do SPEC-005, mas não os edita — se a IA tentasse "atualizar" um gate aceito ou um ADR, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — e os testes têm que passar sem rede e sem DB. O default do Insighter é o fake determinístico, então o CI roda offline; a integração (seed de holdings + cotações + perfil, gates end-to-end, candidatos aterrados) fica atrás do gate de TEST_DATABASE_URL + -p 1, pulando limpo.

12. Conexão com AI Engineering

O SPEC-105 É uma feature de IA — e uma das mais delicadas, porque é onde a saída do modelo está mais perto de "parecer um conselho". Por isso ele exercita, de forma concreta, vários conceitos centrais de engenharia de IA confiável. Vamos a eles, todos ancorados no código desta spec.

12.1 — Grounding levado ao limite: o modelo nem produz o número, nem o nome

O grounding do SPEC-104 dizia "o LLM raciocina sobre fatos computados". O SPEC-105 vai além: a própria recomendação numérica (a % por área) é computada por código, e os nomes são validados contra um universo real. O modelo recebe o split como fato e o explica; recebe a lista de candidatos e só sobrevivem os que existem. É a forma mais forte de grounding: o modelo aponta e raciocina, mas o sistema decide os números e confirma os nomes. Isso torna a saída inteiramente auditável — toda fatia tem uma origem mecânica, todo ticker tem uma cotação real.

12.2 — As guardas como middleware de verificação (FR-013 + FR-014)

O padrão arquitetural aqui é o de verification middleware: entre o modelo e o usuário há uma cadeia que verifica cada saída. No SPEC-105 há dois verificadores em série: (1) os gates herdados do SPEC-005 (explicabilidade rejeita texto sem explicação; não-conselho rejeita ordens/quantidades/preços-alvo), por onde todo texto passa; e (2) a trava de aterramento, um verificador determinístico específico desta feature que descarta candidatos sem ticker real. Verificadores empilhados, cada um fail-closed.

12.3 — Defesa contra alucinação como código de primeira classe

A alucinação de ticker é um dos modos de falha mais perigosos de IA em finanças — nomear um ativo que não existe (ou existe mas o sistema não conhece) destruiria a confiança. A mitigação aqui não é um prompt "por favor não invente tickers" (frágil); é uma checagem determinística que usa o universo como fonte da verdade e descarta o que não bate. É a aplicação da regra "o LLM é falível, então toda saída passa por um verificador determinístico" — desta vez ao conteúdo factual (o nome), não só à forma (a explicação).

12.4 — Separação número/narrativa (D7): determinismo onde importa, riqueza onde ajuda

A decisão D7 é uma lição de engenharia de IA: onde a saída precisa ser determinística e auditável (o número), use código; onde a saída se beneficia de contexto e fluidez (a explicação), use o LLM. O split é função de alocação + perfil (reproduzível); o mercado ao vivo (SELIC/CDI/IPCA, yields) entra na narrativa que justifica o número. Você ganha o melhor dos dois mundos: um cálculo confiável e uma explicação rica — sem deixar o modelo mexer no que precisa ser exato.

12.5 — Caching, custo & observabilidade sem PII

A postura "zero custo" (binding nº 4) continua valendo: o withFocus dá a cada chamada (por área, por candidatos) uma chave de cache distinta via os fatos, e como o Fact Builder é determinístico, repetições batem no cache do SPEC-005 sem chamar o LLM de novo. E os spans (rebalancing.facts, insight.generate) registram latência e outcome sem nenhum aporte, figura ou texto gerado (BR-505) — observabilidade de IA que respeita privacidade. Há ainda a superfície de prompt-injection: holdings/perfil + o aporte fluem para os fatos; a mitigação é dupla — o gate de não-conselho é o backstop fail-closed, e a trava de aterramento limita os nomes a tickers reais.

A ideia que conecta tudo IA confiável em finanças é números computados + nomes aterrados + texto verificado. O SPEC-105 entrega os três: o Split produz a % (computada, em bps, reconciliada), o grounding guard confirma os nomes (contra o universo real) e o Insighter verifica a prosa (explicabilidade + não-conselho, por construção). O motor, no meio, faz o mínimo: monta fatos, junta o número à explicação e passa pela única porta gated. É essa modéstia arquitetural — deixar a matemática à matemática e o LLM à linguagem — que torna o momento mais acionável do produto seguro por design.
