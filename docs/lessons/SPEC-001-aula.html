1. A visão geral: o que está rodando agora

No fim do SPEC-001, você tem um servidor HTTP em Go que compila, roda, responde, desliga com elegância, roda dentro do Docker e tem testes automatizados — mas sem nenhuma regra de negócio ainda. Isso é de propósito: a fundação primeiro.

Se você roda task run e abre o navegador:

GET /healthz  →  {"status":"ok"}      (o app está vivo?)
GET /readyz   →  {"status":"ready"}   (o app está pronto pra receber tráfego?)
GET /version  →  {"version":"dev", "commit":"none", "built_at":"unknown"}
Por quê começar assim? Um "esqueleto que roda no dia 1" te dá um ponto de partida confiável. Cada feature futura (carteira, dashboard, IA) vai se encaixar nessa estrutura já testada, em vez de você descobrir problemas de fundação no meio do caminho.

2. O método: Spec-Driven Development (SDD)

Antes de escrever código, escrevemos documentos que descrevem o que e por quê. O código nasce para satisfazer esses documentos — não o contrário.

IDEIA → PRD → SPEC → PLAN → CÓDIGO → TESTES → DOCS │ │ │ │ │ │ │ └─ implementação em fases │ │ └─ COMO construir (plano com fases, riscos, "pronto") │ └─ O QUE construir (comportamento concreto, critérios de aceite) └─ POR QUê / pra quem (visão do produto)
DocumentoRespondeNo nosso projeto
PRDVisão do produto, escopo, requisitosdocs/01-product/PRD.md
SPECComportamento de uma capacidadedocs/02-specs/SPEC-001-...md
PLANComo construir (fases)docs/03-plans/PLAN-001-...md
ADRDecisões de arquitetura (e o porquê)docs/04-architecture/adr/
A regra de ouro do SDD Uma feature não começa sem um SPEC; um SPEC não é construído sem um PLAN. E fazemos em fatias verticais (um spec por vez, do início ao fim), nunca "todos os specs e depois todo o código".

3. A arquitetura: hexagonal + orientada a pacotes

Combinamos duas ideias que respondem perguntas diferentes:

ConceitoRespondeRegra
Hexagonal (Ports & Adapters)Para onde as dependências apontamO núcleo (regra de negócio) não conhece banco, HTTP nem LLM. Eles ficam atrás de interfaces (ports).
Orientada a PacotesComo agrupamos os arquivosPor feature (portfolio, insight), não por camada técnica.

Na prática, cada feature é uma "caixinha" que contém tudo dela — entidade, casos de uso, o contrato (port) e os adaptadores concretos ao lado:

internal/ portfolio/ portfolio.go ← entidade + regras de negócio (domínio puro) service.go ← casos de uso (o "service") repository.go ← interface Repository (o PORT — definido por quem usa) postgres/ ← adaptador: implementação Postgres do Repository platform/ ← infra transversal: config, logging, httpserver, buildinfo transport/http/ ← adaptador de entrada: router + handlers + DTOs

O fluxo de uma requisição (e quem é quem)

HTTP → Handler (transport/http) → Service (feature) → Repository (port) → Postgres controller + DTO casos de uso interface adaptador conhece HTTP/JSON conhece domínio não conhece nada conhece SQL
Termo clássicoOnde mora aquiConhece o quê?
Controller / Handlertransport/http/*.goHTTP + JSON
DTO (mapeamento)junto do handlersó transporte
Service (casos de uso)<feature>/service.godomínio + a interface do repositório
Entidade de domínio<feature>/<feature>.gosó regra de negócio

Por que isso importa pro YieldForge? Trocar o LLM grátis (Gemini/Ollama) por um pago (Claude) vai ser só escrever um novo adaptador e mudar uma linha. O service.go da IA nunca muda. Essa é a "promessa" da arquitetura — e ela já está montada desde o esqueleto.

A regra de dependência aponta sempre para o centro: o handler conhece o service; o service conhece a interface; o Postgres implementa a interface. O núcleo não importa HTTP nem SQL. No SPEC-001 só existem as "caixinhas vazias" (placeholders) — o conteúdo de cada feature vem nos specs futuros.

4. As 7 fases, uma a uma

Fase 1 — Esqueleto do módulo e layout estrutura

Criamos o módulo Go (go mod init) e a árvore de pacotes. Cada pacote ganhou um doc.go só com a documentação, pra existir e compilar.

Conceitos

  • Módulo Go: o go.mod define o "nome" do projeto (github.com/biel-ferreira/yield-forge) — é o prefixo de todos os imports.
  • Pasta internal/: o Go proíbe que código de fora do módulo importe o que está em internal/. Isso tranca a arquitetura como privada.
  • cmd/api/main.go: o ponto de entrada (a função main).

Fase 2 — Configuração 12-factor

Criamos config.Load(), que lê variáveis de ambiente, aplica valores padrão, valida, e devolve um Config tipado.

type Config struct {
    AppEnv          string        // dev | staging | prod
    Port            int
    LogLevel        string        // debug | info | warn | error
    LogFormat       string        // json | text
    ReadTimeout     time.Duration
    ShutdownTimeout time.Duration
    Warnings        []string      // avisos não-fatais
}

A decisão importante: erro fatal vs. aviso

  • Fatal (devolve erro, app não sobe): porta não-numérica, duração inválida.
  • Aviso (usa o padrão e segue): LOG_LEVEL inválido vira info.
Por que um log inválido não derruba o app? Configuração de log não é crítica — não faz sentido o app inteiro não subir por causa de um valor de log digitado errado. Já um segredo obrigatório faltando (ex.: a URL do banco, que chega no SPEC-002) é fatal. Isso é o princípio 12-factor: configuração vem do ambiente, e o app falha rápido só no que é essencial.

Fase 3 — Logging estruturado slog

Usamos o log/slog (biblioteca padrão do Go) para logs estruturados: em vez de texto solto, cada log é um conjunto de campos (chave=valor ou JSON).

logger.Info("http request",
    slog.String("method", "GET"),
    slog.Int("status", 200))
// dev  → time=... level=INFO msg="http request" method=GET status=200
// prod → {"time":"...","level":"INFO","msg":"http request","method":"GET","status":200}
Injeção em vez de global O logger é passado para quem precisa (injeção de dependência), não usado como variável global. Isso deixa o código testável e explícito sobre suas dependências.

Fase 4 — Servidor HTTP, endpoints e desligamento gracioso a fase grande

Aqui tudo se conectou. Usamos só a biblioteca padrão (net/http), sem framework web.

O roteador (Go 1.22+)

mux := http.NewServeMux()
mux.HandleFunc("GET /healthz", h.healthz)
mux.HandleFunc("GET /version", h.version)
mux.HandleFunc("/", h.notFound)   // catch-all → 404 em JSON

Middleware (camadas que envolvem cada requisição)

Encadeamos: requestID → logRequests → mux. Cada requisição ganha um ID único (devolvido no header X-Request-Id) e gera uma linha de log com método, rota, status e duração.

Desligamento gracioso (graceful shutdown)

// espera um sinal de parada (Ctrl-C / SIGTERM) OU o cancelamento do context
ctx, stop := signal.NotifyContext(ctx, os.Interrupt, syscall.SIGTERM)
...
select {
case err := <-serveErr:   // servidor caiu (ex.: porta ocupada)
    return err
case <-ctx.Done():        // recebeu sinal → vamos drenar
    srv.Shutdown(shutdownCtx) // espera requisições em andamento terminarem
}
O que é "gracioso"? Quando você manda o app parar, ele para de aceitar novas requisições mas deixa as que já estão rodando terminarem (dentro de um prazo). Sem isso, um deploy poderia cortar uma requisição no meio. O context.Context é o mecanismo do Go para propagar "cancelamento/prazo" por todo o fluxo.

Fase 5 — Containerização Docker

Um Dockerfile multi-stage: um estágio compila o binário, outro roda ele numa imagem mínima.

# estágio 1: compila (imagem grande, com o Go)
FROM golang:1.23-alpine AS build
...
RUN go build -o /out/yield-forge ./cmd/api

# estágio 2: roda (imagem minúscula, sem shell, usuário não-root)
FROM gcr.io/distroless/static:nonroot
COPY --from=build /out/yield-forge /app/yield-forge
ENTRYPOINT ["/app/yield-forge"]
TécnicaBenefício
Multi-stageA imagem final não carrega o compilador Go — fica pequena (~10–20 MB).
Binário estático (CGO_ENABLED=0)Roda sozinho, sem dependências de sistema.
Distroless + non-rootSem shell e sem gerenciador de pacotes = pouca superfície de ataque. Segurança.

O docker-compose.yml já tem o serviço do Postgres comentado, pronto pra ser ligado no SPEC-002 sem reestruturar nada.

O teste pegou um bug de verdade! Um dos testes travou no Windows porque eu tinha usado um context que nunca era cancelado. Isso revelou um problema real de portabilidade. 6. Configuração: dev vs prod (12-factor)

Você perguntou "onde estão as condições de dev/prod?". A resposta foi um conceito-chave:

O VALOR muda por ambiente. O CÓDIGO não muda. Não espalhamos if env == "dev" pelo código. O "banco de dev vs banco de prod" é só um DATABASE_URL diferente em cada ambiente — o código lê a variável e conecta, sem if. Só usamos cfg.IsDev() para pouquíssimas diferenças de comportamento real (ex.: formato do log: texto em dev, JSON em prod).

Em dev, o .env (ignorado pelo git) preenche as variáveis; variáveis reais do sistema sempre têm prioridade. Tudo documentado no .env.example.
