1. A visão geral: o que mudou no SPEC-003

No SPEC-001 ganhamos um esqueleto que roda; no SPEC-002, um banco de verdade. Mas o app ainda não sabia quem estava falando com ele — qualquer requisição era anônima. O SPEC-003 é a terceira pedra da fundação: o app passa a ter identidade e isolamento por usuário.

A partir daqui existe uma tabela users, sessões no servidor, os endpoints register/login/logout/me, um middleware de autenticação que protege as rotas, e um valor "usuário atual" que viaja no context — a costura (o seam) que toda feature vai usar pra escopar dados com WHERE user_id = $1.

Importante, no mesmo espírito das aulas anteriores: ainda não há feature de negócio. Nenhuma carteira, nenhum perfil. O SPEC-003 prova o isolamento mecanicamente — ele constrói o cofre e a fechadura, mas o cofre ainda está vazio.

ANTES (SPEC-002) DEPOIS (SPEC-003) toda requisição é anônima requisição carrega um UserID autenticado qualquer um chama qualquer rota deny-by-default: só a allowlist é pública sem usuários, sem senhas users + sessions; senha só como hash bcrypt user_id era só uma convenção (0001) user_id virou uma fronteira de verdade

Os endpoints novos:

POST /auth/register  {"email","password"}   →  201 {"id","email"}         | 400 | 409
POST /auth/login     {"email","password"}   →  200 {"id","email"} + cookie | 401
POST /auth/logout    (autenticado)          →  204  (apaga a sessão + limpa o cookie)
GET  /auth/me        (autenticado)          →  200 {"id","email"}          | 401
Por que "sessões no servidor" e não só um token solto? Uma sessão no servidor é uma linha no banco que podemos apagar. Isso dá logout de verdade (revogação imediata) e expiração controlada por nós. O cliente só guarda um token opaco; quem decide se ele ainda vale é o servidor. Veja a seção 3.

2. As 4 decisões (e por que NÃO JWT, por que NÃO OAuth)

Toda spec tem bifurcações reais. Aqui foram quatro, mais uma decisão fina sobre logout. Vale entender o trade-off de cada uma — é aqui que mora o aprendizado de segurança.

#DecisãoEscolhaPor quê
D1Como autenticar? E-mail + senha, com sessões no servidor Autossuficiente, zero custo, revogável, alto valor de aprendizado. Cabe num MVP que é, na prática, single-user (você).
D2Como guardar a senha? bcrypt atrás do port PasswordHasher Já vem no golang.org/x/crypto, é testado em batalha e tem comparação em tempo constante embutida. Trocar por argon2id depois é um detalhe atrás do port.
D3Como transportar a sessão? Cookie HttpOnly + Secure + SameSite O padrão mais seguro: HttpOnly impede que JavaScript (e um XSS) leia o token; SameSite ajuda contra CSRF; Secure só relaxa em dev local.
D4Como escopar dados por usuário? No nível da aplicação (WHERE user_id = $1) Transparente e fácil de raciocinar. O Row-Level Security do Postgres fica como defesa-em-profundidade futura por cima disto.
Por que NÃO JWT agora? Um JWT é autocontido: o servidor valida a assinatura e confia no conteúdo sem consultar o banco. Soa ótimo — até você precisar de logout. Como o token vale por si só, revogá-lo antes da expiração exige uma "blacklist"... que é, justamente, uma sessão no servidor. Para um app que quer logout imediato e simplicidade, sessão no banco é mais direto. JWT continua sendo plugável depois, atrás do mesmo seam auth.
Por que NÃO OAuth/login social agora? "Entrar com o Google" exige cadastrar a aplicação num provedor externo, gerenciar client secrets e callbacks. Isso fere a posture de zero custo / zero dependência externa e adiciona complexidade que o MVP não precisa. O esquema deixa espaço pra adicionar depois sem redesign.
A decisão fina: logout = DELETE de verdade Logout apaga a linha da sessão (DELETE), não um soft-delete. A expiração natural (expires_at) já cobre as sessões velhas; apagar no logout mantém a tabela enxuta. Simples e correto.

3. Como uma sessão vive no banco

Esta é a parte mais importante de segurança da aula. O fluxo de uma sessão é, no fundo, dois comandos SQL: login = INSERT, logout = DELETE. Mas o truque está em o que guardamos.

LOGIN 1. usuário manda email+senha 2. servidor confere a senha com bcrypt.Compare 3. gera um token aleatório de 256 bits → "abc123..." (o token CRU) 4. INSERT na tabela sessions guardando sha256("abc123...") ← só o HASH 5. devolve o token CRU pro cliente, dentro de um cookie HttpOnly REQUISIÇÃO PROTEGIDA 1. o cookie chega com o token cru "abc123..." 2. servidor calcula sha256("abc123...") e faz SELECT ... WHERE token_hash = ... 3. achou + não expirou → injeta UserID no context → handler roda LOGOUT 1. DELETE FROM sessions WHERE token_hash = sha256(token) → sessão revogada 2. cookie é limpo no cliente (MaxAge = -1)

Repare na assimetria: o cookie do cliente guarda o token cru; o banco guarda só o sha256 do token. É exatamente o que a BR-303 exige. O código que faz isso são duas funções pequenas:

// internal/auth/token.go
func NewSessionToken() (string, error) {
    b := make([]byte, 32)            // 32 bytes = 256 bits de entropia
    if _, err := rand.Read(b); err != nil { ... }   // crypto/rand, não math/rand
    return base64.RawURLEncoding.EncodeToString(b), nil   // token CRU → cliente
}

func HashToken(token string) string {
    sum := sha256.Sum256([]byte(token))
    return hex.EncodeToString(sum[:])   // o HASH → banco
}
Por que guardar só o hash protege contra vazamento do banco? (BR-303) Imagine que alguém rouba um pg_dump do banco. Se guardássemos o token cru, o atacante poderia colar cada token num cookie e se passar por qualquer usuário logado. Guardando só o sha256, o dump contém apenas hashes — e sha256 é de mão única. O atacante teria que inverter o hash de um valor de 256 bits aleatórios, o que é inviável. O vazamento não rende nenhuma sessão usável. É a mesma ideia das senhas, aplicada aos tokens.

4. As 7 fases, uma a uma

Construímos de baixo pra cima, cada fase compilando e testável sozinha — domínio puro primeiro, adaptadores e HTTP por último. Você revisou cada uma.

Fase 1 — Domínio, ports e config

O coração do pacote internal/auth: os tipos User e Session, os erros-sentinela, a validação de e-mail/senha, os três ports (UserRepository, SessionRepository, PasswordHasher) e o helper de contexto. Tudo puro — sem SQL, sem HTTP, sem bcrypt.

// internal/auth/auth.go — erros como sentinelas, checados com errors.Is
var (
    ErrEmailTaken         = errors.New("email already registered")
    ErrInvalidCredentials = errors.New("invalid email or password")  // GENÉRICO de propósito
    ErrSessionNotFound    = errors.New("session not found or expired")
    ErrUserNotFound       = errors.New("user not found")
)
Por que MinPasswordLength = 8 e MaxPasswordLength = 72? O mínimo é política óbvia. O máximo de 72 não é arbitrário: o bcrypt ignora silenciosamente tudo além de 72 bytes. Sem esse teto, duas senhas longas com o mesmo prefixo de 72 bytes seriam intercambiáveis — um buraco sutil. Rejeitamos na fronteira. (Isso virou um achado de review — veja a seção 6.)

Também nasceu aqui o Clock port (internal/platform/clock): a sessão precisa saber a hora pra calcular a expiração, e o CLAUDE.md proíbe time.Now() direto no core. O SPEC-001 adiou esse port "até o primeiro consumidor de tempo" — a expiração de sessão é ele.

Fase 2 — Hashing de senha & tokens de sessão

O adaptador internal/auth/bcrypt implementa o port PasswordHasher com custo documentado (12), e o token.go gera/hasheia o token de sessão (visto na seção 3). Dois tipos de hash, dois propósitos diferentes — a distinção mais importante da aula está na seção 5.

Fase 3 — Persistência (migration + adaptadores)

A migration 0002_auth cria as duas primeiras tabelas com dono — seguindo as convenções que o 0001 documentou (UUID PK, timestamptz/UTC):

CREATE TABLE users (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    email text NOT NULL, password_hash text NOT NULL, ...);
CREATE UNIQUE INDEX users_email_key ON users (email);

CREATE TABLE sessions (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id uuid NOT NULL REFERENCES users (id) ON DELETE CASCADE,   -- apaga o user, somem as sessões
    token_hash text NOT NULL, expires_at timestamptz NOT NULL, ...);
CREATE UNIQUE INDEX sessions_token_hash_key ON sessions (token_hash);

E os adaptadores Postgres em internal/auth/postgres implementam os ports com SQL parametrizado — INSERT ... RETURNING, busca por e-mail, busca por token_hash, DELETE.

Fase 4 — O serviço de aplicação

O Service orquestra os ports — sem saber que existe banco ou HTTP. Register, Login, Logout, Authenticate. É puro o suficiente pra ser testado com fakes escritos à mão. Olhe o Login, onde mora a defesa anti-enumeração:

func (s *Service) Login(ctx, email, password string) (User, string, error) {
    user, err := s.users.GetUserByEmail(ctx, NormalizeEmail(email))
    if err != nil {
        if errors.Is(err, ErrUserNotFound) {
            _ = s.hasher.Compare(s.dummyHash, password) // gasta o MESMO tempo de um Compare real
            return User{}, "", ErrInvalidCredentials    // erro IDÊNTICO ao de senha errada
        }
        return User{}, "", fmt.Errorf("login: %w", err)
    }
    if err := s.hasher.Compare(user.PasswordHash, password); err != nil {
        return User{}, "", ErrInvalidCredentials
    }
    // sucesso → gera token, guarda sha256(token) + expiração, devolve o token CRU
    ...
}

Fase 5 — Transporte (handlers + middleware + seam)

Os handlers em transport/http/auth.go traduzem HTTP↔serviço: DTOs separados do domínio, o envelope genérico {"error":"..."}, e o cookie de sessão moldado com HttpOnly/Secure/ SameSite. O middleware authmw.go faz o deny-by-default. A senha nunca volta na resposta — o userResponse só tem id e email.

Fase 6 — Testes

Unitários com fakes (serviço, handlers, middleware) que rodam sem banco; e de integração contra Postgres real, com o mesmo gating do SPEC-002 (-short + TEST_DATABASE_URL). Os três destaques: o fluxo completo register→login→me→logout sobre httptest; uma asserção de que nada de texto puro é guardado (nem senha, nem token); e a prova do seam de isolamento (FR-306) com dois usuários.

Fase 7 — Documentação & aula

Atualizamos CHANGELOG.md, README.md (endpoints de auth, posture protegido-por-padrão, novas variáveis SESSION_TTL e AUTH_COOKIE_NAME), marcamos SPEC-003 e PLAN-003 como Done. E esta aula. 🙂

5. Conceitos-chave que você aprendeu

Dois hashes, dois propósitos: bcrypt (lento) vs sha256 (rápido)

Parece contraditório: por que hashear senha com um algoritmo lento e token com um rápido? A resposta está na entropia da entrada.

Senha (bcrypt)Token de sessão (sha256)
EntradaEscolhida por humano — baixa entropia ("senha123")32 bytes de crypto/rand — 256 bits de entropia
AmeaçaAtacante testa milhões de senhas comunsNão há o que "adivinhar" — o token já é aleatório
DefesaSer lento de propósito (custo 12) + salt embutidoSer rápido e determinístico, pra buscar no banco
Por que bcrypt é lento DE PROPÓSITO? E o que é salt? Uma senha humana tem pouca entropia: um atacante com o hash pode testar dicionários inteiros. bcrypt é deliberadamente caro (custo 12 ≈ dezenas/centenas de ms por tentativa) — pra você o login demora um piscar de olhos, mas pro atacante cada chute custa caro, inviabilizando força bruta em massa. O salt é um valor aleatório que o bcrypt gera e embute em cada hash, então duas pessoas com a mesma senha têm hashes diferentes — mata as "rainbow tables". Já o token de sessão não precisa nada disso: ele já é aleatório, e a busca tem que ser rápida — por isso sha256 simples.

Deny-by-default e a allowlist pública (BR-301)

O middleware nega tudo que não está numa lista pública explícita. Adicionar uma rota nova é seguro-por-padrão: ela exige login até você liberá-la conscientemente.

// internal/transport/http/authmw.go
func isPublicRoute(method, path string) bool {
    switch path {
    case "/healthz", "/readyz", "/version":  return true
    case "/auth/register", "/auth/login":    return method == http.MethodPost
    default:                                 return false   // ← negar é o padrão
    }
}

Identidade vem do contexto, nunca do payload (BR-304) — o seam auth.UserID(ctx)

Esta é a regra que torna o isolamento real. O middleware resolve a sessão, descobre quem é o usuário e injeta o ID no context. Handlers e (futuros) repositórios leem dali — nunca de um user_id que o cliente mandou no corpo.

// internal/auth/context.go — chave de contexto privada, impossível de colidir
type ctxKey int
const userIDKey ctxKey = iota

func WithUserID(ctx, userID string) context.Context { return context.WithValue(ctx, userIDKey, userID) }

func UserID(ctx context.Context) (string, bool) {     // o "seam" que as features vão usar
    id, ok := ctx.Value(userIDKey).(string)
    if !ok || id == "" { return "", false }           // não-autenticado é um ERRO, não um zero silencioso
    return id, true
}
Por que confiar num user_id do payload seria catastrófico? Se o handler lesse o dono do dado de um campo enviado pelo cliente, eu poderia mandar {"user_id": "id-do-vizinho"} e ler ou apagar a carteira de outra pessoa — é a vulnerabilidade clássica de IDOR. Tirando a identidade sempre da sessão autenticada (que o cliente não controla), esse ataque some por construção. Quando o SPEC-101 (Profile) chegar, todo SELECT vai fazer WHERE user_id = $1 com o ID vindo do context — e nada mais.

O Clock port pra expiração determinística

A expiração da sessão compara expires_at com "agora". Mas "agora" via time.Now() deixaria os testes não-determinísticos. Por isso o Authenticate usa o Clock injetado: nos testes, passamos um Clock fake que devolve uma hora fixa, e conseguimos provar "sessão expirou → 401" sem sleep nenhum.

6. Os achados da review que viraram correções 🐛

Esta é a seção mais didática. O subagente revisor (e o gate de qualidade) pegaram quatro coisas que, sozinhas, teriam virado bugs sutis em produção. Cada uma ensina algo sobre onde os 500 escondem 401 e vice-versa.

(a) Cast de UUID inválido virava 500 — deveria ser "não encontrado" Os IDs são string e o SQL faz WHERE id = $1::uuid. Se alguém passar uma string que não é um UUID, o Postgres devolve o erro 22P02 (invalid_text_representation) — e isso borbulharia como um 500. Mas uma string que nunca poderia casar com nenhuma linha é, na prática, "não encontrado". Mapeamos 22P02 → auth.ErrUserNotFound: 
var pgErr *pgconn.PgError
if errors.As(err, &pgErr) && pgErr.Code == invalidTextRepresentation { // "22P02"
    return auth.User{}, auth.ErrUserNotFound   // não-encontrado, não erro de servidor
}
(b) Distinguir 401 (sem sessão) de 500 (banco fora) No middleware, é tentador tratar qualquer falha do Authenticate como "não autenticado → 401". Errado: se o banco caiu, isso não é problema de credencial — é infraestrutura, e deve ser um 500 (logado!). Só ErrSessionNotFound vira 401; o resto vira 500. 
if errors.Is(err, auth.ErrSessionNotFound) {
    writeError(w, http.StatusUnauthorized, "authentication required")  // 401: sessão ausente/expirada
    return
}
logger.Error("authentication check failed", ...)
writeError(w, http.StatusInternalServerError, "internal error")        // 500: o banco falhou
Mascarar um 500 como 401 esconde incidentes reais e confunde o cliente ("minhas credenciais estão certas, por que 401?!").
(c) bcrypt trunca em 72 bytes → validar o tamanho máximo Como vimos na Fase 1, o bcrypt descarta bytes além de 72 sem avisar. Sem um teto, senhas longas com prefixo comum colidiriam. A correção foi a constante MaxPasswordLength = 72 e a rejeição explícita no ValidatePassword — um limite de segurança virou uma regra de validação visível, com erro ErrPasswordTooLong → 400.
(d) Corrida entre testes de integração que compartilham um banco Os testes de integração rodam contra um Postgres. Por padrão o Go roda pacotes de teste em paralelo — então dois pacotes mexiam nas mesmas tabelas ao mesmo tempo e davam falhas intermitentes (flaky). A correção foi serializar os binários de teste com -p 1 no Taskfile.yml: 
# task test:integration
- go test ./... -count=1 -p 1   # -p 1 serializa: os testes compartilham um banco
Lição: testes que compartilham estado externo (um banco) não são independentes — ou você isola o estado por teste, ou serializa a execução. Aqui, serializar foi o mais simples e honesto.

7. Harness Engineering: como a máquina construiu isto

Este projeto tem um propósito duplo: além de aprender Go e backend, aprender engenharia de harness — como dar à IA trilhos que baixam a taxa de erro. O SPEC-003 foi construído por esse harness, e vale ver os mecanismos concretos que atuaram.

O fluxo /spec-implement, fase a fase

A IA não atacou "faça auth" de uma vez. O PLAN-003 já fatiou o trabalho em 7 fases bottom-up (domínio → primitivas → persistência → serviço → transporte → testes → docs), e cada fase foi implementada, compilada e revisada por você antes da próxima. Fatiar reduz a janela onde um erro pode se propagar: se a Fase 2 (hashing) estiver errada, você pega ali, não três camadas depois.

O subagente hexagonal-reviewer

Os quatro achados da seção 6 não apareceram por sorte — vieram de um revisor especializado rodando sobre o diff. Ele conhece as regras da casa: dependência só aponta pra dentro, auth não importa SQL/HTTP, erros embrulhados com %w, dinheiro não é float. Foi ele que questionou "e se o id não for um UUID?" (achado a) e "isto é 401 ou 500?" (achado b). Um revisor com contexto fixo pega classes inteiras de bug que um humano cansado deixa passar.

A prova de que o core ficou puro Há uma verificação mecânica de que internal/auth (o núcleo) não importa nenhum tipo de SQL, HTTP ou SDK (BR-306). O bcrypt mora em internal/auth/bcrypt; o SQL em internal/auth/postgres; o middleware em transport/http. E os adaptadores têm compile-time checks (var _ auth.UserRepository = UserRepository{}) que quebram a build se o adaptador deixar de satisfazer o port. O compilador vira parte do harness.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs, a review foca no que importa.
  • Bloqueio de imutabilidade: ADRs aceitos não podem ser editados. O SPEC-003 referencia ADR-0002/0003 mas não os altera — se a IA tentasse "corrigir" um ADR aceito, o hook barraria.
  • O backstop do Stop: ao fechar a tarefa, o gate exige build/vet/test limpos. Foi esse backstop que tornou a corrida dos testes (achado d) visível em vez de passar batido.

8. Conexão com AI Engineering

O SPEC-003 não é uma feature de IA — é fundação de segurança. Mas ele prepara o terreno da arquitetura de IA que vem por aí, e vários padrões aqui têm um espelho direto lá na frente.

O que construímos agoraComo serve a IA depois
Isolamento por user_id (FR-306) O Fact Builder que alimenta o LLM vai computar fatos da carteira de um usuário só. O WHERE user_id = $1 de hoje é o que garante que os fatos (e logo os insights) de A nunca vazem pra B. Sem esse isolamento, um insight personalizado seria um vazamento de dados.
O port PasswordHasher (swappable) É o mesmo formato de costura do futuro port Insighter (o LLM). Trocar bcrypt→argon2id sem tocar no serviço é o ensaio de trocar o provedor de LLM (local→hospedado) sem tocar nas features. A disciplina de "provedor atrás de um port" é a base do ADR-0003.
Identidade no context Quando um agente CIO multi-agente e o MCP chegarem, a identidade vai precisar fluir pelas chamadas de ferramenta sem ser confiada à entrada. O auth.UserID(ctx) de hoje é o mesmo canal por onde a autorização vai viajar amanhã.
Gates de verificação como middleware O deny-by-default é, em essência, um gate de verificação na borda. É a mesma forma dos gates de IA que vêm: o de explicabilidade (FR-013, todo insight carrega explicação) e o de não-aconselhamento (FR-014, nunca dar ordem de compra/venda) serão middlewares que negam por padrão uma saída do LLM que não passe na regra.
A ideia que conecta tudo Segurança e IA confiável compartilham a mesma postura: negar por padrão e provar conformidade na borda. Aqui foi "sem sessão válida, sem acesso". Lá será "sem explicação e sem disclaimer, sem resposta". A mesma engenharia de gate, propósitos diferentes.
