1. A visão geral: o que é observabilidade e o que mudou

Nas três aulas anteriores o app ganhou esqueleto (SPEC-001), banco (SPEC-002) e identidade (SPEC-003). Ele funciona — mas era uma caixa-preta em produção: se uma requisição ficasse lenta, você só saberia "está lento em algum lugar", sem onde. O SPEC-004 é a quarta pedra da fundação: o app passa a poder ser observado por dentro.

Observabilidade é a capacidade de responder "o que está acontecendo aqui dentro?" só olhando o que o sistema emite — sem precisar abrir um debugger em produção. Ela se apoia nos chamados três pilares (ou três sinais): logs, métricas e traces. O SPEC-001 já nos deu logs estruturados; o SPEC-004 acrescenta os outros dois e, crucialmente, os costura juntos.

ANTES (SPEC-003) DEPOIS (SPEC-004) logs soltos (sem correlação) logs com trace_id + span_id "está lento em algum lugar" trace mostra: o SELECT em /auth/me levou 240ms nenhuma métrica de latência histograma de duração por rota (mede o p95) caixa-preta em produção HTTP span -> DB span, ponta a ponta zero dependência de telemetria ...e CONTINUA zero: tudo desligado por padrão

Concretamente, o SPEC-004 entregou: o pacote internal/platform/observability com um Setup() que liga o SDK do OpenTelemetry; instrumentação HTTP (um span e métricas por requisição) via otelhttp; instrumentação do banco via otelsql (cada query vira um span filho); a correlação log↔trace (um slog.Handler que injeta trace_id/span_id); e o seam Tracer()/Meter() que as features de IA vão usar. Nenhuma tabela, nenhuma migration, nenhum endpoint novo — é pura infraestrutura transversal.

Por que isso importa agora, antes das features? O PRD define um SLO concreto: p95 < 300 ms para leituras não-IA. Mas você não consegue cumprir uma meta de latência que não mede. Sem o histograma de duração por rota que o SPEC-004 instala, "p95 < 300ms" é só um desejo. Construindo a observabilidade como fundação, toda feature que vier (carteira, perfil, insights) nasce observável de graça — ninguém precisa lembrar de instrumentar.

2. Os três sinais — e como o trace_id costura tudo

Os três pilares respondem perguntas diferentes. A mágica está em poder pular de um pro outro usando um identificador comum.

SinalO que éPergunta que respondeNo SPEC-004
Traces A jornada de uma requisição, quebrada em spans pai/filho com tempos "onde o tempo foi gasto nesta requisição?" span HTTP do servidor → span filho do DB
Métricas Números agregados ao longo do tempo (contadores, histogramas) "como está a saúde no todo? qual o p95?" histograma de duração + contagem por rota/status
Logs Eventos pontuais e textuais ("usuário X logou", "erro Y") "o que exatamente aconteceu neste ponto?" slog (do SPEC-001) agora com trace_id/span_id

Um trace é uma árvore de spans. Cada span é uma unidade de trabalho com início, fim e um nome — e tem um pai. Quando a requisição entra, o otelhttp abre o span raiz (o servidor HTTP); quando esse handler chama o banco, o otelsql abre um span filho dentro dele. Visualmente:

UM TRACE de GET /auth/me (trace_id = 4bf92f3577b34da6...) │ ├─ SPAN "GET /auth/me" ◄ span raiz, criado pelo otelhttp [0ms ─────────── 248ms] │ http.method=GET http.route=/auth/me http.status_code=200 │ │ │ └─ SPAN "db.Query" ◄ span FILHO, criado pelo otelsql [12ms ──── 240ms] │ db.system=postgresql │ db.statement="SELECT ... FROM sessions WHERE token_hash = $1" ◄ SÓ o $1, nunca o valor E a linha de log daquela requisição (mesmo trace_id): {"level":"INFO","msg":"request","method":"GET","path":"/auth/me","status":200, "request_id":"req-abc","trace_id":"4bf92f3577b34da6...","span_id":"00f067aa..."}

Repare: o trace_id aparece tanto no trace quanto na linha de log. Esse é o pulo do gato da correlação — você vê uma linha de log de erro, copia o trace_id, cola no Jaeger/Grafana e enxerga a cascata inteira daquela requisição: quais spans rodaram, onde o tempo foi, qual query foi lenta. Os três sinais deixam de ser ilhas.

3. As 5 decisões (D1–D5) e por quê

Toda spec tem bifurcações. Aqui foram cinco decisões, todas confirmadas com você antes do PLAN e agora binding. O trade-off de cada uma é onde mora o aprendizado.

#DecisãoEscolhaPor quê
D1Quais sinais agora? Traces + métricas + correlação de logs — os três de uma vez O SLO de p95 só fica mensurável com métricas; os traces dão o "onde". Fazer os três juntos evita uma segunda passada de retrabalho (PRD §10).
D2Como exportar? OTLP/HTTP, via config, desligado por padrão; stdout pra dev OTLP é vendor-neutro (ADR-0003): troca de backend é só config. Sem endpoint ⇒ no-op (custo zero). Recusamos /metrics pull do Prometheus (superfície pública extra).
D3Instrumentar o HTTP como? O middleware otelhttp da contrib oficial É a forma padrão de instrumentar net/http. Spans escritos à mão foram recusados — reinventar a roda e errar nos detalhes (status, duração, propagação).
D4E os logs? Manter o slog e injetar trace_id/span_id Preserva o baseline de logging do SPEC-001. O SDK de logs do OTel é mais novo/pesado e dá pouco ganho no MVP — adiado (e revisitável).
D5Instrumentar o banco como? otelsql no nível do pool (platform/database) Spans de DB "de graça", sem nenhum import de OTel nos repositórios. Spans manuais por repositório foram recusados — repetitivo e fácil de esquecer.
Por que OTLP/HTTP e não OTLP/gRPC? O OpenTelemetry exporta via dois transportes: gRPC ou HTTP. O gRPC traz uma árvore de dependências pesada (todo o stack do gRPC + protobuf). Como nosso protocolo é o mesmo OTLP nos dois casos e a vazão de telemetria de um MVP é modesta, escolhemos OTLP/HTTP de propósito: menos dependências, build mais leve, mesma compatibilidade com qualquer collector. É a postura "stdlib-first / justifique cada dependência" aplicada até dentro do OTel.

4. O coração do design: "no-op por padrão / custo zero" (BR-401)

Esta é a regra mais importante da aula e o que torna o SPEC-004 honesto com a postura de custo zero. A BR-401 diz: telemetria nunca é obrigatória pra rodar. Sem um endpoint configurado, o app roda idêntico — sem backend, sem erro, sem custo, sem latência extra. E uma falha de exportação nunca chega no caminho da requisição.

Como isso funciona na prática? O OpenTelemetry tem um truque elegante: os providers globais já começam como "no-op" (não fazem nada). O Setup() só os substitui por providers reais se houver um exportador configurado. Olhe o início do Setup:

// internal/platform/observability/observability.go
func Setup(ctx context.Context, cfg config.Config, version string) (shutdown func(context.Context) error, err error) {
    // O propagador W3C é SEMPRE instalado — é baratíssimo e deixa o trace context
    // fluir pelo context mesmo quando NÃO estamos exportando nada.
    otel.SetTextMapPropagator(propagation.NewCompositeTextMapPropagator(
        propagation.TraceContext{},
        propagation.Baggage{},
    ))

    if !cfg.TelemetryEnabled() {        // kind == "none" (o padrão sem endpoint)
        return noopShutdown, nil        // não troca os providers globais: ficam no-op
    }
    // ...daqui pra baixo só roda se houver exportador: monta Resource, exporters,
    //    TracerProvider (com batcher + sampler) e MeterProvider, e os instala.
}

E TelemetryEnabled() é uma linha só na config:

// internal/platform/config/config.go
func (c Config) TelemetryEnabled() bool { return c.OTELExporterKind != "none" }
Por que o otelhttp e o otelsql não quebram quando está tudo no-op? Porque eles falam só com os providers globais. Quando o Setup deixa o global como no-op, cada span.SetName(...), span.SetAttributes(...) e span.End() vira uma chamada que não faz nada (e o compilador/runtime otimiza a quase-zero). Por isso o routeNamer e o handler de log podem rodar sempre, sem nenhum if telemetria_ligada espalhado pelo código — o "ligado/desligado" mora num lugar só (o Setup), e o resto do código é cego pra isso. Menos ramificação = menos bug.

O efeito é que a observabilidade é aditiva e inerte: mergeada na main, sem endpoint, ela não muda absolutamente nada no comportamento do app. Liga-se quando (e onde) você quiser, só setando OTEL_EXPORTER_OTLP_ENDPOINT.

5. As 7 fases, uma a uma

Como nas specs de infra anteriores, construímos de baixo pra cima — cada fase compilando e revisável sozinha. A ordem segue o fluxo de uma requisição: primeiro a base (config + providers), depois HTTP, depois DB, depois a costura dos logs, depois testes e docs.

Fase 1 — Configuração & dependências

Os campos OTEL_* na Config, validados no Load() (ratio fora de 0..1 ou kind inválido ⇒ falha rápida, como o resto da config). E os módulos novos do OTel (sdk, exporters OTLP/HTTP + stdout, otelhttp, otelsql).

// internal/platform/config/config.go (trechos)
OTELServiceName      string  // resource service.name (default "yield-forge")
OTELExporterKind     string  // otlp | stdout | none
OTELExporterEndpoint string  // URL do OTLP; vazio => kind vira "none"
OTELExporterHeaders  string  // headers OTLP (SEGREDO), ex. "authorization=Bearer xyz"
OTELTraceSampleRatio float64 // probabilidade de amostragem 0.0..1.0 (NÃO é taxa financeira)
Detalhe de convenção: OTELTraceSampleRatio é um float64 — e tudo bem O CLAUDE.md proíbe float64 pra dinheiro e taxas financeiras (lá é int64 em centavos / basis points). Mas a razão de amostragem não é um valor financeiro — é uma probabilidade de engenharia (0.25 = "guarde 25% dos traces"). Por isso o comentário no campo grita "NÃO é taxa financeira": deixa explícito pro próximo leitor (e pro revisor) que a exceção é deliberada, não um escorregão.

Fase 2 — Bootstrap da observabilidade (providers + exporter)

O Setup() (visto na seção 4): monta o Resource que descreve o serviço, escolhe os exporters pelo kind, e devolve um shutdown que dá flush + fecha os providers. O Resource carrega service.name, service.version (do buildinfo) e deployment.environment (o APP_ENV) — para todo span/métrica saber de qual serviço/versão veio.

Fase 3 — Instrumentação HTTP (traces + métricas)

O router é embrulhado por otelhttp.NewHandler como camada mais externa, gerando um span de servidor + métricas de duração/contagem por requisição. A ordem da cadeia importa muito:

// internal/transport/http/router.go
var handler http.Handler = routeNamer(mux)
handler = requireAuth(d.Auth, d.CookieName, d.Logger)(handler)
handler = logRequests(d.Logger)(handler)
handler = requestID(handler)
// Mais externo: cria o span do servidor + métricas HTTP. Probes filtradas pra não
// poluir os traces. Métricas vêm de graça do otelhttp via o MeterProvider global.
return otelhttp.NewHandler(handler, "http.server", otelhttp.WithFilter(traceableRequest))
Por que otelhttp é o mais externo, e não o mais interno? Porque o span precisa já existir no context quando os middlewares de dentro rodam. O logRequests lê o trace_id do span pra colocar na linha de log (Fase 5) — se o span fosse criado por dentro do logging, a linha sairia sem trace_id. Colocando o otelhttp por fora, o span "embrulha" tudo: o tempo medido inclui a autenticação e o logging, que é o tempo real do servidor. As probes (/healthz, /readyz) são filtradas com WithFilter pra não afogarem o sinal.

Fase 4 — Instrumentação do banco

No platform/database.Connect, o pool é aberto via otelsql.Open — então toda query emite um span filho sob o span da requisição, sem tocar em nenhum repositório:

// internal/platform/database/database.go
db, err := otelsql.Open(driverName, cfg.DatabaseURL,
    otelsql.WithAttributes(attribute.String("db.system", "postgresql")),
    otelsql.WithSpanOptions(otelsql.SpanOptions{OmitConnResetSession: true}),
)
// ... e métricas do pool (conexões abertas/ociosas/em uso) no MeterProvider global:
otelsql.RegisterDBStatsMetrics(db, otelsql.WithAttributes(attribute.String("db.system", "postgresql")))

Detalhe importante: o otelsql devolve os erros do driver sem embrulhar, então o mapeamento de erros do pgconn que fizemos no SPEC-003 (o 23505 de e-mail duplicado, o 22P02 de UUID inválido) continua funcionando. A instrumentação não vazou pra dentro da lógica de erro.

Fase 5 — Correlação log/trace & o seam

Um slog.Handler "embrulhador" que, quando há um span ativo no context, anexa trace_id/span_id ao registro. É a chave de tudo que vimos na seção 2:

// internal/platform/logging/logging.go
type traceHandler struct{ slog.Handler }

func (h traceHandler) Handle(ctx context.Context, r slog.Record) error {
    if sc := trace.SpanContextFromContext(ctx); sc.IsValid() {   // há span ativo?
        r.AddAttrs(
            slog.String("trace_id", sc.TraceID().String()),
            slog.String("span_id", sc.SpanID().String()),
        )
    }
    return h.Handler.Handle(ctx, r)   // delega pro handler de baixo (JSON/text)
}

E o seam (seam.go): dois acessores Tracer(name) e Meter(name) que as features futuras vão injetar pra criar spans/métricas próprios — sem nunca apelar pra globais espalhados.

Fase 6 — Testes

Unitários com um exportador em memória (sem rede): Setup no-op-safe; um request pelo handler embrulhado produz um span com o nome da rota esperado; probes filtradas; um traceparent de entrada é continuado; o log dentro de um span carrega os ids. E um de integração (com Postgres real, mesmo gating do SPEC-002/003) provando o span pai HTTP com o span filho de DB.

Fase 7 — Documentação & aula

CHANGELOG.md, README.md (como rodar com um collector local / Jaeger e como ele no-opa sem um), .env.example com as novas vars, SPEC-004 e PLAN-004 marcados Done. E esta aula. 🙂

6. Conceitos-chave de Go + OTel

Span: a unidade de um trace (pai e filho)

Um span mede uma operação: tem nome, início, fim, atributos (chave=valor) e um span pai. Spans aninhados formam a árvore do trace. No nosso caso, o span HTTP é o pai, o span do DB é o filho. A regra de ouro: todo span aberto tem que ser fechado, idiomaticamente com defer span.End() na borda — um span aberto e nunca fechado é um bug clássico (vaza memória e some do trace).

Propagação no context.Context (BR-405)

Como o span filho "sabe" quem é seu pai? Pelo context.Context. Quando o otelhttp cria o span raiz, ele guarda o span dentro do ctx da requisição. Esse ctx é passado adiante (a convenção "ctx é sempre o primeiro parâmetro" do CLAUDE.md). Quando o otelsql vai abrir o span do DB, ele lê o span ativo daquele mesmo ctx e se pendura como filho. É por isso que a BR-405 é "o context carrega o estado do trace, nunca uma global" — o ctx é o fio que costura a árvore inteira, sem estado global nem thread-local.

O routeNamer: como o span fica com o nome da rota O otelhttp está por fora do mux e, no momento em que cria o span, ainda não sabe qual rota casou — só tem o caminho cru. Por isso o routeNamer (o middleware mais interno) resolve a rota pelo próprio mux e renomeia o span já existente: 
func routeNamer(mux *http.ServeMux) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if _, pattern := mux.Handler(r); pattern != "" {   // ex.: "GET /auth/me"
            span := trace.SpanFromContext(r.Context())
            span.SetName(pattern)
            span.SetAttributes(attribute.String("http.route", pattern))
        }
        mux.ServeHTTP(w, r)
    })
}
Num span no-op (telemetria off ou probe filtrada), SetName/SetAttributes são no-ops inofensivos.

O cabeçalho W3C traceparent (FR-404)

Como um trace atravessa serviços diferentes? Pelo padrão W3C Trace Context: um cabeçalho HTTP traceparent que carrega o trace_id e o span_id pai. Se uma requisição chega com esse header, o otelhttp continua o trace existente (o span do nosso servidor vira filho do span do chamador); se chega sem, começa um trace-raiz novo. É o que o propagador instalado no Setup (propagation.TraceContext{}) faz — e por ser um padrão aberto, funciona entre serviços de linguagens diferentes.

Sampling (amostragem)

Em produção, gravar todo trace é caro e estoura cota de free-tier. O sampler decide quais traces guardar. Usamos ParentBased(TraceIDRatioBased(ratio)): respeita a decisão do pai (se o chamador amostrou, a gente também — o trace não fica pela metade) e, na raiz, guarda uma fração ratio deles. Em dev o padrão é 1.0 (guarda tudo); em prod, abaixa-se. Isso é a BR-406: custo limitado e configurável.

O seam Tracer()/Meter() (FR-407)

O ponto onde o SPEC-004 olha pro futuro. Em vez de cada feature chamar otel.Tracer(...) direto, ela pega o Tracer/Meter por esses acessores e instrumenta nas bordas (adaptadores/transporte), nunca no domínio puro. Quando a telemetria está off, ambos são no-op. O doc-comment do seam já documenta as convenções para os próximos:

// internal/platform/observability/seam.go
// Conventions for callers (SPEC-004 BR-402/BR-403):
//   - Span names are low-cardinality (operation, not raw ids/values).
//   - Never put secrets or PII (passwords, tokens, raw emails, SQL args) on spans.
//   - Instrument at the edges (adapters/transport) — never inside the pure domain core.
//   - End spans with defer span.End().
func Tracer(name string) trace.Tracer { return otel.Tracer(name) }
func Meter(name string) metric.Meter  { return otel.Meter(name) }

7. Momentos reais de engenharia 🐛

Esta é a seção mais didática. Quatro situações concretas desta spec, cada uma ensinando uma lição de backend/observabilidade que você não acha num tutorial.

(decisão) O bump obrigatório pro Go 1.25 — e por que dessa vez não dava pra fugir No SPEC-002, o pgx tinha uma versão antiga compatível com o Go 1.24 que já usávamos — o risco do bump foi evitável. Aqui foi diferente: todo o ecossistema OTel (SDK + exporters + contrib) e as golang.org/x/* que ele arrasta exigem Go 1.25. Não havia versão "velha o suficiente" que cumprisse o SPEC-004. A lição honesta de engenharia de dependências: às vezes o bump é inevitável, e a escolha real é "aceitar o bump (e atualizar o Dockerfile builder pra golang:1.25)" vs. "ficar preso a dependências obsoletas e sem suporte". Escolhemos aceitar — mas a regra do PLAN era sinalizar o bump pra aprovação, não deixá-lo "vazar" silenciosamente num go mod tidy. Bump consciente, não acidental.
(problema técnico) Nomear o span pela ROTA, mas o otelhttp está por FORA do mux Queríamos o span chamado GET /auth/me (a rota). Mas o otelhttp embrulha o handler por fora e, na hora de criar o span, ainda não sabe qual rota vai casar — ele só tem o caminho cru da URL. Se nomeássemos pelo caminho cru, uma rota com id (imagine /users/{id}) viraria milhares de nomes de span distintos — /users/abc, /users/def, /users/123... Isso é explosão de cardinalidade: cada nome único vira uma série temporal nas métricas, estoura a memória do backend e a cota do free-tier, e ainda quebra a agregação ("qual o p95 de /users/{id}?" fica impossível se cada id é um nome). A solução foi o routeNamer (seção 6): ele roda dentro, resolve a rota com mux.Handler(r) e renomeia o span pro padrão de rota — baixa cardinalidade, agregação sã. É exatamente a BR-406 virando código.
(segurança) BR-402: nunca colocar segredo/PII na telemetria Telemetria vai pra um backend externo e fica visível pra quem opera dashboards — é o último lugar onde você quer uma senha, um token de sessão, um e-mail cru ou um valor de query. A BR-402 proíbe isso em spans, atributos e labels de métrica. O risco mais sutil é o span de DB: é tentador gravar a query com os valores. Por isso o otelsql grava só o statement parametrizado — WHERE token_hash = $1, e nunca o valor do $1 (que é justamente o hash da sessão!). O comentário no database.go documenta a garantia explicitamente. A mesma postura de redação dos SPEC-002/003, agora estendida à telemetria.
(convenção) O nome certo dos arquivos de teste: por arquivo, nunca por conceito Um teste do span HTTP nasceu como otel_test.go — nomeado pelo conceito ("isto testa OTel"). Errado pela convenção da casa: arquivo de teste leva o nome do arquivo sob teste (foo.go → foo_test.go; integração → foo_integration_test.go), nunca por tema. O span HTTP mora no router.go, então o teste virou router_test.go. Parece bobo, mas a regra mantém o teste colado ao código que ele cobre: quem abre o router.go acha o teste do lado, não precisa caçar um otel_test.go genérico. Convenção mecânica = menos atrito de navegação. (Ficou documentado no CLAUDE.md.)

8. Harness Engineering: como a máquina construiu isto

Este projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. O SPEC-004 foi construído por esse harness; vale ver os mecanismos concretos que atuaram.

O fluxo /spec-implement, fase a fase

A IA não atacou "faça observabilidade" de uma vez. O PLAN-004 fatiou em 7 fases bottom-up (config → providers → HTTP → DB → correlação/seam → testes → docs), e cada fase foi implementada, compilada e revisada por você antes da próxima. Fatiar encurta a janela de propagação de erro: se a Fase 3 (HTTP) estiver errada, você pega ali, não três camadas depois.

O subagente hexagonal-reviewer

O revisor especializado conhece as regras da casa e foi ele que cobrou as duas mais sensíveis aqui: a direção de dependência (OTel só pode viver em platform/observability, platform/database, platform/logging e transport/http — nenhum core de feature importa OTel, BR-403) e o no-PII (BR-402: "este atributo pode vazar um valor de query?"). Foi essa lente que garantiu que o otelsql ficasse só com o statement parametrizado, e que o domínio puro continuasse cego ao OTel.

A prova de que o core ficou puro A garantia de que internal/auth (o núcleo) não importa OTel é mecânica: os repositórios em auth/postgres continuaram sem nenhuma mudança — toda a instrumentação de DB foi parar num lugar só, o otelsql.Open dentro de platform/database. Se a IA tivesse "ajudado" colocando um otel.Tracer dentro de um repositório, o revisor barraria. O fato de o diff dos repositórios ser vazio é a melhor prova de que a costura ficou na borda certa.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs, a review foca no que importa.
  • Bloqueio de imutabilidade: o SPEC-004 referencia ADR-0002/0003 (que já abençoam OTel) mas não os edita — se a IA tentasse "atualizar" um ADR aceito, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar a tarefa, o gate exige build/vet/test/gofmt limpos com e sem banco. Foi esse backstop que garantiu que os testes no-op-safe (rodam sem nenhum backend de telemetria) realmente passassem — provando a BR-401 na prática, não só no papel.

9. Conexão com AI Engineering

O SPEC-004 não é uma feature de IA — é a fundação que torna a IA depurável e auditável. E a IA confiável que o YieldForge promete (explicável, copiloto-não-conselheiro) só é possível se cada interação com o LLM for observável ponta a ponta. É aqui que os trilhos foram postos.

O que construímos agoraComo serve a IA depois
O seam Tracer()/Meter() (FR-407) O SPEC-005 (port Insighter/LLM) vai registrar, por chamada, um span com prompt/modelo/latência e métricas de custo de tokens — usando exatamente este Meter. Quando uma resposta da IA demorar ou custar caro, o trace dirá onde e quanto. O SPEC-006 (market data) registra métricas de ingestão pelo mesmo seam. Sem rework.
Métrica monetária como int64 O custo de tokens é dinheiro — e o seam já documenta que essa métrica futura segue a convenção int64 em unidades menores / basis points, nunca float. A disciplina financeira da casa já está reservada na telemetria, antes da primeira chamada de IA existir.
Correlação log↔trace (trace_id) Quando um insight sair errado, você vai partir da linha de log, pegar o trace_id e ver a cascata inteira: o Fact Builder que computou os números, a chamada ao LLM, a resposta. É a base da explicabilidade operacional que dá suporte ao FR-013 ("todo insight carrega explicação") — antes de explicar pro usuário, você precisa conseguir reconstruir o que aconteceu.
No-PII na telemetria (BR-402) Quando os spans de IA gravarem o prompt/contexto, a mesma regra protege contra vazar dados da carteira do usuário pro backend de observabilidade. A postura de redação que blindou o $1 da query é a mesma que vai blindar o conteúdo enviado ao LLM.
A ideia que conecta tudo IA confiável exige medir tudo o que ela faz: latência, custo, e o caminho dos fatos até a resposta. O SPEC-004 não chama nenhum LLM — mas instala a régua, o cronômetro e o fio (trace_id) que vão tornar cada decisão da IA reconstruível. Sem essa fundação, "explicável" e "auditável" seriam só palavras no PRD.
