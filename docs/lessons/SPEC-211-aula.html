SPEC-211 — Telas de Gestão de Carteira (Carteira)

O SPEC-210 ensinou o Perfil — uma tela pequena, sem dinheiro, sem IA. O SPEC-211 é maior e mais arriscado: a tela Carteira, onde Rafael e Carla cadastram o que realmente possuem — FIIs e renda fixa. É aqui que "facts are computed, not generated" (a regra binding #3 do CLAUDE.md) começa a valer de verdade: sem estes dados, o Dashboard, os Insights, o Health Score e as Projeções não têm nada para calcular sobre. Mas o valor maior desta aula não é o CRUD em si — é uma coleção de bugs reais, pegos de cinco jeitos diferentes (uma auditoria, rodar num navegador de verdade, escrever um teste ponta a ponta, e duas revisões de código), que ensina por que "compila e os testes passam" não é o mesmo que "funciona". Você não precisa dominar React para acompanhar.

1. A visão geral: duas verticais de CRUD, unificadas pelo indexador do SPEC-109

O SPEC-211 transforma a rota /portfolio — até aqui um EmptyState stubado pelo SPEC-200 — na tela real Carteira: a frente de cliente do backend SPEC-102 (o CRUD de holdings) somado ao SPEC-109 (o indexador de renda fixa — % do CDI, IPCA + spread). São, na prática, duas verticais independentes que compartilham os mesmos padrões:

  • FIIs — ticker, quantidade, preço médio. Rafael registra suas compras mensais de fundos imobiliários.
  • Renda fixa — nome, instituição, valor investido, e um indexador (Prefixado / % do CDI / IPCA + spread) em vez de uma taxa anual fixa forçada. Carla mantém sua escada de CDBs e Tesouro Direto do jeito que o extrato do banco realmente mostra: "120% do CDI", não "10,50% ao ano" fingindo ser uma taxa fixa.
ANTES (SPEC-200 — a rota stubada) DEPOIS (SPEC-211 — a tela Carteira real) /portfolio -> EmptyState("chega depois") /portfolio -> duas secoes: FiiSection + FixedIncomeSection nenhum holding e lido/escrito 8 endpoints SPEC-102 + GET /market/indicators (SPEC-109) 0 modais no design system components/ui/dialog.tsx (nativo) + confirm-dialog.tsx parsing pt-BR so em uma direcao (exibir) parseCentavos/parseBps: a direcao INVERSA tambem (digitar) 0 telas de feature de verdade alem do Perfil a 1a tela que grava os FATOS que a IA vai usar depois
Por que "Carteira antes de Dashboard/Insights" é a ordem certa A PRD é explícita: o produto raciocina sobre fatos computados, nunca inventa números (regra binding #3). Mas um fato computado precisa primeiro ter sido registrado por alguém. Sem a Carteira, o Dashboard (SPEC-212) não tem holdings para somar, os Insights (SPEC-213) não têm concentração para avaliar, o Health Score não tem nada para pontuar. A Carteira é o ponto de entrada dos fatos — a tela mais "chata" (formulário, tabela, CRUD) e, ao mesmo tempo, a mais estrutural de todo o track SPEC-21x depois da fundação.

Isso é uma decisão de escopo deliberada (BR-2113, seção 4) que mantém esta tela honesta sobre o que ela sabe: fatos de custo, não avaliação.

Repare de novo no padrão "404 não é erro" que o SPEC-210 já tinha ensinado para /profile — aqui ele aparece duas vezes, para editar e para excluir, e com uma nuance nova: um 404 ao excluir é tratado como sucesso (BR-2111), porque o estado desejado — "esse holding não existe mais na lista" — já foi alcançado, independente de quem o removeu primeiro.

// web/lib/portfolio/holdings.ts — um 404 ao excluir É sucesso (BR-2111)
export function useDeleteFIIHolding() {
  return useMutation({
    mutationFn: async (id: string) => {
      const { error, response } = await api.DELETE("/holdings/fii/{id}", { params: { path: { id } } });
      if (error && response.status !== 404) {
        throw new ApiError(response.status, backendError(error) ?? "delete failed (" + response.status + ")");
      }
      // 404 aqui não lança — a lista é recarregada e o holding já não está nela, que é o objetivo.
    },
    onSettled: () => qc.invalidateQueries({ queryKey: FII_KEY }),
  });
}

4. As decisões de produto e UX — e como as travas do produto ficam visíveis

O SPEC-200 ensinou que a explicabilidade (FR-013) e a não-consultoria (FR-014) viram componentes em telas que produzem saída de IA. A Carteira não produz saída de IA — ela só registra fatos — então essas travas específicas não se aplicam aqui (BR-2115, o mesmo raciocínio do Perfil, BR-2103). Mas isso não significa que a Carteira não tem disciplina de produto própria. Ela tem três, visíveis no cliente:

DecisãoOnde aparecePor quê
Custo, nunca valor de mercado (BR-2113) A tabela mostra average_price_centavos / invested_amount_centavos — o que foi pago — nunca um valor atual calculado nem um ganho/perda Calcular valor de mercado exige preço atual + a matemática de ganho/perda — isso é trabalho do Dashboard (SPEC-212), que cruza holdings com dados de mercado. Misturar as duas responsabilidades aqui seria a Carteira "inventando" um número que não é seu para calcular.
Dinheiro em centavos inteiros, pt-BR só na borda — nas duas direções (BR-2112, FR-2119) O usuário digita "157,50"; parseCentavos converte para 15750 centavos antes do POST. Na edição, centavosToInputString reconverte 15750 de volta para "157,50" — sem passar por float em nenhum dos dois sentidos O SPEC-200 já formatava centavos → string pt-BR para exibir. O SPEC-211 adiciona a direção inversa: string digitada pelo usuário → inteiro, antes de qualquer requisição. É a mesma disciplina "dinheiro nunca é float", agora bidirecional.
Exclusão explícita, nunca um clique só (D2, FR-2114/FR-2118) Excluir sempre passa por um ConfirmDialog com o nome/ticker do holding e o aviso "Esta ação não pode ser desfeita" — nunca um botão de exclusão direta É dinheiro real do usuário sendo registrado; um clique acidental não pode apagar uma posição. E, como a seção 5.5 mostra, essa confirmação é reforçada por texto explícito, não por uma cor de alarme — outra trava do produto ficando visível no cliente. 
// web/lib/money.ts — a direção INVERSA da formatação (FR-2119): string digitada -> inteiro, nunca float
export function parseCentavos(input: string): number | null {
  const value = parsePtBrDecimal(input);   // "1.234,56" -> 1234.56 (ainda decimal, mas validado)
  if (value === null) return null;          // malformado/vazio -> null, NUNCA coagido para 0
  return Math.round(value * 100);           // -> 123456 centavos, inteiro
}
Por que "nenhum bloqueio de botão de compra/venda" nem precisa ser mencionado aqui Vale notar o que a Carteira não tem: nenhum botão "Comprar mais" ou "Vender". Ela é puramente um formulário de registro — o usuário conta ao sistema o que já fez, fora do app, na corretora ou no banco. A ausência de qualquer affordance de execução de ordem não é nem uma trava explícita desta spec: é simplesmente consequência de o design system do SPEC-200 nunca ter criado esse componente (seção 4 da Aula 200). A Carteira herda essa ausência de graça, como qualquer outra tela do produto.

5. LIÇÃO PRINCIPAL — cinco bugs reais, pegos de cinco jeitos diferentes

Esta é a aula central do SPEC-211. Diferente do SPEC-210 (cuja lição foi sobre quando uma trava não se aplica), o SPEC-211 ensina algo sobre processo: nenhum destes cinco bugs foi pego só por escrever código com cuidado. Cada um foi pego porque alguma coisa foi de fato executada — uma auditoria, o app rodando num navegador real, um teste ponta a ponta real, ou um revisor lendo a intenção do código, não só sua sintaxe.

5.1 — O bug pego ANTES de qualquer código de feature: tipos desatualizados

O primeiro item do PLAN-211 (D9) não é uma feature — é uma auditoria. Ao planejar o SPEC-211, ficou claro que web/lib/api/schema.ts tinha ficado desatualizado em relação a api/openapi.yaml desde que o SPEC-109 (o indexador, um PR só de backend) foi mesclado: o próprio gate daquele PR não cobria a regeneração do frontend, e npm run check:api estava falhando na main havia dias.

Por que isso é uma boa lição de engenharia, não uma vergonha O bug não foi descoberto por sorte — foi descoberto porque alguém rodou check:api antes de escrever qualquer linha nova, em vez de assumir que o chão sob os pés estava firme. Virou literalmente a Tarefa 1 da Fase 1 do PLAN-211: npm run gen:api e commit do resultado, antes de qualquer hook de dados do SPEC-211 ser escrito — porque nada tipa contra o contrato real do SPEC-109 (indexer_type, effective_annual_rate_bps, MarketIndicatorResponse) enquanto os tipos gerados estiverem desatualizados. A lição: verifique o chão antes de construir em cima dele — um check:api vermelho no início da spec é informação valiosa, não um detalhe a ignorar.

5.2 — O bug pego rodando o app num navegador de verdade: estado de formulário vazando entre sessões

Os modais de adicionar/editar (FiiForm/FixedIncomeForm) ficam sempre montados — só o prop open alterna entre mostrar/esconder o <dialog>. Só que o estado inicial do formulário vem de useState(initial?.ticker ?? "") — e o inicializador de useState só roda uma vez, na primeira montagem. Resultado: trocar de editar um holding para editar outro, ou reabrir "Adicionar" depois de uma tentativa abandonada, deixava os dados da sessão anterior ainda no formulário.

Isso não foi encontrado lendo o código — foi encontrado rodando a Fase 4 ao vivo num navegador de verdade (Playwright contra o servidor de desenvolvimento), quando um seletor ambíguo do script chamou atenção para algo estranho no formulário.

// web/app/(app)/portfolio/fii-table.tsx — o `key` força um remount limpo por alvo
<FiiForm
  // Um `key` por alvo força um remount (e assim um useState novo em folha) em vez de reusar
  // a mesma instância — senão trocar de editar um holding para outro (ou reabrir "add" depois
  // de uma tentativa abandonada) vazaria o estado da sessão anterior, já que este componente
  // fica sempre montado (só `open` alterna).
  key={formTarget === null ? "closed" : formTarget === "add" ? "add" : formTarget.id}
  open={formTarget !== null}
  onClose={() => setFormTarget(null)}
  initial={formTarget === "add" ? null : formTarget}
/>
Por que este bug é invisível a typecheck, lint e a um teste ingênuo O código é perfeitamente válido em TypeScript. useState(initial?.ticker ?? "") compila, não gera nenhum warning de lint, e um teste que só monta o formulário uma vez e verifica o valor inicial passaria tranquilamente. O bug só existe na sequência de interações — abrir, fechar, abrir de novo com um alvo diferente — algo que só aparece rodando o componente de verdade, num DOM de verdade, clicando de verdade. É o padrão React "a key reseta o estado" aplicado a um caso concreto: a correção é de uma linha, mas achar o problema exigiu executar o app, não apenas escrevê-lo.

5.3 — O bug pego escrevendo um teste ponta a ponta de verdade: dois botões idênticos

Cada seção (FII e renda fixa) tinha, no estado vazio, dois botões rotulados exatamente "Adicionar FII" — um no cabeçalho da seção (persistente) e outro dentro do próprio EmptyState. Para o olho humano isso é inofensivo — os dois fazem a mesma coisa, e visualmente um substitui o outro. Mas é uma ambiguidade real de acessibilidade e automação: dois botões, o mesmo nome acessível.

Isso não apareceu escrevendo o componente, nem revisando o código — apareceu escrevendo o teste e2e/portfolio.spec.ts: um localizador Playwright em modo estrito (getByRole) se recusou a adivinhar qual dos dois botões clicar, e quebrou com um erro claro em vez de silenciosamente escolher um.

// web/app/(app)/portfolio/fii-table.tsx — os dois CTAs agora são mutuamente exclusivos POR CONSTRUÇÃO
const isEmpty = !isLoading && !isError && sorted.length === 0;
// ...
{!isEmpty && (
  <Button size="sm" onClick={() => setFormTarget("add")}>Adicionar FII</Button>
)}
// ...
{isEmpty && (
  <EmptyState title="Nenhum FII cadastrado" description="Adicione sua primeira posição em FIIs.">
    <Button size="sm" onClick={() => setFormTarget("add")}>Adicionar FII</Button>
  </EmptyState>
)}
A correção certa: não afrouxar o teste, tornar a UI inequívoca A tentação fácil seria "consertar" o teste — usar .first() no localizador, ou um seletor mais específico que ignorasse a duplicação. Isso teria escondido o problema, não resolvido. A correção real foi tornar os dois CTAs mutuamente exclusivos por construção: o botão do cabeçalho só existe quando isEmpty é falso. Agora não há ambiguidade nem para uma pessoa lendo a tela na diagonal, nem para um script, nem para um leitor de tela. É o mesmo princípio de "trava por construção" que o SPEC-200 aplicou ao InsightCard — só que aqui aplicado a uma questão de UX/acessibilidade, não de governança de IA.

5.4 — O bug pego pela revisão de código: aritmética de float num valor monetário, só de exibição

Ao pré-preencher o formulário de edição com o valor já salvo, o código original fazia (centavos / 100).toFixed(2) — divisão de ponto flutuante sobre um valor monetário, exatamente o padrão que o projeto bane em todo lugar, com a justificativa precisa registrada no CLAUDE.md: arredondamento de float/decimal pode acertar o centavo errado para valores específicos. O frontend-reviewer pegou isso — mesmo sendo "só" um valor de pré-preenchimento de exibição, não um cálculo que vai para o servidor.

// web/lib/money.ts — os novos helpers de pré-preenchimento: inteiro entra, string pt-BR sai, sem float
/** Integer centavos -> "157,50" (sem "R$ "). e.g. centavosToInputString(15750) === "157,50". */
export function centavosToInputString(centavos: number): string {
  const abs = Math.abs(Math.trunc(centavos));
  const reais = Math.trunc(abs / 100);   // divisão INTEIRA, não /100 com toFixed
  const cents = abs % 100;               // resto INTEIRO
  return reais.toLocaleString("pt-BR") + "," + String(cents).padStart(2, "0");
  // (nunca `(centavos / 100).toFixed(2)` — isso seria float arredondando um valor monetário)
}
Por que "é só pré-preenchimento" não é desculpa toFixed(2) sobre um float parece inofensivo porque o resultado costuma parecer certo — a imprecisão do ponto flutuante em JavaScript só aparece para valores específicos, imprevisíveis a olho nu. É exatamente esse comportamento intermitente que faz do float uma escolha ruim para dinheiro em qualquer contexto, inclusive exibição: se um valor der errado uma vez a cada muitas centenas, ainda é um bug de confiança para o usuário que ver seu próprio dinheiro exibido errado. A correção — centavosToInputString/bpsToInputString, construídos com a mesma matemática inteira de formatCentavos/formatBps — fecha a lacuna nos dois sentidos: exibir (SPEC-200) e agora também pré-preencher para edição (SPEC-211), com testes cobrindo o round-trip exato via parseCentavos/parseBps.

5.5 — O bug pego pela revisão de código: uma cor reservada usada como preenchimento de botão

O primeiro rascunho do botão de confirmação de exclusão usava uma nova variante destructive do Button, estilizada com border-loss/50 bg-loss/5 text-loss — usando o token loss (reservado para indicar perda financeira em uma figura numérica) como preenchimento de fundo de um botão, ainda que em baixa opacidade. O frontend-reviewer apontou: o design system Aurora é inequívoco — "None is ever brand voltage or a card fill" e "gain/loss are figure colors, not actions", sem exceção para baixa opacidade.

A correção: não um novo tom de vermelho, e sim reaproveitar o que já existe A resposta não foi inventar uma nova regra de exceção nem um tom de vermelho "mais seguro" — foi remover a variante inteira e usar o tratamento neutro secondary que o Button já tinha, confiando no texto explícito do ConfirmDialog — "Excluir?", "Esta ação não pode ser desfeita" — para comunicar o peso da ação, em vez de uma cor. É uma lição pequena com uma generalização grande: nem toda nova interação precisa de uma nova linguagem visual; muitas vezes o vocabulário que já existe (aqui, texto claro + um passo de confirmação explícito) já carrega o peso certo, e cores semânticas reservadas devem continuar reservadas.

5.6 — O primitivo <dialog> nativo, e o teste que precisou de um polyfill mínimo

Em vez de importar uma biblioteca de modal, o SPEC-211 usa o elemento <dialog> do próprio navegador — a mesma filosofia "primitivo nativo antes de dependência" que o SPEC-210 usou para o slider (D5). O método showModal() dá foco preso, fechar com Escape e bloqueio de fundo de graça, sem nenhuma dependência nova.

// web/components/ui/dialog.tsx — o modal É o <dialog> nativo; showModal()/close() dão foco-trap +
// Escape + bloqueio de fundo sem nenhuma dependência nova (D1, espelha o slider nativo do SPEC-210)
useEffect(() => {
  const dialog = ref.current;
  if (!dialog) return;
  if (open && !dialog.open) dialog.showModal();
  if (!open && dialog.open) dialog.close();
}, [open]);

Um detalhe interessante, não um bug: antes de escrever qualquer teste do Dialog, verificou-se empiricamente — não por suposição — que o jsdom (o navegador falso usado pela suíte de testes rápida) não implementa showModal()/close() de jeito nenhum. Um pequeno polyfill foi adicionado só para que a lógica própria do componente fosse testável (ele chama showModal quando deveria? um clique no fundo chama onClose?) — mas o foco-preso e o Escape-para-fechar de verdade são garantias nativas do navegador real, provadas rodando o app de verdade no Playwright várias vezes ao longo da construção, não fingidas num navegador falso.

7. Conexão com o backend: o contrato como fronteira

Esta seção ensina a costura frontend↔backend desta spec — o que substitui, do lado do cliente, a "ponte hexagonal" que a aula de backend ensinaria. O SPEC-211 não abre nenhum endpoint novo: consome oito endpoints do SPEC-102 e um do SPEC-109.

7.1 — Tipos gerados, nunca DTOs à mão

Como nas telas anteriores, nenhum tipo FIIHoldingRequest/FixedIncomeResponse/MarketIndicatorResponse foi escrito à mão — todos vêm de web/lib/api/schema.ts, gerado de api/openapi.yaml:

// web/lib/portfolio/holdings.ts — os tipos vêm do contrato GERADO, nunca escritos à mão
export type FIIHolding = components["schemas"]["FIIHoldingResponse"];
export type FIIHoldingInput = components["schemas"]["FIIHoldingRequest"];
export type FixedIncomeHolding = components["schemas"]["FixedIncomeResponse"];
export type FixedIncomeInput = components["schemas"]["FixedIncomeRequest"];

E, como no SPEC-210, os enums do indexador e da liquidez também vêm do contrato, com um Record tipado como estrutura de garantia — uma nova opção de indexador quebraria o build, não silenciosamente:

// web/lib/portfolio/labels.ts — o Record é tipado do enum GERADO; toda chave é obrigatória
type FixedIncomeReq = components["schemas"]["FixedIncomeRequest"];

export const INDEXER_LABELS: Record<Indexer, string> = {

7.2 — O guardião de deriva, agora exercitado a sério pela primeira vez

A seção 5.1 já contou a história: check:api estava vermelho na main quando o SPEC-211 começou, por causa da deriva deixada pelo SPEC-109. Os dois subagentes de revisão — e o veredito real: CHANGES REQUESTED

Diferente do SPEC-210, onde os dois revisores retornaram PASS de primeira, o SPEC-211 teve um veredito mais interessante de estudar: um PASS limpo e um CHANGES REQUESTED corrigido.

react-correctness-reviewer — PASS (com duas notas não-bloqueantes, ambas corrigidas mesmo assim) Rodou typecheck/lint/build e leu o código de perto. Duas observações, endereçadas por serem baratas de corrigir: (1) todayISO()'s segurança em SSR era implícita, dependendo de um curto-circuito de estado padrão — agora é computada uma vez via useState(() => todayISO()), explícito e imune a uma virada de meia-noite na mesma sessão. (2) Os diálogos não bloqueiam Escape/fechar-pelo-fundo enquanto uma mutação está pendente — confirmado seguro (o TanStack Query desanexa listeners no unmount/reset(), verificado lendo o código-fonte instalado de @tanstack/query-core) e deixado como UX intencional: bloquear o fechamento durante uma requisição lenta/travada seria pior, não melhor.
frontend-reviewer — CHANGES REQUESTED → corrigido → reverificado limpo Este é o caso mais didático desta aula sobre revisão de código: o revisor não deu um PASS educado com notas de rodapé — ele bloqueou a mudança até duas questões reais serem corrigidas. São exatamente os bugs 5.4 (float no pré-preenchimento) e 5.5 (cor reservada como preenchimento de botão) descritos acima. Depois das correções, o gate completo (typecheck/lint/test/check:api/build) e o E2E foram re-executados do zero e vieram limpos — não foi um "conserta e assume que está certo", foi conserta e reverifica.
Por que um "CHANGES REQUESTED" corrigido é uma vitória do harness, não uma falha dele É tentador achar que um PASS de primeira é o resultado desejável e um CHANGES REQUESTED é um problema. O oposto é o ponto: o frontend-reviewer pegou dois bugs reais — um de correção monetária, um de violação do design system — antes de qualquer um deles chegar à main. Isso é o harness fazendo exatamente o que deveria: baixar a taxa de erro antes do merge, não depois. Um PASS de primeira só seria melhor se o código já estivesse certo; aqui, o processo de revisão foi o que tornou o código certo.

10. Conexão com AI Engineering & Produto

A Carteira parece "só um formulário", mas é onde a promessa central do produto — facts are computed, not generated — deixa de ser uma frase no CLAUDE.md e vira dados reais no banco.

10.1 — O primeiro fato de verdade que o Fact Builder vai ler

SPEC-104 (motor de insights), SPEC-105 (rebalanceamento), SPEC-106 (Health Score) e SPEC-107 (projeções) já sabem como ler holdings — mas até o SPEC-211, nenhum usuário tinha uma tela web para escrever esses holdings de verdade. Agora existe: cada FII e cada posição de renda fixa cadastrados aqui são literalmente os fatos de entrada que o Fact Builder do backend vai considerar antes de qualquer Insighter gerar uma explicação. Sem esta tela, toda a Fase 1 de IA do backend estaria calculando sobre um portfólio vazio.

10.2 — Por que a ausência das travas de IA aqui prepara o SPEC-213 (de novo)

O mesmo par didático do SPEC-210 se repete: a Carteira captura fatos sem as travas FR-013/FR-014 (BR-2115) — não porque a governança de IA seja opcional, mas porque não há saída de IA aqui para gatear. Quando o SPEC-213 (Insights/Dashboard) ler estes holdings e gerar uma sugestão de rebalanceamento, é lá que o InsightCard com explanation obrigatório e o NonAdviceDisclaimer reaparecem em força total — agora tendo dados reais de Rafael e Carla para explicar.

10.3 — A taxa efetiva resolvida no servidor: uma pequena prova da disciplina "facts are computed"

A exibição da taxa efetiva (FR-2120 — "120% do CDI · ≈12,60% a.a.") é um exemplo em miniatura da regra binding #3 sendo aplicada literalmente: o cliente nunca calcula essa matemática. Ele lê effective_annual_rate_bps, já resolvido pelo SPEC-109 no backend (BR-2117). Se o produto algum dia ganhar um app nativo ou um servidor MCP, eles vão ler o mesmo número resolvido — nenhuma lógica de indexador duplicada em nenhum cliente.

10.4 — A costura para a visão multi-agente / MCP

Como nas telas anteriores, a Carteira reforça a mesma propriedade: o frontend não sabe o que "120% do CDI" significa em termos de matemática — ele só renderiza o que o backend já resolveu, e submete o que o usuário digitou sem reinterpretar. Isso significa que um futuro CIO multi-agente ou servidor MCP pode ler e escrever exatamente os mesmos oito endpoints de holdings que esta tela usa, sem nenhuma lógica de domínio duplicada esperando para ser migrada.

A ideia que conecta tudo Engenharia de IA responsável não termina no Insighter gated do backend — ela também depende de os fatos de entrada serem registrados com a mesma disciplina: dinheiro sempre inteiro (nas duas direções), taxas sempre resolvidas no servidor, nunca recalculadas no cliente. E a lição desta aula generaliza além de IA: a taxa de erro de um sistema não cai só porque alguém escreveu código com cuidado — ela cai quando alguém executa esse código de verdade, de vários jeitos diferentes, e trata cada discrepância encontrada como informação, não como um incômodo a contornar.
