1. A visão geral: por que um port de dados de mercado

O SPEC-005 entregou a costura por onde a IA fala (o port Insighter). Mas um LLM não pode raciocinar no vácuo: a regra binding nº 3 do PRD diz que "fatos são computados, não gerados" — o modelo raciocina sobre números determinísticos, nunca os inventa. Faltava, então, a fonte desses números: o preço atual de cada FII, seu dividend yield, P/VP, setor e último dividendo (FR-006), mais os indicadores macro brasileiros — SELIC, CDI, IPCA, IFIX (FR-007).

O SPEC-006 é a segunda pedra de provedor externo. Como o Insighter, ele não entrega nenhuma tela nem feature que o usuário veja. Entrega a infraestrutura de dados: o port MarketDataProvider, os adaptadores que falam com as fontes grátis, as tabelas, os repositórios, e um worker de ingestão que periodicamente busca e guarda esses dados como referência último-dado-bom. Nada consome ainda — o dashboard (SPEC-103), o Fact Builder (SPEC-104) e as projeções (SPEC-107) leem isto depois, pelos ports de repositório.

ANTES (SPEC-005) DEPOIS (SPEC-006) IA tem onde falar (Insighter) IA tem o que ler: dados de FII + macro determinísticos nenhum dado de mercado port MarketDataProvider — a 2ª costura de provedor externo fonte de dados = decisão futura troca por config: MARKETDATA_PROVIDER=fake|live nada busca preço/DY/P-VP/setor worker de ingestão (off do request path, via Clock) nada protege o "último dado bom" upsert idempotente: fetch ruim NUNCA sobrescreve dado bom sem tabelas de mercado migration 0003: fii_quotes (snapshot) + macro_indicators (série)

Concretamente, o SPEC-006 entregou o pacote internal/marketdata com: o core puro (value objects Ticker/Sector/Indicator/Unit, as entidades FIIQuote/MacroIndicator, os ports e os sentinelas); os adaptadores de borda (fundamentus/, yahoo/, bcb/, mais o fii/ que compõe os dois primeiros); o repositório postgres/; o subpacote ingest/ (worker + scheduler + factory); a migration 0003_market_data; o comando cmd/ingest; e a configuração MARKETDATA_*.

Por que esconder a fonte de dados atrás de um port, igual ao LLM? Porque o maior risco do produto (PRD §14) é "não existe fonte de dados de FII confiável e gratuita". Se o código chamasse uma API específica diretamente, espalhada por todo lado, trocar de fonte seria uma reescrita. Colocando a ingestão atrás do MarketDataProvider — exatamente como o Insighter esconde o LLM — a fonte vira detalhe de configuração. Hoje raspamos uma tabela HTML grátis; amanhã, se ela quebrar, basta plugar uma fonte licenciada (brapi Pro) sem tocar no domínio. O port é o seguro contra o risco nº 1.

2. O problema: dados de mercado + custo zero + fonte trocável

Três tensões, resolvidas de uma vez — e elas rimam com as do SPEC-005, de propósito:

TensãoRestrição binding (PRD)Solução do SPEC-006
Precisamos de preço/DY/P-VP/setor + macro, mas sem gastar dinheiro Custo zero (ADR-0003): só fontes free/públicas Fundamentus (HTML grátis, sem chave) + Yahoo (.SA, grátis) para FIIs; BCB SGS (público, sem chave) para macro. Default é o Fake — zero rede no CI e no app sem config.
Hoje Fundamentus; amanhã pode ser brapi Pro / outra Provedor plugável (FR-018) Tudo atrás do port MarketDataProvider. Trocar de fonte é MARKETDATA_PROVIDER no .env, nunca mudança de código (BR-601).
Os dados são a fonte de verdade sobre a qual a IA raciocina Fatos computados/ingeridos, não gerados (BR-605) Os números são ingeridos e guardados de forma determinística; o LLM nunca os inventa (sustenta o SPEC-005 BR-502 a montante). Mesmos inputs → mesmo Health Score.

A peça-chave, de novo, é o port — uma interface no core da feature, sem importar nenhum HTTP nem SQL. Repare que ele tem dois métodos e que a leitura de FII é em lote:

// internal/marketdata/ports.go
// MarketDataProvider reads current data from an external source — the FR-018 seam that keeps
// the provider swappable by config (BR-601). FII reads are batched because the MVP source
// (Fundamentus) returns every FII in one request; a ticker absent from the returned map is
// a per-item miss the worker treats as "keep last-known-good" (BR-602).
type MarketDataProvider interface {
    FetchFIIQuotes(ctx context.Context, tickers []Ticker) (map[Ticker]FIIQuote, error)
    FetchMacroIndicator(ctx context.Context, ind Indicator) (MacroIndicator, error)
}
Por que a leitura de FII é em lote ([]Ticker → map) e a de macro é uma por vez? A forma do método segue a forma da fonte. O Fundamentus devolve a tabela inteira de ~1500 FIIs em uma única requisição — então pedir ticker por ticker seria desperdício (e abuso da fonte raspada). Já o BCB SGS é uma série por código; cada indicador é uma chamada separada. O port espelha o que é eficiente em cada caso. Um ticker ausente do mapa não é um erro: é um "miss" que o worker trata mantendo o último dado bom.

3. A história da decisão D4: compor fontes grátis

Esta é a decisão mais cara e mais honesta da spec. Não existe uma única API oficial e gratuita com todos os campos de FII que precisamos. A brapi (a candidata óbvia) moveu P/VP e segmento para o plano pago — o tier grátis dela não os expõe mais. O Yahoo, sozinho, dá preço e dividendos, mas não dá P/VP nem setor. A única forma de obter tudo a custo zero é compor:

FonteO que ela dáComo falamos com ela
Fundamentus (FII)Preço, dividend yield, P/VP, segmento — para todos os FIIs, em 1 requisiçãoRaspagem de uma tabela HTML (não há API oficial)
Yahoo Finance (FII)Último dividendo (valor + data) por tickerAPI de chart JSON, sufixo .SA, melhor-esforço
BCB SGS (macro)SELIC, CDI, IPCA por código de sérieAPI JSON pública, sem chave
O que significam .php e .SA? (perguntas reais de quem está aprendendo)
  • .php em fii_resultado.php é só o endpoint do servidor do Fundamentus — o nome do script que, ao ser chamado por GET, devolve a página HTML com a tabela de FIIs. Não é um formato de dado; é o "endereço" da página que raspamos.
  • .SA em HGLG11.SA é o sufixo de bolsa do Yahoo: ele cataloga tickers do mundo todo, então .SA (de São Paulo / B3) diz "este papel é da bolsa brasileira". Sem o sufixo, o Yahoo não saberia em qual bolsa procurar HGLG11.

A composição acontece em duas camadas, e ambas usam o padrão de combinar fontes. Primeiro, o subpacote fii/ junta Fundamentus + Yahoo num só FetchFIIQuotes; o último dividendo é melhor-esforço — se o Yahoo falhar, fica um cotação parcial, nunca uma linha falha:

// internal/marketdata/fii/composite.go
func (c Composite) FetchFIIQuotes(ctx context.Context, tickers []marketdata.Ticker) (map[...]..., error) {
    quotes, err := c.fundamentals.FetchFIIQuotes(ctx, tickers)  // dependência DURA
    if err != nil {
        return nil, err                                          // falha → degrada o lote (BR-602)
    }
    for t, q := range quotes {
        cents, date, derr := c.dividends.FetchLastDividend(ctx, t)  // MELHOR-ESFORÇO
        if derr == nil && cents > 0 {
            q.LastDividendCentavos = cents
            q.LastDividendDate = date
            quotes[t] = q
        }
        // Yahoo falhou? Deixa last-dividend vazio — cotação parcial, não linha falha.
    }
    return quotes, nil
}

Depois, o subpacote ingest/ combina o lado FII (composto) com o lado macro (BCB) para satisfazer o port inteiro — um segundo nível de composição:

// internal/marketdata/ingest/factory.go
if cfg.MarketDataProvider == "live" {
    fiiSource := fii.New(
        fundamentus.New(cfg.MarketDataFundamentusBaseURL, cfg.MarketDataTimeout, clk),
        yahoo.New(cfg.MarketDataYahooBaseURL, cfg.MarketDataTimeout),
    )
    macroSource := bcb.New(cfg.MarketDataBCBBaseURL, cfg.MarketDataTimeout, clk)
    return combined{fii: fiiSource, macro: macroSource}   // FII + macro → 1 MarketDataProvider
}
return marketdata.Fake{At: clk.Now()}                      // default: zero rede
O trade-off honesto: grátis-mas-raspado vs. pago-mas-limpo Raspar uma tabela HTML é frágil: se o Fundamentus mudar a marcação, o parser quebra. Pagar a brapi Pro (~R$116/mês) daria dados limpos com API estável — mas viola o custo zero. A escolha do MVP é grátis e raspado, com a fragilidade mitigada por quatro travas: (1) o port torna a troca para uma fonte licenciada um detalhe de config; (2) o Fake é o default, então dev/CI nunca dependem da fonte real; (3) testes de parser baseados em fixtures; e (4) degradação graciosa — uma falha de parse mantém o último dado bom, nunca escreve lixo. Quando/se a raspagem provar fraca demais, graduar para a fonte licenciada é uma troca de config, rastreada como dívida técnica (§15).

E a robustez não é só no contorno — é dentro do parser. O Fundamentus localiza as colunas por palavra-chave no cabeçalho, não por posição fixa, então um reordenamento de colunas não quebra nada; e um layout irreconhecível vira ErrProviderUnavailable (degrada), não uma escrita corrompida:

// internal/marketdata/fundamentus/fundamentus.go
rows, err := parseTableRows(io.LimitReader(resp.Body, maxBytes))  // body cap (hardening do SPEC-005)
if err != nil || len(rows) < 2 {
    return nil, fmt.Errorf("%w: fundamentus parse: empty or unreadable table", marketdata.ErrProviderUnavailable)
}
cols := indexColumns(rows[0])                 // acha "papel"/"cota"/"dividend yield"/"p/vp"/"segmento"
if cols.ticker < 0 || cols.price < 0 {        // colunas essenciais sumiram = layout mudou
    return nil, fmt.Errorf("%w: fundamentus parse: expected columns not found (layout changed?)", marketdata.ErrProviderUnavailable)
}

4. Disciplina de dinheiro: int64 centavos + basis points (BR-604)

A convenção mais inegociável da casa: dinheiro nunca é float64. Preço e dividendos são int64 em centavos (unidades menores); dividend yield, SELIC, CDI, IPCA e P/VP são inteiros em basis points (1 bp = 0,01%). A migration não tem uma coluna de ponto flutuante:

-- migrations/0003_market_data.up.sql
CREATE TABLE fii_quotes (
    ticker                 text        PRIMARY KEY,
    price_centavos         bigint      NOT NULL,   -- R$ em centavos
    dividend_yield_bps     integer     NOT NULL,   -- yield em basis points
    p_vp_bps               integer     NOT NULL,   -- razão P/VP ×10000 (0.95 -> 9500)
    sector                 text        NOT NULL,
    last_dividend_centavos bigint      NOT NULL DEFAULT 0,
    ...
);
Por que tanto rigor? Porque float não é determinístico 0.1 + 0.2 != 0.3 em ponto flutuante binário — o erro de arredondamento se acumula. Mas o PRD exige "mesmos inputs → mesmo Health Score": o cálculo tem que ser reproduzível bit a bit. Com inteiros, 15,75 vira exatamente 1575 centavos, sempre. Toda a conversão de string decimal → inteiro mora num lugar só, com uma regra documentada de arredondamento (half-up), para que cada adaptador produza o mesmo número a partir da mesma string da fonte: 
// internal/platform/money/money.go
//	DecimalToMinor("15,75", 2) -> 1575   // R$15,75 -> centavos
//	DecimalToMinor("8,50", 2)  -> 850    // 8.50%   -> basis points
//	DecimalToMinor("0,95", 4)  -> 9500   // razão P/VP -> ratio basis points
func DecimalToMinor(s string, scale int) (int64, error) { ... }

Note que ele aceita as duas notações: a brasileira ("1.234,56" — ponto como separador de milhar, vírgula decimal) e a simples ("0.11") — porque o Fundamentus serve PT-BR e o Yahoo serve en-US. Centralizar isso é o que permite os dois adaptadores compartilharem a mesma regra de arredondamento sem duplicar lógica frágil.

Dois bugs de correção pegos em revisão (e por que o helper é centralizado e testado a fundo) Justamente porque todo número monetário do projeto passa por DecimalToMinor, um bug ali contaminaria preço, yield, P/VP e todos os indicadores macro de uma vez. A revisão pegou dois:
  • Overflow de int64 no carry do arredondamento. Quando o half-up precisa somar 1 e o valor já é math.MaxInt64, o val++ daria a volta para negativo silenciosamente. A correção: detectar o caso e devolver um erro de overflow. 
    if roundUp {
    if val == math.MaxInt64 { // o carry estouraria o int64
        return 0, fmt.Errorf("money: %q: overflow: %w", s, ErrInvalidDecimal)
    }
    val++
}
  • Sinal solitário parseando para 0. Uma string como "-" ou "+" (sinal sem dígitos) passava e virava 0 em vez de erro. A correção rejeita explicitamente quando não há nenhum dígito: 
    if intPart == "" && fracPart == "" {
    // Nenhum dígito — ex.: "", um sinal solitário, ou ".".
    return 0, fmt.Errorf("money: %q: %w", s, ErrInvalidDecimal)
}
A lição: um helper compartilhado e crítico merece testes table-driven rigorosos de borda — é o ponto onde um bug pequeno tem alcance gigante.

5. Integridade do "último dado bom" (BR-602)

Esta é a alma de confiabilidade da spec. Um dashboard só vale o quanto valem os dados por trás dele. A regra: um fetch que falhou, veio vazio ou veio malformado NUNCA pode sobrescrever uma linha boa. Melhor servir um dado de ontem (e marcá-lo como velho) do que apagar o bom e mostrar nada — ou pior, mostrar lixo. Três mecanismos garantem isso.

5.1 — Upsert idempotente via ON CONFLICT

A escrita é um único INSERT ... ON CONFLICT ... DO UPDATE — atômico e idempotente. Rodar de novo com o mesmo dado dá a mesma linha; rodar com agendamentos sobrepostos não duplica nem corrompe:

// internal/marketdata/postgres/postgres.go
INSERT INTO fii_quotes (ticker, price_centavos, ...) VALUES ($1, $2, ...)
ON CONFLICT (ticker) DO UPDATE SET
    price_centavos = EXCLUDED.price_centavos,
    ...
    fetched_at     = EXCLUDED.fetched_at

O macro usa a mesma ideia, mas a chave é (indicator, reference_date) — porque macro é uma série temporal (D5): re-buscar a mesma data é um no-op, e datas novas viram linhas novas. O GetLatest só lê a reference_date mais recente por indicador.

5.2 — A chave é: o fetch ruim nunca chega ao banco

O segredo do "último dado bom" não está no SQL — está antes dele. O worker só chama Upsert quando o fetch deu certo. Se a fonte caiu, ele nem tenta escrever — então a linha boa anterior fica intacta por construção:

// internal/marketdata/ingest/worker.go — isolamento por item
quotes, err := w.provider.FetchFIIQuotes(fctx, tickers)
if err != nil {
    // Degrada: mantém o último dado bom de TODOS os tickers (BR-602).
    w.logger.Warn("market data: FII provider unavailable; keeping last-known-good", ...)
    w.recordItem(ctx, "fii", "provider_error")
    sum.FIIFailed += len(tickers)
    return                                  // NÃO escreve nada
}
for t, q := range quotes {
    if err := w.fiiRepo.UpsertFIIQuote(ctx, q); err != nil { ... continue }  // 1 ruim não aborta o resto
}

5.3 — Isolamento de falha por item

Um ticker ruim, um indicador macro indisponível, um erro de gravação — nada disso aborta a rodada. O RunOnce nunca retorna erro: ele isola, registra (log + métrica) e segue. O resultado é uma Summary com contagens de OK e falha, classificada em success / partial / error para a telemetria.

Dado global, SEM user_id (BR-603) — o contraste com o SPEC-003 Na autenticação (SPEC-003), tudo era escopado por usuário: WHERE user_id = $1, com o ID vindo do contexto, nunca do payload. Aqui é o oposto deliberado: o preço do HGLG11 é o mesmo para todo mundo. Dados de mercado são referência global — não há coluna user_id em lugar nenhum desta migration. A regra "identidade vem do contexto" não se aplica a dado compartilhado. Entender quando escopar por usuário e quando não é parte do design: o isolamento por usuário protege dado privado; dado de mercado é público.

6. O worker de ingestão & o agendamento

"Ingestion" e "worker" em português claro (perguntas reais de quem aprende)
  • Ingestão (ingestion) = o ato de puxar dados de uma fonte externa para dentro do nosso armazenamento. Não transformamos nem servimos nada ainda; só buscamos lá fora e guardamos aqui. É o "I" de um pipeline de dados.
  • Worker = um processo de fundo que faz trabalho sem ser disparado por uma requisição HTTP de usuário. Enquanto um handler roda "porque alguém chamou a API", o worker roda "porque chegou a hora" (um relógio). É trabalho desacoplado do request path.

Por que desacoplar da requisição? Porque buscar a tabela inteira do Fundamentus e quatro séries do BCB leva segundos e pode falhar — você não quer que um usuário pedindo o dashboard espere por isso, nem que o request dele dispare uma raspagem. A ingestão roda em fundo, na sua própria cadência, e o dashboard só lê o que já está guardado. Há dois modos de disparar (D2):

ModoO que éQuando usar
cmd/ingest (one-shot) Um binário que faz uma rodada e sai. Sai com código ≠ 0 só em erro fatal (config/DB), nunca por falha de item. O primário — chamado pelo cron do host. Sem risco de ingestão duplicada quando há várias réplicas da API.
Scheduler in-process Uma goroutine dentro do cmd/api que roda no intervalo configurado, via Clock; ligada por flag (MARKETDATA_SCHEDULER_ENABLED). Conveniência do MVP single-instance. Desligada em multi-réplica (senão N réplicas ingerem em paralelo).

O scheduler roda uma vez na partida (dado fresco no boot), depois a cada intervalo, e para limpo quando o contexto é cancelado — drenando a rodada em voo antes de o banco fechar no shutdown:

// internal/marketdata/ingest/scheduler.go
func (s *Scheduler) Run(ctx context.Context) {
    s.worker.RunOnce(ctx)                      // roda já, na partida
    ticker := time.NewTicker(s.interval)
    defer ticker.Stop()
    for {
        select {
        case <-ctx.Done():                      // shutdown: para limpo
            s.logger.Info("market data scheduler stopped")
            return
        case <-ticker.C:
            s.worker.RunOnce(ctx)
        }
    }
}
Cadência (wall time) vs. dado (Clock) — uma distinção fina O time.Ticker usa o relógio de parede para a cadência (de quanto em quanto tempo disparar) — isso é correto, é tempo real. Mas todo dado gravado (observed_at, fetched_at, o cálculo de staleness) usa o Clock injetado, nunca time.Now() — para que os testes sejam determinísticos. "Quando disparar" é wall time; "que carimbo o dado leva" é o Clock.

Observabilidade: só metadados (FR-609)

Cada rodada abre um span marketdata.ingest, com um span-filho por chamada de provedor (marketdata.fetch_fii / marketdata.fetch_macro). Os atributos são provider, kind, outcome, contagens — nunca um token de API (as fontes do MVP nem usam chave) e nunca payload. Há contadores (ingestion_runs, ingestion_items) e um gauge de frescor — segundos desde a última rodada — para alertar se a ingestão travar:

// internal/marketdata/ingest/worker.go
if gauge, err := meter.Int64ObservableGauge("marketdata.seconds_since_last_run"); err == nil {
    _, _ = meter.RegisterCallback(func(_ context.Context, o metric.Observer) error {
        ...
        o.ObserveInt64(gauge, int64(w.clock.Now().Sub(last)/time.Second))  // só uma duração, zero PII
        return nil
    }, gauge)
}

7. As 6 decisões (D1–D6) e por quê

#DecisãoEscolhaPor quê
D1Um port ou dois? Um port conceitual (FII + macro), FII em lote Casa com o nome do PRD (MarketDataProvider); adaptadores ainda diferem por fonte internamente. A leitura de FII é batched porque a fonte devolve tudo de uma vez.
D2Modelo de agendamento? cmd/ingest one-shot (cron) primário + scheduler in-process opcional (flag) O cron evita ingestão duplicada quando a API escala horizontalmente; o scheduler é conveniência do MVP single-instance, desligável.
D3De onde vêm os tickers, antes do SPEC-102? Watchlist configurada (MARKETDATA_WATCHLIST) via TickerSource Um port TickerSource com um adaptador estático agora; quando o SPEC-102 (holdings) chegar, troca-se por uma fonte baseada na carteira — sem mexer no worker.
D4Fonte de FII + macro? Composição: Fundamentus (bulk) + Yahoo .SA (último dividendo); BCB SGS (macro) Não há API grátis única (brapi tornou P/VP e segmento pagos). Compor fontes grátis atrás do port, trocável depois por uma licenciada — sem reescrita.
D5Forma do armazenamento macro? Série temporal (indicator, reference_date); FII fica snapshot A fonte já devolve uma série; útil para projeções/gráficos futuros; GetLatest expõe o mais recente. FII guarda só o snapshot atual (histórico fica para o SPEC-107).
D6Como representar P/VP e razões? Basis points da razão (×10000): 0,95 → 9500 Consistente com a convenção de bps; sem float. Uma razão é só um número adimensional que também merece a disciplina inteira.

8. As 7 fases, uma a uma

Bottom-up, como nas specs anteriores — core puro primeiro, borda depois, cada fase compilando e revisável sozinha. Espelha a cadência do SPEC-005.

Fase 1 — Domínio, ports & config

Value objects com construtor (parse-don't-validate): Ticker (4 letras + 1–2 dígitos), Sector (enum fechado; rótulo desconhecido → Other, acento-insensível), Indicator, Unit. Entidades FIIQuote/MacroIndicator (money int64, rates bps, tempo UTC). Os ports (MarketDataProvider, FIIQuoteRepository, MacroRepository, TickerSource) e os sentinelas (ErrFIIQuoteNotFound, ErrMacroNotFound, ErrProviderUnavailable). Config MARKETDATA_* com validação http(s)/positivos e máscara de token futuro.

Fase 2 — Persistência (migration + repositórios)

A migration 0003_market_data up/down (bigint centavos, int bps, sem float, sem user_id) e o pacote postgres/: upserts idempotentes ON CONFLICT, transacionais, reads que mapeiam sql.ErrNoRows → o sentinela …NotFound, com asserções var _ marketdata.FIIQuoteRepository = … em tempo de compilação.

Fase 3 — Adaptador de FII: composição Fundamentus + Yahoo (+ Fake)

Captura de fixtures reais (a página .php do Fundamentus, o JSON .SA do Yahoo); o fundamentus/ (1 GET → parse de tabela HTML com x/net/html → mapa, User-Agent descritivo, body cap, layout-mudou → degrada); o yahoo/ (último dividendo, melhor-esforço); o fii/ que compõe os dois; e o Fake determinístico, o default.

Fase 4 — Adaptador macro (BCB SGS + IFIX)

O bcb/: busca SELIC/CDI/IPCA por código de série (consts documentados: 432, 4389, 13522), parseando para bps + reference_date. O IFIX não é série do BCB (é índice da B3) — o adaptador devolve ErrProviderUnavailable para ele e o worker degrada; uma fonte grátis de IFIX é um follow-up rastreado (§15). Mesmo contrato de degradação (timeout, body cap, falha → sentinela).

Fase 5 — Worker, scheduler, observabilidade & cmd/ingest

O Worker.RunOnce (resolve a watchlist + o set macro fixo; fetch→valida→upsert por item; isola falhas; devolve Summary); o scheduler in-process (Clock, flag-gated, drenado no shutdown); o cmd/ingest one-shot; a observabilidade (span por rodada + span-filho por chamada, contadores, gauge de frescor, só metadados); o alvo task ingest.

Fase 6 — Testes

Unitários sem rede nem DB: value objects e mapeamento money/bps; ambos adaptadores via httptest (sucesso, payload parcial, 404, 429→degrada, malformado, body cap); o worker com fakes (idempotência, último-dado-bom em falha, sucesso parcial, frescor/staleness com Clock falso). Integração gated: Postgres real (TEST_DATABASE_URL, -p 1) para a idempotência do upsert + round-trip up/down da 0003; um fetch real BCB/FII opcional atrás de flag de env (pula limpo no CI, como o teste do Ollama vivo).

9. Ponte: arquitetura em camadas → hexagonal

Você já conhece a pilha clássica em camadas: controller → DTO → service → repository, com o service importando o repositório concreto. A melhor forma de internalizar os ports & adapters deste projeto é por tradução. (Para a teoria completa, veja a aula conceitual de arquitetura hexagonal; aqui aplicamos ela ao SPEC-006.)

A tabela de-para (com os tipos reais desta spec)

A "grande virada": a inversão de dependência

Em camadas, o service importa o repositório concreto — a seta de dependência aponta para fora, do núcleo rumo à infra. No hexagonal, isso vira: o core define a interface e o adapter a implementa — a seta aponta para dentro. Veja com o par real desta spec:

// O CORE define o port (internal/marketdata/ports.go):
type FIIQuoteRepository interface {
    UpsertFIIQuote(ctx context.Context, q FIIQuote) error
    GetFIIQuoteByTicker(ctx context.Context, t Ticker) (FIIQuote, error)
}

// A BORDA o implementa e PROVA isso em tempo de compilação
// (internal/marketdata/postgres/postgres.go):
var _ marketdata.FIIQuoteRepository = FIIQuoteRepository{}   // o adapter depende do core, não o contrário

O postgres importa marketdata; marketdata jamais importa database/sql nem net/http. É por isso que o domínio é testável com fakes triviais e que trocar Fundamentus por brapi Pro, ou Postgres por outro banco, não toca uma linha do core.

Driving vs. driven nesta spec

  • Driving ports (quem aciona a aplicação): cmd/ingest e o Scheduler chamam Worker.RunOnce. Eles ficam à esquerda do hexágono.
  • Driven ports (o que a aplicação aciona): MarketDataProvider (fontes externas), FIIQuoteRepository/MacroRepository (banco), TickerSource (watchlist). Ficam à direita.
  • Domínio puro: Ticker, Sector, FIIQuote, MacroIndicator, os sentinelas. Borda: tudo que tem HTTP, SQL ou OTel — os adaptadores e o pacote ingest.
┌──────────────── HEXÁGONO (core puro) ───────────────┐ cmd/ingest ──▶ │ Worker.RunOnce (caso de uso) │ Scheduler ──▶ │ │ depende de PORTS (interfaces), não concretos│ (driving) │ ├── MarketDataProvider ◀── fundamentus/yahoo/bcb (driven) │ ├── FIIQuoteRepository ◀── postgres/ (driven) │ ├── MacroRepository ◀── postgres/ (driven) │ └── TickerSource ◀── Watchlist (driven) │ domínio: Ticker, FIIQuote, MacroIndicator (sem SQL/HTTP/OTel) │ └──────────────────────────────────────────────────────┘ A seta de dependência sempre aponta PARA DENTRO: os adaptadores conhecem o core; o core não os conhece.

10. Harness Engineering: como a máquina construiu isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. Veja, no SPEC-006, como o harness construiu e verificou.

O fluxo /spec-implement, fase a fase

A IA não atacou "construa a ingestão" de uma vez. O PLAN-006 fatiou em 7 fases bottom-up, e cada uma fechava com go build + testes verdes antes da próxima. Construir o core puro primeiro (Fase 1) e só então a persistência (2) e os adaptadores de borda (3–4) significa que, quando o worker foi montado (Fase 5), todas as peças que ele compõe já existiam e eram testáveis isoladamente.

O subagente hexagonal-reviewer

O revisor especializado cobrou as regras mais sensíveis daqui: a direção de dependência (BR-601 — o core marketdata não importa net/http nem database/sql; só os subpacotes de borda o fazem), o OTel só na borda (o pacote ingest importa OpenTelemetry; os adaptadores e o core não), e o no-user_id (BR-603 — "este é dado global; não escope por usuário"). O doc-comment do pacote ingest declara essa fronteira explicitamente: "Observability (OTel) lives here, at the edge — the marketdata core stays free of OTel."

A prova de que o core ficou puro É mecânica: internal/marketdata (core) e os adaptadores fundamentus//yahoo// bcb//postgres/ não têm um único import de OpenTelemetry — todo o OTel mora em internal/marketdata/ingest. E as asserções var _ marketdata.FIIQuoteRepository = … em postgres.go fazem o compilador provar que o adapter satisfaz o port. Se a IA tivesse metido um span dentro do bcb, ou esquecido um método do port, o build (ou o revisor) barraria.

O go-correctness-reviewer e os dois bugs de dinheiro

Foi a lente de correção que pegou o overflow do int64 no carry e o sinal solitário parseando para 0 em money.DecimalToMinor (seção 4). Não eram bugs de arquitetura — eram bugs de borda numérica, exatamente o tipo que escapa numa leitura casual e que um revisor focado em correção (e testes table-driven de borda) flagra. Como o helper é centralizado, corrigir num lugar conserta todos os adaptadores.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • Bloqueio de imutabilidade: o SPEC-006 referencia ADR-0002/0003 mas não os edita — se a IA tentasse "atualizar" um ADR aceito, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/ gofmt limpos — e os testes têm que passar sem rede e sem DB. Foi esse backstop que provou, na prática, que o Fake default cobre o CI e que toda a feature roda offline.

11. Conexão com AI Engineering

O SPEC-006 não é uma feature de IA — é fundação de dados. Mas é a fundação que torna a IA honesta. Vale conectar este alicerce à arquitetura de IA futura.

Conceito de AI EngineeringComo o SPEC-006 o prepara
Grounding / fonte de verdade A regra de ouro da IA confiável é "o modelo raciocina sobre fatos, não os inventa". O SPEC-006 produz esses fatos: preço, DY, P/VP, setor, SELIC/CDI/IPCA — ingeridos de forma determinística, nunca gerados (BR-605). É o solo onde o grounding pisa.
Fact Builder (SPEC-104) O Fact Builder vai ler estes repositórios (GetFIIQuoteByTicker, GetLatestMacroIndicator), juntar com a carteira do usuário, e montar o Facts opaco que o Insighter recebe. O SPEC-006 é o "F" de onde os fatos vêm.
Determinismo / reprodutibilidade Money em int64 e rates em bps garantem que "mesmos inputs → mesmo Health Score" (PRD §6). Sem isso, o mesmo prompt produziria scores ligeiramente diferentes a cada rodada — e a explicabilidade (FR-013) perderia o chão.
Frescor & confiança do dado observed_at/fetched_at + o gauge de staleness deixam a IA (e o usuário) saberem quão velho é o fato. Um insight sobre dado de ontem pode ser anotado como tal — em vez de fingir precisão que não tem.
O seam multi-provedor (igual ao Insighter) MarketDataProvider é a segunda costura de provedor externo, espelhando o Insighter. Esse padrão — todo recurso externo atrás de um port config-trocável — é exatamente o que o futuro MCP precisa: cada fonte de dados vira uma "ferramenta" plugável.
A ideia que conecta tudo IA confiável começa antes do modelo: com dados determinísticos e auditáveis. O SPEC-005 instalou a costura por onde a IA fala e os portões que a verificam; o SPEC-006 instala a fonte de verdade sobre a qual ela raciocina — ingerida, nunca inventada, com último-dado-bom protegido, dinheiro em inteiros, e tudo atrás de um port trocável. Sem este chão, o grounding seria ficção.
