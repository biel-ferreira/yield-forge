1. A visão geral: por que a carteira é a fundação de tudo

O SPEC-101 entregou a primeira feature de usuário (o perfil — quem é o investidor). O SPEC-102 entrega a segunda metade do par: o que o investidor tem. Ele é o sistema de registro (system of record) dos FIIs (FR-001) e da renda fixa (FR-002) — a fonte autoritativa, persistida e escopada por usuário, sobre cada posição que a pessoa declarou possuir.

Por que isso é a fundação? Porque "uma visão única e precisa de tudo o que possuo" (Epic 1) é o chão de todo cálculo a jusante. O resumo da carteira, a alocação, a renda passiva, o Health Score e cada insight da IA reconciliam contra os holdings — eles valem exatamente o quanto os holdings valem. Se a base estiver errada, tudo acima dela está errado. Daí o rigor desta spec.

O que ele entrega é o pacote internal/portfolio com dois domínios (FIIHolding e FixedIncomeHolding), os value objects (Quantity, LiquidityType, e o Ticker reusado do marketdata), o serviço, o port Repository e o port consumidor Reader, o adaptador Postgres, a migration 0005_holdings (duas tabelas) e oito endpoints CRUD. Nada calcula valor de mercado ainda — esta spec guarda o custo (cost basis) e expõe um port de leitura.

ANTES (SPEC-101) DEPOIS (SPEC-102) perfil: quem é o investidor carteira: o que ele tem — o system of record (FII + renda fixa) 1 registro 1-para-1 por usuário N holdings por usuário → surge a noção de POSSE (ownership) identidade vem do contexto identidade + POSSE: mutação é DUPLO-ESCOPO (id $1 AND user_id $2) sem dinheiro nenhum a 1ª feature com DINHEIRO: int64 centavos / int bps, até na wire PUT /profile (upsert 1-para-1) 8 endpoints CRUD: POST/GET/PUT/DELETE × {fii, fixed-income} ProfileReader (seam de leitura) Reader.ListHoldings → Holdings{FII, FixedIncome} para SPEC-103/104/107

Os endpoints novos (todos atrás da auth deny-by-default, fora da allowlist pública):

Por que a carteira é o "substrato determinístico" do produto? Porque a regra binding nº 3 do PRD diz que "fatos são computados, não gerados" — a IA raciocina sobre números, nunca os inventa. A carteira é uma das duas fontes desses fatos (a outra é o mercado, SPEC-006). Quando o Fact Builder (SPEC-104) montar o pacote Facts que o Insighter recebe, ele vai ler os holdings deste usuário e cruzá-los com as cotações. Se a carteira não fosse exata e isolada por usuário, o grounding seria ficção e o insight de A poderia vazar dado de B.

2. O problema: "uma visão única e precisa de tudo o que possuo"

O SPEC-102 resolve um enunciado simples e fácil de errar: precisamos guardar N posições por usuário, de dois tipos, de forma que (a) sejam só do dono, (b) sejam válidas por construção, (c) o dinheiro seja exato e (d) outras features possam lê-las sem se acoplar a HTTP ou SQL. Dois tipos de holding, cada um com seus campos:

TipoCamposForma escolhida
FIIHolding ticker, quantity, average_price_centavos Ticker (reusado do marketdata, D1), Quantity inteiro > 0 (D5), preço médio em centavos
FixedIncomeHolding name, institution, invested_amount_centavos, annual_rate_bps, maturity_date, liquidity_type texto não-vazio; valor em centavos; taxa em basis points; data de vencimento opcional; liquidez enum fechado (D3)

Repare a grande diferença em relação ao SPEC-101: lá o registro era um-para-um com o usuário (cada pessoa tem um perfil, com user_id como PK). Aqui é um-para-muitos: cada pessoa tem muitos holdings, cada um com seu próprio id (UUID surrogate). É essa multiplicidade que introduz a noção de posse (ownership) — e que faz o isolamento por usuário ganhar uma segunda camada, como veremos na seção 4.

3. LIÇÃO PRINCIPAL #1 — dinheiro NUNCA é float (BR-1022)

O SPEC-102 é a primeira feature que lida com dinheiro — então é aqui que a convenção mais inegociável da casa sai do papel e se prova de ponta a ponta. A regra: valores monetários (average_price, invested_amount) são int64 em centavos (unidades menores); taxas (annual_rate) são inteiros em basis points (1 bp = 0,01%). Nunca float64 — nem no domínio, nem nas colunas do banco, nem atravessando o JSON.

Por que float para dinheiro é um BUG, não um detalhe de estilo Ponto flutuante binário não consegue representar exatamente a maioria das frações decimais: 0.1 + 0.2 == 0.30000000000000004, não 0.3. Some milhares de posições e o erro de arredondamento contamina o saldo — o investidor vê um total que não bate com a soma das partes. Pior: o PRD exige "mesmos inputs → mesmo Health Score", reprodutível bit a bit. Com float, o mesmo cálculo pode dar resultados ligeiramente diferentes a cada rodada. Com inteiros, R$ 15,75 é exatamente 1575 centavos, sempre, em qualquer máquina. O dinheiro vira exato e reproduzível por construção.

3.1 — Inteiro no domínio

Os campos de dinheiro do domínio já nascem inteiros — sem nenhum float64 à vista:

// internal/portfolio/holding.go
type FIIHolding struct {
    ID                   string
    UserID               string
    Ticker               marketdata.Ticker
    Quantity             Quantity
    AveragePriceCentavos int64          // ← centavos, nunca float
    CreatedAt, UpdatedAt time.Time
}

type FixedIncomeHolding struct {
    // ...
    InvestedAmountCentavos int64        // ← centavos
    AnnualRateBps          int          // ← basis points (1 bp = 0,01%)
    MaturityDate           *time.Time
    LiquidityType          LiquidityType
}

3.2 — Inteiro nas colunas do banco

A migration 0005 não tem uma única coluna de ponto flutuante: dinheiro é bigint, taxa é integer:

-- migrations/0005_holdings.up.sql
CREATE TABLE fii_holdings (

    user_id                uuid        NOT NULL REFERENCES users (id) ON DELETE CASCADE,
    ticker                 text        NOT NULL,

    average_price_centavos bigint      NOT NULL,    -- R$ em centavos, nunca float
    ...

-- fixed_income_holdings: invested_amount_centavos bigint, annual_rate_bps integer

3.3 — Inteiro atravessando a wire (a parte mais fácil de esquecer)

Esta é a sutileza nova do SPEC-102. Não basta ser inteiro por dentro: o JSON também carrega inteiros. Os DTOs nomeiam os campos com o sufixo da unidade (_centavos / _bps) — assim o contrato é explícito e não há lugar para um float entrar:

// internal/transport/http/holdings.go
// money crosses the wire as integer centavos / bps, never a float — FR-1027/BR-1022
type fiiHoldingRequest struct {
    Ticker               string `json:"ticker"`
    Quantity             int    `json:"quantity"`
    AveragePriceCentavos int64  `json:"average_price_centavos"`   // ← int64 na wire
}

type fixedIncomeRequest struct {
    Name                   string  `json:"name"`
    Institution            string  `json:"institution"`
    InvestedAmountCentavos int64   `json:"invested_amount_centavos"`  // ← centavos
    AnnualRateBps          int     `json:"annual_rate_bps"`           // ← basis points
    MaturityDate           *string `json:"maturity_date"`             // "YYYY-MM-DD" ou null
    LiquidityType          string  `json:"liquidity_type"`
}
Por que o nome do campo carrega a unidade (average_price_centavos)? Porque o número sozinho é ambíguo: 1575 são R$ 15,75 ou R$ 1.575,00? O sufixo _centavos resolve a ambiguidade no próprio contrato — o cliente sabe que precisa enviar centavos, e o servidor sabe o que recebeu. É autodocumentação que também previne o erro clássico de "esqueci o fator 100". O mesmo vale para _bps: 850 com sufixo _bps é inequivocamente 8,50%. Quando uma feature futura precisar parsear um decimal de entrada (ex.: "15,75"), ela usa o helper centralizado internal/platform/money.DecimalToMinor (half-up), do SPEC-006 — mas aqui o contrato já é inteiro, então nem isso é necessário.

4. LIÇÃO PRINCIPAL #2 — identidade e posse vêm do contexto (BR-1021)

O SPEC-101 ensinou: identidade vem do contexto, nunca do payload. O SPEC-102 leva isso um degrau acima, porque um usuário tem muitos holdings. Agora não basta saber quem está pedindo; é preciso saber se aquele holding específico é dele. Isso é posse (ownership), e a defesa ganha uma camada extra: as mutações são duplo-escopo.

SPEC-101 (perfil, 1-para-1)SPEC-102 (holdings, 1-para-N)
LeituraWHERE user_id = $1WHERE user_id = $1
Update / DeleteWHERE user_id = $1 (a chave É o usuário)WHERE id = $1 AND user_id = $2 (id + dono)
A regra de ouro: um id de outro usuário é "não encontrado" (404), NUNCA "proibido" (403) Por que não retornar 403 Forbidden quando B tenta editar o holding de A? Porque 403 vaza informação: ele confirma que aquele id existe (só não é seu). Isso é um existence oracle — um atacante poderia varrer ids e mapear quais existem. A defesa correta é tornar "não é seu" indistinguível de "não existe": ambos viram 404 {"error":"holding not found"}. O WHERE id = $1 AND user_id = $2 faz exatamente isso — a linha de A simplesmente não casa para B, e o resultado é zero linhas, igual a um id inexistente. O sentinela do domínio diz isso explicitamente: "absent or not owned — the two are indistinguishable on purpose".

Como no SPEC-101, o errado é tornado estruturalmente impossível, com quatro travas empilhadas — só que agora a quarta é o duplo-escopo. Vamos vê-las no código real.

Trava 1 — nenhum DTO carrega user_id (nem na path, nem no corpo)

Olhe de novo os DTOs da seção 3.3: não há campo user_id em lugar nenhum. O dono nunca é dado; é sempre identidade, vinda da sessão.

Trava 2 — DisallowUnknownFields rejeita um user_id contrabandeado

O decoder de JSON compartilhado (decodeJSON, do SPEC-101) recusa qualquer campo desconhecido — um corpo com user_id intruso vira 400, não é silenciosamente ignorado.

Trava 3 — o handler pega o ID do contexto e o passa ao serviço

Há um helper minúsculo, callerID, que centraliza a leitura da identidade da sessão. Se não houver sessão autenticada, é 401 antes de tocar o serviço — e o user_id entra na lógica por um único caminho:

// internal/transport/http/holdings.go
// callerID returns the authenticated user id from the request context ... Identity is only
// ever read from the context the middleware set, never from request input (BR-1021).
func callerID(w http.ResponseWriter, r *http.Request) (string, bool) {
    id, ok := auth.UserID(r.Context())   // ← do contexto, nunca do corpo/path
    if !ok {
        writeError(w, http.StatusUnauthorized, "authentication required")
        return "", false
    }
    return id, true
}

// updateFIIHolding — o id do RECURSO vem da path; o id do DONO vem do contexto
userID, ok := callerID(w, r)
// ...
holding, err := h.service.UpdateFIIHolding(r.Context(), userID, r.PathValue("id"), fiiInput(req))
//                                                    ↑ dono (contexto)  ↑ recurso (path)
A distinção fina: o {id} da path é o id do RECURSO, não do DONO Repare que o {id} em PUT /holdings/fii/{id} é entrada do cliente — mas ele identifica qual holding, não quem é o usuário. O dono nunca vem da path. Por isso o serviço recebe os dois separados: userID (do contexto, confiável) e id (da path, não confiável). E é o SQL duplo-escopo que reconcilia: o id da path só "existe" se pertencer ao userID do contexto.

Trava 4 — o SQL duplo-escopo, com RETURNING e contagem de linhas

No adaptador, a leitura filtra por user_id; a mutação filtra por id E user_id. Um update que não casa nenhuma linha (sql.ErrNoRows) vira ErrHoldingNotFound; um delete que afeta zero linhas, idem:

// internal/portfolio/postgres/postgres.go — UpdateFIIHolding
UPDATE fii_holdings
SET ticker = $3, quantity = $4, average_price_centavos = $5, updated_at = $6
WHERE id = $1::uuid AND user_id = $2::uuid       -- ← DUPLO-ESCOPO
RETURNING created_at, updated_at                 -- created_at preservado; updated_at avança
// ...
if notFound(err) {                               // sql.ErrNoRows OU uuid malformado (22P02)
    return portfolio.FIIHolding{}, portfolio.ErrHoldingNotFound
}

// deleteHolding — afetou 0 linhas? não era seu (ou não existe) → ErrHoldingNotFound
n, _ := res.RowsAffected()
if n == 0 {
    return portfolio.ErrHoldingNotFound
}
Um detalhe de robustez: o UUID malformado vira 404, nunca 500 E se o cliente mandar PUT /holdings/fii/abc com um id que não é um UUID válido? O cast $1::uuid falharia com o SQLSTATE 22P02 (invalid_text_representation). Sem cuidado, isso borbulharia como erro de banco e viraria um 500. A função notFound trata 22P02 como "não casou nenhuma linha" — afinal, um id malformado nunca poderia casar — devolvendo o mesmo 404 de um id que não existe. Um id inválido não é um erro do servidor; é só mais um "não encontrado". Isso fecha o oracle de existência por completo.

5. A decisão D1: reusar marketdata.Ticker — e a pergunta que ela levantou

O FII tem um ticker (ex.: HGLG11). O SPEC-006 já definiu um value object Ticker que valida o formato B3 (4 letras + 1–2 dígitos). A pergunta: o portfolio cria seu próprio Ticker, ou reusa o do marketdata? A decisão D1 foi reusar:

// internal/portfolio/holding.go
import "github.com/biel-ferreira/yield-forge/internal/marketdata"

type FIIHolding struct {
    Ticker marketdata.Ticker   // ← o MESMO value object do marketdata (D1)
    // ...
}

// internal/portfolio/service.go — usa o ParseTicker do marketdata
ticker, err := marketdata.ParseTicker(in.Ticker)

Mas isso levantou uma pergunta de arquitetura legítima: portfolio importar marketdata não quebra o hexagonal "package-oriented", onde cada feature é autocontida? A resposta é instrutiva e vale guardar.

A regra dura é DIREÇÃO de dependência, não "feature não importa feature" A regra binding do projeto é: o core não importa SQL, HTTP nem framework — a seta de dependência aponta para dentro. Um marketdata.Ticker é um value object puro (só stdlib por baixo); importá-lo não arrasta SQL nem HTTP para dentro do portfolio, então a regra dura continua intacta. Além disso, marketdata é uma costura foundational (tier 0xx), e feature → foundational é a direção pretendida — exatamente o mesmo formato de profile → auth.UserID(ctx) no SPEC-101. O grafo continua acíclico (marketdata não importa portfolio de volta). E o hook block-layering passou: ele barra import de SQL/HTTP/SDK no core, não import de um tipo de domínio puro.

Há um bônus de correção nessa reutilização: existe uma única definição de "ticker B3 válido" no projeto inteiro. Isso garante que um holding de HGLG11 e uma cotação de HGLG11 usem exatamente a mesma noção de ticker — então eles reconciliam quando o dashboard cruzar carteira com mercado. Duas definições divergentes seriam uma fonte silenciosa de bugs ("por que esse holding não acha sua cotação?").

A alternativa reservada para a Regra de Três (shared kernel) E se um terceiro consumidor de Ticker aparecer (além de marketdata e portfolio)? Aí a decisão prevê promover o Ticker para um shared kernel dedicado — internal/b3 — para que nenhuma feature "dona" do tipo seja importada pelas outras. Isso é a Regra de Três: não se cria a abstração compartilhada na primeira reutilização (seria especulação); espera-se a terceira, quando o padrão está provado. Por ora, importar do marketdata é a escolha certa, e a promoção fica rastreada como melhoria futura no PLAN-102.

6. Value objects parse-don't-validate & as regras de vencimento

Como nas specs anteriores, os campos não são string/int crus: são value objects que validam no construtor. Um valor inválido não pode existir.

6.1 — Quantity: inteiro estritamente positivo (D5)

FII se compra em cotas inteiras (não há "1,5 cota"), então a quantidade é um inteiro > 0, com o campo privado para que ninguém construa um valor inválido direto:

// internal/portfolio/quantity.go
type Quantity struct{ value int }   // campo privado: só o construtor cria

func ParseQuantity(n int) (Quantity, error) {
    if n <= 0 {
        return Quantity{}, fmt.Errorf("parse quantity %d: %w", n, ErrInvalidQuantity)
    }
    return Quantity{value: n}, nil
}

6.2 — LiquidityType: enum fechado daily | at_maturity (D3)

A liquidez é um enum fechado, normalizado (trim + lower) no parse. Note que ela não guarda o nome do instrumento — "CDB", "Tesouro Direto", "caixinha" vão no campo name; o LiquidityType captura só se o dinheiro está disponível diariamente ou travado até o vencimento:

// internal/portfolio/liquiditytype.go
const (
    LiquidityDaily      LiquidityType = "daily"       // resgatável qualquer dia
    LiquidityAtMaturity LiquidityType = "at_maturity" // travado até maturity_date
)

func ParseLiquidityType(s string) (LiquidityType, error) {
    lt := LiquidityType(strings.ToLower(strings.TrimSpace(s)))
    if !validLiquidityTypes[lt] {
        return "", fmt.Errorf("parse liquidity type %q: %w", s, ErrInvalidLiquidityType)
    }
    return lt, nil
}

// RequiresMaturity diz se este tipo PRECISA de uma data de vencimento.
func (l LiquidityType) RequiresMaturity() bool { return l == LiquidityAtMaturity }

6.3 — As regras de vencimento: at_maturity exige data futura na criação; daily normaliza para nil

Aqui mora uma lógica de negócio fina, no serviço. Um instrumento at_maturity precisa de uma data de vencimento, e essa data não pode estar no passado no momento da criação (PRD Epic 1). Um instrumento daily não tem vencimento — a data é normalizada para nil. E o "hoje" da comparação vem do Clock injetado, nunca de time.Now():

// internal/portfolio/service.go — buildFixedIncomeHolding(in, isCreate)
if lt.RequiresMaturity() {
    if in.MaturityDate == nil {
        return FixedIncomeHolding{}, fmt.Errorf("build fixed income holding: %w", ErrMaturityRequired)
    }
    maturity := in.MaturityDate.UTC()
    if isCreate && maturity.Before(s.today()) {   // ← só na CRIAÇÃO; s.today() vem do Clock
        return FixedIncomeHolding{}, fmt.Errorf("build fixed income holding: %w", ErrPastMaturity)
    }
    h.MaturityDate = &maturity
}
// Um holding daily NUNCA carrega data de vencimento (normalizado para nil).

func (s *Service) today() time.Time {
    n := s.clock.Now().UTC()   // ← Clock, não time.Now()
    return time.Date(n.Year(), n.Month(), n.Day(), 0, 0, 0, 0, time.UTC)
}
Por que a regra de "data futura" só vale na CRIAÇÃO — e por que via Clock Só na criação porque o tempo passa: um CDB criado hoje com vencimento daqui a 1 ano um dia vai ter o vencimento "no passado" (quando vencer). Bloquear o update por isso seria absurdo — você não conseguiria mais editar um holding vencido. Por isso o buildFixedIncomeHolding recebe um booleano isCreate e só aplica a regra de passado quando é criação.
Via Clock porque "o que é o passado?" depende de "que dia é hoje?" — e amarrar isso a time.Now() tornaria o teste não determinístico (passaria hoje, falharia amanhã com a mesma fixture). Com um Clock falso, o teste fixa "hoje" e a regra fica reproduzível para sempre.

7. Duas tabelas (D2), o upsert com RETURNING e o port Reader

7.1 — Por que duas tabelas (D2)

FII e renda fixa têm campos bem diferentes. Havia dois caminhos: uma tabela polimórfica com um discriminador type e muitas colunas anuláveis (campos de FII nulos nas linhas de renda fixa e vice-versa), ou duas tabelas, uma por tipo, cada uma com campos limpos e NOT NULL. A decisão D2 foi duas tabelas (fii_holdings e fixed_income_holdings):

OpçãoPrósContras
Uma tabela polimórfica (type + colunas anuláveis)Um só CRUD, um só GET /holdingsColunas semanticamente obrigatórias viram anuláveis; o banco não consegue garantir invariantes; consultas cheias de CASE
Duas tabelas (escolhido)Type-safe: cada campo é NOT NULL de verdade; SQL limpo; o domínio reflete o banco 1-para-1Dobra a superfície de CRUD (mitigado com testes table-driven simétricos e nomes de método simétricos)

7.2 — Create com RETURNING: a linha autoritativa sem re-leitura

O insert deixa o banco gerar o id (UUID) e os timestamps, e os devolve no mesmo round-trip com RETURNING — o mesmo padrão que o SPEC-101 usou para fechar a corrida do re-read:

// internal/portfolio/postgres/postgres.go — CreateFIIHolding
INSERT INTO fii_holdings (user_id, ticker, quantity, average_price_centavos, created_at, updated_at)
VALUES ($1::uuid, $2, $3, $4, $5, $6)
RETURNING id::text, created_at, updated_at        -- id e timestamps gerados pelo banco, num só round-trip

E na leitura, o adaptador passa os valores guardados de volta pelos construtores (marketdata.ParseTicker, portfolio.ParseQuantity, ParseLiquidityType) — então uma linha corrompida no banco vira um erro explícito, não um value object inválido circulando solto:

// internal/portfolio/postgres/postgres.go — rebuildFII (chamado por ListFIIHoldingsByUserID)
t, err := marketdata.ParseTicker(ticker)     // revalida o ticker lido do banco
q, err := portfolio.ParseQuantity(quantity)  // revalida a quantidade

7.3 — O port Repository (8 métodos) e o port consumidor Reader

Há dois ports, com a mesma elegância do SPEC-101. O Repository tem 8 métodos (4 por tipo) — o que o serviço usa para falar com o banco:

// internal/portfolio/ports.go
type Repository interface {
    CreateFIIHolding(ctx, h FIIHolding) (FIIHolding, error)
    ListFIIHoldingsByUserID(ctx, userID string) ([]FIIHolding, error)
    UpdateFIIHolding(ctx, h FIIHolding) (FIIHolding, error)   // scoped (id, user_id); ErrHoldingNotFound
    DeleteFIIHolding(ctx, userID, id string) error            // scoped; ErrHoldingNotFound
    // … os mesmos quatro para FixedIncomeHolding …
}

E o Reader é o seam de leitura voltado ao consumidor — o que as features de SPEC-103/104/107 vão usar. Ele expõe um só método que devolve o agregado Holdings{FII, FixedIncome}, nada de escrita. O próprio Service o satisfaz:

// internal/portfolio/ports.go
type Reader interface {
    ListHoldings(ctx, userID string) (Holdings, error) // { FII []FIIHolding; FixedIncome []FixedIncomeHolding }
}

// internal/portfolio/service.go
var _ Reader = (*Service)(nil)   // o Service satisfaz o Reader — provado em compile-time
Cost basis vs. valor de mercado (BR-1024) — a fronteira mais importante do escopo Os holdings guardam o que o usuário pagou: average_price_centavos, invested_amount_centavos — o cost basis. Eles nunca guardam nem calculam o valor de mercado atual, a alocação, a renda passiva ou o rendimento. Isso é deliberado: o valor atual é computado a jusante (dashboard SPEC-103, projeções SPEC-107) cruzando os holdings com as cotações do SPEC-006. Esta spec nunca inventa número de mercado — ela é a fonte do que o investidor declarou ter pago, e nada além disso. Separar "o que paguei" (fato declarado, imutável) de "quanto vale hoje" (derivado, volátil) mantém cada feature com uma responsabilidade só.

8. Por que NÃO há IA aqui — mas as regras de dinheiro se aplicam

Como no SPEC-101, vale ser explícito sobre quais regras binding se aplicam e quais não. Mas o contraste com o perfil é didático: o perfil não tinha nem dinheiro nem IA; aqui, IA continua fora, mas dinheiro entra com força total.

Regra bindingPerfil (SPEC-101)Carteira (SPEC-102)
Dinheiro em int64 centavos / bps N/A — perfil não tem valor monetário CENTRAL — preço médio, valor investido, taxa: tudo inteiro, no domínio, no banco e na wire (BR-1022). É a primeira feature com dinheiro.
Explicabilidade (FR-013) N/A — sem saída de IA N/A — sem saída de IA aqui (BR-1025). Os holdings são fatos determinísticos que o Fact Builder vai entregar ao Insighter depois.

A lição de design: cada regra tem um ESCOPO A disciplina de dinheiro protege cálculo financeiro reproduzível; os gates FR-013/014 protegem saída de IA. O perfil não era nem um nem outro. A carteira é o primeiro caso de dinheiro — então a regra de centavos finalmente "morde" — mas continua sem IA, então os gates seguem inertes. Reconhecer quando cada regra se aplica evita tanto o erro de relaxar (usar float "porque é só um preço") quanto o de cerimônia vazia (tentar anexar um disclaimer a um holding). É o oposto exato do SPEC-006, onde dinheiro e IA-a-jusante conviviam.

9. As 5 decisões (D1–D5) e por quê

#DecisãoEscolhaPor quê
D1Fonte do Ticker do FII? Reusar marketdata.Ticker É value object puro — não arrasta SQL/HTTP, mantém o core puro. marketdata é foundational; feature→foundational é a direção pretendida (igual a profile → auth), grafo acíclico. Uma só definição de "ticker B3 válido" garante reconciliação com as cotações. 3º consumidor → promover a internal/b3 (Regra de Três).
D2Modelagem dos holdings? Duas entidades + duas tabelas Campos limpos e type-safe (NOT NULL de verdade) em vez de uma tabela polimórfica com colunas anuláveis. Dobra o CRUD, mas com nomes/Testes simétricos.
D3Semântica de liquidity_type? Enum de liquidez (daily | at_maturity) O nome do instrumento (CDB/Tesouro/caixinha) vive em name; este campo captura só se o dinheiro está disponível diariamente ou travado. maturity_date obrigatório para at_maturity, nulo para daily.
D4Formato dos endpoints? Rotas por tipo (/holdings/fii, /holdings/fixed-income) DTOs e validação distintos por tipo, mais claros que um /holdings único com discriminador type. Um Reader unificado serve o agregado ao dashboard.
D5Forma da quantidade? Quantity inteiro > 0 Cotas de FII são unidades inteiras; uma quantidade fracionária não faz sentido. Campo privado + construtor tornam o inválido inrepresentável.

10. As 7 fases, uma a uma

Bottom-up, como nas specs anteriores — core puro primeiro, borda depois, cada fase compilando e revisável sozinha. Com disciplina de dinheiro e posse como os dois fios condutores.

Fase 1 — Domínio & ports

Os value objects com construtor: Quantity (> 0, campo privado) e LiquidityType (daily | at_maturity), reusando marketdata.Ticker (D1). As entidades FIIHolding e FixedIncomeHolding (money int64 centavos, rate int bps) e o agregado Holdings. Os sentinelas (ErrHoldingNotFound, ErrInvalidQuantity, ErrInvalidLiquidityType, ErrPastMaturity, ErrMaturityRequired, ErrEmptyField, ErrInvalidAmount, ErrNegativeAmount, ErrInvalidRate). Os dois ports: Repository (8 métodos) e Reader (consumidor).

Fase 2 — Persistência (migration + repositório)

A migration 0005_holdings up/down: duas tabelas (UUID id, FK ON DELETE CASCADE, bigint centavos / integer bps, sem float, índice em user_id). O pacote postgres/: Create com RETURNING, List…ByUserID escopado, Update/Delete duplo-escopo (id + user_id, zero linhas → ErrHoldingNotFound), o tratamento de 22P02 → não-encontrado, o agregado do Reader e a asserção var _ portfolio.Repository = Repository{}.

Fase 3 — Aplicação (serviço)

O portfolio.Service: CRUD para os dois tipos (valida → constrói value objects → chamada escopada ao repo), a regra de vencimento-no-passado via Clock (só na criação), e a implementação do Reader. A identidade entra pelo parâmetro userID, sempre do contexto do handler. Fakes escritos à mão.

Fase 4 — API (transporte)

O transport/http/holdings.go: os 8 handlers, com wildcards de path do Go 1.22 (PUT /holdings/fii/{id} via r.PathValue("id")), DTOs separados do domínio com dinheiro inteiro (*_centavos / *_bps), o helper callerID (401 sem sessão), o mapeamento de erro writeHoldingError (sentinela → status), e o registro das rotas auto-protegidas + fiação no cmd/api (Deps.Portfolio).

Fase 5 — Observabilidade

Os endpoints herdam os spans nomeados por rota do otelhttp (POST /holdings/fii, PUT /holdings/fii/{id} — nomeados pelo padrão da rota, não pelo id cru, mantendo baixa cardinalidade); as chamadas ao banco viram span-filho de query (sem valores de argumento). Contador opcional portfolio.mutations por {type, op, outcome}; nenhum PII além do user_id.

Fase 6 — Testes

Unitários sem banco: value objects (zero/negativo rejeitados, vencimento-no-passado com Clock falso); serviço com fake (create→list→update→delete, not-found, validação) para os dois tipos; handlers (validação de DTO, identidade-do-contexto, round-trip de dinheiro como centavos, ownership 404, 401, e um teste de que o span é nomeado PUT /holdings/fii/{id}, não pelo id cru, via in-memory span exporter). Integração gated: Postgres real (TEST_DATABASE_URL, -p 1) para o round-trip CRUD, o up/down da 0005, o isolamento por usuário, o ownership escopado (B não toca a linha de A) e o cascade.

Fase 7 — Documentação & aula

README.md (endpoints /holdings/*), CHANGELOG.md, SPEC-102 + PLAN-102 marcados Done, índices e status do CLAUDE.md atualizados, e esta aula. 🙂

11. Ponte: arquitetura em camadas → hexagonal

Você já conhece a pilha clássica em camadas: controller → DTO → service → repository, com o service importando o repositório concreto. O SPEC-102 é um caso lindo para a tradução, porque tem todas as peças clássicas — handler, DTO, service, repositório — só que rearranjadas pela inversão de dependência. (Para a teoria completa, veja a aula conceitual de arquitetura hexagonal; aqui aplicamos ela ao SPEC-102.)

A tabela de-para (com os tipos reais desta spec)

DTO de request/responsefiiHoldingRequest / fixedIncomeResponse etc. — continuam DTOs, na borda, com dinheiro inteiroBorda (transport/http)

A "grande virada": a inversão de dependência

Em camadas, o service importa o repositório concreto — a seta de dependência aponta para fora, do núcleo rumo à infra. No hexagonal, isso vira: o core define a interface e o adapter a implementa — a seta aponta para dentro. Veja com o par real desta spec:

// O CORE define o port (internal/portfolio/ports.go):
type Repository interface {
    CreateFIIHolding(ctx context.Context, h FIIHolding) (FIIHolding, error)
    UpdateFIIHolding(ctx context.Context, h FIIHolding) (FIIHolding, error)  // scoped (id, user_id)
    // … e os demais …
}

// A BORDA o implementa e PROVA isso em tempo de compilação
// (internal/portfolio/postgres/postgres.go):
var _ portfolio.Repository = Repository{}   // o adapter depende do core, não o contrário

O postgres importa portfolio; portfolio jamais importa database/sql nem net/http (importa só marketdata.Ticker, value object puro — ver D1). É por isso que o Service é testável com um fake trivial (sem banco), e que trocar Postgres por outro armazenamento não toca uma linha do core.

Driving vs. driven nesta spec

  • Driving port (quem aciona a aplicação): o holdingsHandler chama o serviço através da interface PortfolioService definida no próprio transporte (consumer-defined). Fica à esquerda do hexágono.
  • Driven port (o que a aplicação aciona): portfolio.Repository (banco), implementado pelo adaptador postgres. Fica à direita.
  • Port de saída para outros casos de uso: portfolio.Reader — a fresta por onde futuros casos de uso (dashboard, Fact Builder, projeções) leem o agregado de holdings.
  • Domínio puro: FIIHolding, FixedIncomeHolding, Quantity, LiquidityType, os sentinelas, o Service. Borda: tudo com HTTP ou SQL — o handler e o adaptador Postgres.
┌──────────────── HEXÁGONO (core puro) ───────────────┐ HTTP ──▶ holdingsHandler ──▶ │ portfolio.Service (caso de uso) │ (driving, via PortfolioService)│ │ depende de PORT, não de concreto │ │ └── portfolio.Repository ◀── postgres/ (driven) │ │ SPEC-103/104/107 ◀──────────│── portfolio.Reader (port de leitura) │ (futuros consumidores) │ domínio: FIIHolding, FixedIncomeHolding, Quantity, LiquidityType, Ticker │ └──────────────────────────────────────────────┘ Identidade (auth.UserID(ctx)) + POSSE entram pelo handler como PARÂMETRO — nunca pelo corpo/path. Mutação é DUPLO-ESCOPO: WHERE id = $1 AND user_id = $2. A seta de dependência aponta PARA DENTRO.

12. Harness Engineering: como a máquina construiu isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. Veja, no SPEC-102, como o harness construiu e verificou.

O fluxo /spec-implement, fase a fase

A IA não atacou "faça a carteira" de uma vez. O PLAN-102 fatiou em 7 fases bottom-up (domínio → persistência → serviço → transporte → observabilidade → testes → docs), e cada uma fechava com go build + testes verdes antes da próxima. Construir o core puro primeiro (Fase 1) significa que, quando os 8 handlers foram montados (Fase 4), os value objects, o serviço e os dois ports já existiam e eram testáveis isoladamente — com dinheiro e posse como os dois fios condutores declarados no plano.

O novo hook determinístico block-layering

Uma novidade do [Unreleased]: o hook block-layering (PostToolUse) foi promovido de uma checagem subjetiva do hexagonal-reviewer para um bloqueio duro (exit 2) — ele barra, mecanicamente, qualquer import de SQL/HTTP/SDK de vendor dentro de um core de feature. Foi esse hook que deu a resposta firme à pergunta da D1: portfolio importar marketdata.Ticker (value object puro) passou; se a IA tivesse tentado meter um database/sql ou net/http no core portfolio, o hook teria bloqueado o commit, sem depender de revisão humana.

O subagente hexagonal-reviewer

O revisor especializado cobrou as regras mais sensíveis daqui: a direção de dependência (o core portfolio não importa SQL/HTTP; só o subpacote postgres e o transporte o fazem) e, sobretudo, a identidade + posse do contexto (BR-1021). Foi a lente de revisão que firmou as quatro travas da seção 4 — em especial o duplo-escopo WHERE id = $1 AND user_id = $2 e a exigência de que um id de outro usuário seja 404, nunca 403.

O go-correctness-reviewer e a disciplina de dinheiro / borda

Sendo a primeira feature com dinheiro, a lente de correção vigiou o risco nº 1 do plano: vazamento de float. Ela garantiu que nenhum DTO, coluna ou campo de domínio usasse float64, e que o round-trip de centavos fosse exato na integração. Também cobrou o tratamento do 22P02 (UUID malformado) → 404 em vez de 500 (seção 4), o %w + prefixo minúsculo em todos os erros, e a regra de vencimento amarrada ao Clock, não a time.Now().

A prova de que o core ficou puro É mecânica e dupla: (1) internal/portfolio (core) importa só stdlib + context + o port Clock + o value object puro marketdata.Ticker — nenhum import de database/sql ou net/http, o que o hook block-layering agora garante; e (2) a asserção var _ portfolio.Repository = Repository{} em postgres.go faz o compilador provar que o adapter satisfaz os 8 métodos do port. Esqueça um método ou inverta uma assinatura, e o build barra.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • block-layering: o bloqueio duro que provou que o core portfolio ficou livre de SQL/HTTP, ao mesmo tempo que permitiu o reuso do marketdata.Ticker (domínio puro).
  • Bloqueio de imutabilidade: o SPEC-102 referencia o ADR-0002 mas não o edita — se a IA tentasse "atualizar" um ADR aceito, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — e os testes têm que passar sem rede e sem DB. Os testes de integração (isolamento, ownership, cascade) ficam atrás do gate de TEST_DATABASE_URL + -p 1, pulando limpo no CI.

13. Conexão com AI Engineering

O SPEC-102 não é uma feature de IA — é o substrato de fatos sobre o qual a IA vai raciocinar. Vale conectar este alicerce à arquitetura de IA futura.

O que construímos agoraComo serve a IA depois
O port portfolio.Reader É o mesmo formato de costura do ProfileReader (SPEC-101), do Insighter (SPEC-005) e do MarketDataProvider (SPEC-006): interface pequena, definida pelo domínio. O Fact Builder (SPEC-104) vai receber um Reader e ler o agregado Holdings sem saber que existe HTTP ou SQL — exatamente como recebe o Insighter sem saber qual LLM está por trás.
Holdings como fatos determinísticos (BR-1024) A regra de ouro da IA confiável é "o modelo raciocina sobre fatos, não os inventa". A carteira é a metade desses fatos (a outra é o mercado, SPEC-006): o que o usuário possui e pagou, exato em centavos. O LLM os recebe como parte do Facts opaco; nunca os adivinha nem reescreve um saldo.
Posse + isolamento por user_id do contexto O Fact Builder vai computar fatos da carteira de um usuário só e juntá-los ao perfil daquele mesmo usuário. O duplo-escopo de hoje é o que garante que o holding (e logo o insight) de A nunca vaze para B. Um insight personalizado sem esse isolamento seria um vazamento de dados.
Dinheiro exato → reprodutibilidade int64 centavos garante que "mesmos inputs → mesmo Health Score" (PRD §6). Sem isso, o mesmo prompt produziria scores ligeiramente diferentes a cada rodada — e a explicabilidade (FR-013) perderia o chão, porque a explicação descreveria um número que muda sozinho.
A ideia que conecta tudo IA orientada a objetivo precisa de fatos exatos sobre o investidor. O SPEC-101 instalou o fato de quem ele é; o SPEC-102 instala o fato de o que ele tem — preciso (centavos), isolado por usuário (duplo-escopo), e exposto por um port de leitura (Reader). Quando o Fact Builder chegar (SPEC-104), ele lê os holdings pelo Reader, lê o perfil pelo ProfileReader, cruza com o mercado, e entrega ao Insighter um pacote de Facts determinístico — finalmente respondendo: "sua carteira está alinhada aos seus objetivos?"
