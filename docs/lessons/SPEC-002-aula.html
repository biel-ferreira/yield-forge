1. A visão geral: o que mudou no SPEC-002

No SPEC-001 tínhamos um esqueleto que rodava, mas não guardava nada. O SPEC-002 é a segunda pedra da fundação: o app agora conecta num PostgreSQL de verdade, tem um processo repetível pra mudar o schema, e o /readyz reflete o estado real do banco.

Importante: continuamos sem regra de negócio. Nenhuma tabela de feature (carteira, perfil...) foi criada — isso é de propósito. Cada feature vai trazer suas próprias tabelas. O SPEC-002 só prepara o encanamento.

ANTES (SPEC-001) DEPOIS (SPEC-002) app sobe app sobe → conecta no Postgres (ou falha rápido) /readyz → sempre "ready" /readyz → checa o banco de verdade (200/503) sem banco pool de conexões + migrations versionadas docker compose: só a api docker compose: api + postgres juntos

Os endpoints agora:

GET /healthz  →  {"status":"ok"}                                  (o processo está vivo?)
GET /readyz   →  {"status":"ready","checks":{"db":"up"}}          (consigo servir? o banco responde?)
GET /readyz   →  503 {"status":"not_ready","checks":{"db":"down"}}  (banco caiu → não estou pronto)
GET /version  →  {"version":"dev", ...}
Por que separar "vivo" de "pronto"? /healthz (liveness) responde "o processo está de pé". /readyz (readiness) responde "consigo realmente atender". Se o banco cai, o processo continua vivo (200 no healthz), mas não está pronto (503 no readyz). Um orquestrador (Docker, Kubernetes, etc.) usa essa diferença pra parar de mandar tráfego sem matar o processo desnecessariamente.

2. As 4 decisões que tomamos (e o porquê)

Todo spec tem bifurcações reais. No SPEC-002 foram quatro. Vale entender o trade-off de cada uma — é aqui que mora o aprendizado de engenharia.

#DecisãoEscolhaPor quê
D1O banco é obrigatório? Sim — sem DATABASE_URL o app falha rápido Honestidade: a partir daqui o banco é fundação. Nada de "funciona na minha máquina sem banco".
D2Driver e ferramenta de migration database/sql + driver pgx; migrations com golang-migrate Você aprende a interface padrão e portável da stdlib, usando o melhor driver de Postgres por baixo.
D3Quando aplicar migrations? Manualmente (task migrate:up) — nunca automático Mudança de schema é deliberada. Aplicar no boot automaticamente é uma fonte clássica de acidentes em produção.
D4Banco dos testes de integração Postgres real, atrás de TEST_DATABASE_URL; pula com -short Sem dependência nova pesada; roda no CI/dev quando há banco, e pula limpo quando não há.
Uma lição sobre "stdlib first" Estas foram as primeiras dependências de terceiros do projeto (pgx e golang-migrate). Aceitamos de propósito: Go não tem driver de Postgres na biblioteca padrão. A regra do ADR-0003 não é "nunca use dependência" — é "comece pela stdlib e só adicione uma dependência com motivo claro".

3. As 7 fases, uma a uma

Construímos de baixo pra cima, cada fase compilando e testável sozinha. Você revisou cada uma.

Fase 1 — Configuração do banco

Adicionamos ao Config o DATABASE_URL (segredo, obrigatório) e os parâmetros do pool de conexões. Tudo via variável de ambiente (12-factor).

DatabaseURL       string         // postgres://user:pass@host:5432/db?sslmode=...
DBMaxOpenConns    int            // máx. de conexões abertas    (padrão 10)
DBMaxIdleConns    int            // máx. de conexões ociosas    (padrão 5)
DBConnMaxLifetime time.Duration  // tempo de vida de uma conexão (padrão 30m)
DBConnMaxIdleTime time.Duration  // tempo ocioso até fechar      (padrão 5m)
DBConnectTimeout  time.Duration  // limite do connect/ping inicial (padrão 5s)
Por que uma função RedactedDatabaseURL()? A URL do banco contém a senha. Se logarmos a URL crua no startup, a senha vaza pros logs. Criamos um helper que devolve só esquema://host:porta/banco — nunca usuário/senha — e é ele que vai pro log. Tem até um teste garantindo que a senha nunca aparece.

Fase 2 — A conexão (pacote platform/database)

Uma função Connect que abre o pool com o driver pgx, aplica os parâmetros e verifica a conexão com um ping antes de devolver. Se o banco não responde, retorna erro — e o app falha rápido.

func Connect(ctx, cfg) (*sql.DB, error) {
    db, _ := sql.Open("pgx", cfg.DatabaseURL)
    db.SetMaxOpenConns(cfg.DBMaxOpenConns)
    // ... demais parâmetros do pool ...
    pingCtx, cancel := context.WithTimeout(ctx, cfg.DBConnectTimeout)
    defer cancel()
    if err := db.PingContext(pingCtx); err != nil {
        db.Close()                 // nunca devolve um pool "meio aberto"
        return nil, err
    }
    return db, nil
}
Por que o main virou run() error? O pool precisa ser fechado quando o servidor desliga. Usamos defer db.Close() — mas defer não roda se você chamar os.Exit(). Então transformamos main num run() que retorna erro, e o main só converte o erro em código de saída. Assim o defer executa e o pool fecha depois que o HTTP drenou as requisições em andamento.

Fase 3 — Migrations (golang-migrate + 0001_init)

Mudanças de schema viram arquivos SQL versionados, em pares up/down:

migrations/
  0001_init.up.sql      → CREATE EXTENSION pgcrypto;  (+ convenções documentadas)
  0001_init.down.sql    → DROP EXTENSION pgcrypto;
  embed.go              → //go:embed *.sql  (empacota os SQLs no binário)

O 0001_init não cria nenhuma tabela. Ele só liga a extensão pgcrypto (que dá gen_random_uuid()) e documenta as convenções que toda tabela futura deve seguir:

ConvençãoRegra
Chave primáriaid uuid primary key default gen_random_uuid()
Timestampscreated_at / updated_at timestamptz (sempre UTC)
Dono da linhauser_id uuid (a FK chega no SPEC-003, com auth)
Dinheironumeric ou centavos inteiros — nunca float
Dinheiro nunca é float! float/double têm erros de arredondamento (0.1 + 0.2 ≠ 0.3 em binário). Em finanças isso é inaceitável. Por isso fixamos: valores monetários são numeric ou inteiros em centavos. Essa regra vale pra todas as tabelas que vierem.

E um runner cmd/migrate com 4 comandos, expostos via Task:

task migrate:up                 # aplica tudo que está pendente
task migrate:status             # mostra a versão atual do schema
task migrate:down -- 1          # desfaz 1 migration
task migrate:create -- add_xyz  # cria o par 000N_add_xyz.up/down.sql

Fase 4 — O /readyz reflete o banco

O /readyz, que no SPEC-001 era um "sempre pronto", agora faz um ping de verdade no banco (com timeout próprio) e responde 200 ou 503. Detalhe importante: ele depende de uma interface, não do banco diretamente — veja a seção 5.

Fase 5 — Compose com Postgres

Ligamos o serviço db no docker-compose.yml: imagem postgres:16-alpine, um healthcheck (pg_isready) e depends_on: service_healthy — então a api só sobe depois que o banco está realmente aceitando conexões.

Duas portas, dois "endereços" do mesmo banco Dentro da rede do Compose, a api fala com o banco pelo hostname db:5432. No seu PC, você alcança o mesmo banco em localhost:5433 (escolhemos 5433 pra não conflitar com um Postgres nativo que você já tinha na 5432). Por isso o DATABASE_URL da api (no compose) e o do .env (host) são diferentes — apontam pro mesmo Postgres por caminhos diferentes.

Fase 6 — Testes

Testes de integração contra um Postgres real, com gating: pulam no modo -short e quando TEST_DATABASE_URL não está setado. O destaque é o teste de ida e volta das migrations (up → down → up), que prova que todo down realmente desfaz o up. Foi essa fase que pegou um bug de verdade — veja a seção 6.

Fase 7 — Documentação

Atualizamos o CHANGELOG.md, o README.md (pré-requisitos, como rodar migrations, fluxo docker-up → migrate:up → readyz) e marcamos SPEC-002 e PLAN-002 como Done. E esta aula. 🙂

4. Conceitos novos que você aprendeu

database/sql vs "driver" — qual é a diferença?

database/sql é a interface padrão de Go pra bancos SQL: db.Query, db.Exec, PingContext... Ela não sabe falar com nenhum banco específico. Quem sabe falar "Postgresês" é o driver (o pgx). Você programa contra a interface padrão e troca o driver por baixo.

seu código → database/sql (interface padrão) → driver pgx → PostgreSQL (Query, Exec, Ping...) (fala o protocolo)

Pool de conexões

Abrir uma conexão TCP+autenticação com o banco a cada query é caro. O *sql.DB não é "uma conexão" — é um pool que reaproveita um conjunto de conexões. Os parâmetros que configuramos controlam quantas ficam abertas, quantas ociosas, e por quanto tempo. Guardar esse bug na memória: o pool tem um limite, e estourar esse limite trava (foi o que aconteceu — seção 6).

go:embed — empacotar arquivos no binário

A diretiva //go:embed *.sql faz o compilador colar os arquivos SQL dentro do binário. Resultado: o app/runner de migration carrega as migrations consigo, sem precisar do código-fonte por perto. Um único executável, autossuficiente.

context com timeout

Quase toda operação de I/O recebe um context.Context. Usamos context.WithTimeout pra limitar o ping de conexão e o check do /readyz. Assim, um banco lento ou travado não trava o app pra sempre — depois do prazo, a operação é cancelada e vira erro.

5. O Pinger: uma interface que torna o /readyz testável

O handler do /readyz precisa "perguntar ao banco se ele responde". Mas se ele dependesse do *sql.DB concreto, só dava pra testar com um Postgres de verdade. Solução clássica de Go: dependa de uma interface pequena, definida por quem a usa.

// Definida no pacote transport/http (quem consome)
type Pinger interface {
    PingContext(ctx context.Context) error
}

O *sql.DB real já satisfaz essa interface (tem o método PingContext). E nos testes, criamos um fake de duas linhas:

type fakePinger struct{ err error }
func (f fakePinger) PingContext(context.Context) error { return f.err }
Por que isso é poderoso? Com o fake, testamos os dois caminhos do /readyz sem nenhum banco: fakePinger{} (sem erro) → 200 "up"; fakePinger{err: ...} → 503 "down". É o princípio "aceite interfaces, retorne structs" do SPEC-001, agora aplicado à persistência. A mesma costura (o "port") que vai deixar cada feature plugar seu repositório depois.

6. O bug que os testes pegaram 🐛

Esse é o momento mais didático da aula. Na primeira vez que rodamos os testes de integração, eles travaram (ficaram pendurados pra sempre). O banco mostrava que as migrations tinham funcionado (versão 1, dirty=false) — então o que estava travando?

O diagnóstico

Cada chamada de migration (MigrateUp, MigrateDown, MigrationVersion) pegava uma conexão emprestada do pool compartilhado e nunca devolvia (eu tinha evitado de propósito fechar o "migrator", porque fechá-lo fecharia o pool inteiro). O teste de ida-e-volta faz 8 chamadas, mas o pool tinha MaxOpenConns = 5...

Chamada 1 (up) → pega conexão 1 (não devolve) Chamada 2 (version) → pega conexão 2 (não devolve) Chamada 3 (down) → pega conexão 3 (não devolve) Chamada 4 (version) → pega conexão 4 (não devolve) Chamada 5 (up) → pega conexão 5 (não devolve) ← pool esgotado (5/5) Chamada 6 (version) → espera uma conexão livre... → TRAVA PRA SEMPRE
Por que o teste manual da Fase 3 não pegou isso? Lá, cada task migrate:up era um processo separado que fazia uma operação e morria — abria 1 conexão, usava, e o processo terminava. Nunca acumulava. O bug só aparece quando várias operações rodam no mesmo processo, compartilhando o mesmo pool. Foi exatamente o teste de integração que criou essa condição.

A correção

Migrations passaram a usar uma conexão dedicada e de vida curta, separada do pool que o servidor HTTP usa. Cada operação abre sua conexão, faz o trabalho, e fecha tudo no fim. Sem disputa com o pool de requisições, sem vazamento.

func withMigrator(databaseURL, fn) error {
    db, _ := sql.Open("pgx", databaseURL)   // conexão DEDICADA, só pra esta operação
    // ... monta o migrator ...
    defer m.Close()                         // fecha o migrator E essa conexão dedicada
    return fn(m)
}
A lição Um teste unitário com um fake jamais teria pego isso — o problema era a interação real com o pool de conexões sob múltiplas chamadas. É por isso que testes de integração contra um banco de verdade valem o esforço: eles exercitam o comportamento que mocks escondem. Migrations e o "pool de servir" são preocupações diferentes e merecem conexões diferentes.

7. Ferramentas e o ecossistema

FerramentaPra que serve
golang-migrateAplica/desfaz migrations SQL versionadas; registra a versão aplicada numa tabela schema_migrations.
Docker Composehealthcheck + depends_on: service_healthy garantem ordem de subida: o banco fica pronto antes da api.
DBeaverCliente gráfico pra inspecionar o banco. Conecte em localhost:5433 (o banco do projeto), não 5432 (o nativo).
pgcryptoExtensão do Postgres que fornece gen_random_uuid() pras chaves primárias UUID.
5432 vs 5433 — não confunda! Você tem dois Postgres na máquina: o nativo (instalação Windows, porta 5432) e o do projeto (container Docker, porta 5433). Use sempre 5433 pro YieldForge. Se apontar pra 5432 sem querer, você fala com o banco errado e fica procurando tabelas que não estão lá.

O fluxo completo, do zero

cp .env.example .env        # DATABASE_URL já aponta pra localhost:5433
task docker-up              # sobe api (8080) + postgres (5433)
task migrate:up             # aplica as migrations (noutro terminal)
curl localhost:8080/readyz  # {"status":"ready","checks":{"db":"up"}}
