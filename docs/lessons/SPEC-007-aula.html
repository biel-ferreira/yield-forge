SPEC-007 — Ingestão de Tickers Guiada pela Carteira (Holdings-Driven Ticker Ingestion)

Uma spec pequena, cirúrgica, e nascida de um bug real: um usuário com 10 FIIs na carteira só via cotação para 1. O dashboard não tinha bug nenhum — ele calculava corretamente sobre dados incompletos. A causa raiz era um passo atrás: o worker de ingestão (SPEC-006) só sabia buscar cotação dos tickers de uma lista estática, configurada à mão, e essa lista estava vazia. O SPEC-007 ensina o worker a perguntar "o que os usuários realmente têm?" — direto na tabela de holdings — em vez de depender de alguém lembrar de manter uma lista em dia.

1. O incidente real: 10 FIIs na carteira, 1 cotação no banco

O SPEC-006 (aula anterior) entregou o port MarketDataProvider e um worker de ingestão que busca preço, dividend yield, P/VP e setor de cada FII e guarda como referência "último-dado-bom" em fii_quotes. Funcionou lindamente nos testes. Mas uma investigação real neste ambiente — não hipotética — revelou um problema: um usuário com 10 FIIs cadastrados na carteira (SPEC-102) tinha uma linha em fii_quotes para apenas 1 deles. Os outros 9 nunca tinham sido buscados, nenhuma vez, desde que o worker existia.

A pergunta certa não é "por que o worker falhou?" — ele não falhou. É "o worker sabia, sequer, que esses 9 tickers existiam?" A resposta era não. E a razão é a decisão D3 que o próprio SPEC-006 já tinha documentado como provisória:

A dívida técnica que o SPEC-006 já sabia que tinha (sua própria decisão D3) Quando o SPEC-006 foi escrito, o SPEC-102 (holdings) ainda não existia — não havia carteira nenhuma para consultar. A solução do MVP foi um port TickerSource com um único adaptador: Watchlist, construído a partir da variável de ambiente MARKETDATA_WATCHLIST. O próprio comentário de domínio já avisava: "Configured MARKETDATA_WATCHLIST now via TickerSource; swap to a holdings-backed source when SPEC-102 lands." Isto não era um bug — era uma decisão consciente e documentada de adiar o problema. O SPEC-007 é o dia em que essa dívida vence.

2. A causa raiz: a TickerSource era uma lista estática

Neste ambiente de desenvolvimento, MARKETDATA_WATCHLIST estava simplesmente vazia — ninguém tinha configurado nada. Mas mesmo se estivesse preenchida, o problema de fundo seria o mesmo: uma lista mantida à mão nunca vai acompanhar o que os usuários realmente compram. É um "detalhe de operação" que se tornou um "buraco silencioso de dados" — o pior tipo de bug, porque nenhum log grita erro; o worker simplesmente nunca soube que precisava buscar aqueles tickers.

ANTES (SPEC-006, D3 provisória) DEPOIS (SPEC-007) TickerSource = Watchlist (MARKETDATA_WATCHLIST) TickerSource = union(holdings, watchlist) Fonte de verdade: uma env var mantida à mão Fonte de verdade: a carteira REAL dos usuários Vazia? Worker não sabe de nenhum ticker Vazia? Ainda funciona — holdings bastam Usuário compra HGLG11 → worker nunca fica sabendo Usuário compra HGLG11 → aparece no próximo run 1 de 10 FIIs cotados (o incidente real) 10 de 10 FIIs cotados

Repare que o problema não estava em nenhuma linha de worker.go — o worker sempre foi correto: ele chama tickers.Tickers(ctx) e busca cotação para o que vier. O problema estava inteiramente em o que alimentava essa chamada. É uma lição de diagnóstico importante: antes de suspeitar do código que processa os dados, vale perguntar de onde os dados vieram.

3. Como o dashboard "mentiu" sem mentir

O efeito mais interessante do incidente é que ele não apareceu como um erro. O dashboard (SPEC-103) nunca lançou uma exceção, nunca mostrou um "N/A" óbvio. Ele mostrou números plausíveis, coerentes internamente, e completamente enganosos — porque ele calcula de forma fiel sobre os dados que recebe, e os dados que recebia estavam incompletos. Três sintomas, todos rastreáveis à mesma linha em internal/dashboard/compute.go:

// internal/dashboard/compute.go
for _, h := range holdings.FII {
    qty := int64(h.Quantity.Value())
    invested := h.AveragePriceCentavos * qty
    totalInvested += invested

    var current int64
    var sector marketdata.Sector
    if q, ok := quotes[h.Ticker]; ok {
        current = q.PriceCentavos * qty
        sector = q.Sector
        monthlyIncome += q.LastDividendCentavos * qty
    } else {
        current = invested            // fallback: valor atual = valor investido
        sector = marketdata.SectorOther
        stale = append(stale, h.Ticker.String())
    }
    fiiCurrent += current
    sectorValue[sector] += current
}

Renda passiva mensal subestimadaUm FII sem cotação (quotes[h.Ticker] ausente) nunca soma nada em monthlyIncome — contribui exatamente zero, mesmo que o usuário receba dividendo dele todo mês.
"Valorização" concentrada em 1 tickerSem cotação, current = invested — o fallback de custo-base. Ganho/perda só aparece para o único FII precificado; os outros 9 mostram valorização zero por construção.

Por que isto NÃO é um bug do dashboard — e por que essa distinção importa É tentador "consertar" o sintoma direto no compute.go. Mas o dashboard está fazendo exatamente o que deveria: quando falta um dado, ele aplica um fallback explícito e sinalizado (a lista stale, que vira o aviso de degradação visível no SPEC-212) em vez de inventar um número. O bug não está em como o dashboard reage à ausência de cotação — está em por que a cotação está ausente em primeiro lugar. Corrigir o sintoma teria escondido a causa; o SPEC-007 ataca a causa, uma camada acima, na ingestão.

4. A correção: SystemReader + holdingsSource

A ideia central é simples de enunciar: em vez de perguntar "o que está configurado?", o worker passa a perguntar "o que as pessoas realmente têm?" — uma consulta direta à tabela fii_holdings (já existente desde o SPEC-102, sem nenhuma migration nova). Mas essa pergunta cruza uma fronteira delicada: hoje toda leitura de holdings no projeto é por usuário (WHERE user_id = $1, SPEC-003). Esta leitura precisa ser o oposto — todos os usuários, de uma vez — porque o worker não age em nome de ninguém específico.

Por isso o SPEC-007 não reaproveita portfolio.Repository nem portfolio.Reader (os ports por-usuário existentes). Ele cria um port novo e deliberadamente separado:

// internal/portfolio/ports.go
// SystemReader is a system-scoped read, deliberately separate from Repository/Reader (SPEC-007
// FR-071, BR-071): it is made on behalf of an internal worker, not an authenticated user, so the
// identity-from-context / WHERE user_id = $1 rule does not apply — there is no request user to
// scope to. It returns only public B3 tickers, never user-identifying data, and is reachable only
// from the market-data ingestion edge, never from an HTTP handler.
type SystemReader interface {
    // DistinctFIITickers returns the distinct FII tickers held across ALL users, as raw strings —
    // parsing into marketdata.Ticker happens at the ingestion edge (SPEC-007 FR-071 AC3), keeping
    // this package free of marketdata's ticker semantics.
    DistinctFIITickers(ctx context.Context) ([]string, error)
}
Por que não é só "adicionar um método no Repository que já existe"? Porque nomear as coisas importa. Se DistinctFIITickers(ctx) vivesse dentro de Repository, ao lado de métodos como ListFIIHoldingsByUserID(ctx, userID), o contrato ficaria ambíguo: um dev futuro poderia assumir — razoavelmente — que todo método daquela interface é escopado por usuário, e vazar dado entre contas por engano em algum outro ponto de chamada. Um port separado, com um nome que já diz "isto é diferente" (SystemReader) e um comentário que explica o porquê, torna o contrato honesto por construção. É a mesma filosofia de "parse, don't validate" aplicada a arquitetura: a forma do tipo impede o uso errado, em vez de depender de disciplina.

A implementação do lado Postgres é uma única query, sem parâmetro nenhum — o próprio ausência de um $1 é a prova visual de que isto não é escopado:

// internal/portfolio/postgres/postgres.go
// DistinctFIITickers returns the distinct FII tickers held across ALL users (SPEC-007 FR-071,
// BR-071) — a system read, deliberately parameterless and unscoped, never WHERE user_id = $1.
// An empty result is a valid empty slice, not a "not found" error.
func (r Repository) DistinctFIITickers(ctx context.Context) ([]string, error) {
    const q = `SELECT DISTINCT ticker FROM fii_holdings ORDER BY ticker`
    rows, err := r.db.QueryContext(ctx, q)
    if err != nil {
        return nil, fmt.Errorf("list distinct fii tickers: %w", err)
    }
    defer rows.Close()
    var out []string
    for rows.Next() {
        var ticker string
        if err := rows.Scan(&ticker); err != nil {
            return nil, fmt.Errorf("list distinct fii tickers: %w", err)
        }
        out = append(out, ticker)
    }
    if err := rows.Err(); err != nil {
        return nil, fmt.Errorf("list distinct fii tickers: %w", err)
    }
    return out, nil
}

Do lado da ingestão, um adaptador novo consome esse port e o transforma exatamente na forma que o worker já sabia esperar — marketdata.TickerSource. Repare como cada string crua vira um marketdata.Ticker validado, e como um valor malformado nunca aborta a rodada:

// internal/marketdata/ingest/holdings_source.go
// holdingsSource is a marketdata.TickerSource backed by users' actual FII holdings (SPEC-007
// FR-072) — the primary ticker source, replacing the static MARKETDATA_WATCHLIST.
type holdingsSource struct {
    reader tickerReader
    logger *slog.Logger
}

func (h holdingsSource) Tickers(ctx context.Context) ([]marketdata.Ticker, error) {
    raw, err := h.reader.DistinctFIITickers(ctx)
    if err != nil {
        return nil, fmt.Errorf("holdings ticker source: %w", err)
    }
    tickers := make([]marketdata.Ticker, 0, len(raw))
    for _, s := range raw {
        t, err := marketdata.ParseTicker(s)
        if err != nil {
            h.logger.Warn("market data: skipping malformed stored ticker",
                slog.String("ticker", s), slog.String("error", err.Error()))
            continue    // defensivo: holdings já validam o ticker na escrita (SPEC-102)
        }
        tickers = append(tickers, t)
    }
    return tickers, nil
}
O detalhe pequeno que revela disciplina hexagonal: tickerReader holdingsSource não depende de portfolio.SystemReader — a interface concreta do outro pacote. Ele declara sua própria interface mínima, tickerReader, com apenas o método que precisa. Como Go faz structural typing, qualquer tipo que tenha DistinctFIITickers(ctx) ([]string, error) — inclusive portfolio.postgres.Repository — satisfaz essa interface automaticamente, sem import explícito de portfolio pelo core de marketdata. É "accept interfaces, return structs" levado ao limite: até a fronteira entre duas features vira uma interface pequena, definida por quem consome, não por quem produz.

5. A união graciosa: unionSource

Sozinho, o holdingsSource já resolveria o incidente. Mas duas realidades continuam valendo: (1) ambientes locais/CI podem não ter nenhuma holding cadastrada ainda, e precisam de um jeito de "semear" tickers para testar a ingestão; e (2) qualquer .env que já configura MARKETDATA_WATCHLIST hoje não pode simplesmente parar de funcionar. A solução (decisão D3 do spec) é união, não substituição: o conjunto efetivo de tickers é distinct(holdings ∪ watchlist), calculado por um terceiro tipo pequeno:

// internal/marketdata/ingest/composite_source.go
// unionSource composes N marketdata.TickerSources into the deduplicated union of their output
// (SPEC-007 FR-073). Each underlying source degrades independently: a failing source is logged
// and skipped, never aborting the others, so a holdings-read failure still yields the watchlist
// seed and vice-versa (BR-074). Only a total failure (every source errors) surfaces as an error.
func (u unionSource) Tickers(ctx context.Context) ([]marketdata.Ticker, error) {
    seen := make(map[string]marketdata.Ticker)
    var errs []error
    okCount := 0
    for _, s := range u.sources {
        tickers, err := s.Tickers(ctx)
        if err != nil {
            u.logger.Warn("market data: a ticker source failed; continuing with the others",
                slog.String("error", err.Error()))
            errs = append(errs, err)
            continue                       // uma fonte ruim não derruba as outras
        }
        okCount++
        for _, t := range tickers {
            seen[t.String()] = t           // dedupe por chave textual do ticker
        }
    }
    if okCount == 0 && len(u.sources) > 0 {
        return nil, fmt.Errorf("union ticker source: all sources failed: %w", errors.Join(errs...))
    }
    out := make([]marketdata.Ticker, 0, len(seen))
    for _, t := range seen {
        out = append(out, t)
    }
    sort.Slice(out, func(i, j int) bool { return out[i].String() < out[j].String() })
    return out, nil
}

E a fiação em factory.go é onde as duas fontes viram uma só, exatamente do jeito que o worker já esperava — sem mudar worker.go uma linha sequer:

// internal/marketdata/ingest/factory.go
func New(cfg config.Config, db *sql.DB, logger *slog.Logger, clk clock.Clock) (*Worker, error) {
    watchlist, err := marketdata.NewWatchlist(cfg.MarketDataWatchlist)
    if err != nil {
        return nil, fmt.Errorf("market data watchlist: %w", err)   // erro de config falha rápido
    }
    holdings := newHoldingsSource(portfoliopostgres.New(db), logger)
    tickers := newUnionSource(logger, holdings, watchlist)          // union: holdings ∪ watchlist

    return newWorker(
        buildProvider(cfg, clk), cfg.MarketDataProvider,
        mdpostgres.NewFIIQuoteRepository(db), mdpostgres.NewMacroRepository(db),
        tickers, clk, logger,
    ), nil
}
Por que degradação "por fonte" e não "tudo ou nada"? Pense nos dois jeitos de falhar: um blip momentâneo no banco derruba a leitura de holdings; ou o operador digitou um ticker inválido na watchlist (esse último já falha no startup, antes de qualquer rodada — comportamento herdado do SPEC-006 FR-604, inalterado). Se a união fosse tudo-ou-nada, um blip de banco apagaria também os tickers da watchlist, mesmo que ela estivesse perfeitamente saudável. Fazendo cada fonte degradar independentemente — logando o erro, mas seguindo com o que sobrou — a rodada continua entregando o máximo de valor possível, na mesma filosofia de "último-dado-bom, isolamento por item" que já rege todo o resto do worker desde o SPEC-006 (BR-602).

O único ponto onde worker.go ganhou uma linha (não uma mudança de comportamento, só observabilidade) foi um log do tamanho do conjunto resolvido, para um operador ver, sem consultar o banco, quantos tickers a ingestão aprendeu:

// internal/marketdata/ingest/worker.go
tickers, err := w.tickers.Tickers(ctx)
if err != nil {
    w.logger.Error("market data: ticker source failed", slog.String("error", err.Error()))
    w.recordItem(ctx, "fii", "source_error")   // mesmo caminho de degradação do SPEC-006
    sum.FIIFailed++
    return
}
if len(tickers) == 0 {
    return // macro-only run
}
// Resolved ticker count (SPEC-007 FR-077/observability): no user_id, no PII — just a count.
w.logger.Info("market data: resolved ticker set", slog.Int("count", len(tickers)))

6. As 5 decisões (D1–D5) e por quê

#DecisãoEscolhaPor quê
D1Onde vive o adaptador holdings→ticker? internal/marketdata/ingest — a borda de composição que já importa tanto marketdata quanto o adaptador Postgres de portfolio Mantém os dois cores livres de import um do outro (BR-072) — é o único lugar do projeto onde as duas features legitimamente se encontram.
D2Método novo no Repository ou um port separado? portfolio.SystemReader separado O Repository/Reader existentes são todos por-usuário; misturar uma leitura sem escopo ali borraria o contrato de identidade (BR-071).
D3Substituir, complementar ou semear a watchlist? Semear/complementar — distinct(holdings ∪ watchlist) Preserva retrocompatibilidade de .env e dá ao dev/CI um jeito de testar ingestão antes de existir qualquer holding, a custo zero (FR-073).
D4Cachear o conjunto distinto entre rodadas? Não Um DISTINCT por rodada é trivial no volume do MVP; cache seria otimização prematura (FR-075). Registrado como pergunta em aberto, não construído agora.
D5Adicionar índice em fii_holdings.ticker? Não agora Mantém a mudança livre de migration; a tabela é pequena no MVP. Revisitar se o DISTINCT aparecer em estatísticas de query no futuro.

7. Como testar/rodar esta spec

Assim como o resto do projeto, os testes são table-driven, com testify/require e fakes escritos à mão (nunca gomock/mockery) para os ports — sem rede, sem banco, por padrão:

Rodar localmente segue os comandos padrão do projeto:

task vet              # go vet ./...

task test:integration # integration tests, precisa de TEST_DATABASE_URL
task run              # sobe a API; com MARKETDATA_WATCHLIST vazio, a ingestão já é holdings-driven
Como verificar que o incidente está mesmo resolvido Rode task run com MARKETDATA_WATCHLIST vazio, cadastre alguns FIIs pela API de holdings (SPEC-102), e observe a próxima rodada de ingestão no log: market data: resolved ticker set count=N. N deve bater com o número de tickers distintos que você cadastrou — não zero, mesmo sem watchlist nenhuma configurada. É a prova de ponta a ponta de que o caso "10 de 10", não "1 de 10", agora é o comportamento padrão.

8. Ponte: arquitetura em camadas → hexagonal

Você já conhece a pilha clássica em camadas: controller → DTO → service → repository. A melhor forma de internalizar ports & adapters neste projeto é por tradução. (Para a teoria completa, veja a aula conceitual de arquitetura hexagonal; aqui aplicamos ela ao SPEC-007.) Esta spec não tem service novo nem entidade nova — é quase puramente "port + adapter", o que a torna um exemplo bem limpo do padrão em isolamento.

A "grande virada": a inversão de dependência

Em camadas, um service importaria diretamente o repositório concreto da outra feature — a seta aponta para fora. No hexagonal, o core define a interface e o adapter a implementa — a seta aponta para dentro. Aqui isso acontece duas vezes em sequência, uma coisa rara o bastante para valer a pena visualizar:

// 1ª virada: o CORE de portfolio define o port (internal/portfolio/ports.go)
type SystemReader interface {
    DistinctFIITickers(ctx context.Context) ([]string, error)
}
// ... e a BORDA de portfolio o implementa, provado em tempo de compilação:
var _ portfolio.SystemReader = Repository{}    // internal/portfolio/postgres/postgres.go

// 2ª virada: a BORDA de marketdata (ingest) declara seu PRÓPRIO port de consumo mínimo
type tickerReader interface {
    DistinctFIITickers(ctx context.Context) ([]string, error)
}
// ... satisfeito estruturalmente pelo MESMO Repository acima — sem import nomeado de portfolio
// pelo core de marketdata; só o pacote ingest (a borda de composição) conhece os dois lados.

Note a assimetria interessante: a primeira interface (SystemReader) é definida por quem produz o dado (o core de portfolio, seguindo o padrão usual de repositório). A segunda (tickerReader) é definida por quem consome (o pacote ingest), a aplicação mais estrita do princípio "accept interfaces, return structs" da casa. Nem marketdata nem portfolio jamais importam um ao outro diretamente — só a borda de composição (internal/marketdata/ingest) tem os dois imports, exatamente como já acontecia no SPEC-006 para MarketDataProvider.

Driving vs. driven nesta spec

  • Driving port (quem aciona): inalterado desde o SPEC-006 — cmd/ingest e o Scheduler continuam chamando Worker.RunOnce. O SPEC-007 não toca essa borda.
  • Driven port novo: marketdata.TickerSource — que já existia — ganha uma implementação nova e melhor (unionSource sobre holdingsSource + Watchlist), sem que a interface em si mude uma linha.
  • Domínio puro: marketdata.Ticker, portfolio.SystemReader (a interface). Borda: holdings_source.go, composite_source.go, o método Postgres — todos em pacotes de infraestrutura, nunca no core.
┌── HEXÁGONO de marketdata (core puro) ──┐ ┌── HEXÁGONO de portfolio (core puro) ──┐ cmd/ingest ▶ │ Worker.RunOnce │ │ SystemReader (interface, port) │ Scheduler ▶ │ depende de TickerSource (interface) │ │ DistinctFIITickers(ctx) │ (driving) └──────────────────────────────────────────┘ └─────────────────────────────────────────┘ ▲ ▲ │ implementa │ implementa ┌──────────────┴──────────────────────────────────────────────┴───────────────┐ │ internal/marketdata/ingest (a ÚNICA borda que conhece os dois) │ │ unionSource{ holdingsSource{ tickerReader ◀── portfolio.postgres.Repository },│ │ marketdata.Watchlist } │ └───────────────────────────────────────────────────────────────────────────────┘ A seta de dependência sempre aponta PARA DENTRO de cada hexágono; só a borda de composição cruza os dois.

9. Harness Engineering: como a máquina construiu e verificou isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. O SPEC-007 é um bom estudo de caso justamente por ser pequeno: dá para ver os trilhos agindo sem o ruído de uma feature grande.

O fluxo /spec-implement, fase a fase

O PLAN-007 fatiou o trabalho em 7 fases — domínio (o port) → persistência (o método Postgres) → aplicação (os dois adaptadores de ingestão + a fiação) → API (N/A, explicitamente marcada assim) → observabilidade → testes → documentação — a mesma cadência bottom-up de todas as specs anteriores. Repare que a Fase 4 (API) não foi pulada silenciosamente: o próprio plano registra "N/A. SPEC-007 adds no HTTP endpoint...", deixando explícito que a ausência foi uma decisão, não um esquecimento.

O subagente hexagonal-reviewer

Esta spec existe inteira por causa de uma regra que o hexagonal-reviewer protege em toda spec: nenhum core importa outro core. A tentação óbvia, ao precisar de dados de portfolio dentro de marketdata, seria simplesmente importar internal/portfolio de dentro de internal/marketdata/ports.go. O revisor barra isso — e é exatamente por causa dessa restrição que o design ficou com dois ports pequenos (SystemReader do lado de portfolio, tickerReader do lado de marketdata/ingest) em vez de um import direto. A prova mecânica de que a fronteira ficou intacta: nenhum arquivo em internal/marketdata (fora do subpacote ingest) tem um import de internal/portfolio, e vice-versa.

A prova de que "no core-to-core import" realmente segurou A asserção var _ portfolio.SystemReader = Repository{} em internal/portfolio/postgres/postgres.go faz o compilador provar que o adapter satisfaz o port — não é uma promessa em comentário, é uma linha que quebra o go build se o contrato mudar. Combinada com a interface tickerReader definida dentro de internal/marketdata/ingest (satisfeita estruturalmente, sem import nomeado), qualquer tentativa de "atalho" — importar portfolio direto do core de marketdata — teria saltado aos olhos de um revisor especializado em arquitetura antes mesmo de chegar a testes.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — os três arquivos novos (holdings_source.go, composite_source.go e o método em postgres.go) não geram ruído de estilo no diff.
  • Bloqueio de imutabilidade: o SPEC-007 referencia ADR-0002/ADR-0003 mas não os edita — resolve uma decisão provisória (D3) do SPEC-006 sem reabrir o SPEC-006 em si, que permanece Done e imutável em seu conteúdo original.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — e os testes precisam passar sem rede e sem banco por padrão. Foi esse backstop que provou que o novo caminho holdings-driven não introduziu nenhuma dependência acidental de I/O fora dos testes de integração gated.
  • O hook block-dev-db-test.ps1: nasceu de um near-miss real durante este mesmo trabalho, e merece a seção própria abaixo.
O quase-incidente: quando o próprio teste quase repetiu o problema que estava consertando Durante a implementação, o teste de integração da Fase 6 precisa de dados limpos para semear os dois usuários com ticker sobreposto (o cenário da tabela de testes acima) — e a forma padrão de garantir isso é um TRUNCATE no setup. Em algum momento, TEST_DATABASE_URL acabou apontando para o Postgres de desenvolvimento (porta 5433) em vez de uma instância descartável — e o TRUNCATE apagou dados reais, semeados à mão, deste mesmo ambiente. Isso já tinha acontecido uma vez antes no projeto, e a resposta institucional foi endurecer a regra: o CLAUDE.md agora exige explicitamente uma instância descartável (porta 5434, via docker run --rm) para TEST_DATABASE_URL, e um hook PreToolUse — .claude/hooks/block-dev-db-test.ps1 — passou a bloquear mecanicamente qualquer execução de teste apontada para a porta/DB de desenvolvimento. A lição não é "a IA errou uma vez"; é que engenharia cuidadosa também tem quase-acidentes, e a resposta madura não é confiar mais em disciplina — é transformar a lição em um trilho que não depende de ninguém (humano ou IA) lembrar da regra da próxima vez.

10. Conexão com AI Engineering

O SPEC-007 não é uma feature de IA — não há prompt, LLM, ou Insighter em lugar nenhum deste trabalho. Mas ele fortalece exatamente a fundação sobre a qual a IA do produto se apoia, e vale conectar os pontos.

Conceito de AI EngineeringComo o SPEC-007 o reforça
Grounding / completude do fato O Fact Builder (SPEC-104) monta os Facts que o Insighter recebe a partir de fii_quotes + a carteira do usuário. Um fato "grounded" mas incompleto (9 de 10 FIIs sem cotação) ainda produziria um insight tecnicamente correto sobre dados errados — o equivalente de IA a "garbage in, garbage out". O SPEC-007 fecha esse buraco na origem, antes de qualquer LLM entrar na conversa.
O mesmo bug, em qualquer camada que consuma dado incompleto O incidente apareceu no dashboard, mas teria aparecido idêntico no Health Score (SPEC-106), nas Projeções (SPEC-107) e em qualquer resposta do copiloto conversacional (SPEC-108) sobre a carteira — todos leem os mesmos fii_quotes. Consertar na ingestão corrige todos os consumidores de uma vez, em vez de remendar cada consumidor individualmente.
O seam de "system read" como precedente O portfolio.SystemReader é o primeiro port do projeto deliberadamente sem escopo por usuário fora do próprio marketdata (que já era global). É um precedente útil para a futura arquitetura multi-agente CIO: alguns agentes (um agendador de tarefas, um worker de manutenção) vão legitimamente precisar de leituras cross-usuário — este spec mostra o padrão correto de como nomear e isolar esse tipo de acesso sem comprometer o isolamento por usuário do resto do sistema.
A ideia que conecta tudo A regra binding nº 3 do PRD — "fatos são computados, não gerados" — pressupõe que os fatos existem antes de serem computados. O SPEC-007 não muda como um fato é computado; garante que o fato chegue a existir para todo ticker que importa, sem depender de configuração manual. É infraestrutura invisível, mas é o tipo de infraestrutura sem a qual toda promessa de explicabilidade (FR-013) do produto ficaria construída sobre dados parciais sem que ninguém percebesse.
