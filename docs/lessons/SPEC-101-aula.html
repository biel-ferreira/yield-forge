1. A visão geral: por que um perfil

As specs de fundação (tier 00x) montaram a casa: esqueleto (001), persistência (002), identidade (003), observabilidade (004), o port Insighter da IA (005) e a ingestão de dados de mercado (006). Nenhuma delas era uma feature de usuário — eram cofre, fechadura e encanamento. O SPEC-101 é a primeira pedra do tier 1xx: a primeira coisa que um investidor de verdade cria e vê, e a primeira a consumir a costura de identidade que o SPEC-003 deixou pronta.

O que ele entrega é deliberadamente pequeno: o pacote internal/profile com o domínio Profile e seus value objects (RiskProfile, Objective, Horizon), o serviço, dois ports, o adaptador Postgres, a migration 0004_profiles e dois endpoints — GET /profile e PUT /profile. Nada analisa esse perfil ainda. O SPEC-101 guarda o registro e expõe um port de leitura; as features que raciocinam sobre ele (Insight Engine SPEC-104, Rebalanceamento SPEC-105, Health Score SPEC-106) leem por esse port mais tarde.

ANTES (SPEC-006) DEPOIS (SPEC-101) só fundação — nada que o usuário cria a 1ª feature de usuário: perfil de risco + objetivos + horizonte a costura auth.UserID(ctx) provada, o 1º consumidor real dela: WHERE user_id = $1 com o ID do contexto mas sem nenhuma feature usando análise da IA seria genérica o INPUT determinístico que torna a análise orientada a objetivo sem tabela escopada por usuário migration 0004: profiles, PK user_id, FK ON DELETE CASCADE nenhum port de leitura de perfil ProfileReader — o seam por onde SPEC-104/105/106 vão ler

Os endpoints novos:

PUT /profile  {"risk_profile","objectives":[...],"horizon_years"}  →  200 {perfil} | 400 | 401
GET /profile  (autenticado)                                        →  200 {perfil} | 404 | 401
Por que o perfil é a feature mais importante de chegar primeiro? Porque a premissa inteira do YieldForge (PRD §4) é análise orientada a objetivo: um insight precisa considerar quem é o investidor, não só o que ele tem na carteira. Dizer "seu FII X subiu 3%" é genérico; dizer "isto reforça sua meta de renda passiva dentro do seu horizonte de 10 anos" só é possível se existir um registro estável do perfil. Sem ele, a IA só consegue produzir conselho genérico e sem chão — exatamente o que o produto promete não ser.

2. O problema: saber quem é o investidor

O SPEC-101 resolve uma tensão simples de enunciar e fácil de errar: precisamos guardar três fatos sobre o investidor, de forma que (a) sejam só dele, (b) sejam válidos por construção, e (c) outras features possam lê-los sem se acoplar a HTTP ou SQL. São três campos só:

CampoO que éForma escolhida
Perfil de riscoTolerância a risco do investidorEnum fechado: conservative | moderate | aggressive
ObjetivosMetas financeiras (uma ou mais)Enum fechado de 4 valores; um conjunto não-vazio e deduplicado
HorizontePor quantos anos pretende investirInteiro de anos, limitado a 1–50 (os 5/10/20 do PRD são exemplos)

Repare que isto é um registro um-para-um com o usuário: cada pessoa tem exatamente um perfil. Essa é a primeira diferença grande em relação ao SPEC-006 — lá o dado era global (o preço do HGLG11 é o mesmo para todos, sem user_id em lugar nenhum). Aqui o dado é privado: o perfil de A nunca pode ser visto ou sobrescrito por B. É por isso que a aula inteira gira em torno do isolamento por usuário — e que a costura auth.UserID(ctx) do SPEC-003 finalmente sai do papel.

3. A LIÇÃO PRINCIPAL — identidade vem do contexto, nunca do payload (BR-1012)

Se você levar uma única ideia desta aula, leve esta. Quando uma rota lida com dado privado por usuário, a pergunta "de quem é esse dado?" tem uma resposta correta e uma catastrófica:

  • Correta: o dono vem da sessão autenticada — auth.UserID(ctx), um valor que o middleware do SPEC-003 colocou no context e que o cliente não controla.
  • Catastrófica: o dono vem de um campo user_id enviado pelo cliente no corpo ou na query. Aí qualquer pessoa pode mandar {"user_id":"id-do-vizinho"} e ler ou sobrescrever o perfil de outra. É a vulnerabilidade clássica de IDOR (referência direta e insegura a objeto).

O SPEC-101 não pede que você lembre de fazer certo — ele torna o errado estruturalmente impossível, com quatro travas empilhadas. Vamos ver cada uma no código real.

Trava 1 — o DTO do PUT não tem campo user_id

O corpo da requisição só carrega dado, jamais identidade. Não existe onde colocar um user_id mesmo que você quisesse:

// internal/transport/http/profile.go
// profileRequest is the PUT body. There is deliberately NO user_id field — identity comes
// from the authenticated session (BR-1012), and decodeJSON's DisallowUnknownFields rejects a
// stray user_id, so a client cannot supply an identity.
type profileRequest struct {
    RiskProfile  string   `json:"risk_profile"`
    Objectives   []string `json:"objectives"`
    HorizonYears int      `json:"horizon_years"`
}

Trava 2 — DisallowUnknownFields rejeita um user_id contrabandeado

"Mas e se eu mandar user_id mesmo assim?" O decoder de JSON está configurado para recusar qualquer campo desconhecido. Um corpo com user_id não é silenciosamente ignorado — ele faz o PUT inteiro virar 400:

// internal/transport/http/response.go
func decodeJSON(w http.ResponseWriter, r *http.Request, dst any) error {
    r.Body = http.MaxBytesReader(w, r.Body, maxBodyBytes)
    dec := json.NewDecoder(r.Body)
    dec.DisallowUnknownFields()   // ← um user_id intruso vira erro de decode
    return dec.Decode(dst)
}

Trava 3 — o handler pega o ID do contexto e o passa ao serviço

A identidade entra na lógica de aplicação por um único caminho: o valor que o middleware resolveu da sessão. Se não houver sessão, é 401 antes de tocar no serviço:

// internal/transport/http/profile.go — putProfile
id, ok := auth.UserID(r.Context())     // ← do contexto, nunca do corpo
if !ok {
    writeError(w, http.StatusUnauthorized, "authentication required")
    return
}
// ...decodifica o corpo (só dado)...
p, err := h.service.SetProfile(r.Context(), id, profile.SetProfileInput{  // id passado à parte
    RiskProfile:  req.RiskProfile,
    Objectives:   req.Objectives,
    HorizonYears: req.HorizonYears,
})

E o próprio SetProfileInput deixa isso explícito no comentário: o userID é fornecido separadamente, vindo sempre do contexto autenticado, nunca desse struct.

Trava 4 — toda query é escopada WHERE user_id = $1

Por fim, no adaptador, leitura e escrita só tocam a linha daquele usuário. A escrita usa user_id como conflito do upsert; a leitura filtra por ele:

// internal/profile/postgres/postgres.go — GetProfileByUserID
SELECT user_id::text, risk_profile, objectives, horizon_years, created_at, updated_at
FROM profiles WHERE user_id = $1::uuid
O contraste que fecha a ideia: corpo carrega dado, sessão carrega identidade Numa pilha em camadas mal-feita, é tentador um UpdateProfile(dto) onde o dto traz tudo — inclusive o dono. Aqui a separação é rígida: o corpo da requisição é só o conteúdo do perfil; o dono vem da sessão. Como o cliente controla o corpo mas não controla a sessão, o ataque de trocar de dono some por construção. E há um teste explícito que prova isso: ele manda {"user_id":"hacker",...} e verifica que o DisallowUnknownFields o rejeita — o vizinho está a salvo por design, não por disciplina.

4. Value objects parse-don't-validate

Os três campos do perfil não são string/int crus espalhados pelo código — são value objects que validam no construtor. A consequência é poderosa: um valor inválido não pode existir. Se você tem um RiskProfile em mãos, ele já é válido; não precisa checar de novo.

// internal/profile/riskprofile.go
func ParseRiskProfile(s string) (RiskProfile, error) {
    rp := RiskProfile(strings.ToLower(strings.TrimSpace(s)))   // normaliza
    if !validRiskProfiles[rp] {
        return "", fmt.Errorf("parse risk profile %q: %w", s, ErrInvalidRiskProfile)
    }
    return rp, nil
}

O Horizon vai além: ele esconde o inteiro num struct com campo privado, então a única forma de criar um é pelo construtor que checa os limites 1–50. Não dá nem para construir um horizonte fora da faixa:

// internal/profile/horizon.go
type Horizon struct{ years int }   // campo privado: ninguém constrói direto

func ParseHorizon(years int) (Horizon, error) {
    if years < MinHorizonYears || years > MaxHorizonYears {
        return Horizon{}, fmt.Errorf("parse horizon %d: %w (want %d-%d years)", years, ErrInvalidHorizon, MinHorizonYears, MaxHorizonYears)
    }
    return Horizon{years: years}, nil
}

E os objetivos têm uma regra de conjunto (BR-1014): pelo menos um, sem duplicatas, em ordem estável. O ParseObjectives faz tudo de uma vez — valida cada string, deduplica preservando a ordem da primeira aparição, e recusa um conjunto vazio com um sentinela próprio:

// internal/profile/objective.go
func ParseObjectives(raw []string) ([]Objective, error) {
    seen := make(map[Objective]bool, len(raw))
    out  := make([]Objective, 0, len(raw))
    for _, s := range raw {
        o, err := ParseObjective(s)
        if err != nil {
            return nil, err                 // valor desconhecido → ErrInvalidObjective
        }
        if !seen[o] {                       // deduplica, mantendo a ordem
            seen[o] = true
            out = append(out, o)
        }
    }
    if len(out) == 0 {
        return nil, ErrNoObjectives         // conjunto vazio → sentinela próprio
    }
    return out, nil
}
Por que value objects ganham de string/int crus? Com strings cruas, a validação fica espalhada: cada lugar que recebe um "perfil de risco" tem que lembrar de checar se é um dos três valores. Esqueça num lugar e um "banana" vaza para o banco. Com value objects, a validação mora num lugar só (o construtor) e o sistema de tipos carrega a garantia: uma função que recebe RiskProfile sabe que recebeu algo válido. O erro vira inrepresentável, não "improvável". Repare ainda que o adaptador Postgres, ao ler, passa os valores guardados de volta pelos mesmos construtores — então uma linha corrompida no banco vira um erro explícito, não um value object inválido circulando solto.

5. A decisão D1 (jsonb) e o upsert idempotente com RETURNING

5.1 — Por que jsonb para os objetivos (D1)

Os objetivos são uma lista ([]Objective). Guardar uma lista num banco relacional tem três caminhos, e cada um tem um custo:

OpçãoPrósContras
text[] (array nativo do Postgres)Relacional, indexávelAtrito de driver: database/sql + pgx não têm um scanner limpo de []string sem importar tipos específicos do pgx no adaptador.
Tabela-filha normalizada (profile_objectives)"Certo" pelo livro relacionalExagero para um conjunto fixo de 4 valores: mais uma tabela, mais um JOIN, mais SQL — sem ganho real.
jsonb (escolhido)Stdlib-limpo: json.Marshal para escrever, []byte + json.Unmarshal para ler. Ainda consultável (objectives @> '["retirement"]').Não é o modelo relacional "puro" — mas o custo é teórico aqui.

A escolha foi jsonb: limpa com a stack atual, sem dependência nova, e ainda dá para consultar dentro do JSON se um dia precisar. A escrita e a leitura são duas linhas honestas de stdlib:

// escrever (UpsertProfile):
objectives, err := json.Marshal(p.Objectives)        // []Objective → jsonb

// ler (GetProfileByUserID):
var rawObjectives []string
json.Unmarshal(objectivesJSON, &rawObjectives)        // jsonb → []string
objectives, _ := profile.ParseObjectives(rawObjectives)  // → []Objective revalidado
O trade-off honesto, registrado como dívida jsonb é a escolha certa para um conjunto fixo de 4 valores. Se um dia os objetivos crescerem muito, ou se precisarmos de consultas relacionais pesadas em cima deles (ex.: "quantos usuários têm renda passiva como meta?"), o caminho é graduar para uma tabela normalizada — e isso está rastreado explicitamente como dívida técnica no PLAN-101 §12. Hoje, normalizar seria complexidade sem retorno.

5.2 — O upsert idempotente: ON CONFLICT … RETURNING num só comando

O PUT é um create-or-update (D4 — substituição total): se não há perfil, cria; se há, substitui. Isso é um upsert, e o jeito Postgres de fazê-lo atômico é INSERT … ON CONFLICT … DO UPDATE. Mas há um detalhe lindo aqui — o RETURNING:

// internal/profile/postgres/postgres.go — UpsertProfile
INSERT INTO profiles (user_id, risk_profile, objectives, horizon_years, created_at, updated_at)
VALUES ($1::uuid, $2, $3, $4, $5, $6)
ON CONFLICT (user_id) DO UPDATE SET
    risk_profile  = EXCLUDED.risk_profile,
    objectives    = EXCLUDED.objectives,
    horizon_years = EXCLUDED.horizon_years,
    updated_at    = EXCLUDED.updated_at      -- created_at NÃO entra aqui: é preservado
RETURNING created_at, updated_at             -- a linha autoritativa, sem re-leitura

Três coisas acontecem num único comando atômico:

  1. Preserva created_at: no DO UPDATE, a coluna created_at não é tocada — então, numa atualização, ela continua refletindo a primeira criação. Só o updated_at avança.
  2. Idempotente: rodar o mesmo PUT de novo dá a mesma linha; não duplica (a chave é user_id, e há só um perfil por usuário — BR-1011).
  3. Devolve a verdade sem re-ler: o RETURNING created_at, updated_at traz os timestamps autoritativos que o banco gravou, no mesmo round-trip.
A versão anterior tinha uma janela de corrida — e o RETURNING a fechou Uma primeira tentativa fazia upsert e depois um SELECT separado para descobrir o created_at autoritativo. A revisão de correção apontou o problema: entre o upsert e o re-read, outra escrita do mesmo usuário poderia entrar, e o SELECT leria um estado diferente do que o upsert acabou de gravar — uma janela de corrida sutil. A correção foi colapsar tudo num só comando com RETURNING: a escrita já devolve a linha autoritativa, sem segundo round-trip e sem espaço para corrida. É um padrão SQL que vale guardar: quando você precisa do estado pós-escrita, peça-o ao próprio INSERT/UPDATE via RETURNING em vez de reler.

5.3 — Um perfil por usuário, FK com cascade

A migration 0004 não tem chave substituta (id surrogate): o user_id é a chave primária natural (D3), o que garante um-para-um no nível do banco. E a FK para users tem ON DELETE CASCADE — apagou o usuário, o perfil dele some junto:

-- migrations/0004_profiles.up.sql
CREATE TABLE profiles (
    user_id       uuid        PRIMARY KEY REFERENCES users (id) ON DELETE CASCADE,
    risk_profile  text        NOT NULL,
    objectives    jsonb       NOT NULL,
    horizon_years integer     NOT NULL,
    created_at    timestamptz NOT NULL DEFAULT now(),
    updated_at    timestamptz NOT NULL DEFAULT now()
);

6. A separação hexagonal: core puro, adaptador, handlers

O pacote internal/profile é dividido com a mesma disciplina das specs anteriores — e vale ver quem importa o quê, porque é aí que mora a pureza do domínio.

PeçaOnde viveImporta
Domínio + value objects (profile.go, riskprofile.go, objective.go, horizon.go)Core puroSó stdlib (errors, fmt, strings, time) — zero SQL/HTTP
Ports (ports.go) e serviço (service.go)Core puroSó o context e o port Clock
Adaptador (postgres/postgres.go)Bordadatabase/sql, pgx — e importa o core, nunca o contrário
Handlers (transport/http/profile.go)Bordanet/http, o pacote auth e o core

Há dois ports, e a distinção entre eles é a parte mais elegante da spec:

// internal/profile/ports.go
// ProfileRepository — port DRIVEN (persistência). O adaptador Postgres o implementa.
type ProfileRepository interface {
    UpsertProfile(ctx context.Context, p Profile) (Profile, error)
    GetProfileByUserID(ctx context.Context, userID string) (Profile, error) // ErrProfileNotFound quando ausente
}

// ProfileReader — o port de LEITURA voltado ao consumidor: o seam por onde o Insight Engine,
// o Rebalanceamento e o Health Score (SPEC-104/105/106) leem o perfil sem se acoplar a HTTP/SQL.
type ProfileReader interface {
    GetProfile(ctx context.Context, userID string) (Profile, error) // ErrProfileNotFound quando ausente
}
Por que dois ports, e por que o ProfileReader é tão importante? O ProfileRepository é o que o serviço usa para falar com o banco. O ProfileReader é o que as futuras features de IA vão usar para ler o perfil — e ele expõe só GetProfile, nada de escrita. O próprio Service o satisfaz (var _ ProfileReader = (*Service)(nil)), então o SPEC-104 vai receber um ProfileReader e nunca vai saber que existe HTTP ou SQL por trás. Esse é o mesmo formato de costura do Insighter (SPEC-005) e do MarketDataProvider (SPEC-006): interface pequena, definida pelo domínio, que esconde a borda do consumidor.

7. Por que NÃO há dinheiro e NÃO há IA aqui

Uma das coisas mais didáticas do SPEC-101 é o que ele deixa de fora de propósito — e por quê. Contraste direto com o SPEC-005/006:

Regra bindingSPEC-005/006SPEC-101
Dinheiro em int64 centavos / bpsCentral: preço, DY, P/VP, SELIC — tudo inteiro, nunca floatNão se aplica: o perfil não tem nenhum valor monetário. Risco, objetivos e horizonte são enums e um inteiro de anos.
Explicabilidade (FR-013)Gate: todo insight da IA carrega explicaçãoNão se aplica: não há saída de IA aqui para explicar.
Não-aconselhamento (FR-014)Gate: a IA nunca emite ordem de compra/vendaNão se aplica: o perfil não produz recomendação nenhuma.

Isto não é descuido — está escrito no próprio doc-comment do domínio (BR-1016): "There is no money and no AI output here, so the explainability/non-advice gates do not apply." O perfil é o INPUT determinístico que aquelas features de IA vão ler depois. Ele é o solo; os gates da IA pisam nele lá na frente.

Saber quando uma regra não se aplica é parte do design Uma armadilha comum é aplicar toda regra a tudo, por reflexo — converter o horizonte para centavos "porque é número", ou tentar anexar um disclaimer a um perfil. As regras da casa têm escopo: a disciplina de dinheiro protege cálculo financeiro reproduzível; os gates FR-013/014 protegem saída de IA. O perfil não é nem um nem outro — é dado de entrada estruturado. Reconhecer isso mantém a spec enxuta e evita complexidade cerimonial sem propósito.

8. As 5 decisões (D1–D5) e por quê

#DecisãoEscolhaPor quê
D1Como guardar os objetivos? Array jsonb do enum fechado Limpo com database/sql + pgx (Marshal/Unmarshal, sem atrito de scanner), ainda consultável. Melhor que text[] (atrito de driver) e que uma tabela-filha (exagero para 4 valores fixos).
D2GET quando não há perfil? 404 + ErrProfileNotFound Sinal claro de "não definido" em vez de um 200 com corpo vazio. O sentinela deixa o consumidor tratar a ausência com graça.
D3Chave primária? user_id como PK É a chave natural um-para-um; garante "um perfil por usuário" no nível do banco, sem id substituto.
D4Semântica de update? PUT substituição total (upsert) O perfil é pequeno; substituir por inteiro é mais simples de raciocinar que um PATCH parcial. PATCH fica como melhoria futura se o perfil crescer.
D5Forma do horizonte? Inteiro de anos, 1–50 Os 5/10/20 do PRD são exemplos, não um enum fixo — um inteiro limitado é mais flexível e ainda validável.

9. As 7 fases, uma a uma

Bottom-up, como nas specs anteriores — core puro primeiro, borda depois, cada fase compilando e revisável sozinha.

Fase 1 — Domínio & ports

Os value objects com construtor (parse-don't-validate): RiskProfile, Objective (+ ParseObjectives com dedupe e regra de não-vazio), Horizon (1–50, campo privado). A entidade Profile e os sentinelas (ErrProfileNotFound, ErrInvalidRiskProfile, ErrInvalidObjective, ErrNoObjectives, ErrInvalidHorizon). Os dois ports: ProfileRepository (persistência) e ProfileReader (consumidor).

Fase 2 — Persistência (migration + repositório)

A migration 0004_profiles up/down (PK user_id, FK ON DELETE CASCADE, objectives jsonb, sem dinheiro). O pacote postgres/: o UpsertProfile idempotente com ON CONFLICT … RETURNING (preserva created_at), o GetProfileByUserID escopado que mapeia sql.ErrNoRows → ErrProfileNotFound (e 22P02 → não-encontrado, nunca 500), e a asserção de compilação var _ profile.ProfileRepository = ProfileRepository{}.

Fase 3 — Aplicação (serviço)

O profile.Service: SetProfile(ctx, userID, input) (valida → deduplica → upserte) e GetProfile(ctx, userID) (satisfaz o ProfileReader). A identidade entra pelo parâmetro, vinda do contexto do handler, nunca confiada de uma entrada. Fakes escritos à mão para os testes.

Fase 4 — API (transporte)

O transport/http/profile.go: GET /profile (200/404) e PUT /profile (200, 400 nos inválidos), DTOs separados do domínio, o envelope writeJSON. O user_id vem de auth.UserID(ctx) — um user_id no corpo é rejeitado pelo DisallowUnknownFields. A interface pequena ProfileService definida no consumidor (transporte). Registro das rotas (auto-protegidas, não estão na allowlist pública) e fiação no cmd/api.

Fase 5 — Observabilidade

Os endpoints herdam os spans nomeados por rota do otelhttp (GET /profile / PUT /profile); as chamadas ao banco viram span-filho de query (sem valores de argumento). Nenhuma nova config de telemetria; nenhum PII além do user_id já logado (FR-1018).

Fase 6 — Testes

Unitários sem banco: value objects (válido/inválido/dedupe/limites); serviço com fake (substituição no upsert, preservação do created_at, not-found, validação); handlers (validação de DTO, identidade-do-contexto com user_id do corpo rejeitado, 400/404/401, e um teste de que o span não carrega valores do perfil). Integração gated: Postgres real (TEST_DATABASE_URL, -p 1) para o round-trip do upsert, o up/down da 0004, o isolamento por usuário (A nunca vê B) e o cascade ao apagar o usuário.

Fase 7 — Documentação & aula

README.md (endpoints /profile), CHANGELOG.md, SPEC-101 + PLAN-101 marcados Done, índices e status do CLAUDE.md atualizados, e esta aula. 🙂

10. Ponte: arquitetura em camadas → hexagonal

Você já conhece a pilha clássica em camadas: controller → DTO → service → repository, com o service importando o repositório concreto. O SPEC-101 é um caso lindo para a tradução, porque ele tem todas as peças clássicas — handler, DTO, service, repositório — só que rearranjadas pela inversão de dependência. (Para a teoria completa, veja a aula conceitual de arquitetura hexagonal; aqui aplicamos ela ao SPEC-101.)

A tabela de-para (com os tipos reais desta spec)

Termo em camadasPeça hexagonal no SPEC-101Onde vive
ControllerprofileHandler (driving adapter — traduz HTTP ↔ caso de uso)Borda (transport/http)
DTO de request/responseprofileRequest / profileResponse — continuam DTOs, na bordaBorda (transport/http)
Service / caso de usoprofile.Service — o caso de uso "definir/ler perfil", mas vive no core e depende de ports, não de concretosCore puro
Repository concretoQuebra em port (ProfileRepository, no core) + adapter (postgres.ProfileRepository, na borda)Interface no core; SQL na borda
(novo, sem equivalente direto em camadas)ProfileReader — port de leitura para outros casos de uso (SPEC-104/105/106) lerem sem tocar HTTP/DBInterface no core
Entidade de domínioProfile, RiskProfile, Objective, Horizon — puros, sem SQL/HTTP/tempo diretoCore (domínio puro)

A "grande virada": a inversão de dependência

Em camadas, o service importa o repositório concreto — a seta de dependência aponta para fora, do núcleo rumo à infra. No hexagonal, isso vira: o core define a interface e o adapter a implementa — a seta aponta para dentro. Veja com o par real desta spec:

// O CORE define o port (internal/profile/ports.go):
type ProfileRepository interface {

O postgres importa profile; profile jamais importa database/sql nem net/http. É por isso que o Service é testável com um fake trivial (sem banco), e que trocar Postgres por outro armazenamento não toca uma linha do core.

Driving vs. driven nesta spec

  • Driving port (quem aciona a aplicação): o profileHandler chama o serviço através da pequena interface ProfileService definida no próprio transporte (consumer-defined). Fica à esquerda do hexágono.
  • Driven port (o que a aplicação aciona): ProfileRepository (banco), implementado pelo adaptador postgres. Fica à direita.
  • Port de saída para outros casos de uso: ProfileReader — não aciona nem é acionado pela infra; é a fresta por onde futuros casos de uso (a IA) leem o perfil.
  • Domínio puro: Profile, RiskProfile, Objective, Horizon, os sentinelas, o Service. Borda: tudo com HTTP ou SQL — o handler e o adaptador Postgres.

A seta de dependência sempre aponta PARA DENTRO: o adaptador conhece o core; o core não o conhece.

11. Harness Engineering: como a máquina construiu isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. Veja, no SPEC-101, como o harness construiu e verificou.

O fluxo /spec-implement, fase a fase

A IA não atacou "faça o perfil" de uma vez. O PLAN-101 fatiou em 7 fases bottom-up (domínio → persistência → serviço → transporte → observabilidade → testes → docs), e cada uma fechava com go build + testes verdes antes da próxima. Construir o core puro primeiro (Fase 1) significa que, quando o handler foi montado (Fase 4), os value objects e o serviço já existiam e eram testáveis isoladamente — o erro tem menos espaço para se propagar três camadas adiante.

O subagente hexagonal-reviewer

O revisor especializado cobrou as regras mais sensíveis daqui: a direção de dependência (o core profile não importa net/http nem database/sql; só o subpacote postgres e o transporte o fazem) e, sobretudo, a identidade do contexto (BR-1012 — "nunca confie num user_id do payload"). Foi a lente de revisão que firmou as quatro travas da seção 3 e exigiu o teste explícito que manda {"user_id":"hacker",...} e prova que o DisallowUnknownFields o rejeita.

O go-correctness-reviewer e a corrida do re-read

Foi a lente de correção que pegou a janela de corrida entre o upsert e o re-read (seção 5.2): não era um bug de arquitetura, era um bug de concorrência sutil — exatamente o tipo que escapa numa leitura casual. A correção (colapsar tudo num ON CONFLICT … RETURNING) não só fechou a corrida como ficou mais simples. O mesmo revisor garantiu o mapeamento de 22P02 (UUID malformado) → não-encontrado em vez de 500, e o %w + prefixo minúsculo em todos os erros.

A prova de que o core ficou puro É mecânica: internal/profile (core) importa só stdlib + context + o port Clock — nenhum import de database/sql ou net/http. E a asserção var _ profile.ProfileRepository = ProfileRepository{} em postgres.go faz o compilador provar que o adapter satisfaz o port. Se a IA tivesse metido um sql.DB dentro do core, ou esquecido um método do port, o build barraria.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • Bloqueio de imutabilidade: o SPEC-101 referencia o ADR-0002 mas não o edita — se a IA tentasse "atualizar" um ADR aceito, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — e os testes têm que passar sem rede e sem DB. Os testes de integração (isolamento, cascade) ficam atrás do gate de TEST_DATABASE_URL + -p 1, pulando limpo no CI.

12. Conexão com AI Engineering

O SPEC-101 não é uma feature de IA — é o input que torna a IA orientada a objetivo. Vale conectar este alicerce à arquitetura de IA futura.

O que construímos agoraComo serve a IA depois
O port ProfileReader É o mesmo formato de costura do Insighter (SPEC-005) e do MarketDataProvider (SPEC-006): interface pequena, definida pelo domínio. O Insight Engine (SPEC-104) vai receber um ProfileReader e ler o perfil sem saber que existe HTTP ou SQL — exatamente como vai receber o Insighter sem saber qual LLM está por trás.
Isolamento por user_id do contexto O Fact Builder (SPEC-104) vai computar fatos da carteira de um usuário só e juntá-los ao perfil daquele mesmo usuário. O WHERE user_id = $1 de hoje é o que garante que o perfil (e logo o insight) de A nunca vaze para B. Um insight personalizado sem esse isolamento seria um vazamento de dados.
Perfil como fato determinístico A regra de ouro da IA confiável é "o modelo raciocina sobre fatos, não os inventa". O perfil é um desses fatos: risco, objetivos, horizonte — dados declarados pelo usuário, válidos por construção. O LLM os recebe como parte do Facts opaco; nunca os adivinha.
Grounding orientado a objetivo É o perfil que transforma "seu FII subiu 3%" em "isto reforça sua meta de renda passiva dentro do seu horizonte de 10 anos". Sem o registro de objetivos/horizonte, a explicação (FR-013) seria genérica. O perfil é o que dá à explicabilidade um alvo contra o qual avaliar a carteira.
A ideia que conecta tudo IA orientada a objetivo começa por saber qual é o objetivo. O SPEC-005 instalou a costura por onde a IA fala; o SPEC-006 instalou a fonte de fatos de mercado; o SPEC-101 instala o fato sobre o investidor — pequeno, estável, válido por construção, isolado por usuário e exposto por um port de leitura. Quando o Insight Engine chegar, ele lê o perfil pelo ProfileReader, junta com a carteira e o mercado, e finalmente responde: "sua carteira está alinhada aos seus objetivos?"
