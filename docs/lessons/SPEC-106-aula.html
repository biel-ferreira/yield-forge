SPEC-106 — Portfolio Health Score (Nota Computada + Narrativa do Professor)

Esta é a spec que dá ao investidor um número único e rastreável: uma nota de saúde de 0 a 100 para a carteira, com um detalhamento por fator (diversificação, concentração, liquidez, alinhamento de metas, exposição a risco). A lição central — e o que a diferencia do SPEC-104 e do SPEC-105 — é que aqui a NOTA é COMPUTADA, não gerada: o LLM nunca produz nem altera o número. A métrica de sucesso do PRD exige "mesmos inputs → mesma nota + mesma explicação", então o cálculo é determinístico e reproduzível. E há uma sutileza linda: a nota é ciente do mercado (o macro é um input) e ainda assim reprodutível — porque "mesmos inputs" inclui o snapshot de mercado. Em cima da nota computada, uma narrativa do "professor" (o Insighter gated) explica o resultado usando o mercado ao vivo — aterrada, gated, degradável, e que nunca toca no número.

1. A visão geral: uma nota 0–100 + um detalhamento por fator (FR-012)

Os SPECs anteriores deram ao investidor observações (insights, SPEC-104) e orientação acionável (rebalanceamento, SPEC-105). O SPEC-106 entrega uma terceira coisa, complementar: um termômetro. Uma nota 0–100 que resume "quão saudável está a minha carteira para o meu perfil, neste mercado" — um sinal único, que o investidor pode acompanhar no tempo, sempre acompanhado de um detalhamento de por quê (FR-012). Sem auditoria manual, sem caixa-preta.

O que ele entrega é um novo pacote-feature internal/health e um endpoint:

  • O HealthScore — a nota (int 0–100) + a fatia Factors []FactorScore (o detalhamento) + os campos opcionais da narrativa (Narrative, NarrativeAvailable, Disclaimer).
  • A Compute(Inputs) HealthScore — o coração: uma função pura que calcula os cinco sub-scores e os combina pela média ponderada (money.WeightedMeanBps, Σ pesos = 10 000, meio-para-cima). Nenhum LLM aqui.
  • O Service.Score — compõe quatro reads (dashboard, perfil, holdings, macro) na borda, roda a Compute, e depois pede a narrativa do professor ao Insighter gated (que só explica o número já decidido).
  • Um único endpoint GET /health-score. Nenhuma tabela, nenhuma migration — a nota é computada sob demanda (histórico fica fora de escopo).

Por que esta spec é a "prova de fogo" da reprodutibilidade — e por que ela é computada As três specs de IA do produto têm posturas diferentes diante do número. No SPEC-104 o LLM explica fatos computados. No SPEC-105 o LLM explica uma % computada e nomeia candidatos aterrados. No SPEC-106 a ambição é maior ainda: a nota e o detalhamento inteiro são computados — o LLM não chega nem perto do número. Por quê? Porque a métrica de sucesso do PRD é literal: "mesmos inputs → mesma nota + a mesma explicação". Um LLM, por natureza, não é determinístico — a mesma pergunta dá respostas levemente diferentes. Se a nota viesse do modelo, ela flutuaria entre chamadas e a métrica de reprodutibilidade quebraria. A regra binding nº 3 do PRD ("o LLM nunca inventa números") fecha o argumento: a nota é matemática, auditável e estável. O LLM entra depois, só para narrar.

2. LIÇÃO PRINCIPAL — a NOTA é COMPUTADA, não gerada: Compute + WeightedMeanBps

Esta é a ideia mais importante da aula. A Compute (em internal/health/compute.go) é uma função pura: ela recebe um snapshot estruturado e plano (Inputs) e devolve a nota + o detalhamento, sem tocar em SQL, HTTP, LLM ou relógio. Cada um dos cinco fatores é uma regra determinística que produz um sub-score 0–100 + uma explicação; a nota final é a média ponderada desses sub-scores.

2.1 — A primitiva de combinação: money.WeightedMeanBps (meio-para-cima)

No coração da nota está uma primitiva nova em internal/platform/money: WeightedMeanBps calcula a média ponderada de inteiros, onde os pesos são basis points que somam 10 000 — exatamente a mesma disciplina de inteiros + arredondamento meio-para-cima que rege todo o dinheiro do projeto (nada de float):

// internal/platform/money/ratio.go
// WeightedMeanBps retorna a média ponderada de values, onde weightsBps são pesos inteiros em
// basis points que somam 10000 — Σ(valueᵢ × weightᵢ) / 10000, arredondado meio-para-cima
// (SPEC-106 BR-1061 — a nota de saúde é a média ponderada reprodutível dos seus sub-scores).
func WeightedMeanBps(values, weightsBps []int) int {
    // ... pareia values[i] × weightsBps[i] ...
    var num int64
    for i := 0; i < n; i++ {
        num += int64(values[i]) * int64(weightsBps[i])
    }
    if num <= 0 { return 0 }
    // meio-para-cima: (2*num + 10000) / (2*10000)
    return int((2*num + 10000) / 20000)
}
Por que inteiros + meio-para-cima = reprodutibilidade "ao ponto" A reprodutibilidade do PRD não é "mais ou menos a mesma nota"; é a mesma nota, ao ponto, e a mesma explicação, byte a byte. float64 sabota isso de duas formas: a aritmética binária acumula erros (0,1 + 0,2 ≠ 0,3) e a ordem das operações importa. A solução do projeto, aplicada aqui, é nunca usar float: os sub-scores são int 0–100, os pesos são int bps (Σ = 10 000), e a média tem uma regra de arredondamento documentada (meio-para-cima). O resultado é uma função de inteiros — totalmente determinística — então a mesma entrada sempre dá a mesma saída, sem deriva. É a mesma filosofia da ShareBps/ApplyBps do dashboard.

2.2 — A Compute: o detalhamento e a renormalização dos pesos

A Compute monta a lista de fatores, deixa cada regra calcular seu sub-score, e combina. Há dois detalhes espertos embutidos: a carteira vazia faz short-circuit para nota 0 (D6), e quando o perfil não está setado os dois fatores que dependem dele são omitidos e os pesos restantes são renormalizados para voltar a somar 10 000 (D4) — reusando o mesmo AllocateBps do SPEC-105:

// internal/health/compute.go — a nota é a média ponderada dos sub-scores (SEM LLM)
func Compute(in Inputs) HealthScore {
    if in.CurrentValueCentavos <= 0 {
        return emptyScore()                       // carteira vazia → nota 0 + "adicione posições" (D6)
    }

    factors := []FactorScore{
        diversification(in), concentration(in), liquidity(in),
    }
    if in.HasProfile {
        factors = append(factors, goalAlignment(in), riskExposure(in))  // só com perfil
    }

    // Renormaliza os pesos dos fatores PRESENTES para somar EXATAMENTE 10000 (reusa o allocator):
    weights := make([]int64, len(factors))
    for i, f := range factors { weights[i] = int64(defaultWeightBps[f.Factor]) }
    normalised := money.AllocateBps(weights)      // ← Σ = 10 000 exato, mesmo sem 2 fatores

    values := make([]int, len(factors))
    for i := range factors {
        factors[i].WeightBps = normalised[i]
        values[i] = factors[i].Score
    }
    return HealthScore{Score: money.WeightedMeanBps(values, normalised), Factors: factors}
}

Os pesos default (D3) são uma heurística documentada e ajustável: diversificação 25% · concentração 25% · liquidez 15% · alinhamento de metas 20% · exposição a risco 15% (em bps, Σ = 10 000). Quando o perfil falta, os 20% + 15% dos dois fatores omitidos não somem — eles são redistribuídos proporcionalmente entre os três fatores que sobraram, então a nota continua significativa e os pesos continuam fechando em 10 000.

Por que reusar AllocateBps para renormalizar (e não dividir por 0,65) Dividir os pesos por "0,65" para reescalar reintroduziria float e deriva de arredondamento — exatamente o que estamos evitando. Reusar o money.AllocateBps (a mesma primitiva de maior-resto do SPEC-105) garante que os pesos renormalizados somam exatamente 10 000 bps por construção, com inteiros, de forma determinística. É o invariante de reconciliação aplicado aos pesos: os pesos sempre fecham em 100%, com ou sem perfil — então a média ponderada nunca fica "fora de escala".

3. Ciente do mercado, ainda assim reprodutível: o macro como INPUT

Aqui está a sutileza mais elegante da spec, e ela responde a uma pergunta natural: "como a nota pode mudar com o mercado E continuar reprodutível?". A resposta é que o macro (SELIC) é um INPUT da computação, não uma fonte de aleatoriedade. A nota é uma função determinística de (carteira, perfil, snapshot de mercado) — e "mesmos inputs" inclui o snapshot de mercado. Então:

  • Se o mercado mudar (SELIC sobe), a nota se ajusta — porque um dos inputs mudou.
  • Se você rodar duas vezes com o mesmo snapshot, a nota é idêntica — porque a função é determinística.

O mercado entra via um tilt modesto, documentado e limitado no alvo de alocação dos fatores de meta/risco: conforme a SELIC sobe acima de uma banda neutra (~10%), o mix "saudável" inclina para a renda fixa (pós-fixados ficam mais atraentes). O tilt é determinístico (mesma SELIC → mesmo tilt) e limitado (no máximo 10 pontos percentuais), então nunca vira um chamado de market-timing:

// internal/health/compute.go — o tilt de mercado: determinístico e limitado
const (
    selicNeutralBps = 1000 // ~10% de taxa básica (a banda neutra)
    maxTiltBps      = 1000  // teto: desloca no máximo 10 p.p. do alvo de FII
)

// targetFIIShareBps: o alvo de FII ciente do mercado — o alvo do perfil deslocado para renda fixa
// conforme a SELIC sobe acima da banda neutra (SPEC-106 FR-1064). Determinístico e limitado.
func targetFIIShareBps(risk profile.RiskProfile, selicBps int, hasMacro bool) int {
    base := baseFIIShareBps(risk)                 // a regra de direção do perfil (herdada do SPEC-105)
    tilt := 0
    if hasMacro && selicBps > selicNeutralBps {
        tilt = clamp((selicBps-selicNeutralBps)/2, 0, maxTiltBps)  // modesto, limitado
    }
    return maxInt(0, base-tilt)                    // SELIC alta → menos FII no alvo "saudável"
}
A distinção que resolve o paradoxo "muda com o mercado E é reprodutível" Não-determinismo e dependência-de-input são coisas diferentes. Uma função f(x) que dá saídas diferentes para o mesmo x é não-determinística (ruim para reprodutibilidade — é o LLM cru). Uma função que dá saídas diferentes para x diferentes é apenas sensível ao input (perfeitamente reprodutível). O macro é um argumento da nota, então: trocar o macro muda a nota (sensibilidade desejada), mas fixar o macro fixa a nota (reprodutibilidade preservada). É por isso que o teste de reprodutibilidade compara duas chamadas com o mesmo (carteira, perfil, macro) e exige um detalhamento byte-idêntico — e ainda assim a nota responde às condições. Se o macro estiver ausente, o tilt é neutro (cai no alvo só-do-perfil), explicado, e a nota continua sendo computada.

4. A narrativa do "professor": a camada gated que NUNCA muda o número

Em cima da nota computada vem a experiência do professor: uma narrativa em prosa, rica e contextual, que explica o porquê da nota usando o mercado ao vivo. Ela é produzida pelo Insighter gated (SPEC-005, com uma nova task health_score) e tem quatro propriedades que vale fixar — porque elas são o que torna a camada segura:

  1. Aterrada nos fatos computados. A narrativa recebe a própria nota e os sub-scores como FATOS — ela explica o número que foi dado, nunca inventa um.
  2. Gated. Passa pelos mesmos gates do SPEC-005 — explicabilidade (FR-013) e não-conselho (FR-014) — sem re-implementá-los.
  3. Degradável. Se o LLM estiver fora, a narrativa some (narrative_available:false) e a nota + detalhamento continuam de pé.
  4. Nunca muda o número. Esta é a garantia binding (D1), e ela é provada por teste.
// internal/health/narrative.go — a narrativa é ADITIVA: só explica, nunca toca a nota
func (s *Service) addNarrative(ctx context.Context, userID string, in Inputs, hs HealthScore) HealthScore {
    res, err := s.insighter.Generate(ctx, insight.InsightRequest{   // ← a ÚNICA via do texto de IA
        Facts:  narrativeFacts(in, hs),   // a nota COMPUTADA + sub-scores + mercado vão como FATOS
        Task:   TaskHealthScore,
        UserID: userID,
    })
    if err != nil {
        return hs   // degrada: a nota + o detalhamento computados ficam de pé (FR-1063)
    }
    text := narrativeText(res)
    if text == "" { return hs }
    hs.Narrative = text                  // ← só ANEXA prosa; Score e Factors permanecem intocados
    hs.NarrativeAvailable = true
    hs.Disclaimer = res.Disclaimer
    return hs
}

Repare em narrativeFacts: a nota e cada sub-score viram fatos inteiros que alimentam o prompt. O LLM recebe "a nota é 72; diversificação 60, concentração 80..." e explica isso com o contexto de SELIC/CDI ao vivo — ele nunca calcula a nota, só a justifica:

// internal/health/narrative.go — aterra a narrativa na nota COMPUTADA (inteiros, sem float)
func narrativeFacts(in Inputs, hs HealthScore) insight.Facts {
    factorScores := make(map[string]int, len(hs.Factors))
    for _, f := range hs.Factors { factorScores[string(f.Factor)] = f.Score }
    facts := insight.Facts{
        "health_score":  hs.Score,        // ← a nota JÁ decidida, passada como fato
        "factor_scores": factorScores,    // ← os sub-scores JÁ decididos
        "fii_value_centavos": in.FIIValueCentavos, /* ... */
    }
    if in.HasProfile { facts["risk_profile"] = string(in.Risk) }
    if in.HasMacro   { facts["selic_bps"]   = in.SelicBps }     // o mercado tempera a EXPLICAÇÃO
    return facts
}
A prova de que a narrativa nunca muda o número: TestService_Score_NumberUnchangedByNarrative É a estrutura que prova, mas há também um teste que crava a garantia binding. Ele roda a Score duas vezes com os mesmos inputs: uma com a narrativa OK e outra com o Insighter totalmente fora (ErrInsightsUnavailable). Depois assevera require.Equal(off.Score, on.Score, "a narrativa nunca muda o número") e require.Equal(off.Factors, on.Factors, "nem o detalhamento"). O outage não é um erro duro — a nota fica de pé com NarrativeAvailable:false. Some a isso o teste de que uma carteira vazia não chama o LLM (stub.calls == 0) e você tem a separação número/narrativa garantida por construção e por teste, não por boa-fé.
O caminho da nota → da narrativa (e por que o número está blindado): Inputs (carteira + perfil + macro) ─▶ Compute(Inputs) [PURO, sem LLM] │ produz Score + Factors (determinístico) ▼ HealthScore{Score, Factors} ◀── ESTE número é final │ (carteira vazia? → para aqui, sem chamar o LLM) ▼ insighter.Generate(facts = nota + sub-scores + mercado, "health_score") │ cadeia de decorators do SPEC-005 ┌───────────────────────────────────────────────┐ │ cache → gate(FR-013 explica + FR-014 não │ │ aconselha) → telemetria → provider (LLM) │ └───────────────────────────────────────────────┘ │ só ANEXA: hs.Narrative = text (Score/Factors intocados) ▼ GET /health-score → nota + detalhamento + narrativa (+ disclaimer) O LLM recebe a nota como FATO e a explica. Ele nunca a produz nem a altera.

5. Os cinco fatores, um a um (regras determinísticas)

Cada fator é uma função pura documentada dos fatos: recebe os Inputs, devolve um sub-score 0–100 + uma explicação que cita os fatos que a moveram (FR-1062). Todas usam clamp para ficar no intervalo, e nenhuma toca float.

FatorPeso (D3)Regra (determinística)O sinal que ele dá
Diversificação25% Recompensa mais posições espalhadas por mais setores de FII (~5 posições + ~3 setores saturam) Carteira fina ou de setor único pontua baixo
Concentração25% 100 − maiorFatiaIsolada/100 — o inverso da maior fatia de classe/setor Muita concentração isolada → nota baixa
Liquidez15% Fatia do patrimônio prontamente líquida (FIIs + renda fixa de liquidez diária) / patrimônio total Tudo travado no vencimento → nota baixa
Alinhamento de metas ciente do mercado20% Quão perto o mix FII/RF atual está do alvo implícito no perfil + tilt de mercado Longe do alvo "saudável" para o perfil → nota baixa
Exposição a risco ciente do mercado15% Penaliza risco (fatia de FII + concentração) acima da tolerância do perfil Mais risco do que o perfil suporta → nota baixa

Veja a diversificação e a concentração — as duas mais simples, mas que mostram o padrão "explicação cita os fatos":

// internal/health/compute.go
func diversification(in Inputs) FactorScore {
    holdingsPart := clamp(in.HoldingsCount*12, 0, 60)  // ~5 posições saturam a parte de posições
    sectorPart   := clamp(in.FIISectorCount*13, 0, 40) // ~3 setores saturam a parte de setores
    return FactorScore{Factor: FactorDiversification, Score: clamp(holdingsPart+sectorPart, 0, 100),
        Explanation: fmt.Sprintf("%d posição(ões) em %d setor(es) de FII — mais posições e setores elevam este fator.",
            in.HoldingsCount, in.FIISectorCount)}
}

func concentration(in Inputs) FactorScore {
    worst := maxInt(in.LargestClassBps, in.LargestSectorBps)
    return FactorScore{Factor: FactorConcentration, Score: clamp(100-worst/100, 0, 100),
        Explanation: fmt.Sprintf("maior concentração isolada: %d%% — quanto menor, mais saudável.", worst/100)}
}

5.1 — A derivação de liquidez (FII + renda fixa de liquidez diária)

O fator de liquidez tem uma derivação digna de nota. "Líquido" = todos os FIIs (negociados em bolsa) + a parcela de liquidez diária da renda fixa (um CDB de liquidez diária é líquido; um CDB no vencimento, não). O service calcula a parcela diária da RF pela proporção do valor investido e a aplica ao valor atual de RF:

// internal/health/service.go — a parcela líquida da renda fixa (split por valor investido)
func liquidValue(fiiValue, fiValue int64, fi []portfolio.FixedIncomeHolding) int64 {
    var totalInvested, dailyInvested int64
    for _, h := range fi {
        totalInvested += h.InvestedAmountCentavos
        if h.LiquidityType == portfolio.LiquidityDaily {   // só os de liquidez DIÁRIA contam
            dailyInvested += h.InvestedAmountCentavos
        }
    }
    liquidFI := int64(0)
    if totalInvested > 0 {
        liquidFI = money.ApplyBps(fiValue, money.ShareBps(dailyInvested, totalInvested))
    }
    return fiiValue + liquidFI   // FIIs (sempre líquidos) + a parcela diária da RF
}

6. O service: compõe os reads na borda, o core fica puro

A regra de pureza (BR-1063) diz: o core de scoring é puro (sem SQL/HTTP/LLM/time); os reads e a chamada ao Insighter acontecem na borda (o service). O Service.Score faz exatamente essa divisão de trabalho: monta os Inputs a partir de quatro reads, roda a Compute pura, e depois anexa a narrativa:

// internal/health/service.go — a borda compõe; o core computa
func (s *Service) Score(ctx context.Context, userID string) (HealthScore, error) {
    computeCtx, span := s.tracer.Start(ctx, "health.compute")   // span SEM conteúdo (BR-505)
    in, err := s.buildInputs(computeCtx, userID)                // 4 reads: dashboard/holdings/profile/macro
    if err != nil { span.End(); return HealthScore{}, err }
    hs := Compute(in)                                           // ← o core PURO decide a nota
    span.End()
    if in.CurrentValueCentavos <= 0 {
        return hs, nil                                          // carteira vazia: nada a narrar
    }
    return s.addNarrative(ctx, userID, in, hs), nil            // ← a narrativa gated, na borda
}

E o buildInputs mostra a degradação graciosa dos reads opcionais: um perfil não setado (ErrProfileNotFound) e uma macro ausente (ErrMacroNotFound) não são erros — eles deixam os fatores afetados se virarem (renormalização / tilt neutro). Só um erro de dashboard/holdings sobe:

// internal/health/service.go — perfil/macro ausentes degradam, não falham
func (s *Service) addProfile(ctx, userID, in *Inputs) error {
    p, err := s.profiles.GetProfile(ctx, userID)
    if errors.Is(err, profile.ErrProfileNotFound) {
        return nil   // meta + risco são omitidos (renormalizados) — NÃO é erro
    }
    if err != nil { return fmt.Errorf("health inputs: profile: %w", err) }
    in.Risk = p.Risk; in.HasProfile = true; return nil
}

Os Inputs são flat e primitivos de propósito — só inteiros (centavos, bps, contagens) e enums. É isso que torna a Compute uma função pura e fácil de testar table-driven: ela não conhece dashboard.Dashboard nem portfolio.Holdings; ela conhece apenas o snapshot numérico que o service extraiu.

7. O endpoint GET /health-score e o contrato

O handler é deliberadamente magro: lê a identidade do contexto (nunca do payload, BR-1064), chama o serviço, mapeia para o DTO. Não há body de request — é um GET:

// internal/transport/http/health.go
func (h healthHandler) getHealthScore(w http.ResponseWriter, r *http.Request) {
    userID, ok := callerID(w, r)   // identidade do contexto, nunca do payload (BR-1064)
    if !ok { return }
    hs, err := h.service.Score(r.Context(), userID)
    if err != nil { /* erro REAL de serviço → 500 */ }
    writeJSON(w, http.StatusOK, toHealthResponse(hs))   // degradado (narrativa fora) ainda é 200
}

O DTO de resposta carrega a nota, o detalhamento (cada fator com name/score/weight_bps/explanation) e os campos da narrativa. A nota e os pesos são inteiros — o ban do float se estende ao contrato:

// internal/transport/http/health.go — DTO de resposta (inteiros, nunca float)
type healthResponse struct {
    Score              int                    `json:"score"`               // 0–100, computado (nunca LLM)
    Factors            []healthFactorResponse `json:"factors"`
    Narrative          string                 `json:"narrative"`           // prosa opcional; "" se indisponível
    NarrativeAvailable bool                   `json:"narrative_available"` // false no outage total do LLM
    Disclaimer         string                 `json:"disclaimer"`          // presente com a narrativa (FR-014)
}
type healthFactorResponse struct {
    Name string `json:"name"`; Score int `json:"score"`              // 0–100

8. As 6 decisões (D1–D6) e por quê

#DecisãoEscolhaPor quê
D1LLM ou determinístico? Em camadas: nota + detalhamento computados; narrativa gated opcional por cima Reprodutibilidade do PRD + "o LLM nunca inventa números" exigem nota computada; o macro é input (ciente do mercado, ainda reprodutível); a narrativa é uma camada aditiva rotulada que nunca toca o número.
D2Onde mora o motor? Novo pacote-feature internal/health — Compute pura + um service magro Espelha a forma do dashboard (SPEC-103); o core fica puro, os reads na borda.
D3Pesos dos fatores? div 25% · conc 25% · liq 15% · meta 20% · risco 15% (bps, Σ = 10 000) Heurística documentada e ajustável; a personalização por perfil fica como follow-up (§15).
D4Perfil não setado? Renormalizar — omite meta + risco e reescala os pesos restantes a 10 000, cada omitido anotado A nota continua significativa sem perfil; reusa AllocateBps (sem float, fecha em 10 000).
D5Reusar fontes estruturadas ou BuildFacts? Ler o dashboard/perfil/holdings/macro estruturados direto Mais limpo para uma computação numérica que parsear o mapa opaco de fatos; a seam BuildFacts é para grounding de LLM, não necessária aqui.
D6Carteira vazia? 200 com score: 0 + explicação "adicione posições" Consistente com o padrão de estado amigável; não é um 404; e short-circuita sem chamar o LLM.

9. As 7 fases, uma a uma

Bottom-up, como nas specs anteriores — core/domínio primeiro, borda depois, cada fase compilando e revisável sozinha. Com três fios condutores: "a nota é computada, não gerada" (a Compute pura antes de qualquer LLM), "só inteiros" (sub-scores/nota em [0,100], pesos em bps, half-up, Σ = 10 000) e "guardas por construção" (a narrativa só pelo Insighter, e nunca tocando o número).

Fase 1 — Domínio & as cinco regras de fator + a nota ponderada (o coração)

O enum fechado Factor (+ ParseFactor), os tipos FactorScore/HealthScore, os pesos default (D3); a primitiva money.WeightedMeanBps (média ponderada inteira, half-up, Σ = 10 000); as cinco regras puras sobre os Inputs (meta + risco cientes do mercado); e a Compute (combina, renormaliza se sem perfil — D4, carteira vazia → 0 — D6, macro ausente → tilt neutro). A nota foi provada antes de qualquer LLM — testes table-driven de fronteiras, do tilt de mercado, da reprodutibilidade byte-idêntica e do zero-float.

Fase 2 — Inputs & o service (compõe os reads)

Os ports consumer-defined DashboardReader/ProfileReader/HoldingsReader/MacroReader; o health.Service compondo os quatro reads nos Inputs e chamando a Compute (perfil/macro degradam graciosamente). Ainda sem LLM. Testes da composição + dos caminhos de degradação.

Fase 3 — A narrativa de IA gated (guardas por construção)

Depois de computar a nota, pede a narrativa via Insighter.Generate (task health_score) com os fatos = nota computada + sub-scores + macro + fatos-chave; anexa o texto gated + o disclaimer. Degrada gracioso (NarrativeAvailable:false), aborta em ctx.Err(), e é emitida só via o Insighter (teste-marcador). Inclui o teste-chave: a nota é idêntica com a narrativa ligada ou desligada.

Fase 4 — API (transporte)

transport/http/health.go: GET /health-score → DTO (nota + fatores + narrativa + available + disclaimer), identidade via auth.UserID(ctx); erro de serviço → 500. Deps.HealthScore, registro na routeTable, path + schema em api/openapi.yaml (drift verde) e a fiação no cmd/api (reusa o Insighter compartilhado).

Fase 5 — Observabilidade

A rota herda o span do otelhttp; um span health.compute sem conteúdo (nenhuma nota/holdings/perfil vai para telemetria); o Insighter grava seu span insight.generate (sem conteúdo — BR-505). Um teste garante nenhum valor de fato / nota / holdings no span.

Fase 6 — Testes

Unitários sem banco: cada regra de fator (fronteiras: vazio, setor-único, tudo-ilíquido, perfil-fora); a média ponderada + half-up; o tilt de mercado (mesmo macro → mesmo tilt, ausente → neutro); reprodutibilidade (mesmos inputs → mesma nota + detalhamento byte-idêntico); a narrativa (só-via-Insighter, degrada, número-inalterado); vazio + perfil-fora; handler (identidade-do-contexto, forma 200, 401). Integração gated: Postgres real (TEST_DATABASE_URL, -p 1) + o fake Insighter — semeia holdings + perfil + cotações + macro, chama GET /health-score, asserta que a nota ∈ [0,100], todo fator é explicado, o resultado reconcilia, duas chamadas devolvem nota + explicação idênticas (reprodutibilidade ponta a ponta) e o isolamento por usuário vale.

Fase 7 — Documentação & aula

README.md (endpoint /health-score), CHANGELOG.md, api/openapi.yaml (path + schema), SPEC-106 + PLAN-106 marcados Done, índices e status do CLAUDE.md atualizados, e esta aula. 🙂

11. Harness Engineering: como a máquina construiu isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. Veja, no SPEC-106, como o harness construiu e verificou.

O fluxo /spec-implement, fase a fase — a nota antes do LLM

A IA não atacou "faça o health score" de uma vez. O PLAN-106 fatiou em 7 fases bottom-up, e a ordenação foi deliberada: a Fase 1 lança a Compute + as cinco regras + WeightedMeanBps antes de tudo, com testes table-driven provando fronteiras, o tilt de mercado, a reprodutibilidade byte-idêntica e zero-float — sem nenhum LLM por perto. Isso significa que a nota já estava provada e mecânica quando o service (Fase 2) e a narrativa (Fase 3) entraram. É o oposto de pedir a nota ao modelo: o harness força a matemática a ser escrita e testada primeiro, então não há como o LLM "vazar" para dentro do cálculo.

O hook determinístico block-layering

O hook block-layering (PostToolUse, exit 2) barra mecanicamente qualquer import de SQL/HTTP/SDK de vendor dentro de um core de feature. Importar os tipos de domínio puros de outras features (dashboard.Dashboard, profile.RiskProfile, portfolio.Holdings, marketdata.MacroIndicator, insight.Facts) passa; se a IA tivesse tentado meter um database/sql ou um net/http dentro de internal/health, o hook teria bloqueado — sem revisão humana. É o que garante, na prática, que a Compute ficou pura e que os reads ficaram na borda (atrás dos readers), não embutidos no domínio.

O subagente hexagonal-reviewer

O revisor especializado cobrou as regras mais sensíveis daqui: a direção de dependência (o grafo acíclico health → {dashboard, profile, portfolio, marketdata, insight, money}, sem ciclo de volta), a leitura direta das fontes estruturadas em vez do mapa opaco de fatos (D5) e — o ponto mais crítico — a pureza da Compute: que o core de scoring não chama SQL/HTTP/LLM/time, e que a narrativa de usuário sai só via insighter.Generate, nunca construindo strings de IA por outro caminho.

O go-correctness-reviewer e a disciplina de "computado, não gerado"

Sendo a feature onde a nota é produzida, a lente de correção vigiou os riscos nº 1 do plano (PLAN-106 §8): que o LLM influenciasse a nota (quebra de reprodutibilidade), que o detalhamento fosse não-reprodutível (float / ordem-de-mapa / time) e que a narrativa burlasse os gates. Ela garantiu que a nota vem da Compute antes de qualquer chamada ao LLM (a narrativa recebe a nota como fato); que tudo é inteiro (sem float); que o detalhamento é em ordem fixa de fatores (reprodutível); que ctx.Err() aborta; e o %w + prefixo minúsculo nos erros ("health inputs: dashboard: %w") e o ctx como primeiro parâmetro.

A prova de que a separação vale — a nota não se move É a estrutura que prova. A Compute roda antes de a narrativa ser pedida, e addNarrative só faz hs.Narrative = text — ela nem recebe acesso de escrita ao Score/Factors de forma a alterá-los. Some a isso o teste TestService_Score_NumberUnchangedByNarrative (nota idêntica com o LLM ligado vs. totalmente fora), o teste de que a carteira vazia não chama o LLM, o teste de reprodutibilidade byte-idêntica e o hook block-layering (sem provider cru no pacote) e você tem a regra binding garantida por construção, não por boa-fé — exatamente o que o PRD exige.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • block-layering: o bloqueio duro que provou que o health ficou livre de SQL/HTTP, ao mesmo tempo que permitiu o consumo dos tipos de domínio puros das features vizinhas.
  • O teste de drift do OpenAPI: ao adicionar GET /health-score na routeTable, o openapi_test.go exigiu a entrada correspondente em api/openapi.yaml — senão o build falharia (seção 7).
  • Bloqueio de imutabilidade: o SPEC-106 referencia ADR-0002 e herda os gates do SPEC-005, mas não os edita — se a IA tentasse "atualizar" um gate aceito ou um ADR, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — e os testes têm que passar sem rede e sem DB. O default do Insighter é o fake determinístico, então o CI roda offline; a integração (seed de holdings + perfil + cotações + macro, reprodutibilidade ponta a ponta) fica atrás do gate de TEST_DATABASE_URL + -p 1, pulando limpo.

12. Conexão com AI Engineering

O SPEC-106 é uma feature de IA com uma postura distinta das anteriores: aqui o LLM tem o papel mais restrito de todos — ele não produz nenhum número, só narra. Justamente por isso, ele ensina alguns dos conceitos mais importantes de engenharia de IA confiável: onde NÃO usar o LLM, e como verificar o que ele produz. Vamos a eles, todos ancorados no código desta spec.

12.1 — Determinismo onde importa: a nota como cálculo, não como geração

A lição central de engenharia de IA aqui é negativa e poderosa: nem tudo deve passar pelo LLM. Uma nota que precisa ser reprodutível ao ponto (a métrica do PRD) é exatamente o tipo de saída que não deve sair de um modelo probabilístico — porque um LLM, por construção, não dá a mesma resposta duas vezes. A Compute resolve isso com código determinístico; o LLM entra depois, para a parte em que ele é insubstituível (a linguagem). Saber desenhar essa fronteira — cálculo determinístico vs. geração — é metade da engenharia de IA de produto.

12.2 — Grounding levado ao extremo: o modelo recebe a resposta pronta

No SPEC-104 o grounding era "o LLM raciocina sobre fatos computados". Aqui ele é ainda mais forte: o LLM recebe a própria conclusão (a nota e os sub-scores) como fato e só a explica. O narrativeFacts passa health_score: hs.Score e factor_scores — o modelo não tem nem a oportunidade de derivar um número, porque ele já vem decidido. É a forma mais segura de grounding: o sistema computa, o modelo verbaliza. Isso torna a narrativa inteiramente justificável — toda frase explica um número que tem origem mecânica.

12.3 — A narrativa como camada gated e degradável (FR-013 + FR-014)

A narrativa passa pelos mesmos gates herdados do SPEC-005 (explicabilidade rejeita texto sem explicação; não-conselho rejeita ordens/quantidades/preços-alvo), sem re-implementá-los — a propriedade arquitetural é que o motor só emite texto de usuário pela porta que tem os gates. E ela é degradável: um outage do LLM não derruba a feature, só remove a prosa (narrative_available:false). Esse é o padrão de graceful degradation que mantém a IA como um enhancement, nunca um ponto único de falha do produto.

12.4 — Defesa contra prompt-injection: a nota é blindada por design

A narrativa passa holdings e a nota pelo LLM, então há uma superfície de prompt-injection. Mas a mitigação mais forte é estrutural: a nota é computada, então uma injeção não pode mover o número — no pior caso a narrativa degrada. O gate de não-conselho é o backstop fail-closed sobre o texto. É a aplicação da regra "o LLM é falível, então o que importa (o número) não depende dele, e o que ele produz (o texto) passa por um verificador". A injeção esbarra num número que ela não controla.

12.5 — Mercado como input determinístico vs. market-timing (uma lição de calibração)

O tilt de mercado é uma lição de engenharia de IA aplicada a finanças: como deixar uma métrica responder ao ambiente sem virar um chamado de market-timing? A resposta é mantê-lo modesto, documentado e limitado (maxTiltBps = 1000), e determinístico (mesma SELIC → mesmo tilt). A narrativa então explica "a nota mudou por causa do mercado, não da sua carteira" — separando deriva de mercado de deriva de carteira. É calibração deliberada: input suficiente para ser útil, limitado o bastante para ser seguro e reproduzível.

A ideia que conecta tudo IA confiável em finanças é saber o que NÃO entregar ao modelo. O SPEC-106 entrega a lição mais limpa: a nota é computada (determinística, reproduzível, ciente do mercado via input), e o LLM faz só o que ele faz melhor — explicar em linguagem natural, com o mercado ao vivo, atrás dos gates. O service, no meio, faz o mínimo: compõe os reads, computa a nota e anexa a narrativa pela única porta gated. É essa modéstia arquitetural — deixar a matemática à matemática e o LLM à linguagem — que torna a "nota de saúde" um sinal em que o investidor pode confiar e acompanhar no tempo.
