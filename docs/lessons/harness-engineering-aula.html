1. O que é Harness Engineering

Harness engineering é a engenharia do sistema em volta do modelo — não do modelo. O modelo (Opus, Gemini, etc.) é fixo: você não o treina. O que você controla é tudo o que o cerca. E é aí que mora a qualidade do resultado.

São cinco alavancas:

AlavancaPergunta que ela respondeExemplo no que fizemos
ContextoO que o modelo enxerga?CLAUDE.md, os docs de SDD
FerramentasO que ele pode fazer?Permissões no settings.json
MemóriaO que persiste entre sessões?CLAUDE.md (carregado sempre)
Controle de fluxoQuem orquestra quem?Commands chamando subagents
VerificaçãoComo sei que deu certo?Hooks, reviewers, gates de teste
Por que isso importa pro seu aprendizado Você quis aprender "tudo de ponta do uso de IA". A virada de chave é entender que melhorar o resultado raramente é "prompt mágico" — é desenhar contexto, ferramentas e verificação ao redor do modelo. Essa é a disciplina. O resto da aula é ela, na prática.

2. Os dois harnesses

Um detalhe que organiza tudo: no YieldForge existem dois harnesses, e os primitivos são os mesmos nos dois. Aprender um te ensina o outro.

HARNESS DE DESENVOLVIMENTO HARNESS DO PRODUTO (o Claude Code construindo o app) (o app usando LLM pra gerar insights) CLAUDE.md, hooks, subagents, Insighter (port), Fact Builder, commands, reviewers gates FR-013/FR-014, evals, CIO multi-agente → te deixa eficiente → é o CORAÇÃO do produto \ / └──── MESMOS conceitos ───────────────┘ contexto · ferramentas · verificação · orquestração
A sacada do "dogfooding" Quando você monta um subagent reviewer com um gate de verificação no harness de dev, está praticando exatamente o que vai implementar no produto: o gate de non-advice (FR-014) e o de explicabilidade (FR-013) são verificadores que envolvem cada chamada de LLM — o mesmo padrão, agora protegendo o usuário final.

3. O princípio-chave: o espectro soft → hard rule

Se você levar uma ideia desta aula, que seja esta:

Coloque cada regra no nível mais baixo capaz de garanti-la. A mesma "regra" tem valor totalmente diferente dependendo de onde você a coloca. "Sempre rodar gofmt" como uma frase no CLAUDE.md é uma soft rule — o modelo pode esquecer. A mesma regra como um hook é uma hard rule — roda toda vez, independente do modelo.

O espectro, do mais fraco (probabilístico) ao mais forte (determinístico):

Tipo de regraOnde viveEnforcement
Instrução que eu leio e devo seguirCLAUDE.mdProbabilístico (posso esquecer)
Instrução por papelsystem prompt de subagentProbabilístico, isolado
Instrução por subpastaCLAUDE.md aninhadoCarregado só quando relevante
GarantiahooksDeterminístico
O que posso executarsettings.json (permissões)Determinístico
Proteção do usuário finalgates em código (FR-013/014)Determinístico, no produto

4. Tier 0 — a fundação

Tudo começou por aqui, porque é o de maior retorno: sem a fundação, todo o resto rende menos.

4.1 CLAUDE.md — a memória do projeto

Um arquivo na raiz, carregado em toda sessão. Antes dele, eu redescobria a cada conversa que o projeto é hexagonal, que o quality gate é task vet + task test:short, que existe a constraint de non-advice. Agora isso é memória persistente. É a alavanca contexto + memória em um arquivo.

4.2 Permissões — settings.json vs settings.local.json

Permissões são a alavanca ferramentas: o que eu posso rodar sem te perguntar. Separamos em dois:

ArquivoO que vai neleGit
.claude/settings.jsonRegras do projeto (ex.: go build/test/vet, task *) — valem pra qualquer colaboradorCommitado
.claude/settings.local.jsonEspecífico da sua máquina (paths de leitura, etc.)Gitignorado
Precedência settings.local.json (sua máquina) > .claude/settings.json (projeto) > user global. Hard rules de projeto vão no versionado pra todo mundo herdar. Limpamos ~60 regras auto-geradas e bagunçadas para um conjunto pequeno e seguro.

4.3 Hooks — os três modos: bloquear, agir, avisar

Hooks são a estrela do Tier 0: comandos que o harness executa em eventos do ciclo de vida — não eu. É como uma soft rule vira hard rule. Criamos um de cada modo:

EventoModoO que criamosArquivo
PreToolUseBloquear (antes da ação)Impede editar migration/ADR já commitado (exit 2)block-immutable.ps1
PostToolUseAgir (depois da ação)gofmt no arquivo .go recém-editadogofmt-edited.ps1
StopAvisar (ao terminar)go vet + lembrete de CHANGELOG, não-bloqueanteon-stop.ps1
A regra elegante do hook de imutabilidade O working agreement diz "ADRs são imutáveis once accepted" e "migrations são append-only". Em vez de parsear status, escolhemos uma regra única e robusta: "uma vez commitado no git, é imutável". Arquivo novo (não rastreado) pode editar; arquivo já commitado → bloqueio. Testamos os 4 cenários antes de confiar nele.
Por que escrevemos os hooks em PowerShell? Sondamos seu ambiente: jq não estava instalado, então parsear o JSON do stdin com jq estava fora. PowerShell já tinha ConvertFrom-Json nativo. Lição de harness: o harness se adapta à máquina real — você verifica o terreno antes de construir.

5. "Rules": por que não criamos um arquivo de rules

Você perguntou se deveríamos pôr "Rules" no projeto (estilo .cursor/rules). A resposta ensina a diferença do Claude Code: rules são o harness — mas não são um arquivo só. São o espectro inteiro da seção 3. Copiar um arquivo "rules" seria cargo cult. Em vez disso, roteamos cada regra:

  • Soft rules → CLAUDE.md (e aninhado por pacote quando for local).
  • Hard rules → hooks + permissões (o que dá pra garantir, garanta).
  • Regras de produto (explicabilidade/non-advice) → gates em código (a forma mais forte, porque protege o usuário).

6. As code conventions (e como as decidimos)

Aqui teve um momento didático importante: o código existente parecia ter convenções, mas você esclareceu que era só "como saiu no começo". Então decidimos deliberadamente. E o jeito certo foi derivar do código real e debater os trade-offs — não inventar.

O debate do testify

Sua intuição: "sem testify/mock não fica mais difícil?". A resposta separou duas coisas que costumam ser confundidas:

 AsserçõesMocks
O medotestify reduz boilerplate"sem framework fica confuso"
A realidadeVerdade — mas a diferença é pequenaInverte: a confusão vem dos frameworks (gomock), não dos fakes
Por quê—Hexagonal deixa os ports pequenos → fake à mão é ~15 linhas óbvias
A decisão testify/require para asserções (ergonomia) + fakes à mão para os ports (sem gomock/mockery). O conforto onde importa, o idioma limpo onde framework atrapalha.

As convenções fixadas

ConvençãoRegra

Por que dinheiro é a convenção nº 1 do seu produto Você lida com BRL, dividend yield, juros compostos nas projeções. float64 acumula erro e quebra o critério do PRD "mesmos inputs → mesmo Health Score". int64 centavos + regra de arredondamento explícita é o que torna o resultado determinístico e reproduzível.

7. Tier 1 — o fluxo SDD virou harness

O SDD de vocês (PRD → SPEC → PLAN → CODE) já era ótimo context engineering. Faltava conectá-lo ao Claude Code como peças reutilizáveis. Transformamos cada etapa em um primitivo:

/spec-new ──▶ /plan-new ──▶ /spec-implement ──▶ /pr-review (rascunha (rascunha (constrói fase a (gate final o SPEC) o PLAN) fase, com review) do PR) │ ├─▶ hexagonal-reviewer (arquitetura) ├─▶ go-correctness-reviewer (bugs) └─▶ lesson-writer (a aula PT-BR)
ArtefatoTipoPapel
/spec-newcommandRascunha um SPEC do template, ancorado numa fatia do PRD
/plan-newcommandRascunha o PLAN espelhando o número do SPEC
/spec-implementcommandImplementa fase a fase; roda quality gate; fecha com docs + aula
/pr-reviewcommandGate final do PR (2 reviewers + closeout SDD)
hexagonal-reviewersubagentLayering, guards de non-advice/explicabilidade, conventions
go-correctness-reviewersubagentnil deref, errors não-checados, concorrência, leaks, edge cases
lesson-writersubagentGera a aula PT-BR HTML (com seções de Harness + AI Engineering)
O gate só vale se for executado Você pegou uma falha real: o /spec-implement só rodava testes unitários. Os de integração estavam no PLAN como prosa, mas não eram disparados. Movemos "rodar integração" pra passo obrigatório do command (com Postgres real, ou reporta explicitamente — nunca pula em silêncio). Isso é, de novo, promover soft rule → enforcement.

8. Subagent vs slash command — o critério

A pergunta "isso é subagent ou command?" apareceu várias vezes. O critério que usamos:

 SubagentSlash command
O que éContexto isolado, paralelizávelTemplate de prompt no contexto principal
CustoManutenção; só compensa se grande/paraleloBarato
Use quandoTrabalho isolado e reutilizável (review, aula)Fluxo que você acompanha (implementar)
Aplicando Por isso o "go-implementer" virou o command /spec-implement (você acompanha), e reviewer/lesson-writer viraram subagents (contexto isolado: o review não polui o implementer). Esse discernimento evita "subagent sprawl" — criar subagent pra tudo.
E o "PR reviewer"? E plugin? Plugin é só um empacotamento pra distribuir entre projetos — não precisa num projeto só. O /pr-review ficou: command (o plumbing do GitHub via gh) + subagents (o cérebro reutilizável). Cérebro = subagent; casca/orquestração = command.

9. Verificação por perspectivas diversas

Você notou que o passe de correção do /pr-review estava raso. A correção virou um padrão de harness de verdade: duas lentes independentes, cada uma com foco próprio.

diff do código │ ┌───────┴────────┐ ▼ ▼ hexagonal-reviewer go-correctness-reviewer (arquitetura, (nil deref, races, leaks, guards, conventions) SQL injection, edge cases) │ │ └───────┬────────┘ ▼ veredito combinado
Por que duas e não uma maior? Diversidade pega o que redundância não pega. Um reviewer focado em arquitetura e outro focado em bugs cobrem modos de falha diferentes — e cada um trabalha com contexto enxuto, sem se dispersar. É o mesmo princípio dos "céticos independentes" que se usa pra validar achados de IA: perspectivas diversas > um olhar só.

10. A ponte para AI Engineering

Aqui fecha o ciclo: cada conceito do harness de dev tem um gêmeo no produto. É por isso que montar o harness te ensina AI Engineering de verdade.

Conceito de harness (que praticamos)Onde reaparece no YieldForge
Grounding / contexto controladoFact Builder dá fatos computados ao LLM (não liberdade de inventar números)
Gate de verificação (hook que bloqueia)FR-013 (explicabilidade) e FR-014 (non-advice) como middleware em cada chamada
Verificação por perspectivas diversas (2 reviewers)Validar insights com múltiplos verificadores antes de mostrar ao usuário
Subagents orquestrados por um commandFase 2: CIO multi-agente (Macro/FI/FII/Risk) atrás do port Insighter
Ferramentas atrás de permissãoFase 3: capacidades expostas como MCP tools
Determinismo (gofmt sempre, money int64)"Mesmos inputs → mesmo Health Score" (reprodutibilidade do PRD)
O port Insighter é o seu harness de produto Assim como envolvemos cada edição com hooks, o produto envolve cada chamada de LLM com gates. Projete o Insighter com os mesmos princípios — pluggable, observável, com verificação — e você terá aprendido harness engineering construindo o coração do YieldForge. O mergulho fundo nisso (grounding, structured outputs, evals, prompt injection, observabilidade de custo/latência) vem no SPEC-005.
