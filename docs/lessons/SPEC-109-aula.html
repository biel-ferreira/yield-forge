Aula 109 · Indexadores de Renda Fixa · "120% do CDI" vira um número que se move sozinho

SPEC-109 — Indexadores de Renda Fixa (% do CDI / IPCA+)

Esta é uma spec pequena e cirúrgica, mas ensina uma lição arquitetural grande. Até aqui, um holding de renda fixa guardava uma taxa anual fixa — mas quase nenhuma renda fixa brasileira de varejo funciona assim: um CDB comum é quase sempre "120% do CDI", um Tesouro IPCA+ é "IPCA + 5,80%". O SPEC-109 deixa o investidor digitar a taxa do jeito que o extrato do banco mostra, e resolve a taxa efetiva atual automaticamente, usando o SELIC/CDI/IPCA que o SPEC-006 já ingere — sem cadastro manual toda vez que o Copom se reúne. A lição central: essa taxa efetiva é um fato computado, nunca persistido — a mesma disciplina "fatos são computados, não gerados" que rege as features de IA do projeto, aplicada aqui a uma aritmética simples, sem LLM nenhum.

1. O problema real: "120% do CDI" não é uma taxa fixa

O SPEC-102 (Gestão de Carteira) modelou FixedIncomeHolding com um campo único, AnnualRateBps — uma taxa anual fixa, em basis points. Isso funciona perfeitamente para renda fixa prefixada ("10% a.a., ponto final"). Mas ao desenhar o SPEC-211 (as telas de gestão de carteira do frontend), ficou claro um problema: a maioria da renda fixa de varejo brasileira não é prefixada. Um CDB comum é pós-fixado ("120% do CDI") — a taxa real muda toda vez que o CDI muda. Um Tesouro IPCA+ é híbrido ("IPCA + 5,80%") — a parte de inflação também se move.

Perguntar ao usuário "qual é a taxa anual?" quando ele só sabe "120% do CDI" é uma pergunta errada. E converter isso uma vez, no cadastro, para um número fixo seria pior: o número ficaria errado silenciosamente a cada decisão do Copom, sem que ninguém percebesse — quebrando exatamente a promessa de reprodutibilidade que o Dashboard (SPEC-103) e o Health Score (SPEC-106) já seguem. A solução do SPEC-109: deixar o usuário cadastrar a taxa do jeito que ela é cotada (indexador + valor) e resolver a taxa efetiva a cada leitura, contra o SELIC/CDI/IPCA mais recente que o SPEC-006 já ingere.

ANTES (SPEC-102) DEPOIS (SPEC-109) FixedIncomeHolding{ FixedIncomeHolding{ AnnualRateBps: 1000 // "10% a.a." AnnualRateBps: 12000 // "120%" (do CDI) } IndexerType: cdi_percentual → sempre 10%, para sempre EffectiveAnnualRateBps: 1260 // COMPUTADO, nunca salvo } Só serve para prefixado → hoje CDI=10,50% ⇒ efetiva 12,60%; muda sozinho amanhã Dashboard/Projections leem AnnualRateBps direto Dashboard/Projections leem EffectiveAnnualRateBps (para prefixado os dois são idênticos — regressão OK)

O que a spec entrega é deliberadamente pequeno:

  • Um enum fechado novo, portfolio.Indexer (prefixado | cdi_percentual | ipca_spread), e uma coluna nova indexer_type na tabela existente — nenhuma tabela nova.
  • Um método puro em cima do tipo de domínio já existente, FixedIncomeHolding.ResolveEffectiveRate — sem I/O, sem SQL, sem HTTP.
  • Um campo novo computado, nunca persistido: EffectiveAnnualRateBps, populado pelo portfolio.Service em todo caminho de leitura/escrita.
  • Um endpoint novo, só-leitura: GET /market/indicators — reaproveita o repositório de macro do SPEC-006, sem nenhuma ingestão nova.
Por que isto não é "adicionar um campo" — é reafirmar um princípio do PRD A tentação óbvia seria converter "120% do CDI" em um número fixo no momento do cadastro e seguir em frente. O SPEC-109 resiste a essa tentação exatamente pela mesma razão que o Health Score (SPEC-106) resiste a deixar o LLM "sugerir" a nota: um número que deveria mudar com a realidade, mas está congelado, é pior do que nenhum número — ele passa segurança falsa. A regra binding nº 3 do PRD ("fatos são computados, não gerados") não fala só de LLM; fala de qualquer número que o sistema apresenta como verdade atual. Uma taxa "120% do CDI" só é verdadeira hoje se for recalculada hoje.

2. LIÇÃO PRINCIPAL — a taxa efetiva é COMPUTADA, nunca persistida (BR-1092)

Este é o coração da aula, e é uma variação — sem LLM nenhum — do padrão que as specs de IA do projeto já ensinam repetidas vezes: o dado bruto é armazenado; o fato derivado é recalculado a cada leitura. Veja o comentário de domínio no próprio tipo, em internal/portfolio/holding.go:

// internal/portfolio/holding.go
// FixedIncomeHolding is a registered fixed-income position (SPEC-102 FR-002). ... IndexerType
// determines how AnnualRateBps is interpreted (SPEC-109) — see ResolveEffectiveRate.
// EffectiveAnnualRateBps is a computed, NEVER-PERSISTED field: it starts zero-value and is
// populated by the Service (via ResolveEffectiveRate) on every read/write path before the
// holding reaches its caller — the repository never selects or writes it (FR-1092/BR-1092).
type FixedIncomeHolding struct {
    ID                     string
    UserID                 string
    Name                   string
    Institution            string
    InvestedAmountCentavos int64
    AnnualRateBps          int
    IndexerType            Indexer
    EffectiveAnnualRateBps int // computed, never persisted — see doc comment above
    MaturityDate           *time.Time
    LiquidityType          LiquidityType
    CreatedAt              time.Time
    UpdatedAt              time.Time
}

Repare no que não existe: não há coluna effective_annual_rate_bps na migration, não há INSERT nem SELECT tocando esse campo em internal/portfolio/postgres/postgres.go. O campo vive só na struct Go, sai do zero-value (0) toda vez que uma linha é lida do banco, e é preenchido pelo Service — nunca pelo repositório. Isso torna o "nunca persistir" uma propriedade estrutural: um dev futuro teria que escrever código novo, explícito, para persistir esse campo por engano; não é algo que acontece "por acidente" de um SELECT *.

A mesma disciplina das specs de IA, aplicada a aritmética pura Compare com o Health Score (SPEC-106): a nota 0–100 é computada pela função pura Compute a cada GET /health-score, nunca guardada numa tabela de histórico — porque uma nota "congelada" mentiria assim que a carteira ou o mercado mudassem. O SPEC-109 aplica exatamente essa regra a um número muito mais simples: a taxa efetiva de um CDB pós-fixado. Não há IA aqui, mas há o mesmo motivo — "fatos são computados, não gerados" não é uma regra sobre LLMs; é uma regra sobre o que conta como verdade atual num sistema financeiro. Isso é um ótimo lembrete de que o princípio central do projeto é mais amplo que "IA": ele é sobre nunca deixar um número parecer atual quando na verdade é uma foto antiga.

3. A resolução pura: ResolveEffectiveRate, os três indexadores

O enum fechado Indexer segue exatamente o padrão parse-don't-validate que o LiquidityType (SPEC-102) já estabeleceu — type X string + um ParseX que normaliza (trim + lower) e devolve um sentinela via %w para um valor desconhecido:

// internal/portfolio/indexer.go
type Indexer string

const (
    IndexerPrefixado     Indexer = "prefixado"      // flat annual rate
    IndexerCDIPercentual Indexer = "cdi_percentual" // % of CDI (pós-fixado)
    IndexerIPCASpread    Indexer = "ipca_spread"    // IPCA + spread (híbrido)
)

// ParseIndexer normalizes (trim + lower) and validates s. An empty string defaults to
// Prefixado (SPEC-109 BR-1093 backward compatibility — existing holdings/callers that never
// supply an indexer keep today's flat-rate behavior).
func ParseIndexer(s string) (Indexer, error) {
    trimmed := strings.ToLower(strings.TrimSpace(s))
    if trimmed == "" {
        return IndexerPrefixado, nil   // ← o detalhe que garante retrocompatibilidade "de graça"
    }
    idx := Indexer(trimmed)
    if !validIndexers[idx] {
        return "", fmt.Errorf("parse indexer %q: %w", s, ErrInvalidIndexer)
    }
    return idx, nil
}

O verdadeiro trabalho acontece num método puro, colocado ao lado do tipo que ele resolve (internal/portfolio/holding.go). Ele recebe um mapa de leituras de mercado já buscadas — nunca faz a busca ele mesmo — e devolve a taxa efetiva, um int em basis points, sem nunca errar:

// internal/portfolio/holding.go
// ResolveEffectiveRate resolves the holding's current effective annual rate from its stored
// AnnualRateBps + IndexerType and the latest macro readings (SPEC-109 FR-1092). Pure — no I/O;
// the caller fetches macro (keyed by Indicator) via its own MacroReader port before calling
// this. Never errors: an Indexer other than Prefixado whose reference indicator is absent from
// macro degrades to the raw stored value, unresolved (BR-1094/PLAN-109 D3) — a transient,
// self-healing gap (e.g. before the first ingestion run), never a crash or a silent zero.
func (h FixedIncomeHolding) ResolveEffectiveRate(macro map[marketdata.Indicator]marketdata.MacroIndicator) int {
    switch h.IndexerType {
    case IndexerCDIPercentual:
        if cdi, ok := macro[marketdata.IndicatorCDI]; ok {
            return int(money.ApplyBps(cdi.Value, h.AnnualRateBps))   // rate × CDI / 10000, half-up
        }
    case IndexerIPCASpread:
        if ipca, ok := macro[marketdata.IndicatorIPCA]; ok {
            return h.AnnualRateBps + int(ipca.Value)                  // spread + IPCA
        }
    }
    // Prefixado, an unrecognized/zero-value indexer, or a missing macro reading: pass through.
    return h.AnnualRateBps
}

Três decisões pequenas, todas coerentes com o resto do projeto:

  • Nenhum float. O cálculo de cdi_percentual reaproveita money.ApplyBps — a mesma primitiva half-up com big.Int por dentro que já serve o Dashboard e a Projeção — em vez de escrever uma divisão nova. Zero código novo de arredondamento nesta spec.
  • Método, não função solta. ResolveEffectiveRate é um método em FixedIncomeHolding, colocado no mesmo arquivo do tipo que ele resolve — a assinatura fica h.ResolveEffectiveRate(macro), legível no ponto de chamada, e a lógica mora perto do dado que ela interpreta.
  • Nunca erra. Um indexador desconhecido/zero-value ou uma leitura ausente do mapa caem no mesmo return h.AnnualRateBps — não existe um caminho de erro que o chamador precise tratar. É a mesma filosofia de "casos de borda são estados, não exceções" das outras specs computacionais (SPEC-106, SPEC-107).

Os testes table-driven (internal/portfolio/holding_test.go) provam a aritmética com os números do exemplo canônico da spec — 120% do CDI a 10,50% a.a. vira 12,60% a.a.:

// internal/portfolio/holding_test.go
t.Run("CDIPercentual resolves rate x CDI / 10000, half-up", func(t *testing.T) {
    // 120% do CDI, CDI = 10.50% a.a. -> 1050 * 12000 / 10000 = 1260 (12.60% a.a.)
    h := FixedIncomeHolding{IndexerType: IndexerCDIPercentual, AnnualRateBps: 12000}
    require.Equal(t, 1260, h.ResolveEffectiveRate(macroWithCDIAndIPCA))
})

4. Retrocompatibilidade como não-negociável (BR-1093)

Toda holding de renda fixa cadastrada antes desta spec precisa continuar se comportando exatamente igual — não "quase igual", byte a byte. A migration garante isso do lado do banco, e um teste garante isso do lado do código.

-- migrations/0007_fixed_income_indexer.up.sql
ALTER TABLE fixed_income_holdings
    ADD COLUMN indexer_type text NOT NULL DEFAULT 'prefixado'
        CHECK (indexer_type IN ('prefixado', 'cdi_percentual', 'ipca_spread'));

Uma ALTER TABLE ... ADD COLUMN ... DEFAULT é aditiva: toda linha existente ganha indexer_type = 'prefixado' sem precisar de backfill, e o CHECK espelha o mesmo enum fechado do lado Go — a validação vive nos dois lugares, como convenção do projeto para enums fechados persistidos. Com IndexerPrefixado, ResolveEffectiveRate sempre devolve AnnualRateBps intacto — então uma holding antiga passa pelo caminho novo de resolução e sai idêntica ao que era antes desta spec.

Do lado dos consumidores, o teste que fecha o argumento é uma fixture no dashboard/compute_test.go:

// internal/dashboard/compute_test.go
// EffectiveAnnualRateBps (SPEC-109) is what Compute reads for accrual; here it equals
// the raw stored AnnualRateBps (a prefixado-shaped fixture — ListHoldings resolves ...
{InvestedAmountCentavos: 1_000_000, AnnualRateBps: 1_200, EffectiveAnnualRateBps: 1_200,
    LiquidityType: portfolio.LiquidityAtMaturity, CreatedAt: created}

A fixture crava AnnualRateBps e EffectiveAnnualRateBps com o mesmo valor (1200) e verifica que o resultado do Compute do dashboard não mudou nada — é a prova de regressão exigida pela BR-1093. O mesmo padrão se repete na integração: o teste de repositório (TestRepository_FixedIncomeIndexer_Integration) prova que "uma linha pré-existente (sem indexer_type setado) lê de volta como prefixado".

Por que retrocompatibilidade "quase igual" não bastaria Neste projeto, o Dashboard e as Projeções carregam a promessa de reprodutibilidade ("mesmos inputs → mesmo resultado", a métrica-mãe do PRD nas specs 106/107). Se a nova coluna mudasse o cálculo de qualquer holding pré-existente — mesmo um centavo — isso quebraria a confiança do investidor no número que ele já via ontem. É por isso que a spec exige explicitamente "byte-for-byte identical" e não apenas "sem regressão visível": a prova precisa ser um teste que compara valores exatos, não uma inspeção manual.

5. Degradação graciosa — duas posturas diferentes num só feature (BR-1094)

Esta é a parte mais sutil e mais interessante da spec: o SPEC-109 tem dois lugares que leem dados macro, e eles degradam de formas propositalmente diferentes — porque o papel de cada um no sistema é diferente. Entender essa diferença é entender quando "engolir o erro" é a decisão certa e quando não é.

5.1 — O caminho de leitura de holdings (dashboard/projeções/chat): engole QUALQUER falha de macro

A função latestMacro, em internal/portfolio/service.go, busca CDI e IPCA e monta o mapa que ResolveEffectiveRate consome. Repare no comentário — ele não se limita a tratar "ainda não ingerido" (marketdata.ErrMacroNotFound); ele trata qualquer erro da mesma forma:

// internal/portfolio/service.go
// latestMacro best-effort-fetches the CDI/IPCA readings SPEC-109's indexers need. An indicator
// whose read fails is simply omitted — ResolveEffectiveRate's own degradation (BR-1094) takes it
// from there; this never returns an error, never blocks a holdings read on a market-data hiccup.
//
// Deliberately not limited to marketdata.ErrMacroNotFound (PLAN-109 D3's literal scope): any
// GetLatestMacroIndicator failure, including an infra error (e.g. the macro subsystem's DB is
// briefly unreachable), degrades the same way. The holdings/dashboard/projections read path must
// not become coupled to macro-subsystem availability — unlike GET /market/indicators (whose sole
// purpose is serving macro data, and which does distinguish + log real errors), this method
// cannot log from the domain core (no I/O in core, per CLAUDE.md) and has no per-call way to
// surface an infra failure without also risking failing the read entirely, which would be a
// worse outcome for a page whose primary content is the holdings, not the macro reference rate.
func (s *Service) latestMacro(ctx context.Context) map[marketdata.Indicator]marketdata.MacroIndicator {
    out := make(map[marketdata.Indicator]marketdata.MacroIndicator, 2)
    for _, ind := range [...]marketdata.Indicator{marketdata.IndicatorCDI, marketdata.IndicatorIPCA} {
        if m, err := s.macro.GetLatestMacroIndicator(ctx, ind); err == nil {
            out[ind] = m
        }
    }
    return out
}

Note também o teste que prova a degradação é por indicador, não tudo-ou-nada — se o CDI estiver disponível mas o IPCA não, a holding cdi_percentual ainda resolve corretamente enquanto a ipca_spread cai no valor bruto:

// internal/portfolio/service_test.go
// TestService_ListHoldings_ResolvesEffectiveRatePerIndicatorIndependently proves the macro fetch
// degrades per indicator (SPEC-109 BR-1094), not all-or-nothing: CDI present + IPCA absent must
// resolve the cdi_percentual holding while the ipca_spread holding falls back to its raw rate.

5.2 — GET /market/indicators: distingue "ainda não ingerido" de uma falha real, e LOGA a última

O handler novo em internal/transport/http/market.go tem o trabalho oposto: seu único propósito é servir dados macro, então engolir uma falha real de infraestrutura em silêncio seria esconder exatamente o problema que esse endpoint existe para revelar. Por isso ele distingue os dois casos:

// internal/transport/http/market.go
func (h marketHandler) getMarketIndicators(w http.ResponseWriter, r *http.Request) {
    if _, ok := callerID(w, r); !ok {
        return
    }
    out := make([]marketIndicatorResponse, 0, len(publishedIndicators))
    for _, ind := range publishedIndicators {
        m, err := h.reader.GetLatestMacroIndicator(r.Context(), ind)
        switch {
        case errors.Is(err, marketdata.ErrMacroNotFound):
            continue // not yet ingested — omit, don't fail the request (BR-1094)
        case err != nil:
            h.logger.ErrorContext(r.Context(), "get latest macro indicator failed",
                slog.String("indicator", string(ind)), slog.String("error", err.Error()))
            continue   // ← ainda omite (não derruba a resposta), mas agora fica OBSERVÁVEL
        }
        out = append(out, marketIndicatorResponse{
            Indicator: string(ind), ValueBps: m.Value, ReferenceDate: m.ReferenceDate.Format(dateLayout),
        })
    }
    writeJSON(w, http.StatusOK, out)
}

Em ambos os casos o request continua devolvendo 200 (a resposta nunca falha por causa de um indicador faltando) — mas só o segundo caso vai para o log estruturado via slog. Um outage silencioso de banco de dados agora deixa rastro; um ambiente novo sem ingestão ainda ainda não gera ruído nenhum no log.

O bug real que o go-correctness-reviewer encontrou nesta sessão A primeira versão de getMarketIndicators engolia qualquer erro do mesmo jeito — inclusive uma falha real de infraestrutura — sem diferenciar de "ainda não ingerido" e sem logar nada. O go-correctness-reviewer pegou isso na revisão (PLAN-109 §9, Quality Validation): um endpoint cujo único propósito é servir dados de mercado não pode deixar uma falha real indistinguível de "ambiente novo". A correção acrescentou o switch acima e passou a usar o campo logger do handler, que já existia mas estava sem uso. É um ótimo exemplo de "o mesmo padrão de tratamento de erro não serve para dois contextos diferentes" — e de como um revisor especializado em correção (não em arquitetura) pega esse tipo de detalhe.

6. O seam único: por que só o portfolio.Service tem um MacroReader

Aqui está o momento mais instrutivo desta spec sobre como usar um plano. O PLAN-109 (decisão D2, na seção 4) sugeriu que cada serviço que precisasse da taxa resolvida — portfolio, dashboard e projection — ganhasse seu próprio port MacroReader, espelhando o padrão já usado em internal/health e internal/insight/engine. Isso teria significado três ports novos, três fiações novas no cmd/api, e três lugares fazendo I/O de macro para o mesmo request.

Durante a Fase 3, essa suposição foi revisitada contra o seam que já existia — e a resposta mudou. O dashboard.Service e o projection.Service já leem os holdings do usuário através do port portfolio.Reader.ListHoldings. Se o portfolio.Service resolver a taxa efetiva uma vez, dentro do próprio ListHoldings, antes de devolver os dados, então o dashboard e a projeção recebem a taxa já resolvida — de graça, sem precisar saber que macro existe:

// internal/portfolio/service.go
// ListHoldings returns the caller's full set of holdings (SPEC-102 FR-1025). Each fixed-income
// holding carries its resolved EffectiveAnnualRateBps (SPEC-109 FR-1092/1093/1094) — this is the
// single seam the dashboard, projections, health, and chat all read holdings through, so they
// consume the resolved rate automatically with no macro dependency of their own.
func (s *Service) ListHoldings(ctx context.Context, userID string) (Holdings, error) {
    fii, err := s.repo.ListFIIHoldingsByUserID(ctx, userID)
    if err != nil {
        return Holdings{}, fmt.Errorf("list holdings: %w", err)
    }
    fixedIncome, err := s.repo.ListFixedIncomeHoldingsByUserID(ctx, userID)
    if err != nil {
        return Holdings{}, fmt.Errorf("list holdings: %w", err)
    }
    return Holdings{FII: fii, FixedIncome: s.withEffectiveRates(ctx, fixedIncome)}, nil
}

O resultado: um único port MacroReader, dono só do portfolio.Service; nenhum port novo em dashboard ou projection; e o Compute de ambos continua exatamente tão puro quanto antes — eles apenas leem h.EffectiveAnnualRateBps de um campo que já chega preenchido, em vez de h.AnnualRateBps. A mudança no dashboard, por exemplo, foi uma linha:

// internal/dashboard/compute.go
// EffectiveAnnualRateBps (SPEC-109), not the raw stored AnnualRateBps: for a prefixado
// holding they're equal; for cdi_percentual/ipca_spread it's the resolved current rate.
accrued := money.AccrueSimpleInterest(h.InvestedAmountCentavos, h.EffectiveAnnualRateBps, daysBetween(h.CreatedAt, now))

O próprio PLAN-109 registra essa mudança de rota honestamente — não como um erro corrigido às pressas, mas como uma decisão documentada na Fase 3 e na checklist de validação técnica:

// PLAN-109.md, Phase 3
- [x] internal/dashboard / internal/projection: implemented as a simplification of D2 —
      rather than each service owning its own MacroReader, portfolio.Service.ListHoldings
      (the single seam dashboard/projection/health/chat all already read holdings through)
      resolves EffectiveAnnualRateBps on every FI holding before returning Holdings.

// PLAN-109.md, §9 Technical Validation
- [x] Hexagonal layering intact ... (Note: dashboard/projection don't have their own
      MacroReader as D2 literally described — see Phase 3 note — they consume the
      already-resolved rate through the existing HoldingsReader seam instead.)
Por que isto é a lição mais valiosa da spec, e não um detalhe de rodapé Um PLAN é uma hipótese sobre como implementar uma SPEC, escrita antes de a IA olhar de perto para os seams que já existem no código. É perfeitamente correto — e é exatamente o comportamento que se espera do harness — que a Fase 3 verifique essa hipótese contra a realidade e ajuste o design quando encontra um caminho mais simples. "Copiar o padrão do health/insight/engine três vezes" era a suposição inicial porque esse padrão já tinha funcionado duas vezes; mas o health e o insight/engine precisam do macro para sua própria lógica de negócio (o tilt de mercado, os fatos do LLM), enquanto o dashboard/projeção só querem o número já pronto — o mesmo número que o portfolio.Service já calcula para responder GET /holdings/fixed-income. Multiplicar um padrão sem checar se o problema é realmente o mesmo é como copiar-e-colar sem entender a diferença — e o resultado bom aqui é fruto de não ter feito isso.

7. O endpoint GET /market/indicators e as DTOs estendidas

O handler novo é deliberadamente magro — só transporta o que o marketdata.MacroRepository já sabe fazer, sem lógica de negócio nova:

// internal/transport/http/market.go
// MarketDataReader is the slice of the market-data reader the transport needs (consumer-
// defined); *marketdata/postgres.MacroRepository satisfies it directly — the same port
// portfolio/health/insight-engine already depend on (SPEC-109 FR-1095).
type MarketDataReader interface {
    GetLatestMacroIndicator(ctx context.Context, ind marketdata.Indicator) (marketdata.MacroIndicator, error)
}

var publishedIndicators = []marketdata.Indicator{
    marketdata.IndicatorSELIC, marketdata.IndicatorCDI, marketdata.IndicatorIPCA,
}

Como toda spec que toca um endpoint, o contrato vive em lockstep na routeTable (internal/transport/http/routes.go) e em api/openapi.yaml, guardado pelo teste de drift (openapi_test.go) — sem user_id, porque SELIC/CDI/IPCA são dados de referência globais (BR-1095, espelha a BR-603 do SPEC-006), mas ainda assim atrás do gate de auth deny-by-default padrão do projeto:

GET /market/indicators
   → 200 [ {indicator, value_bps, reference_date}, ... ]  (possivelmente parcial, ou vazio)
   | 401 (sem sessão)

As DTOs de renda fixa (internal/transport/http/holdings.go) ganharam o mesmo par assimétrico que qualquer campo computado do projeto tem — aceito só na leitura, nunca na escrita:

// internal/transport/http/holdings.go
type fixedIncomeRequest struct {
    Name                   string  `json:"name"`
    Institution            string  `json:"institution"`
    InvestedAmountCentavos int64   `json:"invested_amount_centavos"`
    AnnualRateBps          int     `json:"annual_rate_bps"`
    IndexerType            string  `json:"indexer_type"`  // "" defaults to prefixado (SPEC-109 BR-1093)
    MaturityDate           *string `json:"maturity_date"`
    LiquidityType          string  `json:"liquidity_type"`
}

type fixedIncomeResponse struct {
    // ...
    AnnualRateBps int    `json:"annual_rate_bps"`
    IndexerType   string `json:"indexer_type"`
    // EffectiveAnnualRateBps is computed, never persisted (SPEC-109 FR-1092): the resolved
    // current rate for cdi_percentual/ipca_spread holdings; equal to AnnualRateBps for prefixado.
    EffectiveAnnualRateBps int `json:"effective_annual_rate_bps"`
    // ...
}

fixedIncomeRequest não tem effective_annual_rate_bps — um cliente que tentasse mandar esse campo no POST/PUT simplesmente teria o valor ignorado, porque fixedIncomeInput (a função que mapeia o request para portfolio.FixedIncomeInput) nunca lê esse campo. É o mesmo padrão "torne o estado ruim irrepresentável" das DTOs de outras specs computacionais: o valor computado só existe do lado da resposta.

#DecisãoEscolhaPor quê

D3Degradação quando o indicador falta? D5Número da migration nova? 0007_fixed_income_indexer (próxima livre após 0006_chat) Aditiva, com DEFAULT e CHECK; o down derruba a coluna com segurança a qualquer momento.

Bottom-up, como as specs anteriores — domínio primeiro, borda depois, cada fase compilando e revisável sozinha.

10. Ponte: arquitetura em camadas → hexagonal

Você já conhece a pilha clássica em camadas: controller → DTO → service → repository, com o service importando o repositório concreto. O SPEC-109 é um bom exemplo de como uma feature pequena ainda segue a tradução ponto a ponto — e de como o "grande giro" (inversão de dependência) aparece mesmo num port de uma linha. (Para a teoria completa, veja a aula conceitual de arquitetura hexagonal; aqui aplicamos ela ao SPEC-109.)

A "grande virada": a inversão de dependência

Em camadas, o service importa o cliente concreto que fala com o banco de macro — a seta aponta para fora, rumo à infra. No SPEC-109 o portfolio define a interface mínima de que precisa (ports.go) e o adapter que já existia para outra feature (SPEC-006) a satisfaz, sem saber que o portfolio existe:

// internal/portfolio/ports.go — o service DEFINE o port de que precisa (consumer-defined):
type MacroReader interface {
    GetLatestMacroIndicator(ctx context.Context, ind marketdata.Indicator) (marketdata.MacroIndicator, error)
}

// A peça que já existia o satisfaz — sem saber que o portfolio precisa dela:
//   marketdata/postgres.MacroRepository (SPEC-006)   já tem GetLatestMacroIndicator
//       → satisfaz portfolio.MacroReader, health.MacroReader e o MacroReader do internal/transport/http
//         (a MESMA instância concreta, passada em TRÊS slots de interface distintos no cmd/api)
// Todas as setas de dependência apontam PARA DENTRO.

Repare que o MacroReader desta spec é o quarto lugar do projeto a definir essa mesma interface mínima (health, insight/engine, e agora portfolio e transport/http/market.go) — todos consumer-defined, todos satisfeitos pela mesma instância concreta de MacroRepository. É a prova viva de "interfaces pequenas, definidas pelo consumidor": em vez de um port de macro "genérico" e grande compartilhado por todo o código, cada consumidor pede exatamente o método que usa.

Driving vs. driven nesta spec

  • Driving ports (quem aciona a aplicação): marketHandler (novo) e a extensão de holdingsHandler — ambos na borda, chamando os services por interfaces consumer-defined do próprio transporte.
  • Driven port novo (o que a aplicação aciona): portfolio.MacroReader. Fica à direita do hexágono — nenhum adapter novo foi escrito para satisfazê-lo, só uma nova fiação no cmd/api reaproveitando o MacroRepository do SPEC-006.
  • Domínio / core puro: Indexer, ResolveEffectiveRate, money.ApplyBps — sem SQL/HTTP/LLM/time. Aplicação: portfolio.Service (busca macro, resolve, anexa). Borda: os handlers HTTP, as DTOs, o repositório Postgres de portfolio (que não toca EffectiveAnnualRateBps).
┌──────── internal/portfolio (core da feature) ────────────────┐ HTTP ─▶ ListHoldings / │ Service.ListHoldings / withEffectiveRate(s) │ CreateFixedIncomeHolding / ─────▶│ │ │ UpdateFixedIncomeHolding │ ├─▶ Repository (Postgres) — NUNCA lê/escreve │ (driving) │ │ EffectiveAnnualRateBps (computado, não salvo) │ │ ├─▶ MacroReader.GetLatestMacroIndicator ◀── marketdata (006)│ │ └─▶ h.ResolveEffectiveRate(macro) [PURO, sem I/O] │ └─────────────────────────────────────────────────────────┘ │ dashboard.Service / projection.Service ◀── leem via portfolio.Reader/HoldingsReader (SEM MacroReader próprio — consomem EffectiveAnnualRateBps já resolvido) Acíclico: portfolio → {marketdata, money}. dashboard/projection → portfolio (sem novo port).

11. Harness Engineering: como a máquina construiu isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. Esta spec é um bom estudo de caso justamente por ser pequena: dá para ver o processo inteiro sem ruído.

O fluxo /spec-implement, fase a fase — o domínio antes da fiação

O PLAN-109 fatiou o trabalho em 7 fases bottom-up (§9), e a Fase 1 lançou o enum Indexer + ResolveEffectiveRate puros e testados antes de qualquer migration, service ou handler existir. Isso significa que quando a Fase 3 revisitou a decisão D2 (§6 desta aula), a matemática já estava provada — a mudança de plano foi só sobre onde chamar uma função já correta, nunca sobre reescrever o que ela calcula.

O hook determinístico block-layering

O hook block-layering (PostToolUse, exit 2) barra mecanicamente qualquer import de database/sql, net/http ou SDK de vendor dentro de um core de feature. É esse hook que garante, em tempo de escrita, que internal/portfolio's ResolveEffectiveRate ficou puro — nenhum import de database/sql aparece no arquivo — enquanto o import de internal/marketdata (o tipo Indicator, puro) segue permitido, porque portfolio já legitimamente depende de tipos de domínio do marketdata desde o SPEC-102 (D1, para o Ticker).

O subagente hexagonal-reviewer

Segundo o PLAN-109 §9, o hexagonal-reviewer revisou a spec e devolveu PASS, sem bloqueios — mas o detalhe mais interessante está no comentário que ele deixou passar registrado na checklist: a nota explícita de que "dashboard/projection não têm seu próprio MacroReader como D2 literalmente descrevia" — o revisor validou a simplificação de D2 em vez de exigir que o código seguisse o plano ao pé da letra, porque a regra de camadas (nenhum port de infra vazando do core, direção de dependência acíclica) continuava intacta com o seam único.

O go-correctness-reviewer e o bug real que ele encontrou

É o achado mais concreto desta sessão: o go-correctness-reviewer pegou getMarketIndicators engolindo qualquer erro de GetLatestMacroIndicator indiscriminadamente — inclusive uma falha real de infraestrutura — sem log e sem distinguir de "ainda não ingerido" (§5.2 desta aula). A correção foi pequena (um switch com dois casos + usar o campo logger que já existia no handler, mas estava sem uso), mas o achado é exatamente o tipo de bug que passa despercebido numa revisão manual apressada — os dois caminhos de erro parecem iguais até você perguntar "o que esse endpoint existe para fazer?".

Um segundo achado do revisor: a lacuna de cobertura por indicador O mesmo ciclo de revisão também sinalizou uma lacuna de teste: nada provava que a degradação de macro acontecia por indicador (CDI presente + IPCA ausente) em vez de tudo-ou-nada. A resposta foi o teste TestService_ListHoldings_ResolvesEffectiveRatePerIndicatorIndependently (§5.1) — fechando exatamente o buraco que o revisor apontou, antes do fechamento da spec.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • block-layering: o bloqueio duro que provou que ResolveEffectiveRate ficou livre de SQL/HTTP, ao mesmo tempo que permitiu o import legítimo de internal/marketdata (tipos puros).
  • O teste de drift do OpenAPI: ao adicionar GET /market/indicators na routeTable, o openapi_test.go exigiu a entrada correspondente em api/openapi.yaml — senão o build falharia (seção 7).
  • Bloqueio de imutabilidade: o SPEC-109 estende o SPEC-102 (a migration 0007 é aditiva, uma ALTER TABLE nova) e reaproveita o SPEC-006 sem editar nenhuma migration ou ADR já aceitos — se a IA tivesse tentado "editar" a migration original de renda fixa em vez de somar uma nova, o hook barraria.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — os testes unitários passam sem rede e sem DB (a resolução é pura, os fakes de macro são escritos à mão). A integração (migration round-trip, resolução contra um MacroRepository real) fica atrás do gate de TEST_DATABASE_URL + -p 1, pulando limpo.

12. Conexão com AI Engineering

O SPEC-109 não é uma feature de IA — zero chamada a LLM, zero prompt, zero Insighter. É por isso que a conexão mais honesta aqui não é "como isso melhora um insight", mas sim "como essa fundação prepara e reforça o que já existe do lado de IA".

12.1 — O mesmo princípio ("fatos são computados, não gerados"), sem LLM nenhum

A lição mais direta já apareceu na seção 2: EffectiveAnnualRateBps segue a mesma disciplina que governa a nota do Health Score (SPEC-106) e as séries de Projeção (SPEC-107) — um valor derivado nunca é gravado como se fosse permanente; ele é recalculado a cada leitura contra o estado atual do mundo. Ver essa regra aplicada a uma multiplicação simples (taxa × CDI ÷ 10000) ajuda a entender que "fatos computados, não gerados" é uma regra sobre honestidade temporal de um número, não uma regra específica de LLM — o que torna mais fácil reconhecer o mesmo padrão quando ele aparece disfarçado em outro contexto.

12.2 — Um fato mais rico que a IA futura vai ler de graça

O portfolio.Reader.ListHoldings é exatamente o seam que o Fact Builder (SPEC-104) e o Insighter/chat (SPEC-108) já consomem para montar os fatos que vão ao LLM. Como o SPEC-109 resolve a taxa efetiva dentro desse mesmo seam (§6), qualquer feature de IA futura que já lê holdings passa a enxergar a taxa correta e atual de um CDB pós-fixado automaticamente — sem precisar ser reescrita, sem precisar saber que indexadores existem. É um exemplo concreto de como investir num port bem desenhado paga dividendos em features que ainda nem foram planejadas.

12.3 — Sem LLM, sem superfície de prompt-injection

Como no SPEC-107 (Projeções), a ausência total de LLM nesta spec significa que não existe vetor de prompt-injection aqui: nenhum dado de holding ou de mercado é enviado a um modelo. A resolução da taxa é uma função pura de três inputs conhecidos — indexador, valor armazenado, snapshot de macro. É outro lembrete de que a decisão de não usar IA numa parte do sistema é, por si só, uma decisão de segurança.

A ideia que conecta tudo O SPEC-109 não ensina nada de novo sobre prompts ou grounding — ensina que o princípio de fundo por trás do grounding ("o modelo raciocina sobre fatos computados, nunca inventa números") é o mesmo princípio que já vale para qualquer número derivado no sistema, com ou sem LLM por perto. Uma taxa "120% do CDI" resolvida hoje e uma explicação de Health Score gerada hoje compartilham a mesma regra: o que é apresentado como verdade atual precisa ser recalculado, não lembrado.
