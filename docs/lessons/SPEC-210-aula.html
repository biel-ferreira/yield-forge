SPEC-210 — Tela de Perfil do Investidor

O SPEC-200 entregou a fundação — zero telas, todas as travas. O SPEC-210 é a primeira feature a preencher um slot: a tela Perfil, onde Rafael e Carla contam ao YieldForge quem são — o perfil de risco, os objetivos e o horizonte de investimento. É uma tela pequena e sem dinheiro, sem IA, sem gráfico. E é exatamente por isso que ela ensina uma lição importante: nem toda tela precisa das travas de explicabilidade e não-conselho — só as telas que produzem saída de IA precisam. O Perfil só captura dado; ele é o combustível que a Fase 1 do backend (SPEC-104/105/106) vai queimar depois. Você não precisa dominar React para acompanhar — o que importa aqui é produto, contrato e disciplina de harness.

1. A visão geral: a primeira tela a preencher um slot da fundação

O SPEC-200 prometeu que cada tela SPEC-21x "preenche um slot preparado" e "herda as guardas de graça". O SPEC-210 é a prova: ele pega a rota /profile, que até aqui era um EmptyState honesto dizendo "chega depois", e a transforma na tela real — a frente de cliente do SPEC-101, o backend que já existe e já funciona.

Em código real, o que mudou:

  • A rota web/app/(app)/profile/page.tsx deixou de renderizar um placeholder e passou a buscar GET /profile, mostrar o formulário (preenchido ou vazio) e salvar via PUT /profile.
  • Três controles de formulário novos e reutilizáveis entraram no design system: Segmented (single-select), ChipToggleGroup (multi-select) e Slider (range nativo), todos em web/components/ui/.
  • Um mapeamento centralizado enum↔pt-BR (web/lib/profile/labels.ts) tipado direto do contrato gerado.
  • Dois hooks de dados (web/lib/profile/profile.ts): useProfile (trata 404 como "sem perfil ainda") e useSaveProfile (PUT, invalida o cache no sucesso).
  • 6 testes Vitest/RTL novos + um scaffold Playwright, e uma verificação manual contra o backend Go rodando.
ANTES (SPEC-200 — a rota stubada) DEPOIS (SPEC-210 — a tela Perfil real) /profile → EmptyState("chega depois") /profile → formulário real, ligado ao SPEC-101 nenhum dado do usuário é lido/escrito GET /profile (prefill ou 404→vazio) + PUT /profile design system tinha só as guardas + primitivas +3 controles: Segmented, ChipToggleGroup, Slider nenhum hook de domínio existia lib/profile/{profile,labels}.ts, tipados do contrato 0 telas de produto 1a tela de produto -- o "SPEC-001" de feature do front
Por que começar pelo Perfil, e não pelo Dashboard O Perfil é a tela mais simples possível: sem dinheiro, sem gráfico, sem saída de IA. Começar por ela testa a fundação do SPEC-200 (cliente tipado, shell, sessão) contra o caso mais fácil antes de arriscar o caso mais difícil (Dashboard, com AllocationBar, InsightCard e dinheiro real). E, no produto, o Perfil é literalmente um pré-requisito: sem risco/objetivos/horizonte salvos, as telas de IA (SPEC-104/105/106, consumidas depois via SPEC-213) não têm personalização nenhuma para usar.

2. O que a tela entrega e para quem (Rafael & Carla)

O Perfil é a costura de personalização de todo o produto. Sem ele, cada insight, cada sugestão de rebalanceamento e cada Health Score seriam genéricos — calculados sem saber se o usuário tolera risco ou quer preservar patrimônio. É uma tela pequena, mas o valor dela é alavancado: tudo que a IA vai dizer depois depende do que esta tela captura hoje.

PersonaA dorO que a tela Perfil resolve
Carla, a planejadora de longo prazo focada em patrimônio Quer confirmar que o app "entende" que seu objetivo é preservação de patrimônio e um horizonte longo — não quer ser tratada como uma trader de curto prazo Um formulário direto onde ela marca Preservação de Patrimônio (e pode somar Crescimento de Longo Prazo) e desliza o horizonte para 20-30 anos — uma confirmação explícita e revisável, não uma suposição do sistema
Rafael, o investidor de FIIs que quer orientação sob medida Já usou apps que tratam todo investidor igual, sem levar em conta apetite a risco ao sugerir algo Ao declarar Agressivo + Renda Passiva, ele planta a semente para que, quando a SPEC-213 ligar os insights nesta tela, as sugestões considerem esse perfil — não um "meio-termo" genérico

E há uma dor comum às duas personas que a tela resolve com uma decisão simples: nada é salvo até um clique explícito em "Salvar". Ninguém quer que um deslize acidental no slider "grave sozinho" um horizonte errado.

3. A jornada do usuário: do 404 de primeiro acesso ao perfil salvo

A jornada é curta, mas cada bifurcação é uma decisão de produto:

1. Usuario abre /profile --> GET /profile 200 (perfil ja existe) 404 (perfil nunca foi definido) --> formulario PREENCHIDO --> formulario VAZIO "Seu perfil" "Defina seu perfil" risco/objetivos/horizonte ja marcados nenhum campo marcado; horizonte default = 10 2. Usuario ajusta risco (segmented), objetivos (chips), horizonte (slider) --> Salvar fica DESABILITADO ate: risco escolhido E >=1 objetivo E horizonte em [1,50] 3. Clica "Salvar" --> PUT /profile { risk_profile, objectives, horizon_years } (SEM user_id) 200 -> "Perfil salvo (check)"; cache invalidado 400 -> mensagem do servidor inline; dado permanece na tela formulario PRESERVA o que o usuario digitou Falha transitoria no meio do caminho (rede/5xx no load): estado de "tentar novamente", nunca uma tela em branco ou quebrada

Repare que 404 não é tratado como erro — é tratado como "primeiro acesso". A mesma ideia do SPEC-200 para sessão (useSession trata 401 como "deslogado", não como falha), aqui reaparece para dado de domínio:

// web/lib/profile/profile.ts — 404 é "sem perfil ainda", não uma falha (D3, espelha useSession)
queryFn: async () => {
  const { data, response } = await api.GET("/profile");
  if (response.status === 404) return null; // no profile yet — first run
  if (!data) throw new Error("failed to load profile");
  return data;
},
Por que "404 vira null" é o mesmo princípio duas vezes O SPEC-200 já tinha ensinado esse padrão para autenticação: um 401 confirmado em /auth/me vira "usuário deslogado" (null), nunca um erro genérico que dispararia um retry ou uma tela quebrada. O SPEC-210 aplica a mesma disciplina a um dado de domínio: um 404 confirmado em GET /profile vira "usuário sem perfil ainda" (null), que a tela interpreta como estado de primeiro acesso, não como falha. É o mesmo raciocínio replicado — status code específico e esperado ≠ erro genérico — e é por isso que o campo isError só é verdade para falha de rede/5xx de verdade, nunca para o 404 esperado.

4. As decisões de produto e UX — e por que os três controles

Cada um dos três controles do formulário corresponde a uma decisão de modelagem deliberada, não a um capricho visual:

CampoControle escolhidoPor quê
Perfil de risco Segmented (single-select, role="radiogroup") É mutuamente exclusivo por definição — ninguém é "Conservador e Agressivo" ao mesmo tempo. Um segmented torna essa exclusividade visível, e o valor selecionado é sempre óbvio de relance.
Objetivos ChipToggleGroup (multi-select, cada chip é um role="checkbox") Objetivos não são exclusivos — Carla pode querer Preservação de Patrimônio e Renda Passiva ao mesmo tempo. O contrato exige minItems: 1; a UI espelha isso bloqueando o Salvar com zero chips marcados.
Horizonte Slider nativo (<input type="range">, 1–50) Horizonte é um número contínuo dentro de um intervalo fechado — um slider comunica isso melhor que um campo de texto livre, e um range nativo é impossível de sair do intervalo por construção (min/max do próprio elemento), sem precisar de validação extra de "número fora do range".

E uma quarta decisão, transversal: salvar é um botão explícito, nunca autosave.

// web/app/(app)/profile/page.tsx — Salvar é explícito; nada grava sozinho
const valid = risk !== null && objectives.length >= 1 && horizon >= 1 && horizon <= 50;
// ...
<Button type="submit" disabled={!valid || save.isPending}>
  {save.isPending ? "Salvando…" : "Salvar"}
</Button>
Por que não autosave (a Decisão Resolvida #2 do SPEC-210) Um autosave a cada clique no slider geraria uma sequência de PUTs parciais e ruidosos — e, pior, salvaria um horizonte "de passagem" enquanto o usuário ainda está arrastando o controle em busca do número certo. Um botão "Salvar" explícito, desabilitado até o formulário ser válido e enquanto uma gravação está em voo, garante que o que está no banco é sempre uma escolha deliberada do usuário — nunca um estado intermediário acidental. É uma escolha de UX pequena com uma consequência grande: nenhum dado é gravado sem intenção explícita.

5. LIÇÃO PRINCIPAL — quando as travas do produto NÃO se aplicam (e por quê)

A aula do SPEC-200 mostrou o InsightCard com explanation obrigatório e o NonAdviceDisclaimer presente em toda superfície de IA. Se você abrir a tela Perfil esperando ver essas travas, não vai encontrar nenhuma — e isso é correto, não um esquecimento. Esta é a lição central desta aula.

O raciocínio, formalizado como BR-2103 na spec: "Profile has no monetary values and produces no AI text, so the explainability (FR-013), non-advice (FR-014), and money-format conventions do not apply here". É o espelho exato, no cliente, da mesma regra que o backend SPEC-101 já declara (BR-1016): "esta spec não emite nenhuma saída de IA, então FR-013/FR-014 não se aplicam diretamente".

O que uma tela de SAIDA DE IA precisa (ex.: Dashboard/Insights, SPEC-212/213) O que o Perfil precisa (SPEC-210) <InsightCard explanation={...}> <-- obrigatorio (FR-013) nenhum insight e renderizado aqui <NonAdviceDisclaimer /> <-- presenca e contrato da tela (FR-014) nenhum texto de IA e gerado aqui dinheiro em centavos, formatado so via money.ts nenhum valor monetario existe aqui ganho/perda = cor de numero, nunca acao nao ha numero de ganho/perda aqui ----------------------------------------------------------------------------------------------- A LICAO: FR-013/FR-014 gateiam SAIDA DE IA e DINHEIRO. Uma tela que so CAPTURA preferencia do usuario (risco/objetivos/horizonte) nao tem nada para essas travas gatearem. Aplica-las aqui seria burocracia sem funcao -- o oposto de "trava por construcao".
Por que essa ausência é uma decisão correta, não um buraco Uma trava-por-construção só faz sentido quando existe algo concreto para travar. O explanation obrigatório do InsightCard existe porque existe a possibilidade real de um insight sem "por quê" — e é essa possibilidade que o TypeScript precisa tornar irrepresentável. Na tela Perfil, não há insight, não há saída de IA, não há dinheiro: aplicar as mesmas travas seria como exigir um cinto de segurança num carrinho de compras. O frontend-reviewer sabe disso — o item 3 do seu checklist ("Binding product guards") é escopado explicitamente para "when the change touches AI / insight / suggestion output". O Perfil não toca; o revisor não cobra as guardas aqui, e isso é o comportamento certo, não uma omissão.

Guarde essa contraposição — ela volta na seção 10. A SPEC-213 (quando o Dashboard e os Insights começarem a ler o que o Perfil salvou) é onde as travas de FR-013/FR-014 reaparecem com força total, porque aí sim haverá saída de IA sendo renderizada a partir do que esta tela capturou.

7. Conexão com o backend: o contrato como fronteira

Esta seção ensina a costura frontend↔backend — o que substitui, no lado do cliente, a "ponte hexagonal" que a aula de backend ensinaria. O SPEC-210 não abre nenhum endpoint novo: ele consome dois que já existiam.

Endpoint (api/openapi.yaml)Usado porResposta que a tela trata
GET /profileuseProfile()200 ProfileResponse → prefill · 404 → primeiro acesso · 401 → SPEC-200 redireciona
PUT /profileuseSaveProfile()200 ProfileResponse → salvo, cache atualizado · 400 → envelope {"error":"..."} inline

7.1 — Tipos gerados, nunca escritos à mão

Assim como no SPEC-200, nenhum tipo de ProfileRequest/ProfileResponse foi escrito à mão. Eles vêm do schema gerado a partir de api/openapi.yaml:

// web/lib/profile/profile.ts — os tipos vêm do contrato GERADO, nunca escritos à mão
export type Profile = components["schemas"]["ProfileResponse"];
export type ProfileInput = components["schemas"]["ProfileRequest"];

7.2 — Os enums também vêm do contrato — e uma nova enum quebraria o build, não silenciosamente

Esta é a parte mais didática da costura nesta spec. O contrato declara risk_profile como um enum de três valores e objectives como um array de quatro valores possíveis:

// api/openapi.yaml — a fonte única dos enums
risk_profile:
  type: string
  enum: [conservative, moderate, aggressive]
objectives:
  type: array
  minItems: 1
  items:
    type: string
    enum: [retirement, passive_income, wealth_preservation, long_term_growth]

O mapeamento pt-BR não reimplementa esses valores — ele os deriva do tipo gerado, e usa um Record tipado como estrutura de garantia:

// web/lib/profile/labels.ts — o Record é tipado do enum GERADO; toda chave é obrigatória
type ProfileReq = components["schemas"]["ProfileRequest"];
export type Objective = NonNullable[number];

export const OBJECTIVE_LABELS: Record = {
  retirement: "Aposentadoria",
  passive_income: "Renda passiva",
  wealth_preservation: "Preservação de patrimônio",
  long_term_growth: "Crescimento de longo prazo",
};
O experimento mental: o que aconteceria se o backend adicionasse "esg_focused"? Imagine que amanhã o time de backend adiciona um quinto objetivo ao SPEC-101 — esg_focused — e roda gen:api no frontend para regenerar o schema.ts. O tipo Objective em labels.ts passaria a incluir automaticamente "esg_focused" na sua união (ele é NonNullable<ProfileReq["objectives"]>[number], derivado do schema). Mas OBJECTIVE_LABELS é declarado como Record<Objective, string> — um Record tipado exige todas as chaves da união. Sem a entrada esg_focused: "...", o TypeScript recusaria compilar: "Property 'esg_focused' is missing in type". Não é um teste que passaria silenciosamente e falharia em produção — é o compilador travando o build imediatamente. Essa é a diferença entre um DTO escrito à mão (que aceitaria qualquer string e deixaria a lacuna passar despercebida até o usuário ver um rótulo vazio em produção) e um tipo derivado do contrato: a deriva vira erro de compilação local, não bug de runtime.

7.3 — O frontend continua sendo apenas um cliente

O SPEC-210 não muda essa propriedade central do SPEC-200 — só a exercita de novo. O navegador chega ao Postgres do SPEC-101 pela mesma fronteira:

Navegador (ProfilePage) Next.js (mesma origem) API Go ----------------------- ----------------------- ------------------- Segmented / ChipToggleGroup / Slider /api/profile -- rewrite --> GET/PUT /profile useProfile() / useSaveProfile() proxy same-origin (D1 SPEC-200) (SPEC-101, Postgres) tipos: lib/api/schema.ts (GERADO) cookie HttpOnly Lax, sem CORS identidade = sessao (ctx) hoje: so o cliente web fala esse contrato -- amanha: app nativo / MCP / agente falam o MESMO /profile

7.4 — O guardião de deriva continua vigiando

Como o SPEC-210 não mudou o contrato, o npm run check:api do SPEC-200 rodou verde durante todo o desenvolvimento — a Definition of Done da spec inclui explicitamente "drift check green" como critério de aceite. É o mesmo guardião, agora exercitado por uma feature de verdade, não só pela fundação.

8. Como foi construído (visão geral, sem virar tutorial de React)

Você não precisa dominar React para acompanhar. O suficiente para entender a estrutura:

  • A rota. web/app/(app)/profile/page.tsx — dentro do shell autenticado do SPEC-200 (sidebar + RequireAuth), então herda a sessão e a navegação de graça.
  • Os dados. web/lib/profile/profile.ts — dois hooks TanStack Query sobre o cliente tipado do SPEC-200 (useProfile, useSaveProfile), no mesmo padrão de web/lib/auth/session.ts.
  • Os rótulos. web/lib/profile/labels.ts — o mapeamento enum↔pt-BR centralizado, tipado do schema gerado (seção 7.2).
  • Os controles. web/components/ui/segmented.tsx, chip-toggle.tsx, slider.tsx — três componentes genéricos, novos no design system Aurora, estilizados só com tokens (nada de hex solto).

Se você quiser se aprofundar, os arquivos reais estão apontados acima. A mensagem da aula continua sendo o porquê do produto e do contrato — não a mecânica de hooks ou de renderização do React.

9. Harness Engineering: como a máquina construiu e verificou isto

O SPEC-210 é a primeira vez que o harness de frontend, montado no SPEC-200, é exercitado por uma feature real — não só pela fundação. Vale a pena ver como ele se comportou.

O fluxo /spec-implement 210 em fases

O PLAN-210 fatiou o trabalho em 5 fases pequenas, cada uma deixando o web/ compilando/lintando/type-checando verde: (1) dados e mapeamento de rótulos, (2) os três controles de formulário, (3) a tela Perfil compondo os controles, (4) testes, (5) documentação e fechamento. A ordem é bottom-up deliberado: primeiro o que é testável isolado (hooks tipados, mapeamento), depois os componentes reutilizáveis, só então a tela que os compõe — o mesmo princípio de de-riscar as costuras mais arriscadas primeiro que o SPEC-200 usou para o contrato tipado e a auth.

FaseO que entregou
1 — Dados & rótuloslib/profile/labels.ts (enum↔pt-BR, tipado do contrato) + lib/profile/profile.ts (useProfile 404→null, useSaveProfile).
2 — Controles de formulárioSegmented, ChipToggleGroup, Slider — token-styled, com papéis ARIA (radiogroup/checkbox).
3 — A tela Perfilapp/(app)/profile/page.tsx compondo os três controles; validação de borda; Salvar explícito; feedback de sucesso/erro.
4 — Testes6 Vitest/RTL (rótulos, primeiro acesso vs. prefill, validação, o corpo exato do PUT) + Playwright gated + verificação manual contra o backend.
5 — Docs & fechamentoCHANGELOG, SPEC-210/PLAN-210 → Done, revisão pelos dois subagentes, esta aula.

Os dois subagentes de revisão — e os verdictos reais

Como no SPEC-200, a mudança passou pelos dois revisores de frontend, com divisão de trabalho clara. Ambos retornaram PASS — mas "PASS" não significa "nada a dizer": os dois levantaram observações não-bloqueantes, e vale a pena ler o que eles notaram, porque é um exemplo real de como um revisor cuidadoso pensa mesmo quando não está travando a entrega.

frontend-reviewer — PASS Verificou, em ordem de severidade: contrato-do-OpenAPI (os tipos de Profile/ProfileInput vêm do schema gerado, nenhum DTO à mão), dinheiro-sem-float (não se aplica — não há nenhum valor monetário nesta tela, confirmando a seção 5), as guardas binding (também não se aplicam — nenhuma saída de IA aqui, BR-2103), identidade-do-servidor (o PUT não carrega user_id), tokens-como-código (os três controles novos usam só classes Aurora — bg-primary, border-hairline, text-loss/text-gain, nada de hex solto) e acessibilidade (o Segmented usa role="radiogroup"/role="radio", o ChipToggleGroup usa role="checkbox", o Slider é um input type="range" nativo com aria-label). Verdict: PASS.
react-correctness-reviewer — PASS (com duas notas não-bloqueantes) Rodou typecheck/lint/build e leu o código de perto. Duas observações que não bloqueiam, mas valem a pena entender:
  • A string de fallback "morta" em useSaveProfile. Em throw new Error(backendError(error) ?? "save failed (" + response.status + ")"), o segundo ramo do ?? só dispararia se error existisse mas não tivesse o formato {"error":"..."} esperado — uma situação que, na prática, o openapi-fetch não produz no caminho de falha coberto pelos testes atuais. É um fallback defensivo razoável (não custa nada tê-lo), mas é código que hoje não tem um caminho de teste que o exercite — o tipo de nota que fica registrada para quando a superfície de erro crescer.
  • A suposição de "form-resync" em ProfileForm. O formulário inicializa seu estado a partir de initial via useState(initial?.risk_profile ?? null) — isso é seguro hoje porque ProfilePage só monta ProfileForm depois que isLoading resolve (o comentário no código é explícito sobre isso: "the form initializes from it on mount... it mounts only after the load resolves"). Mas é uma suposição implícita: se initial mudasse depois da montagem (por exemplo, um refetch de fundo trazendo um valor mais novo do servidor), o formulário não se re-sincronizaria sozinho — useState só lê o valor inicial uma vez. Não é um bug hoje, porque nada dispara esse refetch enquanto o formulário está montado; é uma fragilidade a vigiar se a tela ganhar polling ou refetch em background no futuro.
Verdict: PASS.
Por que vale a pena ler notas não-bloqueantes de um PASS É tentador achar que "PASS" quer dizer "nada a aprender aqui". O oposto é verdade: as duas notas acima mostram um revisor que não só roda typecheck/lint/build mecanicamente, mas lê a intenção do código — reconhece um fallback defensivo mas sem cobertura de teste, e reconhece uma suposição correta-mas-implícita sobre o ciclo de vida do componente. Nenhuma das duas é um bug agora. Mas são exatamente o tipo de nota que, se ignorada repetidamente por várias specs, vira dívida técnica silenciosa. Isso é harness engineering na prática: o revisor automatizado captura uma classe de risco que nem o typecheck nem o build jamais pegariam, porque ambos são sintaticamente válidos.

10. Conexão com AI Engineering & Produto

Uma tela sem IA e sem dinheiro pode parecer distante do "AI Engineering" do projeto. Mas o Perfil é, na verdade, o primeiro dado real que qualquer feature de IA futura vai ler — e é onde a lição da seção 5 fecha o círculo.

10.1 — O primeiro dado real para a Fase 1 da IA

SPEC-104 (motor de insights), SPEC-105 (assistente de rebalanceamento) e SPEC-106 (Health Score) já existem no backend e já sabem como ler um perfil de investidor — mas até o SPEC-210, nenhum usuário via um formulário para escrever esse perfil pela web. Agora existe: cada risco/objetivo/horizonte salvo por esta tela é literalmente o fato de entrada que o Fact Builder do backend vai considerar antes de qualquer Insighter gerar uma explicação. "Facts are computed, not generated" (a regra binding #3 do CLAUDE.md) começa, para cada usuário, com o que ele digita aqui.

10.2 — Por que a ausência das travas aqui prepara a SPEC-213

A seção 5 mostrou que FR-013/FR-014 não se aplicam ao Perfil porque ele não produz saída de IA. Mas essa ausência tem uma função pedagógica e de produto: ela isola exatamente onde as guardas precisam reaparecer. Quando a SPEC-213 ligar Insights/Dashboard para ler este perfil e gerar recomendações personalizadas, essas telas vão precisar do InsightCard com explanation obrigatório e do NonAdviceDisclaimer de novo — porque aí sim haverá algo (uma sugestão de IA calibrada pelo perfil de risco de Rafael) para essas travas gatearem. O SPEC-210 e a futura SPEC-213 formam um par didático: captura sem guarda → saída com guarda.

10.3 — A costura para a visão multi-agente / MCP

Como no SPEC-200, o Perfil reforça a mesma propriedade: o frontend não tem lógica de domínio própria sobre o que significa "conservador" ou "agressivo" — ele só renderiza o enum e delega tudo ao contrato. Isso significa que, quando o produto evoluir para um CIO multi-agente ou um servidor MCP, esses agentes vão poder ler o mesmo GET /profile que esta tela lê — sem nenhuma reescrita de frontend. O perfil de um usuário não é "propriedade da tela React"; é propriedade da API, e a tela é só mais um cliente dela.

A ideia que conecta tudo A engenharia de IA responsável não é só "gatear a saída da IA" — é também saber quando não gatear. O SPEC-210 ensina essa segunda metade: uma tela de captura de preferência não precisa das travas de uma tela de saída de IA, e forçar essas travas onde não cabem seria burocracia, não segurança. A disciplina real está em saber identificar, spec a spec, onde a trava é estrutural e onde ela é irrelevante — e documentar essa decisão (BR-2103) para que o próximo desenvolvedor (ou a próxima IA) não a reintroduza por engano.
