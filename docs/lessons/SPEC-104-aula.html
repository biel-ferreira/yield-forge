Aula 104 · O Motor de Insights · A 1ª Feature de IA que CHEGA AO USUÁRIO

SPEC-104 — AI Insight Engine (Fact Builder + Insights Explicáveis)

É aqui que todo o stack vira produto: o motor pega os fatos determinísticos (carteira computada + perfil + macro), entrega ao Insighter (SPEC-005) e devolve insights explicáveis sobre carteira, alocação e contexto de mercado — sem nunca inventar um número. A lição central são as duas travas binding por construção: os fatos são computados, não gerados (o Fact Builder), e o texto de IA sai só pela porta com gates — então explicabilidade (FR-013) e não-aconselhamento (FR-014) são inevitáveis. Não há caminho que os burle.

1. A visão geral: o motor que transforma fatos em insights explicáveis

Tudo até aqui foi fundação. O perfil (SPEC-101) instalou quem é o investidor; a carteira (SPEC-102) guardou o que ele tem (o cost basis); o mercado (SPEC-006) ingeriu cotações e indicadores macro; o dashboard (SPEC-103) computou o patrimônio, a alocação e a exposição setorial; e o port Insighter (SPEC-005) plantou o seam único por onde o LLM fala, já com os gates de explicabilidade e não-aconselhamento embutidos. O SPEC-104 é a feature que amarra tudo e finalmente cumpre a promessa central do produto: "detectar concentração, desequilíbrio e risco sem análise manual".

O que ele entrega é o subpacote internal/insight/engine com três peças e um endpoint:

  • O Fact Builder (FactBuilder.BuildFacts(ctx, userID)) — compõe um snapshot determinístico de fatos a partir dos seams do dashboard + perfil + macro. É a seam publicada que o chat (SPEC-108) e o rebalanceamento/health score (SPEC-105/106) vão reusar.
  • O engine service (engine.Service.Insights(ctx, userID)) — constrói os fatos uma vez e chama o Insighter uma vez por categoria (portfolio, allocation, market_context), agregando os resultados gated.
  • O enum fechado Category que mapeia 1:1 para uma insight.Task, e os ports consumidores (DashboardReader, ProfileReader, MacroReader).
  • Um único endpoint GET /insights. Nenhuma tabela, nenhuma migration, nenhuma escrita.
ANTES (SPEC-005 / 101 / 103 / 006) DEPOIS (SPEC-104) Insighter: seam do LLM + gates (pronto) engine: orquestra fatos → Insighter → insights dashboard: COMPUTA patrimônio/alocação Fact Builder: junta dashboard + perfil + macro perfil: quem é o investidor um fato determinístico, N chamadas por categoria macro: SELIC/CDI/IPCA ingeridos GET /insights → insights explicáveis + disclaimer cada peça pronta, nada conversa 1ª feature de IA que CHEGA AO USUÁRIO fatos computados (SPEC-103) fatos COMPUTADOS viram grounding do LLM

O endpoint novo (atrás da auth deny-by-default, fora da allowlist pública):

GET /insights  → 200 { insights[]{category,title,detail,explanation}, disclaimer, available } | 401
Por que esta é a spec "capstone" — e por que ela é mais sobre TRAVAS do que sobre IA Cada spec anterior existiu para tornar esta confiável. Os fatos são computados (SPEC-103), não inventados; o LLM só raciocina sobre eles; e as guardas binding tornam a saída segura por construção. O segredo do SPEC-104 é que ele quase não escreve código de IA: ele não constrói prompt, não chama provider, não checa explicação nem disclaimer — tudo isso já mora no Insighter (SPEC-005). O motor só monta fatos e passa pela porta certa. É justamente essa economia — fazer o mínimo, pela única via gated — que torna as travas inescapáveis.

2. O Fact Builder (o coração): BuildFacts(ctx, userID)

A peça central do SPEC-104 é o Fact Builder — e ele é a encarnação direta da regra binding nº 3 do PRD: "fatos são computados, não gerados". Sua única responsabilidade é compor um insight.Facts (um map[string]any) determinístico a partir de três seams de leitura, sem nenhum texto livre e sem nenhum float. Dinheiro entra como int64 centavos; taxas e fatias como int em basis points. O LLM recebe esse pacote e raciocina sobre ele — nunca calcula um saldo.

// internal/insight/engine/factbuilder.go
// FactBuilder compõe o snapshot determinístico para um usuário a partir dos seams do dashboard,
// perfil e macro (SPEC-104 FR-1041). É a seam PUBLICADA (BuildFacts) que SPEC-108/105/106 reusam.
// Todo dinheiro é int64 centavos e toda taxa/fatia é int basis points — nunca float (BR-1044);
// os mesmos inputs sempre dão os mesmos fatos (BR-1041). Só valores COMPUTADOS — sem invenção.
func (b *FactBuilder) BuildFacts(ctx context.Context, userID string) (insight.Facts, error) {
    d, err := b.dashboards.GetDashboard(ctx, userID)   // ← reusa a Compute do SPEC-103
    if err != nil {
        return nil, fmt.Errorf("build facts: dashboard: %w", err)
    }

    facts := insight.Facts{
        "total_invested_centavos": d.Summary.TotalInvestedCentavos,
        "current_value_centavos":  d.Summary.CurrentValueCentavos,
        "monthly_income_centavos": d.Summary.MonthlyIncomeCentavos,
        "growth_bps":              d.Summary.GrowthBps,
    }
    // ... allocation_bps, fii_sector_bps, largest_fii_sector_bps, stale_tickers ...
    if err := b.addProfileFacts(ctx, userID, facts); err != nil { return nil, err }
    if err := b.addMacroFacts(ctx, facts); err != nil { return nil, err }
    return facts, nil
}

2.1 — O que entra no snapshot (e de onde vem)

FatoChave(s)Seam (spec)

Por que o Fact Builder REUSA a Compute do dashboard em vez de recomputar (D3) O dashboard (SPEC-103) já cruza holdings × cotações numa função pura e determinística e produz valor atual, alocação e exposição setorial — tudo reconciliado, em centavos e bps. O Fact Builder lê esse resultado via GetDashboard em vez de refazer o cálculo. Por quê? Porque um único caminho de computação significa que o que a tela mostra e o que a IA raciocina são o mesmo número. Se houvesse dois caminhos, eles poderiam divergir por um arredondamento, e a explicação da IA descreveria um valor diferente do que o usuário vê — a explicabilidade perderia o chão. A Compute do SPEC-103 foi desenhada, desde sempre, para ser o substrato deste Fact Builder.

2.2 — Degradação graciosa NA COMPOSIÇÃO dos fatos

O dashboard é a espinha dorsal — se ele falha, o fato falha (o erro borbulha com %w). Mas o perfil e o macro são enriquecimentos opcionais: o investidor pode não ter preenchido o perfil ainda, ou um indicador macro pode estar faltando. A decisão é omitir graciosamente, nunca quebrar:

// internal/insight/engine/factbuilder.go — perfil ausente é ESTADO, não erro
func (b *FactBuilder) addProfileFacts(ctx context.Context, userID string, facts insight.Facts) error {
    p, err := b.profiles.GetProfile(ctx, userID)
    if errors.Is(err, profile.ErrProfileNotFound) {
        return nil   // perfil não setado → omite os campos de perfil, segue em frente
    }
    if err != nil {
        return fmt.Errorf("build facts: profile: %w", err)   // erro REAL borbulha
    }
    facts["risk_profile"] = string(p.Risk)
    facts["objectives"]   = objectives
    facts["horizon_years"] = p.Horizon.Years()
    return nil
}

// addMacroFacts: um indicador faltando (ErrMacroNotFound) é PULADO, não um erro:
for _, ind := range macroFacts {  // SELIC, CDI, IPCA — IFIX é o gap documentado do SPEC-006
    m, err := b.macro.GetLatestMacroIndicator(ctx, ind)
    if errors.Is(err, marketdata.ErrMacroNotFound) { continue }   // omite ESTE fato
    if err != nil { return fmt.Errorf("build facts: macro %s: %w", ind, err) }
    macro[string(ind)+"_bps"] = m.Value
}
A disciplina do errors.Is: distinguir "não tem" de "deu erro" Repare no padrão, idêntico ao do dashboard com ErrFIIQuoteNotFound: a sentinela específica (ErrProfileNotFound, ErrMacroNotFound) é tratada como ausência graciosa (omite o fato, segue), e qualquer outro erro borbulha com %w. Confundir os dois seria desastroso: tratar um erro de banco como "perfil ausente" esconderia uma falha real; tratar "perfil ausente" como erro quebraria os insights de quem ainda não preencheu o perfil. A regra "erros são sentinelas, checadas com errors.Is" é o que mantém essa fronteira limpa.

3. LIÇÃO PRINCIPAL — as travas POR CONSTRUÇÃO: o texto de IA sai SÓ pelo Insighter

Esta é a ideia mais importante da aula, e a razão de o SPEC-104 ser tão pequeno. As duas guardas binding do produto — explicabilidade (FR-013) e não-aconselhamento (FR-014) — não são re-implementadas aqui. Elas já vivem no Insighter (SPEC-005), como decorators de gate. O motor garante que elas valem não por checá-las, mas por uma propriedade arquitetural: ele só emite texto de usuário pela porta que tem os gates. Não existe outro caminho.

// internal/insight/engine/engine.go — o engine NUNCA constrói texto de IA; só passa pela porta
for _, c := range AllCategories {
    res, err := s.insighter.Generate(ctx, insight.InsightRequest{   // ← a ÚNICA via
        Facts:  facts,
        Task:   c.Task(),
        UserID: userID,
    })
    if err != nil { /* trata degradação/rejeição — seção 5 */ continue }
    for _, in := range res.Insights {
        in.Category = string(c)   // só TAGUEIA a categoria; não toca no texto gerado
        items = append(items, in)
    }
}
Por que "por construção" é mais forte que "por verificação" Há duas formas de garantir uma regra. A frágil: o motor gera um texto e depois verifica se ele tem explicação e não tem conselho — fácil esquecer um caminho, fácil um novo dev adicionar um atalho que pula a checagem. A robusta, escolhida aqui: o motor não tem como produzir texto de IA — a única coisa que ele sabe fazer é insighter.Generate(...), e essa chamada já passa por explicabilidade + não-conselho + cache + telemetria (a cadeia de decorators do SPEC-005). A Insight que volta já está gated; o motor apenas a tagueia com a categoria e a agrega. Como não existe um segundo gerador de strings, não existe um caminho para burlar os gates. O go-correctness-reviewer e um teste dedicado provam exatamente isso: nenhum texto de usuário nasce fora do Insighter.
O ÚNICO caminho do texto de IA até o usuário: facts (computados) ─▶ insighter.Generate(facts, task, userID) │ ▼ cadeia de decorators do SPEC-005 ┌──────────────────────────────────────────┐ │ cache → gate(FR-013 explica + FR-014 não │ │ aconselha) → telemetria → provider (LLM) │ └──────────────────────────────────────────┘ │ ▼ Insight JÁ gated (explanation + sem ordem) engine: só TAGUEIA a categoria e agrega ─▶ GET /insights ─▶ usuário Não há um segundo gerador de strings. Burlar os gates é IMPOSSÍVEL por construção.

O endpoint reflete isso no DTO: cada item carrega um campo explanation que está sempre presente (o gate de explicabilidade o garante), e a resposta carrega o disclaimer de não-aconselhamento:

// internal/transport/http/insights.go
type insightItemResponse struct {
    Category    string `json:"category"`
    Title       string `json:"title"`
    Detail      string `json:"detail"`
    Explanation string `json:"explanation"` // sempre presente — o gate de explicabilidade garante
}

4. Um fato, N categorias: portfolio / allocation / market_context (D2)

O motor produz três famílias de insight, cada uma respondendo a um FR do PRD. A decisão D2 foi: construir UM fato determinístico e chamar o Insighter uma vez por categoria, cada chamada com uma Task distinta. O enum Category mapeia 1:1 para a insight.Task:

// internal/insight/engine/category.go
type Category string
const (
    CategoryPortfolio     Category = "portfolio"      // concentração, desequilíbrio setorial, risco (FR-008)
    CategoryAllocation    Category = "allocation"     // alocação vs perfil, concentração (FR-009)
    CategoryMarketContext Category = "market_context" // macro ligado à carteira (FR-010)
)
// AllCategories fixa a ordem em que o motor gera e a resposta apresenta.
var AllCategories = []Category{CategoryPortfolio, CategoryAllocation, CategoryMarketContext}

func (c Category) Task() insight.Task { return insight.Task(c) }   // mesma string
CategoriaO que produzFR
portfolioConcentração, desequilíbrio setorial, exposição a risco e diversificação — sobre os fatos da carteira.FR-008
allocationAlinhamento da alocação (mix de classes vs perfil de risco/objetivos) e concentração em setor/ativo único.FR-009
market_contextLiga condições macro (SELIC alta, inflação) ao split FI/FII da carteira — como considerações, nunca diretivas.FR-010
Por que 3 chamadas ao LLM não saem caras — o cache do SPEC-005 "Um request, três chamadas ao LLM" soa lento e caro (e os free-tiers têm limite). Mas a cadeia do SPEC-005 tem um cache que chaveia por (user, task, facts): como o Fact Builder produz fatos determinísticos, dois requests seguidos com a mesma carteira batem no cache e não chamam o LLM de novo. O determinismo do Fact Builder não é só sobre confiabilidade — é o que torna o cache possível. Se os fatos mudassem sozinhos (um float, um time.Now() escondido, uma ordem de map), a chave do cache mudaria a cada chamada e o cache nunca acertaria. Determinismo → cache → custo zero (a postura binding nº 4 do PRD).

5. Estados graciosos: vazio, outage total, parcial

O mundo real falha de formas variadas, e o motor representa cada uma como um estado, não um erro 500. Há três casos, e o código de Insights os trata explicitamente.

5.1 — Carteira vazia → SEM chamada ao LLM

Se o investidor não tem holdings, não há nada sobre o que raciocinar — e seria desperdício (e custo) chamar o LLM. O motor faz short-circuit com um estado amigável antes de qualquer chamada:

// internal/insight/engine/engine.go
facts, err := s.facts.BuildFacts(factCtx, userID)
// ...
// Carteira vazia: nada a analisar — estado amigável "available, mas sem itens", SEM LLM.
if !hasHoldings(facts) {
    return Insights{Disclaimer: insight.Disclaimer, Available: true}, nil
}
// hasHoldings lê os centavos que o BuildFacts escreveu — int64, nunca float.
func hasHoldings(facts insight.Facts) bool {
    invested, _ := facts["total_invested_centavos"].(int64)
    current, _  := facts["current_value_centavos"].(int64)
    return invested != 0 || current != 0
}

5.2 — Outage total → 200 available:false (D5)

Se o LLM está totalmente fora (rate-limit, provider caído), o motor não devolve 503. A decisão D5 é mais amigável para a UI: devolve 200 com available:false e itens vazios — o dashboard e a carteira continuam usáveis; só a faixa de insights mostra "temporariamente indisponível".

5.3 — Resultado PARCIAL quando só algumas categorias falham

O ponto mais elegante: cada categoria é independente. Se a chamada de market_context degrada mas portfolio e allocation respondem, o motor devolve os dois que deram certo — um resultado parcial, não um tudo-ou-nada. O laço conta os sucessos:

// internal/insight/engine/engine.go — o coração da degradação parcial
var items []insight.Insight
succeeded := 0
for _, c := range AllCategories {
    res, err := s.insighter.Generate(ctx, insight.InsightRequest{Facts: facts, Task: c.Task(), UserID: userID})
    if err != nil {
        // Cancelamento/timeout do request ABORTA — não degrada silenciosamente.
        if ctx.Err() != nil {
            return Insights{}, fmt.Errorf("insights: %w", ctx.Err())
        }
        continue   // esta categoria degradou ou foi rejeitada pelo gate — pula (FR-1047)
    }
    succeeded++
    for _, in := range res.Insights {
        in.Category = string(c)
        items = append(items, in)
    }
}
return Insights{
    Items:      items,
    Disclaimer: insight.Disclaimer,
    Available:  succeeded > 0,   // false SÓ quando TODAS as categorias falharam
}, nil
O detalhe sutil: cancelamento NÃO é degradação Olhe o if ctx.Err() != nil dentro do tratamento de erro. Um erro da categoria normalmente vira continue (degrada graciosamente). Mas se o contexto foi cancelado — o cliente desconectou, ou o request estourou o timeout — isso não é "esta categoria está indisponível"; é "todo o request acabou". Tratar um cancelamento como degradação faria o motor seguir tentando as próximas categorias num contexto morto, e talvez devolver um "parcial" enganoso. A checagem distingue os dois: cancelamento aborta com o erro do contexto; uma falha real da categoria pula para a próxima. É o tipo de cuidado que separa degradação correta de degradação que esconde um problema.

6. A arquitetura (D1): o motor na borda, o core insight puro

A decisão D1 foi pôr o motor dentro da feature internal/insight (que já é dona do port Insighter + gates desde o SPEC-005), mas num subpacote engine. Por quê essa separação? Porque o core de insight (o port, os gates, o domínio Facts/Insight) precisa continuar puro — sem importar dashboard, perfil ou macro. O subpacote engine é a camada de aplicação da feature: é ele que compõe os seams de outras features. O doc.go registra essa fronteira:

// internal/insight/engine/doc.go
// Package engine é o AI Insight Engine (SPEC-104): constrói fatos determinísticos e os
// transforma em insights explicáveis de Portfolio / Allocation / Market-Context.
//
// É a CAMADA DE APLICAÇÃO da feature insight — o core puro de insight (o port Insighter +
// gates, SPEC-005) fica no pacote pai; este subpacote LÊ os seams do dashboard (SPEC-103),
// perfil (SPEC-101) e macro (SPEC-006) e ORQUESTRA o Insighter.

O motor lê os seams através de ports consumidores — interfaces pequenas que o próprio engine define (accept interfaces), satisfeitas na fiação por dashboard.Service, profile.Service e o repo macro:

// internal/insight/engine/ports.go
type DashboardReader interface {
    GetDashboard(ctx context.Context, userID string) (dashboard.Dashboard, error)
}
type ProfileReader interface {
    GetProfile(ctx context.Context, userID string) (profile.Profile, error)
}
type MacroReader interface {
    GetLatestMacroIndicator(ctx context.Context, ind marketdata.Indicator) (marketdata.MacroIndicator, error)
}
Por que engine é um subpacote e não a feature inteira — o grafo continua acíclico e puro Três coisas tornam essa estrutura saudável: (1) O core insight não ganha nenhum import novo de dashboard/perfil/macro — ele continua a costura pura do LLM, reusável por qualquer um. É o subpacote engine que importa essas features, e ele é a aplicação, não o domínio. (2) O grafo é acíclico: engine → {dashboard, profile, marketdata, insight}; nenhum deles importa engine de volta. O motor fica "acima" — um orquestrador que compõe camadas mais baixas. (3) Tanto o Fact Builder quanto o Service importam só os tipos de domínio puros das outras features (e o port Insighter); jamais database/sql ou net/http. O hook block-layering garante isso mecanicamente. A identidade vem sempre de auth.UserID(ctx) — nunca do payload (BR-1043).
┌──────── internal/insight/engine (aplicação, "acima") ─────────┐ HTTP ─▶ getInsights ─▶ │ Service.Insights (orquestra) │ (driving) │ │ │ │ ├─▶ FactBuilder.BuildFacts(ctx, userID) │ │ │ GetDashboard ◀── DashboardReader ◀── dashboard.Service │ │ │ GetProfile ◀── ProfileReader ◀── profile.Service │ │ │ GetLatestMacro◀─ MacroReader ◀── macro repo (006) │ │ │ │ │ └─▶ insighter.Generate(facts, task) (×N categorias) │ │ ▲ a ÚNICA via do texto de IA (gates SPEC-005) │ └────────────────────────────────────────────────────────────┘ Grafo acíclico: engine → {dashboard, profile, marketdata, insight}. Core insight PURO.

7. O endpoint GET /insights e o contrato

O handler é deliberadamente magro: lê a identidade do contexto, chama o serviço, mapeia para o DTO. Repare que ele não conhece estados de erro de IA — vazio e degradado são ambos 200 (D5), distinguidos pelo available no corpo. Só um erro real de serviço vira 500.

// internal/transport/http/insights.go
func (h insightsHandler) getInsights(w http.ResponseWriter, r *http.Request) {
    userID, ok := callerID(w, r)   // identidade do contexto, nunca do payload (BR-1043)
    if !ok { return }
    res, err := h.service.Insights(r.Context(), userID)
    if err != nil {
        h.logger.ErrorContext(r.Context(), "get insights failed", slog.String("error", err.Error()))
        writeError(w, http.StatusInternalServerError, "internal error")
        return
    }
    writeJSON(w, http.StatusOK, toInsightsResponse(res))   // vazio E degradado → 200
}

Como toda spec que toca um endpoint, o contrato vive em lockstep em dois lugares, guardados pelo teste de drift: a routeTable (internal/transport/http/routes.go) e api/openapi.yaml. O dinheiro aparece como inteiro (*_centavos) e taxa como *_bps — o ban do float se estende ao contrato:

# api/openapi.yaml
  /insights:
    get:
      operationId: getInsights
      summary: Generate the caller's explainable portfolio insights
      description: |-
        AI insights across categories (portfolio, allocation, market_context), grounded in
        computed facts and emitted only through the gated Insighter — every insight carries an
        explanation (FR-013) and never a transaction order (FR-014); the response carries the
        non-advice disclaimer. `available` is false when the LLM was fully unavailable; an empty
        portfolio returns `available:true` with no insights.
      responses:
        "200": { ... $ref InsightsResponse ... }
        "401": { $ref: Unauthorized }

8. As 6 decisões (D1–D6) e por quê

#DecisãoEscolhaPor quê
D1Onde mora o motor? Em internal/insight (subpacote engine), a feature que já é dona do Insighter + gates O motor é a camada de aplicação da feature insight; o core (port + gates) fica puro no pacote pai. Acíclico, sem SQL/HTTP no core.
D2Um fato ou N? Um fato determinístico, N chamadas por categoria Cada categoria tem uma Task distinta; o cache do SPEC-005 chaveia por (user, task, facts), então repetições saem baratas.
D3Recomputar a carteira? Não — reusa dashboard.GetDashboard Um único caminho de computação: a tela e a IA raciocinam sobre o mesmo número. Sem divergência de arredondamento.
D4Histórico de insights (FR-022)? Deferido a um follow-up focado A memória conversacional que o usuário queria é o chat (SPEC-108, FR-025) — threads limitadas e limpáveis. Insight-history é concern separado; BuildFacts deixa pronto para reuso.
D5Forma do estado degradado? 200 com available:false + itens vazios Mais amigável para a UI que um 503; parcial 200 quando só algumas categorias falham. Dashboard/carteira seguem usáveis.
D6Fact Builder privado ou publicado? Exportado FactBuilder / BuildFacts(ctx, userID) É a seam de grounding que SPEC-108 (via FactSource) e SPEC-105/106 reusam — não um passo privado do motor. Estável e exportada.

9. As 7 fases, uma a uma

Bottom-up, como nas specs anteriores — core/domínio primeiro, borda depois, cada fase compilando e revisável sozinha. Com "fatos computados, não gerados" e "guardas por construção" como os dois fios condutores.

Fase 1 — Domínio, tasks & ports de entrada

O enum fechado Category (portfolio | allocation | market_context) mapeando 1:1 para insight.Task, com a lista de ordem fixa AllCategories; o tipo de resultado Insights{ Items, Disclaimer, Available }; e os três ports consumidores (DashboardReader, ProfileReader, MacroReader). Adições puras a internal/insight/engine, compilando contra os tipos existentes de dashboard/profile/marketdata/insight.

Fase 2 — O Fact Builder (o coração)

BuildFacts(ctx, userID): lê o dashboard (valor atual, alocação bps, exposição setorial bps, concentração do maior setor, stale tickers), o perfil (risco/objetivos/horizonte) e o macro (SELIC/CDI/IPCA) → monta um insight.Facts determinístico (money centavos / rates bps). Carteira vazia → fatos de estado vazio; perfil não setado → omite campos de perfil; macro faltando → omite o fato. Testes table-driven provando fatos determinísticos esperados, vazio, sem-perfil, macro-faltante e NENHUM float em qualquer valor — sem banco nem HTTP.

Fase 3 — O Engine (guardas por construção)

O Service.Insights(ctx, userID): short-circuit da carteira vazia (sem LLM); senão, BuildFacts uma vez, depois Insighter.Generate por categoria (taguendo cada insight), agregando os resultados gated + disclaimer; categoria degradada/rejeitada → pulada (parcial); Available:false só no outage total. Fakes escritos à mão (o insight.Fake determinístico + fake readers). Inclui o teste-chave: o motor emite texto de IA SÓ via o Insighter.

Fase 4 — API (transporte)

O transport/http/insights.go: GET /insights → DTO (insights taguados por categoria + disclaimer + available), identidade via auth.UserID(ctx) (callerID), vazio/degradado → 200 (D5), erro de serviço → 500. Deps.Insights, registro da rota auto-protegida na routeTable, o path + schema em api/openapi.yaml (drift verde) e a fiação no cmd/api (Insighter via factory + Fact Builder + engine).

Fase 5 — Observabilidade

O endpoint herda o span de rota do otelhttp (GET /insights); um span insight.facts para o fact-building sem PII (nenhum dinheiro/figura/perfil/texto gerado vai para telemetria); o Insighter grava seus spans insight.generate por categoria (provider/model/outcome/cache-hit, sem conteúdo — SPEC-005 BR-505). Um teste garante nenhum fato/PII no span.

Fase 6 — Testes

Unitários sem banco: o Fact Builder table-driven (determinismo/vazio/sem-perfil/macro-faltante); o engine com fakes (agregação, vazio, degradação → indisponível, parcial, só-via-Insighter); handler (identidade-do-contexto, vazio → 200, degradado available:false, 401). Integração gated: Postgres real (TEST_DATABASE_URL, -p 1) + o fake Insighter — semeia holdings + cotações + perfil + macro, chama GET /insights e asserta que todo insight carrega uma explicação e o disclaimer está presente (os gates valem end-to-end), com isolamento por usuário.

Fase 7 — Documentação & aula

README.md (endpoint /insights), CHANGELOG.md, api/openapi.yaml (path + schema), SPEC-104 + PLAN-104 marcados Done, índices e status do CLAUDE.md atualizados, e esta aula. 🙂

10. Ponte: arquitetura em camadas → hexagonal

Você já conhece a pilha clássica em camadas: controller → DTO → service → repository, com o service importando o repositório concreto. O SPEC-104 é um caso rico para a tradução porque o "repositório" do motor não é um banco — são os seams de leitura de outras features e o port do LLM. (Para a teoria completa, veja a aula conceitual de arquitetura hexagonal; aqui aplicamos ela ao SPEC-104.)

A "grande virada": a inversão de dependência

Em camadas, o service importa o repositório concreto e o cliente de LLM concreto — a seta aponta para fora, rumo à infra. No SPEC-104 a virada é dupla: o motor define as interfaces que precisa (os três readers) e features vizinhas as satisfazem; e o texto de IA passa pelo port Insighter que o core insight definiu — a seta aponta para dentro:

// O engine DEFINE os ports de que precisa (internal/insight/engine/ports.go):
type DashboardReader interface { GetDashboard(ctx, userID) (dashboard.Dashboard, error) }
type ProfileReader   interface { GetProfile(ctx, userID)   (profile.Profile, error) }
type MacroReader     interface { GetLatestMacroIndicator(ctx, ind) (marketdata.MacroIndicator, error) }

// As OUTRAS features os satisfazem — sem saber que o engine existe:
//   dashboard.Service  já tem GetDashboard            → satisfaz DashboardReader
//   profile.Service    já tem GetProfile              → satisfaz ProfileReader
//   o repo macro (006) já tem GetLatestMacroIndicator → satisfaz MacroReader
//   o Insighter (005)  é o driven port do LLM, com gates → a única via do texto de IA
// A fiação acontece só no cmd/api. Todas as setas de dependência apontam PARA DENTRO.

Driving vs. driven nesta spec

  • Driving port (quem aciona a aplicação): o insightsHandler chama o serviço pela interface InsightsEngine definida no próprio transporte (consumer-defined). Fica à esquerda do hexágono.
  • Driven ports (o que a aplicação aciona): DashboardReader, ProfileReader, MacroReader (leitura de fatos) e o insight.Insighter (o LLM com gates). Ficam à direita.
  • Domínio / core puro: os tipos insight.Facts/Insight/Task e o port Insighter + gates (no pacote insight). Aplicação: o engine.Service e o FactBuilder (orquestram, compõem seams). Borda: o handler e os DTOs (HTTP); os adapters de LLM e de banco das features vizinhas.
  • A seam publicada: ao contrário do dashboard (puro consumidor), o engine publica uma seam — o FactBuilder exportado (D6) — que SPEC-108 (via FactSource), SPEC-105 e SPEC-106 vão consumir sem re-implementar o grounding.

11. Harness Engineering: como a máquina construiu isto

O projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. Veja, no SPEC-104, como o harness construiu e verificou.

O fluxo /spec-implement, fase a fase

A IA não atacou "faça o motor de insights" de uma vez. O PLAN-104 fatiou em 7 fases bottom-up (domínio → Fact Builder → engine → transporte → observabilidade → testes → docs), e cada uma fechava com go build + testes verdes antes da próxima. Construir o Fact Builder (Fase 2) antes do engine significou que o coração da spec — a composição determinística dos fatos — foi escrito e provado por testes table-driven (determinismo, vazio, sem-perfil, macro-faltante, zero-float) sem nenhum LLM por perto. Quando o engine entrou (Fase 3), ele só orquestrava fatos já comprovados e um port já gated.

O hook determinístico block-layering

Este foi o hook decisivo para a pergunta de arquitetura da D1: o subpacote engine importa dashboard, profile, marketdata e insight — isso é permitido? O hook block-layering (PostToolUse, exit 2) barra mecanicamente qualquer import de SQL/HTTP/SDK de vendor dentro de um core/aplicação de feature. Importar os tipos de domínio puros de outra feature (dashboard.Dashboard, profile.Profile, marketdata.MacroIndicator, insight.Facts) passa; se a IA tivesse tentado meter um database/sql ou um net/http dentro de internal/insight/engine, o hook teria bloqueado — sem revisão humana. É o que garante, na prática, que o motor ficou "acima" e o core insight ficou puro.

O subagente hexagonal-reviewer

O revisor especializado cobrou as regras mais sensíveis daqui: a direção de dependência (o grafo acíclico engine → {dashboard, profile, marketdata, insight}, sem ciclo de volta) e — o ponto mais crítico desta spec — a trava por construção: que o motor emite texto de usuário só via insighter.Generate, nunca construindo strings de IA por outro caminho (BR-1042). Foi a lente que firmou a separação core puro (port + gates) vs. aplicação (engine que orquestra) e a identidade vinda só de auth.UserID(ctx).

O go-correctness-reviewer e a disciplina de fatos / determinismo

Sendo a feature que monta o grounding da IA, a lente de correção vigiou o risco nº 1 do plano (PLAN-104 §8): um caminho que burlasse os gates e o vazamento de float / não-determinismo nos fatos. Ela garantiu que nenhum valor de fato usasse float64 (money int64 centavos / rates int bps); que o Fact Builder fosse uma função pura dos seams (mesmos inputs → mesmos fatos, condição do cache do SPEC-005); que ErrProfileNotFound/ErrMacroNotFound fossem omissão graciosa via errors.Is; e — crítico — que o ctx.Err() abortasse em vez de degradar silenciosamente. Também cobrou o %w + prefixo minúsculo nos erros e o ctx como primeiro parâmetro.

A prova de que as travas valem — e por que isso é a melhor garantia possível É a estrutura que prova: o engine.Service tem um único colaborador capaz de produzir texto de usuário — o insight.Insighter — e a única operação que ele invoca nele é Generate(...), que já passa pelos gates de FR-013/FR-014. O motor não recebe um cliente de LLM cru, não monta prompt, não escreve título/explicação. O compilador impede um atalho que a estrutura não oferece. Some a isso o teste dedicado ("nenhum texto de IA nasce fora do Insighter") e o hook block-layering (sem provider cru no pacote) e você tem as guardas binding garantidas por construção, não por boa-fé — exatamente o que o PRD exige para "nunca uma saída black-box, nunca uma ordem de compra/venda".

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • block-layering: o bloqueio duro que provou que o core insight ficou livre de SQL/HTTP, ao mesmo tempo que permitiu o consumo dos tipos de domínio puros das features vizinhas pelo subpacote engine.
  • O teste de drift do OpenAPI: ao adicionar GET /insights na routeTable, o openapi_test.go exigiu a entrada correspondente em api/openapi.yaml — senão o build falharia (seção 7).
  • Bloqueio de imutabilidade: o SPEC-104 referencia o ADR-0003/ADR-0005 mas não os edita — se a IA tentasse "atualizar" um ADR aceito, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — e os testes têm que passar sem rede e sem DB. O default do Insighter é o fake determinístico, então o CI roda offline; a integração (seed de holdings + cotações + perfil + macro, gates end-to-end) fica atrás do gate de TEST_DATABASE_URL + -p 1, pulando limpo.

12. Conexão com AI Engineering

O SPEC-104 É uma feature de IA — a primeira que chega ao usuário. Por isso ele exercita, de forma concreta, vários conceitos centrais de engenharia de IA confiável. Vamos a eles, todos ancorados no código desta spec.

12.1 — Grounding / Fact Builder: o modelo raciocina sobre fatos, não os inventa

O conceito mais importante de IA confiável aplicada a finanças é grounding: o LLM nunca deve produzir um número; ele recebe fatos computados e raciocina sobre eles. O BuildFacts é exatamente esse mecanismo — ele monta um insight.Facts com valor atual, alocação, exposição setorial, perfil e macro, tudo em centavos/bps determinísticos, e o passa ao Insighter. O modelo explica a concentração; ele não calcula a concentração. Essa separação é o que torna a saída auditável: a explicação descreve fatos que o sistema computou e pode mostrar.

12.2 — As guardas como middleware de verificação (FR-013 + FR-014)

O padrão arquitetural aqui é o de verification middleware: entre o modelo e o usuário há uma cadeia que verifica cada saída. O gate de explicabilidade (FR-013) rejeita qualquer insight sem explicação; o gate de não-aconselhamento (FR-014) rejeita ordens de compra/venda, quantidades, preços-alvo. No SPEC-104 esses gates não são re-codados — o motor os herda por passar só pela porta Insighter. É a aplicação disciplinada de "o LLM é falível, então toda saída passa por um verificador determinístico antes do usuário".

12.3 — Outputs estruturados & categorização

Em vez de uma resposta de texto livre, o motor pede saídas estruturadas por categoria: cada insight tem title, detail e explanation, e o motor o tagueia com category (portfolio/allocation/market_context). Estruturar a saída — N tasks distintas, campos nomeados — é o que permite a UI renderizar, o cache chavear e os gates verificar de forma confiável.

12.4 — Caching, custo & degradação como cidadania de primeira classe

A postura "zero custo" (binding nº 4) é uma decisão de engenharia de IA. O cache por (user, task, facts) só funciona porque os fatos são determinísticos; a degradação graciosa (parcial, ou available:false) mantém o produto usável quando o free-tier do LLM cai. Em IA de produção, o que fazer quando o modelo falha é tão importante quanto o que fazer quando ele acerta — e aqui isso é um estado representado, não um 500.

12.5 — Observabilidade sem PII & defesa contra prompt-injection

Os spans (insight.facts, insight.generate) registram latência e outcome sem nenhum fato, figura ou texto gerado — observabilidade de IA que respeita privacidade (BR-505). E há a superfície de prompt-injection: valores de holdings/perfil fluem para os fatos. A mitigação é dupla — o Fact Builder passa valores estruturados (não narrativa livre) onde possível, e o gate de não-aconselhamento é o backstop fail-closed: mesmo que um valor malicioso induza o modelo a "recomendar comprar X", o gate rejeita a saída antes do usuário.

A ideia que conecta tudo IA confiável em finanças é fatos determinísticos + saída verificada. O SPEC-104 entrega os dois: o Fact Builder produz o grounding (computado, em centavos/bps, reproduzível) e o Insighter é o verificador (explicabilidade + não-conselho, por construção). O motor, no meio, faz o mínimo: monta fatos e passa pela única porta gated. É essa modéstia arquitetural — fazer pouco, pela via certa — que torna a primeira feature de IA do produto segura por design, e que prepara o terreno para o chat (SPEC-108) e o multi-agente CIO reusarem a mesma costura.

O BuildFacts é exatamente isso — uma função determinística que, dado quem o usuário é, o que ele tem e o que o mercado diz, devolve o snapshot completo e auditável.
