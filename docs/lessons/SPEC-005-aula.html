1. A visão geral: por que um port de LLM, e o que mudou

As quatro aulas anteriores construíram a fundação não-IA: esqueleto (SPEC-001), banco (SPEC-002), identidade (SPEC-003) e observabilidade (SPEC-004). O SPEC-005 é a primeira pedra do lado de IA — mas, atenção, ele não entrega nenhuma feature de IA que o usuário veja. Ele entrega a infraestrutura por onde toda IA futura vai passar: o port Insighter, dois adaptadores de LLM, e os dois portões de segurança que tornam o produto honesto com suas promessas.

A ideia central é uma costura só. Em vez de espalhar chamadas a um SDK de LLM por todo o código, todo acesso ao modelo passa por uma única interface — o Insighter. Isso é a arquitetura hexagonal aplicada ao mais volátil dos recursos externos: um LLM. O provedor vira detalhe de configuração; as regras de negócio vivem no centro.

ANTES (SPEC-004) DEPOIS (SPEC-005) nenhum acesso a LLM 1 port Insighter — a costura única de IA "explicável / não-conselheiro" gate decorator: portões INESCAPÁVEIS por construção eram só texto no PRD (todo provider herda os dois guards) provedor de IA = decisão futura troca por config: INSIGHTER_PROVIDER=ollama|groq|fake nada protege contra "compre HGLG11" validador de ordem determinístico (PT-BR + EN), fail-closed risco de custo silencioso cache em memória + degradação graciosa (zero charge)

Concretamente, o SPEC-005 entregou o pacote internal/insight com: o port Insighter e o port Cache; os tipos de domínio (Insight, InsightRequest, InsightResult) e os erros-sentinela; o gate decorator (os dois portões) com seu validador de ordens; o prompt compartilhado (system prompt estável); um fake determinístico; dois adaptadores (ollama/ e groq/); e o subpacote factory/ que compõe tudo — cache em memória (LRU+TTL) e observabilidade de IA. Nenhuma migration, nenhum endpoint HTTP novo — é pura fundação. Nada chama Generate ainda: o primeiro consumidor é o SPEC-104 (o motor de insights).

Por que construir a costura antes de qualquer feature de IA? Porque as duas restrições #1 e #2 do PRD — "explicabilidade primeiro" e "copiloto, nunca conselheiro" — só são confiáveis se forem impossíveis de esquecer. Se cada feature de IA tivesse que se lembrar de validar a saída, uma hora alguém esqueceria. Implementando os portões uma vez, num decorator que embrulha qualquer provider, toda feature futura (Insights, Rebalanceamento, Health Score) os herda de graça e por construção. As promessas do PRD viram código estrutural, não disciplina humana.

2. O problema: copiloto de IA + custo zero + provedor trocável

O YieldForge promete uma IA copiloto para o investidor pessoa física brasileiro. Isso cria três tensões que o SPEC-005 resolve de uma vez:

TensãoRestrição binding (PRD)Solução do SPEC-005
Precisamos de um LLM, mas sem gastar dinheiro Custo zero (ADR-0003): só free/local Ollama local (R$ 0, sem chave) é o default de dev; Groq (free tier generoso) é o caminho hospedado. Cache + degradação garantem nenhuma cobrança silenciosa.
Hoje é Ollama/Groq; amanhã pode ser Claude/OpenAI LLM plugável (FR-018) O LLM vive atrás de um port (Insighter). Trocar de provedor é uma linha no .env (INSIGHTER_PROVIDER), nunca uma mudança de código.
Os dados da carteira são PII financeira sensível Privacidade + custo zero O default de dev é o Ollama local: os fatos da carteira nunca saem da máquina. E a telemetria nunca grava prompt/fatos (BR-505).

A peça-chave é o port. Olhe como ele é minúsculo — uma interface de um método só, no core da feature, sem importar nenhum SDK de LLM nem tipo de HTTP:

// internal/insight/ports.go
// Insighter is the single seam through which the application invokes an LLM
// (SPEC-005 FR-501 / FR-018). Implemented by provider adapters (ollama/, groq/), the
// deterministic fake, and the gate/cache/observability decorators — all behind this one
// interface, so the provider is swappable by config and the guards are inescapable.
type Insighter interface {
    Generate(ctx context.Context, req InsightRequest) (InsightResult, error)
}
Ollama (local) vs. Groq (hospedado) — e por que os dois Ollama roda um modelo na sua máquina (loopback, sem chave, R$ 0): perfeito para desenvolver e — crucial — os fatos da carteira nunca trafegam pela rede. Groq é um serviço hospedado com free tier generoso e, de propósito, uma API compatível com a da OpenAI. Por que isso importa? Porque o adaptador do Groq, construído na forma OpenAI, deixa o OpenAI (pago) a um passo de virar um drop-in futuro — mesmo formato de requisição. Um é o caminho de dev (privacidade), o outro o caminho hospedado (deploy); ambos atrás do mesmo port.

3. Os dois portões binding: explicabilidade & não-aconselhamento

Esta é a alma do SPEC-005. As duas restrições inegociáveis do produto deixam de ser texto no PRD e viram um decorator que embrulha qualquer Insighter. Toda saída de IA passa por ele — não há caminho que escape (BR-501). E ele falha fechado (fail-closed): na dúvida sobre uma ordem, ele rejeita, porque deixar passar uma ordem é pior do que descartar um insight.

Portão 1 — Explicabilidade (FR-013)

Todo insight tem que carregar uma explicação ("por que isto foi levantado?"). Sem explicação, a saída é rejeitada antes de chegar no chamador — nunca passa em silêncio.

Portão 2 — Não-aconselhamento (FR-014)

A saída não pode conter uma ordem de transação: nenhum "compre/venda" imperativo de um ativo, nenhuma quantidade transacionada, nenhum preço-alvo ou ponto de entrada/saída, nenhuma promessa de retorno garantido. Mas — e este "mas" é o pulo do gato — nomear um setor ou ativo como candidato a pesquisar é permitido. A diferença é entre um fato/consideração e uma ordem.

O gate é um decorator. Repare como ele é simples e por que isso é uma virtude:

// internal/insight/gate.go
func (g gated) Generate(ctx context.Context, req InsightRequest) (InsightResult, error) {
    result, err := g.next.Generate(ctx, req)
    if err != nil {
        return InsightResult{}, err
    }
    for _, in := range result.Insights {
        if strings.TrimSpace(in.Explanation) == "" {
            g.reject(ctx, "insight rejected: missing explanation", req.Task)
            return InsightResult{}, fmt.Errorf("gate: %w", ErrMissingExplanation)
        }
        if containsOrder(insightText(in)) {
            g.reject(ctx, "insight rejected: order signature", req.Task)
            return InsightResult{}, fmt.Errorf("gate: %w", ErrAdviceDetected)
        }
    }
    result.Disclaimer = Disclaimer  // sempre anexa o disclaimer de não-aconselhamento
    return result, nil
}
Por que detectar a "assinatura de ordem", e não "ação + ticker por perto"? Um detector ingênuo barraria qualquer frase com um verbo de ação perto de um ticker — e comeria insights legítimos. O insight "HGLG11 é um FII de logística que vale analisar" é exatamente o tipo de inteligência que o produto deve entregar (PRD FR-019). A solução é mirar na assinatura de uma ordem: um verbo imperativo de compra/venda perto de uma quantidade/ticker/preço, um preço-alvo, ou uma promessa de retorno garantido. O validador é bilíngue (PT-BR + EN) porque o LLM pode responder em qualquer língua: 
// internal/insight/advice.go (trecho)
var orderPatterns = []*regexp.Regexp{
    // Imperativo de compra/venda perto de quantidade, ticker B3, ou cotas/ações.
    regexp.MustCompile(`(?i)\b(compre|venda|vende|buy|sell)\b[^.!?\n]{0,30}?(\b\d+\b|\b[a-z]{4}\d{1,2}\b|ações|açao|cotas?|shares?)`),
    // Compra/venda a um preço: "compre a R$ 160", "buy at $100".
    regexp.MustCompile(`(?i)\b(compre|venda|vende|buy|sell)\b[^.!?\n]{0,20}?\b(a|por|at|@)\b\s*(r\$|us\$|\$)?\s*\d`),
    // Preço-alvo / ponto de entrada-saída.
    regexp.MustCompile(`(?i)(pre[çc]o[- ]?alvo|target\s+price|ponto\s+de\s+(entrada|sa[íi]da)|(entry|exit)\s+(point|price))`),
    // ...promessas de retorno garantido (nas duas ordens de palavras).
}
Detalhe sutil documentado no código: um "100 cotas" sozinho NÃO é ordem — lê-se igual a um saldo ("você tem 100 cotas"). É o verbo de transação que distingue uma ordem de um fato.
Normalizar, não mutilar (BR-506) O gate é fail-closed na assinatura de ordem (rejeita) e na falta de explicação (rejeita). Mas para a moldura — quando a única dúvida é se um candidato nomeado foi frasiado como consideração ou como diretiva — a postura é normalizar para uma consideração (e garantir o disclaimer), não deletar o insight. Um default conservador demais que apagasse toda menção a ativos neutraria a inteligência legítima que o produto promete. O disclaimer, aliás, é uma string PT-BR deliberada (o público é o investidor brasileiro): "Estas são considerações educacionais para a sua própria análise — não são recomendações de compra ou venda..."

4. "Fatos são computados, não gerados" (BR-502)

Esta é a terceira restrição binding e a que evita a alucinação financeira. O LLM nunca inventa números. Ele raciocina sobre fatos determinísticos passados na requisição — saldos, percentuais, yields — que foram computados a montante. Quem produz esses fatos é o Fact Builder, que pertence ao SPEC-104; o SPEC-005 trata os fatos de forma opaca:

// internal/insight/insight.go
// Facts is the deterministic, structured snapshot the LLM reasons over (BR-502 — facts
// are computed, not generated). Its concrete shape is owned by the Fact Builder
// (SPEC-104); this port treats it opaquely: it is serialized into the prompt and hashed
// into the cache key.
type Facts map[string]any

O system prompt é o guarda-corpo estável: ele instrui o modelo a raciocinar só sobre os fatos dados, a nunca emitir ordens, e a responder em JSON estruturado com um campo explanation obrigatório. Há uma sutileza de engenharia de IA aqui que vale destacar:

// internal/insight/prompt.go (trecho do systemPrompt)
const systemPrompt = `You are a portfolio-analysis assistant for a Brazilian retail investor.

Hard rules (never violate):
- Reason ONLY over the facts given in the user message. NEVER invent numbers, prices, or percentages...
- Produce explainable observations framed as AREAS / CONSIDERATIONS for the user's own analysis.
- NEVER issue a transaction order: no imperative buy/sell of an asset, no quantities, no price...
- You MAY name a sector or asset as a candidate to research.
- Write all user-facing text (title, detail, explanation) in Brazilian Portuguese (pt-BR).

Respond with ONLY valid JSON in exactly this shape (no text outside the JSON):
{"insights":[{"category":"...","title":"...","detail":"...","explanation":"..."}]}
Every insight MUST include "explanation" (why it was raised).`
Por que o prompt é em INGLÊS, mas a saída é em pt-BR? Por uma razão empírica de engenharia de IA: LLMs (sobretudo os modelos locais menores que o Ollama roda) seguem instruções em inglês com mais confiabilidade. Então as instruções são em inglês, mas o modelo é mandado escrever o texto que o usuário vê (título, detalhe, explicação) em português. E note o que esse prompt não é: ele é mínimo e task-agnóstico — só carrega o contrato de segurança + formato. A inteligência por tarefa (o prompt rico montado a partir do perfil + carteira + mercado) é o Fact Builder + os prompts de tarefa do SPEC-104. O system prompt é o guarda-corpo, não a inteligência.

E o prompt é a primeira linha de defesa; o gate determinístico (seção 3) é o backstop. Se um modelo for ludibriado e mesmo assim emitir uma ordem, o gate ainda rejeita. É defesa em profundidade: prompt + validador determinístico, dois mecanismos independentes para a mesma garantia.

5. A cadeia de decorators: observed(cached(gated(provider)))

Aqui mora o aprendizado de arquitetura mais bonito da spec. Como toda saída é Insighter ⇒ InsightResult, podemos empilhar comportamentos embrulhando um Insighter em outro Insighter — o padrão Decorator. O factory.New monta a cebola:

// internal/insight/factory/factory.go
func New(cfg config.Config, logger *slog.Logger, clk clock.Clock) insight.Insighter {
    provider, model := newProvider(cfg)              // ollama | groq | fake (por config)

    chain := insight.Gated(provider, logger)         // 1. PORTÕES (FR-013/FR-014)
    chain = cached{next: chain, cache: newMemCache(  // 2. CACHE (guarda resultados JÁ gateados)
        cfg.InsighterCacheSize, cfg.InsighterCacheTTL, clk)}
    return newObserved(chain, cfg.InsighterProvider, model) // 3. OBSERVABILIDADE (span + métrica)
}
UMA chamada a Generate desce a cebola e volta: observed ┌─ abre span "insight.generate" (provider, model, task, cost=0) (mais │ ┌─ cache_hit? devolve o resultado JÁ gateado, ZERO chamada de LLM externo) │ │ cached │ │ ┌─ valida explicação + ordem, anexa disclaimer (FAIL-CLOSED) │ │ │ gated │ │ │ ┌─ monta prompt dos FATOS, chama o LLM, parseia JSON │ │ │ │ provider (ollama / groq / fake) │ │ │ └─ re-ask 1x se malformado, senão DEGRADA │ │ └─ rejeita ordem/sem-explicação OU deixa passar + disclaimer │ └─ guarda o resultado gateado no cache (NUNCA guarda erro) └─ classifica o outcome, conta a métrica, fecha o span

A ordem da cebola é deliberada e cada camada tem uma razão de estar onde está:

CamadaPor que nesta posição
gated (mais interno, junto do provider) Tem que rodar logo após a saída crua do LLM — é o que torna o resultado seguro. Tudo acima dela só lida com resultados já gateados.
cached (meio) Embrulha o gated, então o que ele guarda já é explicado + sem-ordem. Um cache hit é seguro por construção. E ele nunca guarda erros (degradações, rejeições) — só sucessos.
observed (mais externo) Precisa cronometrar tudo, inclusive o cache hit e o resultado do gate. Por isso abre o span por fora; o cached grava o atributo cache_hit no mesmo span que o observed criou.

O cache: LRU + TTL, escopado por usuário, via o Clock

O cache é em memória (LRU com TTL), atrás do port Cache trocável. A chave é um sha256 de usuário + tarefa + fatos — o usuário entra na chave para que o cache de um usuário nunca sirva outro (SPEC-003 BR-304). E o TTL é avaliado contra o Clock injetado (não time.Now()), para que a expiração seja determinística nos testes:

// internal/insight/factory/cached.go
func (c cached) Generate(ctx context.Context, req insight.InsightRequest) (insight.InsightResult, error) {
    key := cacheKey(req)
    if res, ok := c.cache.Get(ctx, key); ok {
        trace.SpanFromContext(ctx).SetAttributes(attribute.Bool("insight.cache_hit", true))
        return res, nil                              // HIT: zero chamada de LLM, zero custo
    }
    trace.SpanFromContext(ctx).SetAttributes(attribute.Bool("insight.cache_hit", false))
    res, err := c.next.Generate(ctx, req)
    if err != nil {
        return insight.InsightResult{}, err          // NUNCA cacheia erro/degradação/rejeição
    }
    c.cache.Set(ctx, key, res)
    return res, nil
}
Degradação graciosa + um re-ask limitado (e por que não há retry que cobra) Se o LLM devolve um JSON malformado, o adaptador re-pergunta exatamente uma vez (com um sufixo reforçando "só JSON"); se ainda falhar, degrada para ErrInsightsUnavailable. Mas uma falha que não é malformação — um erro HTTP, um 429 de rate-limit — degrada imediatamente, sem re-ask. Por quê? Porque re-tentar um free tier estourado é justamente o que incorreria em cobrança (BR-504). A política mora num lugar só, compartilhada por todos os adaptadores: 
// internal/insight/prompt.go — GenerateWithReask
result, genErr := call(ctx, system, user)
if genErr == nil { return result, nil }
if errors.Is(genErr, ErrMalformedResponse) {          // SÓ malformação ganha o re-ask
    result, retryErr := call(ctx, system, user+ReAskSuffix)
    if retryErr == nil { return result, nil }
    genErr = retryErr
}
return InsightResult{}, fmt.Errorf("%w: %v", ErrInsightsUnavailable, genErr)  // degrada

Observabilidade de IA: só metadados, nunca conteúdo (BR-505)

O decorator observed usa o seam do SPEC-004 (observability.Tracer/Meter) para abrir um span por Generate e contar gerações por desfecho. Ele grava provider, model, task, outcome, custo — e jamais prompt, fatos ou texto gerado. Dados de carteira são PII financeira; não podem vazar para um backend de observabilidade.

// internal/insight/factory/observed.go
span.SetAttributes(
    attribute.String("insight.provider", o.provider),
    attribute.String("insight.model", o.model),
    attribute.String("insight.task", string(req.Task)),
    attribute.Int64("insight.cost_centavos", 0),  // free providers; custo por modelo é §15
)
// ... depois da chamada:
outcome := classifyOutcome(err)   // success | insufficient_facts | rejected_advice | unavailable | ...
span.SetAttributes(attribute.String("insight.outcome", outcome))
Por que o custo é int64 cost_centavos = 0 agora, se ainda não há custo? Duas decisões deliberadas. Primeiro, o custo de tokens é dinheiro — então segue a convenção da casa: int64 em unidades menores (centavos), nunca float64 (CLAUDE.md). Fixar a forma da métrica agora evita refatorar quando um provedor pago chegar. Segundo, a contagem de tokens foi deliberadamente adiada: ambos os provedores do MVP são gratuitos, logo o custo é literalmente 0. Não vale instrumentar tokens hoje; vale reservar o atributo com o tipo certo. É a postura "faça o mínimo correto agora, deixe o seam pronto" (Open Question §15).

6. As 5 decisões (D1–D5) e por quê

Toda spec tem bifurcações. Aqui foram cinco, confirmadas antes do PLAN e agora binding. O trade-off de cada uma é o aprendizado.

#DecisãoEscolhaPor quê
D1Quais adaptadores? Ollama (local/dev) + Groq (hospedado) + fake Ollama é R$ 0, sem chave, e mantém os fatos na máquina (privacidade); Groq é o caminho de deploy com free tier e API OpenAI-compatível (OpenAI vira drop-in futuro). O fake cobre o CI. Gemini ficou em segundo; Claude/OpenAI são adaptadores pagos futuros.
D2Formato da resposta? JSON estruturado (cada insight tem explanation), parseado + validado Torna o portão de explicabilidade determinístico (campo obrigatório). Um re-ask limitado em caso de JSON malformado. Texto livre + heurísticas foi recusado.
D3Como barrar aconselhamento? Validador determinístico da assinatura de ordem + system prompt não-conselheiro Belt-and-suspenders: prompt (1ª linha) + validador (backstop). Fail-closed na ordem; candidatos nomeados como consideração passam (corpus true-negative). Só-prompt e "limpar no lugar" foram recusados.
D4Como cachear? Cache em memória (LRU+TTL) atrás do port Cache Para um MVP single-user em free tiers, o cache economiza latência/cota (não dinheiro — os provedores são grátis), e o in-memory captura quase tudo com zero infra. Um cache em DB foi adiado atrás do mesmo port — e, quando valer, deve reusar a tabela de histórico do SPEC-104 (FR-022), não criar uma redundante.
D5Onde ficam os portões? Gate decorator embrulhando qualquer Insighter gated(insighter) torna os guards estruturais, agnósticos de provedor, impossíveis de esquecer — e compõe com o cache. Portões "dentro de cada adaptador" foram recusados (fácil de esquecer num provedor novo).
A reversão honesta na D4 (cache em DB → cache em memória) Houve uma decisão breve de fazer o cache em banco. Ela foi revertida quando ficou claro o overlap entre "cache" e "histórico": o SPEC-104 vai ter uma tabela de histórico de insights (FR-022). Uma linha de histórico recente com o mesmo facts-hash dentro do TTL já é um cache hit. Criar uma tabela insight_cache separada duplicaria isso. Então: in-memory agora (zero migration), e o cache persistente fica adiado atrás do port, para reusar a tabela do SPEC-104 quando valer a pena. É a disciplina de "não construa infra antes de a dor existir".

7. As 7 fases, uma a uma

Como nas specs anteriores, bottom-up — cada fase compilando e revisável sozinha. A novidade de ordenação: os portões foram puxados para a Fase 2, antes de qualquer adaptador real, porque são as restrições binding e tudo o mais compõe em torno deles.

Fase 1 — Domínio, ports & config

Os tipos (Insight, InsightRequest, InsightResult), os sentinelas (ErrMissingExplanation, ErrAdviceDetected, ErrInsightsUnavailable, ErrInsufficientFacts), as interfaces Insighter/Cache, e os campos INSIGHTER_* na Config com validação (provider inválido = fatal; chave Groq exigida iff provider=groq).

// internal/platform/config/config.go (trechos)
InsighterProvider      string        // ollama | groq | fake   (default ollama)
InsighterOllamaBaseURL string        // http://localhost:11434
InsighterGroqAPIKey    string        // segredo — exigido iff provider=groq
InsighterTimeout       time.Duration // timeout por geração (default 30s)
InsighterCacheTTL      time.Duration // TTL do cache (default 30m)
InsighterCacheSize     int           // máx. de entradas (LRU, default 256)

Fase 2 — Os portões (as restrições binding)

O advice.go (validador da assinatura de ordem) e o gate.go (o decorator). É a peça central da spec, e a revisada mais duramente — testada contra um corpus PT-BR + EN com true-negatives explícitos (candidatos nomeados que devem passar) ao lado de frases de ordem (que devem rejeitar).

Fase 3 — Adaptador Ollama (local/dev)

O prompt.go (mensagens neutras de provedor + system prompt) e o ollama/: um cliente net/http para /api/chat (modo JSON), limitado por ctx/timeout, com a política compartilhada de re-ask + degradação. Todo o detalhe HTTP fica confinado no subpacote.

Fase 4 — Adaptador Groq (hospedado, OpenAI-compatível)

O groq/: cliente net/http para o /chat/completions OpenAI-compatível (Bearer key, response_format: json_object, modelo via config), reusando o prompt.go. Auth/rate-limit (429)/5xx mapeiam para ErrInsightsUnavailable — degrada, sem retry que cobra, e nunca loga a chave ou o prompt.

Fase 5 — Cache, degradação, observabilidade & factory

O memcache.go (LRU+TTL via Clock), o cached.go (decorator que guarda resultados gateados), o observed.go (span + contador via o seam do SPEC-004, só metadados) e o factory.go que compõe observed(cached(gated(provider))) e seleciona o provider por config.

Fase 6 — Testes

Unitários, sem modelo nem rede: o corpus dos portões; os adaptadores com httptest (sucesso, malformado→re-ask, erro/429/401→degrada); o cache (miss chama uma vez, hit não chama, chave muda com os fatos, TTL expira = miss); o fake determinístico; e a asserção de que nenhum conteúdo vaza para os spans. Mais um teste de integração opcional contra um Ollama local (env-gated como o TEST_DATABASE_URL, pula limpo no CI).

Fase 7 — Documentação & aula

CHANGELOG.md, README.md (o port, as duas garantias, como rodar o Ollama, as INSIGHTER_*), .env.example, SPEC-005 e PLAN-005 marcados Done, e esta aula. 🙂

8. Conceitos-chave de Go & arquitetura

Port & Adapter (hexagonal aplicado a um LLM)

O port é a interface no core da feature (Insighter em internal/insight). Os adapters são as implementações na borda (ollama/, groq/) que falam o protocolo do provedor. A regra de ouro: a dependência aponta para dentro. O core importa zero SDK de LLM e zero tipo de HTTP — só o adaptador conhece net/http. É por isso que trocar de provedor não toca o domínio.

O padrão Decorator

Como tudo é Insighter ⇒ Insighter, comportamentos viram camadas empilháveis: gate, cache, observabilidade. Cada decorator faz uma coisa e delega o resto ao next. O resultado é testável isoladamente (você embrulha um fake) e a composição é uma linha no factory. Compare com a alternativa — um método gigante com if cache then... if gate then... — e veja como o decorator deixa cada responsabilidade independente.

O core puro vs. o composition root (a factory)

Olhe o doc-comment do pacote factory — ele declara a fronteira explicitamente:

// internal/insight/factory/memcache.go
// Package factory is the insight composition root (SPEC-005): it wires the provider
// adapters, the guard gates, the cache, and the AI observability into a single
// Insighter from config. It is the only part of the insight area that imports the
// adapters and OpenTelemetry — the insight core and the adapters stay pure.

O core (insight.go, gate.go, prompt.go) não importa OTel nem os adaptadores. Os adaptadores não importam OTel. Só a factory — o composition root — amarra tudo. Essa é a fronteira que mantém o domínio testável e portável.

Stdlib-only (zero dependência nova)

Os dois adaptadores usam net/http + encoding/json; o cache usa container/list + sync; a facts-hash usa crypto/sha256. Nenhuma dependência nova. A API do Groq é JSON puro OpenAI-compatível — não precisa de SDK. É a postura "stdlib-first / justifique cada dependência" (ADR-0003) aplicada ao recurso mais propenso a arrastar SDKs pesados: um cliente de LLM.

O fake determinístico

O Fake é um Insighter sem modelo, sem rede, sem custo: devolve um insight fixo, explicável e sem-ordem (passa pelo gate), ou ErrInsufficientFacts quando não há fatos — para o caller exercitar o mesmo caminho de fatos-vazios dos adaptadores reais. É o que faz o CI rodar a feature de IA inteira offline e de graça, e é o modo "IA desligada" do app (INSIGHTER_PROVIDER=fake). Note: hand-written fake, não gomock — a convenção da casa.

9. Harness Engineering: como a máquina construiu isto

Este projeto tem propósito duplo: além de Go/backend, aprender engenharia de harness — dar à IA trilhos que baixam a taxa de erro. O SPEC-005, sendo a própria spec de IA-segura, é onde o harness mais se prova.

O fluxo /spec-implement, fase a fase

A IA não atacou "construa o Insighter" de uma vez. O PLAN-005 fatiou em 7 fases bottom-up, e a ordem foi deliberadamente reordenada: os portões vieram na Fase 2, antes de qualquer adaptador real. Construir e testar a restrição binding primeiro significa que todo o resto (adaptadores, cache) já nasce compondo em torno dela — o erro de "esquecer de gatear um provedor novo" fica estruturalmente impossível.

O subagente hexagonal-reviewer

O revisor especializado conhece as regras da casa e cobrou as mais sensíveis aqui: a direção de dependência (BR-503 — o core do insight e os adaptadores não importam SDK de LLM nem OTel; só a factory os amarra), os portões como middleware central (BR-501 — não gateados-dentro-de-cada-adaptador), e o no-PII (BR-505 — "este atributo de span pode vazar um fato da carteira?"). Foi essa lente que garantiu que o observed.go gravasse só metadados, e que o prompt.go ficasse no core neutro de provedor.

A prova de que o core ficou puro É mecânica: o pacote internal/insight (core) e os subpacotes ollama//groq/ não têm um único import de OpenTelemetry. Todo o OTel mora em internal/insight/factory. Se a IA tivesse "ajudado" colocando um span dentro do adaptador do Groq, o revisor barraria. A fronteira está documentada no próprio doc-comment do pacote factory — doc e código contam a mesma história.

/security-review nos guardrails de IA

O PLAN-005 sinalizou que esta spec é security-sensitive (são guardrails de IA) e sugeriu um /security-review antes de fechar. O foco: o validador de ordem não tem false-negatives (deixar uma ordem passar) nem false-positives demais (comer candidatos legítimos). A mitigação virou código: matching só na assinatura de ordem (não "ação + ticker por perto"), um corpus grande com true-negatives, e fail-closed só na assinatura.

Os hooks que agiram

  • gofmt: todo arquivo Go sai formatado — zero ruído de estilo nos diffs.
  • Bloqueio de imutabilidade: o SPEC-005 referencia ADR-0002/0003 (que já abençoam o LLM plugável) mas não os edita — se a IA tentasse "atualizar" um ADR aceito, o hook barraria. ADR é imutável; supersede, nunca edita.
  • O backstop do Stop: ao fechar, o gate exige build/vet/test/gofmt limpos — e, crucialmente, os testes têm que passar sem modelo e sem rede. Foi esse backstop que provou, na prática, que o fake determinístico cobre o CI e que a feature de IA inteira roda offline.

10. Conexão com AI Engineering

Diferente do SPEC-004 (que era fundação para a IA), o SPEC-005 é a fundação da IA — e exercita, em código real, vários conceitos centrais de engenharia de IA confiável. Vale nomear cada um, porque eles vão se repetir em toda feature de IA daqui pra frente.

Conceito de AI EngineeringComo o SPEC-005 o materializa
Grounding / Fact Builder O LLM raciocina sobre fatos determinísticos serializados no prompt — nunca inventa números (BR-502). O contrato Facts torna "fatos de entrada" explícito; fatos vazios = ErrInsufficientFacts, não uma resposta fabricada. (O Fact Builder em si é o SPEC-104.)
Saídas estruturadas (structured outputs) O prompt pede JSON estrito com explanation obrigatório (D2); o ParseResult tolera o modelo embrulhar em prosa/code-fences extraindo o {...} externo. JSON estruturado é o que torna o portão de explicabilidade determinístico.
Verification middleware (os gates) FR-013 e FR-014 são middleware de verificação: a saída do LLM não é confiável até passar pelos portões. O gate é fail-closed — descartar uma ordem é mais seguro do que vazá-la. É a ideia de "nunca confie cegamente na saída do modelo", virada decorator.
Defesa contra prompt-injection Defesa em profundidade: os fatos são estruturados/escapados (não texto livre do usuário), o system prompt é o guarda-corpo, e o gate determinístico é o backstop — um modelo ludibriado a "aconselhar" ainda é rejeitado. Dois mecanismos independentes para a mesma garantia.
Caching de LLM Cache por hash dos fatos (+ tarefa + usuário): entradas idênticas → resposta cacheada, zero chamada de modelo. Economiza latência e cota de free-tier; o cache guarda só resultados já gateados, então um hit é seguro.
Observabilidade de IA Span por Generate com provider/model/latência/cache-hit/outcome/custo — via o seam do SPEC-004 — e nunca prompt/fatos/PII (BR-505). Habilita as métricas do PRD: latência/custo de IA e taxa de sucesso de geração de insights.
Evals (a semente) O corpus PT-BR + EN de testes do gate — com true-negatives explícitos (candidatos que devem passar) e ordens que devem rejeitar — é a semente de um conjunto de avaliação dos guardrails. Cada frase de borda que surgir vira um caso novo, e o gate é re-testado contra todo o corpus.
A ideia que conecta tudo IA confiável não é "o modelo é bom"; é "a saída do modelo é verificada antes de virar produto". O SPEC-005 instala essa disciplina como arquitetura: fatos determinísticos de entrada, saída estruturada, dois portões fail-closed como middleware, cache seguro e telemetria sem PII — tudo atrás de uma costura só. Antes de qualquer insight existir, as garantias do PRD já são código inescapável.
