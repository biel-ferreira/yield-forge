Mapa visual da arquitetura hexagonal

Você não está perdido num monte de arquivos jogados. É a mesma pilha em camadas que você já conhece — handler → DTO → service → repository — só que reorganizada e com uma regra a mais. Este documento te dá um mapa pra achar cada coisa no projeto real.

💡 A frase que destrava tudo: hexagonal = arquitetura em camadas + uma regra (“o núcleo nunca depende da borda”) + um lugar novo onde tudo é montado (o main.go). Se você guardar só isso, o resto encaixa.

As 5 zonas (decore as cores)

Todo arquivo do projeto cai em uma destas cinco zonas. Quando abrir um arquivo e ficar perdido, a primeira pergunta é: “em qual zona isso vive?”

DOMAIN — regras puras PORT — interface (contrato) ADAPTER — implementação na borda COMPOSITION ROOT — main.go monta tudo PLATFORM — utilitários transversais
ZonaO que éEquivalente em “camadas”Pode importar…
DOMAIN Tipos e regras de negócio puros. Sem SQL, sem HTTP, sem time.Now(). As entidades + a lógica que normalmente vaza pro service. só o stdlib e ele mesmo
PORT Uma interface Go. Diz o que precisa acontecer, não como. A assinatura do seu repository / client — mas declarada pelo núcleo. tipos do domínio
ADAPTER O código “sujo” que fala com o mundo: Postgres, HTTP externo, bcrypt. A implementação concreta do repository / o client HTTP. domínio + ports + libs externas
COMPOSITION ROOT O main(). O único lugar que conhece as implementações concretas e as “pluga” nas interfaces. — (em camadas isso fica espalhado / num framework de DI) tudo
PLATFORM Config, logger, clock, pool de DB, servidor HTTP. Apoio que todos usam. Sua pasta infra/ ou common/. stdlib + libs

Por que “hexágono”? Esquerda chama, direita é chamado

Esqueça a forma geométrica — o hexágono só serve pra mostrar que a aplicação tem dois lados. À esquerda quem aciona a app (driving); à direita o que a app aciona (driven). No meio, o núcleo, que só conhece ports.

Repare: as duas pontas são adapters. A diferença é a direção da seta — o handler chama o núcleo; o núcleo chama o repositório/cliente (através de um port).

▼

🔑 A “grande virada” (inversão de dependência). Em camadas, o service dá import no pacote do repositório concreto — ele depende “pra fora”. No hexagonal, o service declara a interface que precisa, e é o adapter Postgres que dá import no núcleo pra implementá-la. A seta de dependência aponta pra dentro. Por isso o domínio “não sabe” que Postgres existe.

A inversão, em código real do projeto

O núcleo de marketdata declara o que precisa — sem citar Postgres, Fundamentus ou Yahoo:

// internal/marketdata/ports.go  — ZONA: PORT (só uma interface)
type MarketDataProvider interface {
    FetchFIIQuotes(ctx context.Context, tickers []Ticker) (map[Ticker]FIIQuote, error)
    FetchMacroIndicator(ctx context.Context, ind Indicator) (MacroIndicator, error)
}

O Worker (o caso de uso) guarda a interface, nunca a implementação. Ele não tem ideia se os dados vêm de Postgres, de HTTP ou de um fake:

// internal/marketdata/ingest/worker.go  — ZONA: núcleo (depende só de ports)
type Worker struct {
    provider  marketdata.MarketDataProvider  // ← interface
    fiiRepo   marketdata.FIIQuoteRepository   // ← interface
    macroRepo marketdata.MacroRepository      // ← interface
    tickers   marketdata.TickerSource        // ← interface
}

E é o adapter que “puxa” a dependência pra dentro — ele importa o domínio e se compromete a satisfazer o contrato:

// internal/marketdata/fii/composite.go  — ZONA: ADAPTER
type Composite struct { fundamentals Fundamentals; dividends LastDividends }

// Satisfaz a metade FII do port MarketDataProvider — junta Fundamentus + Yahoo.
func (c Composite) FetchFIIQuotes(ctx context.Context, tickers []marketdata.Ticker) (...) { ... }

Note o padrão de Go aqui: “aceite interfaces, retorne structs”. O Composite nem depende dos clients concretos — depende de mini-interfaces Fundamentals/LastDividends definidas por ele mesmo. É port & adapter em escala de função.

O main.go desmistificado — é aqui que tudo se conecta

Esse arquivo te confunde porque ele faz um trabalho que em camadas costuma ficar escondido num container de injeção de dependência ou em anotações. No hexagonal ele é explícito e tem nome: Composition Root. É o único lugar que conhece as implementações concretas e as encaixa nos ports. Leia cmd/api/main.go como uma receita de montagem em 4 passos:

1
Liga a infra (platform). Config, logger, telemetria e o pool do Postgres. 
cfg, _ := config.Load()
logger := logging.New(cfg)
db, _ := database.Connect(ctx, cfg)   // *sql.DB — um adapter de infra
2
Monta a feature auth: cria os adapters concretos e os injeta no Service. Aqui os ports recebem suas implementações. 
authService := auth.NewService(
    authpostgres.NewUserRepository(db),    // adapter → port UserRepository
    authpostgres.NewSessionRepository(db), // adapter → port SessionRepository
    authbcrypt.New(),                     // adapter → port PasswordHasher
    clock.System{},                      // adapter → port Clock
    cfg.SessionTTL,
)

Troque authpostgres por authmemory e nada no Service muda. Esse é o ponto inteiro da arquitetura.

3
Monta o transporte (driving): entrega o service pronto ao router HTTP. 
router := transporthttp.NewRouter(transporthttp.Deps{
    Logger: logger, Auth: authService, ...   // o handler recebe a interface AuthService
})
4
Monta marketdata e sobe tudo. A factory esconde a fiação interna (qual provider, quais repos); o main só decide rodar. 
mdWorker, _ := ingest.New(cfg, db, logger, clock.System{}) // factory.go escolhe fake|live
// ... scheduler opcional em goroutine ...
serveErr := httpserver.Run(ctx, cfg, router, logger)

Resumo mental do main.go: “crio as peças concretas → plugo nos buracos (ports) → mando rodar”. Se você ler procurando regra de negócio vai se perder — não tem nenhuma aqui de propósito. É só montagem.

Roteiro: como ler uma feature sem se perder

Quando abrir uma pasta nova (ex.: internal/profile/), leia nesta ordem — de dentro pra fora, seguindo a direção das dependências:

1
O domínio primeiro. Os arquivos soltos na raiz do pacote (ticker.go, quote.go). São os substantivos do negócio. → entende o “o quê”.
2
ports.go. A lista de interfaces. É o índice do que essa feature precisa do mundo externo. → entende as “tomadas”.
3
O service / worker. Como as regras usam os ports. O coração do caso de uso. → entende o “como”.
4
As subpastas = adapters. postgres/, fundamentus/, bcrypt/. Cada uma é um “plugue”. Só leia quando precisar do detalhe técnico. → o “com o quê”.
5
cmd/api/main.go por último. Pra ver quem foi plugado em quem. → o “montado como”.

Regra prática: subpasta = adapter (detalhe externo); arquivo na raiz do pacote = domínio ou port (núcleo). Só essa heurística já te localiza em 90% dos casos.

Essa arquitetura é boa mesmo? Vale a pena?

Resposta honesta: depende do projeto — e pra este projeto, vale. Não é hype; tem trade-off real. Vou ser direto dos dois lados.

A FAVOR

  • Trocar provedor por config. A meta “custo zero” do YieldForge exige trocar LLM / fonte de dados / DB sem reescrever nada. O buildProvider alterna fake↔live numa linha. Isso só é barato por causa dos ports.
  • Testes triviais e sem rede. O Fake implementa o mesmo port — os testes do Worker rodam offline, determinísticos. Sem mock framework.
  • Domínio protegido. Regra de negócio não “pega carona” em detalhe de Postgres/HTTP. Mais fácil de raciocinar e de manter determinístico (mesmo input → mesmo Health Score, exigência do produto).
  • Costura pro futuro. O port Insighter é o mesmo seam por onde o multi-agent CIO + MCP vão entrar depois. Você troca o motor sem mexer no resto.

O CUSTO

  • Mais arquivos e indireção. O que era 1 service + 1 repo vira domínio + port + adapter + factory. Pra um CRUD bobo, é cerimônia a mais — exatamente a sensação de “arquivos jogados” que você teve.
  • Curva de aprendizado. A inversão de dependência não é intuitiva no começo (você está sentindo isso agora). Depois que “vira a chave”, some.
  • Fácil exagerar. Criar port pra coisa que nunca vai ter 2 implementações é over-engineering. A regra sã: port só quando há uma fronteira externa de verdade (DB, rede, relógio, LLM) — que é justamente onde o YieldForge usa.
🎯 Veredito. Hexagonal compensa quando o projeto tem muitas fronteiras externas que precisam ser trocáveis e testáveis — e o YieldForge é a definição disso (vários provedores de dados, LLM swappable, postura custo-zero, scores determinísticos). Pra uma API CRUD simples e descartável, seria peso morto. Aqui, é a escolha certa — e a confusão que você sente é só o pedágio de entrada, não um defeito do desenho.
